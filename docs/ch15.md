## 1. 概述 (Overview)

本章是全書的最後一章 DI 容器專章，聚焦於微軟隨 ASP.NET Core 一同推出的內建 DI 容器 —— `Microsoft.Extensions.DependencyInjection`（以下簡稱 **MS.DI**）。

微軟設計 MS.DI 的初衷是提供一個「最小公約數」（lowest common denominator）的 DI 容器，讓框架與第三方元件開發者在 ASP.NET Core 生態中有統一的依賴管理方式。然而，正因為這個「最小化」的設計哲學，它在功能上存在顯著限制。

**作者的核心立場**：MS.DI 的功能過於有限，不適合用於任何「有一定規模且實踐鬆耦合」的應用程式開發。但由於它表面上與其他成熟 DI 容器（如 Autofac、Simple Injector）極為相似，開發者容易被「內建就好」的想法誘導而忽略其限制，因此本章的目的在於揭露這些限制，幫助你做出明智的技術選型。

> **白話比喻**：MS.DI 就像一把瑞士刀中最基本的款式 —— 只有一把刀片和一個開瓶器。對於日常小需求夠用了，但如果你要進行專業的野外求生（大型鬆耦合應用），你需要更完整的工具組（Autofac 或 Simple Injector）。

## 2. 學習目標 (Learning Objectives)

讀完本章後，你應該能夠：

1. **掌握 MS.DI 的兩階段使用模式**：`ServiceCollection`（配置階段） → `ServiceProvider`（解析階段），並理解為何必須從 `IServiceScope` 解析而非直接從根容器解析
2. **配置三種生命週期**：Transient、Singleton、Scoped，並理解 Torn Lifestyle 陷阱與 Captive Dependency 風險
3. **處理困難的 API 註冊**：原始型別依賴、Parameter Object 萃取、靜態工廠的 code block 註冊
4. **理解多元件配置的複雜度**：序列注入、Decorator 與 Composite 的手動配線
5. **認識 MS.DI 的本質限制**：缺乏原生 Auto-Registration、Decorator、Composite 支援，不偵測 Captive Dependency
6. **做出架構決策**：能明確判斷何時應該選用 MS.DI，何時應換用更成熟的 DI 容器或 Pure DI

## 3. 章節大綱 (Outline)

### 3.1 認識 MS.DI（15.1 Introducing MS.DI）

#### 3.1.1 兩階段模式：ServiceCollection → ServiceProvider

MS.DI 採用嚴格的「配置與消費分離」設計：

```
+-------------------+         +-----------------+
| ServiceCollection |  Build  | ServiceProvider  |
|  （配置階段）      | ------> |  （解析階段）     |
|  .AddTransient()  |         |  .GetRequired    |
|  .AddScoped()     |         |    Service()     |
|  .AddSingleton()  |         |                  |
+-------------------+         +-----------------+
       註冊一次                     解析多次
```

> **白話說明**：`ServiceCollection` 就像一份「食譜清單」，你在上面登記要哪些料理和材料。寫好之後交給廚房（`BuildServiceProvider`），廚房就變成 `ServiceProvider`，開始根據食譜做菜（解析元件）。食譜一旦交出去，就不能再改了。

**最基本的使用範例：**

```csharp
// 1. 配置階段
var services = new ServiceCollection();
services.AddTransient<IIngredient, SauceBéarnaise>();

// 2. 建立容器（開啟 scope 驗證）
ServiceProvider container =
    services.BuildServiceProvider(validateScopes: true);

// 3. 從 Scope 解析（而非直接從根容器）
using (IServiceScope scope = container.CreateScope())
{
    IIngredient sauce = scope.ServiceProvider
        .GetRequiredService<IIngredient>();
}
```

**關鍵規則 — 永遠從 Scope 解析：**

```
+-----------+
| Container |  <-- 不要直接從這裡解析！會造成記憶體洩漏
+-----------+
      |
      | CreateScope()
      v
+---------------+
| IServiceScope |  <-- 從這裡解析才安全
+---------------+
      |
      | Dispose()  --> 釋放 Scoped 與 Transient 元件
      v
```

> **重要警告**：MS.DI 會追蹤大多數元件（包括可棄置的 Transient），若直接從根容器解析，Transient 實例會被根容器持有直到應用程式結束，造成記憶體洩漏。

#### 3.1.2 配置 ServiceCollection

**Configuration as Code（唯一內建方式）：**

```csharp
// 單一抽象對應單一實作
services.AddTransient<IIngredient, SauceBéarnaise>();
services.AddTransient<ICourse, Course>();

// 同一抽象註冊多個實作 — 最後註冊的為預設
services.AddTransient<IIngredient, SauceBéarnaise>();
services.AddTransient<IIngredient, Steak>();  // 解析 IIngredient 時拿到 Steak
```

**Torn Lifestyle 陷阱（撕裂的生命週期）：**

> **白話說明**：Torn Lifestyle 就像你以為家裡只養一隻貓，結果不小心養了兩隻長一樣的貓 —— 你以為是同一隻，其實是兩個獨立個體。

```csharp
// ❌ 錯誤：兩個獨立的 Singleton 快取，產生兩個不同實例
services.AddSingleton<SauceBéarnaise>();
services.AddSingleton<IIngredient, SauceBéarnaise>();

// ✅ 正確：使用委派轉發，確保只有一個實例
services.AddSingleton<SauceBéarnaise>();
services.AddSingleton<IIngredient>(
    c => c.GetRequiredService<SauceBéarnaise>());
```

**手動實作 Auto-Registration（組件掃描）：**

MS.DI 不提供任何內建的 Auto-Registration API。你必須自己用 LINQ + Reflection：

```csharp
Assembly assembly = typeof(Steak).Assembly;

var ingredientTypes =
    from type in assembly.GetTypes()
    where !type.IsAbstract
    where typeof(IIngredient).IsAssignableFrom(type)
    select type;

foreach (var type in ingredientTypes)
{
    services.AddTransient(typeof(IIngredient), type);
}
```

**泛型抽象的 Auto-Registration：**

```csharp
Assembly assembly = typeof(AdjustInventoryService).Assembly;

var mappings =
    from type in assembly.GetTypes()
    where !type.IsAbstract
    where !type.IsGenericType
    from i in type.GetInterfaces()
    where i.IsGenericType
    where i.GetGenericTypeDefinition() == typeof(ICommandService<>)
    select new { service = i, type };

foreach (var mapping in mappings)
{
    services.AddTransient(mapping.service, mapping.type);
}
```

> **警告**：上述手動掃描存在靜默失敗的風險 —— 如果同一個封閉泛型介面有多個實作，MS.DI 會全部註冊但只解析最後一個，且順序未定義（可能在重新編譯或重啟後改變）。

### 3.2 管理生命週期（15.2 Managing Lifetime）

#### 三種 Lifestyle 對照表

| MS.DI 方法 | 模式名稱 | 行為說明 |
|---|---|---|
| `AddTransient` | Transient | 每次請求都建立新實例。實例被容器追蹤，Scope 結束時 Dispose |
| `AddSingleton` | Singleton | 整個容器生命週期只有一個實例。容器 Dispose 時一起釋放 |
| `AddScoped` | Scoped | 同一 Scope 內共用同一個實例。不同 Scope 各自獨立 |

```
+---------------------------------------------+
|              Container (根容器)               |
|  +- Singleton A (唯一一個，活到容器結束)       |
|                                              |
|  +- Scope 1 --------+  +- Scope 2 --------+ |
|  | Scoped B (共用)   |  | Scoped B (新的)   | |
|  | Transient C (新)  |  | Transient C (新)  | |
|  | Transient D (新)  |  | Transient D (新)  | |
|  +-------------------+  +-------------------+ |
+---------------------------------------------+
```

#### Scope 的建立與釋放

```csharp
// 建立 Scope
using (IServiceScope scope = container.CreateScope())
{
    // 從 Scope 解析
    IMeal meal = scope.ServiceProvider
        .GetRequiredService<IMeal>();

    meal.Consume();
}
// using 結束 → Scope 被 Dispose → 所有 Scoped 和 Transient 元件被釋放
```

#### Captive Dependency 問題（俘虜依賴）

> **白話比喻**：Captive Dependency 就像把一條需要每天換水的魚（Scoped/Transient）養在一個永遠不換水的缸裡（Singleton）。魚活著但水越來越髒——元件存活著但可能持有過期的狀態或資源。

**MS.DI 的處理方式：**
- 開啟 `validateScopes: true` 後，注入 Scoped 到 Singleton 會被阻擋
- **但是**：注入 Transient 到 Singleton **不會被阻擋**，因為 MS.DI 的 Transient 被設計為「活得和消費者一樣久」
- MS.DI **不偵測** Captive Dependency，這一點不如 Simple Injector

```csharp
// MS.DI 允許此操作但可能產生問題！
services.AddSingleton<MySingleton>();       // 活到容器結束
services.AddTransient<MyDependency>();      // 預期短命，但被 Singleton 持有
// MyDependency 變成事實上的 Singleton（被俘虜了）
```

### 3.3 註冊困難的 API（15.3 Registering Difficult APIs）

#### 3.3.1 原始型別依賴

```csharp
// 方法一：全域註冊原始型別（所有消費者共用同一個值）
services.AddSingleton(typeof(Spiciness), Spiciness.Medium);
services.AddTransient<ICourse, ChiliConCarne>();

// 方法二（推薦）：萃取為 Parameter Object
public class Flavoring
{
    public readonly Spiciness Spiciness;
    public readonly bool ExtraSalty;

    public Flavoring(Spiciness spiciness, bool extraSalty)
    {
        this.Spiciness = spiciness;
        this.ExtraSalty = extraSalty;
    }
}

var flavoring = new Flavoring(Spiciness.Medium, extraSalty: true);
services.AddSingleton(flavoring);
services.AddTransient<ICourse, ChiliConCarne>();
```

> **架構啟示**：Parameter Object 不僅解決技術上的歧義，更在語意層面提升了設計品質。`Flavoring` 比裸露的 `Spiciness` enum 更能表達領域意圖。

#### 3.3.2 使用 Code Block 註冊（處理靜態工廠等）

```csharp
// 靜態工廠 — 容器無法自動理解，需手動指定
services.AddTransient<IMeal>(
    c => JunkFoodFactory.Create("chicken meal"));

// 好處：
// 1. 仍然維持 IMeal → JunkFood 的鬆耦合映射
// 2. Lifestyle 仍可配置（改 AddSingleton 就只建一次）
```

### 3.4 處理多元件（15.4 Working with Multiple Components）

這是本章最複雜也最重要的段落。DI 容器在面對「同一抽象有多個實作」時最容易出問題。

#### 3.4.1 處理歧義：從多個候選者中選擇

**問題場景 — 同一介面的多參數：**

```csharp
public class ThreeCourseMeal
{
    public ThreeCourseMeal(
        ICourse entrée,        // 這三個型別一樣
        ICourse mainCourse,    // 容器不知道要注入哪個
        ICourse dessert) { }
}
```

**方法一：Code Block（編譯期安全）：**

```csharp
services.AddTransient<IMeal>(c => new ThreeCourseMeal(
    entrée: c.GetRequiredService<Rillettes>(),
    mainCourse: c.GetRequiredService<CordonBleu>(),
    dessert: c.GetRequiredService<CrèmeBrûlée>()));
```

**方法二：ActivatorUtilities（自動配線但脆弱）：**

```csharp
services.AddTransient<IMeal>(c =>
    ActivatorUtilities.CreateInstance<ThreeCourseMeal>(
        c,
        new object[]
        {
            c.GetRequiredService<Rillettes>(),
            c.GetRequiredService<CordonBleu>(),
            c.GetRequiredService<MousseAuChocolat>()
        }));
// ⚠️ 依賴參數順序匹配，重構時容易靜默出錯！
```

> **架構建議**：作者認為歧義本身往往暗示設計可以改進。將 `ThreeCourseMeal` 重構為接受 `IEnumerable<ICourse>` 的 `Meal`，從根本上消除歧義。

#### 3.4.2 序列注入

```csharp
// 自動配線序列 — MS.DI 天生支援
services.AddTransient<ICourse, Rillettes>();
services.AddTransient<ICourse, CordonBleu>();
services.AddTransient<ICourse, MousseAuChocolat>();
services.AddTransient<IMeal, Meal>();
// Meal(IEnumerable<ICourse>) 會自動收到全部三個 ICourse
```

**從集合中挑選子集：**

```csharp
// 先用具體型別註冊所有課程
services.AddScoped<Rillettes>();
services.AddScoped<CordonBleu>();
services.AddSingleton<MousseAuChocolat>();

// 再用委派將具體型別橋接到介面（防止 Torn Lifestyle）
services.AddTransient<ICourse>(c => c.GetRequiredService<Rillettes>());
services.AddTransient<ICourse>(c => c.GetRequiredService<CordonBleu>());
services.AddTransient<ICourse>(c => c.GetRequiredService<MousseAuChocolat>());

// 手動挑選子集注入 Meal
services.AddTransient<IMeal>(c => new Meal(
    new ICourse[]
    {
        c.GetRequiredService<Rillettes>(),
        c.GetRequiredService<CordonBleu>(),
        c.GetRequiredService<MousseAuChocolat>()
    }));
```

#### 3.4.3 配線 Decorator（裝飾者模式）

> **白話說明**：Decorator 就像俄羅斯套娃 —— 每一層都包裹著內層，擁有相同的「外觀」（實作同一介面），但在傳遞請求的過程中加上額外行為。

**MS.DI 沒有任何內建的 Decorator 支援。** 你必須手動用 `ActivatorUtilities` 巢狀呼叫：

```csharp
// 非泛型 Decorator：VealCutlet → HamCheeseGarlic → Breading
services.AddTransient<IIngredient>(c =>
    ActivatorUtilities.CreateInstance<Breading>(
        c,
        ActivatorUtilities.CreateInstance<HamCheeseGarlic>(
            c,
            ActivatorUtilities.CreateInstance<VealCutlet>(c))));

// 等效的 Pure DI 版本（清楚多了）：
// new Breading(
//     new HamCheeseGarlic(
//         new VealCutlet()));
```

**泛型 Decorator — 複雜度爆炸：**

```
+----------------------------------+
| ICommandService<TCommand>        |
+----------------------------------+
         |
         v
+----------------------------------+
| SecureCommandServiceDecorator<T> |  <-- 安全性
+----------------------------------+
         |
         v
+----------------------------------+
| TransactionCommandService        |
|   Decorator<T>                   |  <-- 交易處理
+----------------------------------+
         |
         v
+----------------------------------+
| AuditingCommandServiceDecorator  |
|   <T>                            |  <-- 稽核
+----------------------------------+
         |
         v
+----------------------------------+
| 實際的 CommandService 實作        |  <-- 業務邏輯
+----------------------------------+
```

```csharp
Assembly assembly = typeof(AdjustInventoryService).Assembly;

var mappings =
    from type in assembly.GetTypes()
    where !type.IsAbstract
    where !type.IsGenericType
    from i in type.GetInterfaces()
    where i.IsGenericType
    where i.GetGenericTypeDefinition() == typeof(ICommandService<>)
    select new { service = i, implementation = type };

foreach (var mapping in mappings)
{
    Type commandType = mapping.service.GetGenericArguments()[0];

    Type secureType = typeof(SecureCommandServiceDecorator<>)
        .MakeGenericType(commandType);
    Type transactionType = typeof(TransactionCommandServiceDecorator<>)
        .MakeGenericType(commandType);
    Type auditingType = typeof(AuditingCommandServiceDecorator<>)
        .MakeGenericType(commandType);

    services.AddTransient(mapping.service, c =>
        ActivatorUtilities.CreateInstance(c, secureType,
            ActivatorUtilities.CreateInstance(c, transactionType,
                ActivatorUtilities.CreateInstance(c, auditingType,
                    ActivatorUtilities.CreateInstance(
                        c, mapping.implementation)))));
}
```

**此方案的五大缺陷：**
1. 當 Decorator 的泛型參數不完全匹配抽象時，建構封閉泛型型別會變得困難
2. 無法為開放泛型實作套用 Decorator
3. 條件式套用 Decorator 極為複雜
4. 不同 Lifestyle 下容易產生 Torn Lifestyle
5. 無法為鏈中的不同 Decorator 配置不同 Lifestyle

> **作者建議**：不要在 MS.DI 上自建 Decorator 支援，那等於在 MS.DI 之上重新發明一個 DI 容器。直接使用 Autofac 或 Simple Injector。

#### 3.4.4 配線 Composite（組合模式）

> **白話說明**：Composite 像一個「團隊經理」—— 它跟團隊成員做同一份工作（實作同一介面），但做法是把工作分派給所有成員去做。

**非泛型 Composite：**

```csharp
public class CompositeNotificationService : INotificationService
{
    private readonly IEnumerable<INotificationService> services;

    public CompositeNotificationService(
        IEnumerable<INotificationService> services)
    {
        this.services = services;
    }

    public void OrderApproved(Order order)
    {
        foreach (var service in this.services)
        {
            service.OrderApproved(order);
        }
    }
}
```

**註冊方式（含 Auto-Registration）：**

```csharp
Assembly assembly = typeof(OrderFulfillment).Assembly;

Type[] types = (
    from type in assembly.GetTypes()
    where !type.IsAbstract
    where typeof(INotificationService).IsAssignableFrom(type)
    where type != typeof(CompositeNotificationService) // 排除自己！
    where !IsDecoratorFor<INotificationService>(type)  // 排除 Decorator
    select type)
    .ToArray();  // 物化為陣列，避免重複掃描

foreach (Type type in types)
{
    services.AddTransient(type);
}

services.AddTransient<INotificationService>(c =>
    new CompositeNotificationService(
        types.Select(t =>
            (INotificationService)c.GetRequiredService(t))
            .ToArray()));

// 輔助方法
static bool IsDecoratorFor<T>(Type type)
{
    return typeof(T).IsAssignableFrom(type) &&
        type.GetConstructors()[0].GetParameters()
            .Any(p => p.ParameterType == typeof(T));
}
```

> **陷阱**：如果不排除 Composite 自身，會產生循環依賴，導致 `StackOverflowException`。而且 MS.DI 的錯誤訊息品質差，不像 Autofac/Simple Injector 能清楚說明循環原因。

**泛型 Composite — 極端複雜：**

由於 MS.DI 的限制，泛型 Composite 需要：
1. 改寫 Composite 使其依賴 `IServiceProvider` 和一個 `CompositeSettings` Parameter Object
2. Composite 自行負責型別篩選與實例建立（本應是容器的職責）
3. Composite 被迫放在 Composition Root 中（因為它依賴容器本身）

```csharp
public class CompositeSettings
{
    public Type[] AllHandlerTypes { get; set; }
}

public class CompositeEventHandler<TEvent> : IEventHandler<TEvent>
{
    private readonly IServiceProvider provider;
    private readonly CompositeSettings settings;

    public CompositeEventHandler(
        IServiceProvider provider,
        CompositeSettings settings)
    {
        this.provider = provider;
        this.settings = settings;
    }

    public void Handle(TEvent e)
    {
        foreach (var handler in GetHandlers())
            handler.Handle(e);
    }

    IEnumerable<IEventHandler<TEvent>> GetHandlers()
    {
        return
            from type in settings.AllHandlerTypes
            where typeof(IEventHandler<TEvent>).IsAssignableFrom(type)
            select (IEventHandler<TEvent>)
                ActivatorUtilities.CreateInstance(provider, type);
    }
}
```

> **這是一個不得已的妥協方案**：Composite 承擔了容器該做的工作，且對 MS.DI 產生了硬依賴。

## 4. 關鍵框架與公式 (Key Frameworks)

### 4.1 MS.DI 的核心架構模型

```
+====================================================+
|                應用程式啟動流程                       |
+====================================================+
|                                                      |
|  [1] ServiceCollection (食譜)                        |
|      .AddTransient<TService, TImpl>()                |
|      .AddScoped<TService, TImpl>()                   |
|      .AddSingleton<TService, TImpl>()                |
|      .AddTransient<TService>(factory delegate)       |
|                                                      |
|              | BuildServiceProvider(true)             |
|              v                                       |
|                                                      |
|  [2] ServiceProvider (廚房)                           |
|      +--> CreateScope() --> IServiceScope            |
|                              .ServiceProvider        |
|                              .GetRequiredService<T>()|
|                              .Dispose()              |
|                                                      |
+====================================================+

     配置不可變              解析與釋放
     (Immutable)             (Resolve & Release)
```

### 4.2 Lifestyle 選擇決策樹

```
你的元件需要什麼樣的生命週期？
    |
    +-- 全應用程式共用一個實例？
    |       --> AddSingleton
    |       ⚠️ 注意：其依賴也必須是 Singleton 或無狀態
    |
    +-- 每個請求（Scope）共用一個實例？
    |       --> AddScoped
    |       ⚠️ 注意：不可被 Singleton 消費
    |
    +-- 每次注入都要新實例？
            --> AddTransient
            ⚠️ 注意：被 Singleton 消費時會變成事實上的 Singleton
```

### 4.3 MS.DI 功能比較矩陣

```
功能                          | MS.DI | Autofac | Simple Injector
------------------------------|-------|---------|----------------
Configuration as Code         |  ✅   |   ✅    |      ✅
Auto-Registration API         |  ❌   |   ✅    |      ✅
Open Generic Registration     |  ✅   |   ✅    |      ✅
Decorator 原生支援            |  ❌   |   ✅    |      ✅
Composite 原生支援            |  ❌   |   ✅    |      ✅
Captive Dependency 偵測       |  ❌   |   ❌    |      ✅
Scope Validation              |  ✅   |   ✅    |      ✅
Keyed / Conditional 註冊      |  ❌   |   ✅    |      ✅
錯誤訊息品質                  |  差   |   好    |      優
```

### 4.4 三種處理歧義的策略比較

| 策略 | Autofac | Simple Injector | MS.DI |
|---|---|---|---|
| 具名/鍵值註冊 | `Keyed` | N/A | ❌ 不支援 |
| 條件式註冊 | `OnlyIf` | `RegisterConditional` | ❌ 不支援 |
| Code Block | `Register(c => ...)` | `Register(() => ...)` | `Add...(c => ...)` |
| ActivatorUtilities | N/A | N/A | `CreateInstance<T>()` |

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點一：Torn Lifestyle（撕裂的生命週期）

**為什麼難**：表面上看兩種寫法效果相同，但底層行為完全不同。

```csharp
// === 情境 A：兩個獨立的 Singleton 快取 ===
services.AddSingleton<SauceBéarnaise>();              // 快取 #1
services.AddSingleton<IIngredient, SauceBéarnaise>(); // 快取 #2
// 解析 SauceBéarnaise → 實例 A
// 解析 IIngredient   → 實例 B（不同實例！）

// === 情境 B：委派轉發，共用同一快取 ===
services.AddSingleton<SauceBéarnaise>();              // 快取 #1
services.AddSingleton<IIngredient>(
    c => c.GetRequiredService<SauceBéarnaise>());     // 轉發到快取 #1
// 解析 SauceBéarnaise → 實例 A
// 解析 IIngredient   → 實例 A（同一實例！）
```

```
情境 A（Torn）:                    情境 B（正確）:

ServiceCollection                  ServiceCollection
  |                                  |
  +-- Key: SauceBéarnaise            +-- Key: SauceBéarnaise
  |   Cache: [Instance A]            |   Cache: [Instance A]
  |                                  |
  +-- Key: IIngredient               +-- Key: IIngredient
      Cache: [Instance B] ❌             Delegate --> 轉發到上面 ✅
      (不同實例！)                       (同一實例！)
```

**精通要訣**：在 MS.DI 中，每一個 `Add...` 呼叫都會建立獨立的快取。只有使用委派轉發模式才能確保跨服務型別共用同一實例。

---

### 難點二：泛型 Decorator 的配線

**為什麼難**：需要結合 Reflection、LINQ、`MakeGenericType` 和巢狀 `ActivatorUtilities` 呼叫。

**思維拆解：**

```
步驟 1：掃描組件 → 找到所有 ICommandService<T> 實作
        例如：AdjustInventoryService : ICommandService<AdjustInventory>

步驟 2：對每個實作，萃取 TCommand 型別
        例如：AdjustInventory

步驟 3：用 MakeGenericType 建構每一層 Decorator 的封閉泛型型別
        SecureCommandServiceDecorator<AdjustInventory>
        TransactionCommandServiceDecorator<AdjustInventory>
        AuditingCommandServiceDecorator<AdjustInventory>

步驟 4：由內而外巢狀包裹

  ActivatorUtilities.CreateInstance(SecureDecorator,
      ActivatorUtilities.CreateInstance(TransactionDecorator,
          ActivatorUtilities.CreateInstance(AuditingDecorator,
              ActivatorUtilities.CreateInstance(AdjustInventoryService))))

步驟 5：結果物件圖

  +-- Secure<AdjustInventory>
       +-- Transaction<AdjustInventory>
            +-- Auditing<AdjustInventory>
                 +-- AdjustInventoryService
```

**為什麼作者不推薦**：這段程式碼等同於自建一個迷你 DI 容器。一旦需求變得更複雜（條件式套用、混合泛型約束、不同 Lifestyle），就會崩潰。

---

### 難點三：泛型 Composite 需要改寫 Composite 實作

**為什麼難**：MS.DI 的限制迫使你「汙染」Composite 的設計 —— 它必須知道容器的存在。

**正常的 Composite（理想版本）：**

```csharp
public class CompositeEventHandler<TEvent> : IEventHandler<TEvent>
{
    private readonly IEnumerable<IEventHandler<TEvent>> handlers;

    public CompositeEventHandler(IEnumerable<IEventHandler<TEvent>> handlers)
        => this.handlers = handlers;

    public void Handle(TEvent e)
    {
        foreach (var handler in handlers) handler.Handle(e);
    }
}
```

**MS.DI 被迫的版本：**

```csharp
public class CompositeEventHandler<TEvent> : IEventHandler<TEvent>
{
    private readonly IServiceProvider provider;    // ❌ 依賴容器
    private readonly CompositeSettings settings;   // ❌ 需要額外 Parameter Object

    // Composite 被迫自己做型別篩選與實例建立
    IEnumerable<IEventHandler<TEvent>> GetHandlers()
    {
        return
            from type in settings.AllHandlerTypes
            where typeof(IEventHandler<TEvent>).IsAssignableFrom(type)
            select (IEventHandler<TEvent>)
                ActivatorUtilities.CreateInstance(provider, type);
    }
}
```

```
理想版本:                          MS.DI 版本:

Container                          Container
  |                                  |
  +-- 負責篩選型別                    +-- 只負責建立 Composite
  +-- 負責建立所有 handler             |
  +-- 注入 handler 序列               Composite
  |                                    |
  v                                    +-- 自己持有 IServiceProvider
Composite                             +-- 自己篩選型別
  |                                    +-- 自己呼叫 ActivatorUtilities
  +-- 只負責迭代與委派                  +-- 還要負責迭代與委派
      (職責單一)                        (職責混亂)
```

**精通要訣**：這個案例完美展示了為什麼 MS.DI 不適合大型應用 —— 當容器功能不足時，你被迫把容器的職責塞進業務元件中，違反了 Single Responsibility Principle 和 Composition Root 的封裝原則。

## 6. 本章小結與架構啟示

### 核心結論

1. **MS.DI 採用嚴格的兩階段模式**：`ServiceCollection`（不可解析）配置元件，`ServiceProvider`（不可再配置）解析元件。這個分離本身是好的設計。

2. **永遠從 Scope 解析**：直接從根容器解析會導致記憶體洩漏與並發問題。

3. **三種 Lifestyle**：Transient、Singleton、Scoped。MS.DI 的 Transient 特殊之處在於被設計為「活得跟消費者一樣久」，這意味著注入 Singleton 時不會報錯但會被俘虜。

4. **MS.DI 的功能限制是根本性的**：
   - 無 Auto-Registration API
   - 無 Decorator 原生支援
   - 無 Composite 原生支援
   - 無 Captive Dependency 偵測
   - 無 Keyed/Conditional 註冊
   - 錯誤訊息品質低落

5. **不要在 MS.DI 上自建進階功能**：這等同於在它之上重新發明一個 DI 容器，既不生產也不可靠。

### 架構決策指引

```
你的應用程式規模與設計取向？
    |
    +-- 小型應用 / 少量服務 / 不需要 Decorator-Composite
    |       --> MS.DI 足矣
    |
    +-- 中大型應用 / 實踐 SOLID / 需要 Cross-Cutting Concerns
    |       --> 使用 Autofac 或 Simple Injector
    |
    +-- 想完全掌控物件圖 / 不想依賴任何容器
            --> Pure DI（手動 new）
```

### 全書終極啟示

> DI 不依賴於任何特定技術或容器。一個好的應用程式應該運用本書介紹的 DI 友善模式與原則來設計。當你成功做到這一點，DI 容器的選擇就變得不那麼重要了 —— 容器只是組裝應用程式的工具，理想情況下，你應該能夠將一個容器替換為另一個容器，而不需要重寫 Composition Root 以外的任何部分。

**換言之：先學會原則（Parts 1-3），再選擇工具（Part 4）。** 無論你選 Autofac、Simple Injector、MS.DI，還是 Pure DI，原則不變。容器只是手段，鬆耦合、可測試、可維護才是目的。
