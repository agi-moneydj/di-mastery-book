# 第 2 章：撰寫緊耦合程式碼 (Writing Tightly Coupled Code)

## 章節精通指南 | Chapter Mastery Guide

> **書籍**：Dependency Injection Principles, Practices, and Patterns
> **作者**：Steven van Deursen, Mark Seemann
> **導讀視角**：資深系統架構師
> **程式碼範例**：C#

---

## 1. 概述 (Overview)

本章透過一位名為 Mary Rowan 的資深 .NET 開發者建構電商應用程式的完整過程，具體示範「緊耦合（Tight Coupling）」是如何在看似遵循最佳實踐（三層式架構）的情況下悄悄產生的。章節的核心訊息是：畫出三層架構圖很容易，但真正落實鬆耦合卻需要刻意的設計決策。透過分析 Mary 的程式碼，我們將看到緊耦合如何摧毀可組合性（Composability），導致應用程式喪失晚期綁定、可擴展性、可維護性、可測試性與平行開發等關鍵能力。

---

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

- **辨識** 緊耦合程式碼的典型特徵，即使它表面上看起來像是分層架構
- **理解** 為什麼直接使用 `new` 關鍵字建立 Volatile Dependency 會破壞可組合性
- **繪製** 模組之間的依賴圖（Dependency Graph），並分析依賴方向是否正確
- **評估** 一個應用程式的可組合性——能否在不修改現有模組的情況下替換某一層
- **解釋** 緊耦合導致喪失的五大能力：Late Binding、Extensibility、Maintainability、Testability、Parallel Development
- **實作** 依賴反轉的思維——理解依賴箭頭應該「指向」Domain Layer，而非從 Domain Layer 指出去

## 3. 章節大綱 (Outline)

### 3.1 建構緊耦合應用程式 (Building a Tightly Coupled Application)

#### 3.1.1 認識 Mary Rowan
- Mary 是一位有 11 年經驗的 .NET 資深開發者
- 被要求用 ASP.NET Core MVC + Entity Framework Core + SQL Server 建構電商應用
- 要求三層架構，第一個功能是「精選產品列表」（Featured Products）
- 若使用者是「優先客戶」（Preferred Customer），所有產品價格打 95 折

#### 3.1.2 建立資料存取層 (Data Access Layer)
- **Product 類別**：定義在資料存取層（這本身就是一個問題）
- **CommerceContext 類別**：繼承 `DbContext`，直接在內部讀取設定檔（又一個問題）
- 資料存取層完成後，架構進度為 1/3

#### 3.1.3 建立領域層 (Domain Layer)
- **ProductService 類別**：在建構函式中直接 `new CommerceContext()`（關鍵耦合點）
- Mary 的同事 Jens 給了錯誤的建議：把判斷「是否為優先客戶」的邏輯放在 UI 層
- Jens 濫用了 Separation of Concerns 和 Single Responsibility Principle 等術語
- **Single Responsibility Principle（SRP）**：一個類別應只有一個理由需要改變

#### 3.1.4 建立 UI 層 (UI Layer)
- **HomeController**：直接 `new ProductService()`（又一個耦合點）
- UI 層必須同時引用 Domain Layer 和 Data Access Layer 才能編譯
- Index View 包含過多邏輯（型別轉換、字串格式化）

### 3.2 評估緊耦合應用程式 (Evaluating the Tightly Coupled Application)

#### 3.2.1 評估依賴圖 (Dependency Graph)
- UI 層同時依賴 Domain Layer 和 Data Access Layer
- UI 層可以繞過 Domain Layer 直接存取 Data Access Layer（違反分層原則）

#### 3.2.2 評估可組合性 (Composability)
- **替換 UI 層**：可行，因為沒有模組依賴 UI 層
- **替換資料存取層**：不可行，Domain Layer 和 UI 層都依賴它，移除後整個應用崩解
- **其他組合**：替換 Domain Layer 通常沒有意義，因為它是應用的核心

### 3.3 分析缺失的可組合性 (Analysis of Missing Composability)

#### 3.3.1 依賴圖分析
- 即使移除 UI 對 Data Access Layer 的直接依賴，問題仍未解決
- 根本原因：Domain Layer 依賴 Data Access Layer 的方向錯了

#### 3.3.2 資料存取介面分析
- `ProductService` 建構函式中 `new CommerceContext()` 是核心問題
- `GetFeaturedProducts` 回傳 `Product` 型別定義在 Data Access Layer 中
- 喪失的五大能力：Late Binding、Extensibility、Maintainability、Parallel Development、Testability

#### 3.3.3 其他問題
- Product 類別應定義在 Domain Layer 而非 Data Access Layer
- 判斷優先客戶是商業邏輯，不該放在 UI 層
- `CommerceContext` 不該自行讀取設定檔，應由呼叫端注入設定
- View 包含過多邏輯，應移至 Model

## 4. 關鍵框架與公式 (Key Frameworks)

### 框架一：三層式架構 (Three-Layer Architecture)

**何時使用**：幾乎所有需要區分展示、業務邏輯、資料存取的應用程式。

**核心概念**：將應用程式分為三個獨立層次，各層有明確職責。

```
  ┌─────────────────────────────┐
  │      User Interface Layer   │  <-- 處理使用者互動、HTTP 請求
  │   (Controllers, Views)      │
  └──────────────┬──────────────┘
                 │ 呼叫
                 ▼
  ┌─────────────────────────────┐
  │        Domain Layer         │  <-- 商業邏輯、規則、運算
  │   (Services, Entities)      │
  └──────────────┬──────────────┘
                 │ 呼叫
                 ▼
  ┌─────────────────────────────┐
  │     Data Access Layer       │  <-- 資料庫存取、ORM、SQL
  │  (DbContext, Repositories)  │
  └─────────────────────────────┘
```

> **重點**：畫出這張圖很容易，但真正落實「每一層只依賴下面那一層」卻需要刻意的設計。Mary 的問題就是在「落實」這一步出了差錯。

---

### 框架二：Mary 的實際依賴圖 vs. 理想依賴圖

**Mary 的實際依賴圖（有問題的）：**

```
  ┌──────────────────┐
  │    UI Layer       │
  │  (HomeController) │
  └───┬──────────┬────┘
      │          │
      │ 依賴     │ 依賴（不該有！）
      ▼          │
  ┌──────────┐   │
  │  Domain  │   │
  │  Layer   │   │
  └────┬─────┘   │
       │         │
       │ 依賴    │
       ▼         ▼
  ┌──────────────────┐
  │  Data Access     │
  │  Layer           │
  │ (CommerceContext) │
  └──────────────────┘

  問題：
  1. UI Layer 直接依賴 Data Access Layer（跨層依賴）
  2. Domain Layer 依賴 Data Access Layer（依賴方向錯誤）
  3. 無法替換 Data Access Layer，因為所有層都依賴它
```

**理想的依賴圖（第 3 章將實現）：**

```
  ┌──────────────────┐     ┌──────────────────┐
  │    UI Layer       │     │  Data Access     │
  │  (HomeController) │     │  Layer           │
  └───────┬──────────┘     └───────┬──────────┘
          │                        │
          │ 依賴                   │ 依賴（方向反轉了！）
          ▼                        ▼
       ┌──────────────────────────────┐
       │        Domain Layer          │
       │  (介面定義在這裡)             │
       └──────────────────────────────┘

  關鍵差異：
  - Data Access Layer 反過來依賴 Domain Layer
  - UI Layer 只依賴 Domain Layer
  - Domain Layer 不依賴任何人（它是核心）
  - 替換 Data Access Layer 變得可能！
```

### 框架三：可組合性評估法 (Composability Evaluation)

**何時使用**：當你需要評估一個系統的模組化程度時。

**核心概念**：逐一嘗試「移除」每個模組，看看剩下的部分是否還能編譯和運作。

**評估步驟**：

| 步驟 | 操作 | 問自己 |
|------|------|--------|
| 1 | 畫出依賴圖 | 誰依賴誰？箭頭方向對嗎？ |
| 2 | 嘗試移除模組 A | 移除後其他模組還能編譯嗎？ |
| 3 | 嘗試替換模組 A | 能否用新模組替換，而不修改其他模組？ |
| 4 | 檢查跨層依賴 | 有沒有「跳過中間層」的依賴？ |

**以 Mary 的應用為例**：

```
  場景：替換 UI Layer           場景：替換 Data Access Layer
  ┌────────────┐                ┌────────────┐
  │  Web UI    │ ──移除──>      │  UI Layer  │ ──受影響
  └────────────┘                └────────────┘
                                ┌────────────┐
  ┌────────────┐                │  Domain    │ ──受影響
  │  WPF UI   │ ──替換成功！    └────────────┘
  └────────────┘                ┌────────────┐
                                │  Data      │ ──移除──> 全部崩解！
  結論：可以替換 UI              │  Access    │
                                └────────────┘
                                結論：無法替換 Data Access
```

### 框架四：緊耦合導致喪失的五大能力

```
  ┌─────────────────────────────────────────────────────────────────┐
  │                  緊耦合的代價 (Cost of Tight Coupling)           │
  ├─────────────────┬───────────────────────────────────────────────┤
  │ Late Binding    │ 無法在部署時期才決定要用哪個資料存取層          │
  │ (晚期綁定)       │ 例：同一應用無法一版用 SQL Server、             │
  │                 │     一版用 Azure Table Storage                 │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ Extensibility   │ 無法輕鬆插入橫切關注點 (Cross-Cutting Concerns)│
  │ (可擴展性)       │ 例：加入日誌、快取、權限檢查需要改動大量程式碼   │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ Maintainability │ 每次新增功能都讓現有類別更複雜                   │
  │ (可維護性)       │ 程式碼越來越難讀、越來越難改                     │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ Testability     │ 無法抽換資料存取層 = 無法做單元測試               │
  │ (可測試性)       │ 測試永遠需要真實資料庫，速度慢又不穩定           │
  ├─────────────────┼───────────────────────────────────────────────┤
  │ Parallel Dev    │ 多人開發時容易踩到彼此的程式碼                   │
  │ (平行開發)       │ 合併衝突 (merge conflict) 頻繁發生              │
  └─────────────────┴───────────────────────────────────────────────┘
```

---

### 框架五：Single Responsibility Principle (SRP)

**何時使用**：設計任何類別時都該考慮。

**核心概念**：一個類別應只有一個「改變的理由」。也可以從「內聚性（Cohesion）」的角度思考——類別中各元素的功能相關性越高，內聚性越高，越符合 SRP。

> **白話比喻**：想像一把瑞士刀，它同時是刀、剪刀、開瓶器、螺絲起子。如果你要升級「刀」的部分，你必須把整把瑞士刀拆開。SRP 就是告訴你：不如做四個獨立的工具，各自可以獨立升級和替換。

**程式碼示範——違反 SRP**：

```csharp
// CommerceContext 違反 SRP：
// 理由一：資料庫結構改變時需要修改
// 理由二：設定檔格式改變時需要修改
public class CommerceContext : DbContext
{
    public DbSet<Product> Products { get; set; }

    protected override void OnConfiguring(
        DbContextOptionsBuilder builder)
    {
        // 這段讀取設定檔的邏輯不該在這裡！
        var config = new ConfigurationBuilder()
            .SetBasePath(Directory.GetCurrentDirectory())
            .AddJsonFile("appsettings.json")
            .Build();

        string connectionString =
            config.GetConnectionString("CommerceConnectionString");

        builder.UseSqlServer(connectionString);
    }
}
```

**改善方向**（預告第 3 章）：

```csharp
// CommerceContext 只負責資料庫存取（單一理由）
// 連線字串由外部（呼叫端）注入
public class CommerceContext : DbContext
{
    public DbSet<Product> Products { get; set; }

    public CommerceContext(
        DbContextOptions<CommerceContext> options)
        : base(options)
    {
        // 不再自己讀設定檔，連線字串從外部傳入
    }
}
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：為什麼 Domain Layer 依賴 Data Access Layer 是致命的？

**為什麼難**：
直覺上，Domain Layer 需要從資料庫拿資料，所以「依賴」Data Access Layer 似乎很合理。畢竟你不引用它，怎麼拿資料呢？這種「自然而然」的想法正是緊耦合的溫床。

**白話解釋**：
想像你是一家餐廳的主廚（Domain Layer）。你需要食材，所以你直接跟「李大叔的有機農場」（Data Access Layer）簽了獨家合約，而且合約上寫死了只能跟他買。有一天，李大叔的農場倒了，你整間餐廳就停擺了——因為你的食譜上寫的都是「李大叔的番茄」「李大叔的洋蔥」。

正確做法是：你只定義「我需要番茄、洋蔥」這些規格（介面），然後讓管理層（Composition Root）決定跟哪家農場買。這樣李大叔倒了，你換一家就好，食譜完全不用改。

**程式碼示範——問題所在**：

```csharp
public class ProductService
{
    private readonly CommerceContext dbContext;

    public ProductService()
    {
        // 寫死了！就像主廚合約上寫死只跟李大叔買
        this.dbContext = new CommerceContext();
    }

    public IEnumerable<Product> GetFeaturedProducts(
        bool isCustomerPreferred)
    {
        // 直接使用 CommerceContext（Entity Framework）的 API
        // 如果要換成 Azure Table Storage，這裡全部要改
        var featuredProducts =
            from product in this.dbContext.Products
            where product.IsFeatured
            select product;

        decimal discount = isCustomerPreferred ? .95m : 1;

        return
            from product in featuredProducts.AsEnumerable()
            select new Product
            {
                Id = product.Id,
                Name = product.Name,
                Description = product.Description,
                IsFeatured = product.IsFeatured,
                UnitPrice = product.UnitPrice * discount
            };
    }
}
```

**依賴方向反轉的概念圖**：

```
  【錯誤方向】                     【正確方向】

  Domain Layer                     Domain Layer
       │                                ▲
       │ new CommerceContext()           │ 實作介面
       ▼                                │
  Data Access Layer                Data Access Layer

  主廚寫死跟李大叔買                主廚定義食材規格
                                   農場來符合規格
```

### 難點 2：Product 類別定義在 Data Access Layer 為什麼是個問題？

**為什麼難**：
Entity Framework 需要 `Product` 類別來對應資料庫表格，把它放在 Data Access Layer 看起來很合理。而且它確實能正常運作，Mary 的應用跑得好好的。那為什麼書上說這是個問題？

**白話解釋**：
`Product` 是一個「業務概念」——它描述的是「你的生意在賣什麼」，不是「你的資料庫長什麼樣」。想像你經營一間書店。「書」這個概念屬於你的「生意」，而不是屬於你的「倉庫管理系統」。

如果「書」的定義寫在「倉庫管理系統」裡，那麼你的收銀台、你的網站、你的行銷部門，全部都要去依賴倉庫管理系統才能知道「書是什麼」。哪天你換了倉庫管理系統，所有部門都炸了。

**依賴連鎖反應**：

```
  Product 類別定義在 Data Access Layer：

  ┌──────────┐
  │ UI Layer │──需要顯示 Product──┐
  └──────────┘                    │
                                  ▼
  ┌──────────┐              ┌───────────────┐
  │ Domain   │──需要操作     │ Data Access   │
  │ Layer    │──Product──>  │ Layer         │
  └──────────┘              │               │
                            │ class Product │ <-- 所有人都被迫依賴這裡
                            │ { ... }       │
                            └───────────────┘

  Product 類別定義在 Domain Layer（正確做法）：

  ┌──────────┐     ┌───────────────┐
  │ UI Layer │     │ Data Access   │
  └─────┬────┘     └───────┬───────┘
        │                  │
        │ 依賴             │ 依賴
        ▼                  ▼
  ┌────────────────────────────────┐
  │         Domain Layer           │
  │                                │
  │  class Product { ... }         │  <-- 業務概念定義在業務層
  └────────────────────────────────┘
```

### 難點 3：CommerceContext 自行讀取設定檔為什麼是陷阱？

**為什麼難**：
「把連線字串寫在 `appsettings.json` 裡，然後類別自己去讀」是極為常見的做法。很多教學文章甚至推薦這樣做。為什麼書上說這是 Hidden Dependency（隱藏的依賴）？

**白話解釋**：
想像你叫了一台 Uber，司機說：「你不用告訴我地址，我自己會去翻你家信箱找到你最近收到的信，然後根據信上的地址去送你。」

聽起來很方便？但問題是：
1. 你不知道司機會去翻你的信箱（**隱藏行為**）
2. 如果你搬家了但信箱還有舊信，司機會送錯地方（**錯誤來源不明**）
3. 你沒辦法在不同情境用不同地址（**缺乏彈性**），例如你有時想去公司、有時想去健身房
4. 每次叫車都翻一次信箱，明明地址不會一直變（**浪費效能**）

正確的做法是：你直接告訴 Uber 司機地址。在程式中，就是把連線字串當參數傳進去。

**程式碼示範——問題**：

```csharp
public class CommerceContext : DbContext
{
    protected override void OnConfiguring(
        DbContextOptionsBuilder builder)
    {
        // 隱藏的依賴！
        // 1. 依賴檔案系統（Directory.GetCurrentDirectory()）
        // 2. 依賴特定設定檔名稱（"appsettings.json"）
        // 3. 依賴特定設定鍵（"CommerceConnectionString"）
        // 4. 每次建立實例都重新讀取，浪費效能
        var config = new ConfigurationBuilder()
            .SetBasePath(Directory.GetCurrentDirectory())
            .AddJsonFile("appsettings.json")
            .Build();

        string connectionString =
            config.GetConnectionString("CommerceConnectionString");

        builder.UseSqlServer(connectionString);
    }
}
```

**程式碼示範——正確做法（預告）**：

```csharp
// 連線字串由外部傳入，依賴變成「顯式」的
public class CommerceContext : DbContext
{
    public CommerceContext(
        DbContextOptions<CommerceContext> options)
        : base(options)
    {
        // 什麼都不用做，連線字串已經在 options 裡了
    }
}

// 在應用程式的入口點 (Composition Root) 設定
// 只有這裡讀設定檔，只讀一次
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddDbContext<CommerceContext>(options =>
    options.UseSqlServer(
        builder.Configuration.GetConnectionString(
            "CommerceConnectionString")));
```

**核心原則**：
> 只有最終的應用程式入口點（Composition Root）才應該讀取設定檔。可重用的程式庫應該由呼叫端以「命令式」的方式設定，而非自行讀取設定檔。

## 6. 本章小結與架構啟示

### 小結

本章透過 Mary 的電商應用案例，揭示了一個殘酷的事實：**緊耦合是自然產生的，鬆耦合是刻意設計的**。即使開發者有最好的意圖（建立三層架構），語言特性（`new` 關鍵字）和工具（Visual Studio 的「加入參考」功能）都會不知不覺地把你推向緊耦合。

Mary 的三個致命錯誤：

| 錯誤 | 位置 | 後果 |
|------|------|------|
| `new CommerceContext()` | ProductService 建構函式 | Domain Layer 緊耦合 Data Access Layer |
| `new ProductService()` | HomeController.Index() | UI Layer 緊耦合 Domain Layer |
| `Product` 定義在 Data Access Layer | Product.cs | 所有層都被迫依賴 Data Access Layer |

### 架構啟示

**啟示一：`new` 是一把雙面刃**
每次你對 Volatile Dependency 使用 `new`，你就在水泥裡灌入一根鋼筋——它把兩個模組永久焊死了。對 Stable Dependency（如 `List<T>`、`string`）使用 `new` 完全沒問題，但對 Volatile Dependency（需要執行環境配合的，如資料庫、檔案系統、外部服務）使用 `new` 就是在製造緊耦合。

**啟示二：依賴方向決定架構品質**
正確的架構不是「上層呼叫下層」這麼簡單。關鍵是依賴的方向：所有外層都應該依賴內層（Domain Layer），而不是內層依賴外層。這就是「依賴反轉原則（Dependency Inversion Principle）」的精髓。

```
  【傳統思維：誰呼叫誰，誰就依賴誰】

  UI ──呼叫──> Domain ──呼叫──> Data Access
  UI ──依賴──> Domain ──依賴──> Data Access   （依賴方向 = 呼叫方向）

  【DI 思維：依賴方向可以與呼叫方向相反】

  UI ──呼叫──> Domain ──呼叫──> Data Access   （呼叫方向不變）
  UI ──依賴──> Domain <──依賴── Data Access   （依賴方向反轉！）

  如何做到？Domain Layer 定義介面 (Interface)，
  Data Access Layer 去實作它。
```

**啟示三：可組合性是架構品質的試金石**
評估架構品質的最佳方式就是問：「能不能在不改現有程式碼的情況下，替換掉某個模組？」如果答案是「不行」，那你的架構就有問題。這不代表你一定會去替換，但這個能力本身代表了你的程式碼具備低耦合和高內聚。

**啟示四：設定應該是顯式的，不是隱式的**
任何類別都不該偷偷去讀設定檔。設定值應該像參數一樣明確傳入，這樣：
- 依賴是透明的（看 constructor 就知道需要什麼）
- 測試是容易的（直接傳入測試用的值）
- 彈性是最大的（不同情境傳入不同值）

**啟示五：團隊溝通中的「術語陷阱」**
Jens 用 Separation of Concerns、SRP、Agile 等術語來包裝錯誤的建議。身為架構師，你必須能識破「用正確術語說錯誤的話」這種情況。專業術語是溝通工具，不是論證武器。
