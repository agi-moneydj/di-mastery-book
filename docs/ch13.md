## 1. 概述 (Overview)

本章是 Autofac DI 容器的實戰操作手冊。在前面的章節中，我們已經學會了 DI 的原則與模式（Pure DI、Composition Root、Lifestyle 管理、Decorator、Composite 等），本章則將這些知識「落地」到 Autofac 這個具體的 DI 容器上。

Autofac 的核心使用節奏非常簡單——**先註冊（Register），後解析（Resolve）**。你可以把它想像成一本「食譜目錄」：先把所有食材與烹飪方式寫進目錄（ContainerBuilder），然後根據目錄建造一間廚房（Container），之後每次需要上菜時，就從廚房裡取出成品。

```
┌─────────────────────────────────────────────────────────┐
│                   Autofac 使用流程                        │
│                                                         │
│   ┌──────────────────┐    Build()    ┌──────────────┐   │
│   │ ContainerBuilder │──────────────>│  IContainer   │   │
│   │   (註冊階段)      │              │  (容器)        │   │
│   └──────────────────┘              └──────┬───────┘   │
│         ▲                                  │            │
│         │ RegisterType<T>()                │ BeginLifetimeScope() │
│         │ RegisterAssemblyTypes()          ▼            │
│         │ Register(delegate)        ┌──────────────┐   │
│         │ RegisterDecorator<T>()    │ILifetimeScope │   │
│                                     │  (生命週期範圍) │   │
│                                     └──────┬───────┘   │
│                                            │            │
│                                            │ Resolve<T>()│
│                                            ▼            │
│                                     ┌──────────────┐   │
│                                     │  元件實例      │   │
│                                     │  (Component)  │   │
│                                     └──────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**重點提醒**：永遠不要直接從根容器（root container）Resolve，必須從 `ILifetimeScope` 解析。直接從根容器解析會導致記憶體洩漏和並行錯誤。

## 3. 章節大綱 (Outline)

### 3.1 Autofac 入門 (Section 13.1)

#### 3.1.1 解析物件 (Resolving Objects)

Autofac 的最基本用法——建立 `ContainerBuilder`、註冊型別、建立容器、從 Lifetime Scope 解析：

```csharp
// 最簡單的 Autofac 使用方式
var builder = new ContainerBuilder();

// 註冊具體型別
builder.RegisterType<SauceBéarnaise>();

// 建造容器
IContainer container = builder.Build();

// 建立生命週期範圍 (絕對不要跳過這步！)
ILifetimeScope scope = container.BeginLifetimeScope();

// 從 scope 解析
SauceBéarnaise sauce = scope.Resolve<SauceBéarnaise>();
```

> **白話解釋**：`ContainerBuilder` 就像一張建築藍圖，`Build()` 是蓋好房子，`BeginLifetimeScope()` 是開一間臨時工作間。你在工作間裡取材料（Resolve），工作間結束時（Dispose），裡面暫用的東西會被妥善清理。

**將抽象映射到具體型別**：

```csharp
var builder = new ContainerBuilder();

// 將具體型別映射到抽象介面
builder.RegisterType<SauceBéarnaise>()
    .As<IIngredient>();  // 從具體型別 → 抽象介面

IContainer container = builder.Build();
ILifetimeScope scope = container.BeginLifetimeScope();

// 透過抽象介面解析
IIngredient sauce = scope.Resolve<IIngredient>();
```

> **注意**：Autofac 的方向與多數 DI 容器相反——它從具體型別開始，再映射到抽象。而多數容器是從抽象開始映射到具體型別。

**弱型別解析**（當你只有 `Type` 物件時）：

```csharp
// 適用於 ASP.NET Core MVC 的 IControllerActivator 場景
Type controllerType = context.ActionDescriptor.ControllerTypeInfo.AsType();
return scope.Resolve(controllerType);  // 弱型別重載
```

#### 3.1.2 配置 ContainerBuilder

Autofac 支援三種配置策略：

```
        明確性 (Explicit)
           ▲
           │
    ┌──────┴──────┐
    │ Config Files │ ◄─── 晚期綁定 (Late Binding)
    └─────────────┘
           │
    ┌──────┴──────────────┐
    │ Configuration       │
    │ as Code             │ ◄─── 早期綁定 (Early Binding)
    └─────────────────────┘
           │
    ┌──────┴──────────────┐
    │ Auto-Registration   │ ◄─── 慣例 (Convention-based)
    └─────────────────────┘
           │
           ▼
        隱含性 (Implicit)
```

**Configuration as Code（程式碼配置）**：

```csharp
// 基本映射
builder.RegisterType<SauceBéarnaise>().As<IIngredient>();

// 映射到多個抽象（避免 Torn Lifestyle）
builder.RegisterType<SauceBéarnaise>()
    .As<SauceBéarnaise, IIngredient>();  // 一次註冊，兩個服務

// 多重註冊：最後一個勝出
builder.RegisterType<SauceBéarnaise>().As<IIngredient>();
builder.RegisterType<Steak>().As<IIngredient>();
// Resolve<IIngredient>() → 得到 Steak
```

> **Torn Lifestyle（撕裂的生命週期）**：如果你對同一個型別呼叫了兩次 `RegisterType`，每次呼叫會有自己獨立的快取。即使都設為 Singleton，也會產生兩個不同的實例。這就像同一道菜在兩張菜單上各有一份——廚房會分別做兩盤。

**Auto-Registration（自動註冊）**：

```csharp
// 掃描組件，註冊所有 IIngredient 實作
Assembly ingredientsAssembly = typeof(Steak).Assembly;
builder.RegisterAssemblyTypes(ingredientsAssembly)
    .As<IIngredient>();

// 加入篩選條件
builder.RegisterAssemblyTypes(ingredientsAssembly)
    .Where(type => type.Name.StartsWith("Sauce"))
    .As<IIngredient>();

// 泛型抽象的自動註冊 (AsClosedTypesOf)
builder.RegisterAssemblyTypes(assembly)
    .AsClosedTypesOf(typeof(ICommandService<>));
```

> **白話解釋**：`RegisterAssemblyTypes` 就是 `RegisterType` 的複數版本。把它想像成：「去這個倉庫裡，把所有符合條件的零件都登記進來。」

**AsClosedTypesOf** 適用於泛型抽象。例如 `ICommandService<TCommand>` 是開放泛型，而 `AdjustInventoryService` 實作了 `ICommandService<AdjustInventory>`（封閉泛型）。`AsClosedTypesOf` 會自動找出並註冊所有封閉泛型實作。

**Configuration Files（設定檔配置）**：

```json
{
  "defaultAssembly": "Ploeh.Samples.MenuModel",
  "components": [
    {
      "services": [{
        "type": "Ploeh.Samples.MenuModel.IIngredient"
      }],
      "type": "Ploeh.Samples.MenuModel.Steak"
    }
  ]
}
```

```csharp
// 載入設定檔 (需要 Autofac.Configuration NuGet 套件)
var configuration = new ConfigurationBuilder()
    .AddJsonFile("autofac.json")
    .Build();

builder.RegisterModule(new ConfigurationModule(configuration));
```

> **設計指引**：設定檔只用在需要晚期綁定的場合（例如不重新編譯就能換實作）。主要配置應使用 Code 或 Auto-Registration。

### 3.2 管理生命週期 (Section 13.2)

#### Instance Scopes 概覽

| Autofac 名稱 | 通用模式名稱 | 方法 | 說明 |
|---|---|---|---|
| Per Dependency | **Transient** | `InstancePerDependency()` | 預設值，每次解析都產生新實例 |
| Single Instance | **Singleton** | `SingleInstance()` | 整個容器生命週期只有一個實例 |
| Per Lifetime Scope | **Scoped** | `InstancePerLifetimeScope()` | 每個 Lifetime Scope 內共享同一實例 |

```
┌─────────────────────────────────────────────────────────────┐
│                     Container (根容器)                        │
│                                                             │
│   ┌─────────────────┐                                       │
│   │   Singleton      │  ◄── 整個應用程式只有一份              │
│   └─────────────────┘                                       │
│                                                             │
│   ┌──────────────────────┐   ┌──────────────────────┐       │
│   │  ILifetimeScope A    │   │  ILifetimeScope B    │       │
│   │                      │   │                      │       │
│   │  ┌────────────────┐  │   │  ┌────────────────┐  │       │
│   │  │ Scoped (實例 X) │  │   │  │ Scoped (實例 Y) │  │       │
│   │  └────────────────┘  │   │  └────────────────┘  │       │
│   │                      │   │                      │       │
│   │  Transient: 每次新的  │   │  Transient: 每次新的  │       │
│   │  Transient: 每次新的  │   │  Transient: 每次新的  │       │
│   └──────────────────────┘   └──────────────────────┘       │
│                                                             │
│   Scope A 的 Scoped 實例 X ≠ Scope B 的 Scoped 實例 Y        │
│   但 Singleton 在 A、B 中都是同一個                            │
└─────────────────────────────────────────────────────────────┘
```

#### 3.2.1 配置 Instance Scopes

```csharp
// Singleton
builder.RegisterType<SauceBéarnaise>()
    .As<IIngredient>()
    .SingleInstance();

// Transient (預設，可明確宣告)
builder.RegisterType<SauceBéarnaise>()
    .InstancePerDependency();

// Scoped
builder.RegisterType<SauceBéarnaise>()
    .As<IIngredient>()
    .InstancePerLifetimeScope();

// Auto-Registration 也能設定 Lifestyle
Assembly ingredientsAssembly = typeof(Steak).Assembly;
builder.RegisterAssemblyTypes(ingredientsAssembly)
    .As<IIngredient>()
    .SingleInstance();
```

設定檔中也能配置：

```json
{
  "type": "Ploeh.Samples.MenuModel.Steak",
  "services": [{ "type": "Ploeh.Samples.MenuModel.IIngredient" }],
  "instance-scope": "single-instance"
}
```

#### 3.2.2 釋放元件 (Releasing Components)

Autofac 沒有明確的 `Release` 方法，而是使用 **Lifetime Scope** 的 `Dispose` 來釋放資源：

```csharp
// 建立 Lifetime Scope 並使用 using 區塊自動釋放
using (var scope = container.BeginLifetimeScope())
{
    IMeal meal = scope.Resolve<IMeal>();
    meal.Consume();
}
// 離開 using 區塊 → scope.Dispose() → 釋放該 scope 建立的所有元件
```

> **關鍵觀念**：
> - Transient 元件會被 Autofac 追蹤（即使是 Transient！），所以必須透過 Lifetime Scope 解析
> - Scoped 元件在 Scope Dispose 時被釋放
> - Singleton 在根容器 Dispose 時才被釋放
> - 從根容器直接解析 Scoped → 永遠得到同一個實例 → **並行錯誤**
> - 從根容器直接解析 Transient → 實例被根容器追蹤，直到應用結束才釋放 → **記憶體洩漏**

### 3.3 註冊困難的 API (Section 13.3)

#### 3.3.1 原始型別相依性 (Primitive Dependencies)

當建構子需要 string、int、enum 等基本型別時：

```csharp
public class ChiliConCarne
{
    public ChiliConCarne(Spiciness spiciness) { ... }
}

public enum Spiciness { Mild, Medium, Hot }
```

**方案一：全域註冊原始值**

```csharp
builder.Register<Spiciness>(c => Spiciness.Medium);
builder.RegisterType<ChiliConCarne>().As<ICourse>();
// 所有需要 Spiciness 的元件都會得到 Medium
```

**方案二：WithParameter 針對特定元件**

```csharp
builder.RegisterType<ChiliConCarne>().As<ICourse>()
    .WithParameter("spiciness", Spiciness.Hot);
```

**方案三（推薦）：Parameter Object（參數物件）**

```csharp
public class Flavoring
{
    public readonly Spiciness Spiciness;
    public readonly bool ExtraSalty;

    public Flavoring(Spiciness spiciness, bool extraSalty)
    {
        this.Spiciness = spiciness;
        this.ExtraSalty = extraSalty;
    }
}

// 註冊 Parameter Object
var flavoring = new Flavoring(Spiciness.Medium, extraSalty: true);
builder.RegisterInstance<Flavoring>(flavoring);
builder.RegisterType<ChiliConCarne>().As<ICourse>();
```

> **為什麼推薦 Parameter Object？** 因為它消除了歧義（不會有「哪個 string 對應哪個參數」的問題），同時也更好地表達了業務語意。`Flavoring` 比一個孤零零的 `Spiciness` enum 更有表達力。這也符合介面隔離原則（ISP）——每個消費者只注入自己需要的配置。

#### 3.3.2 用程式碼區塊註冊 (Code Blocks)

當需要使用靜態工廠或內部建構子時：

```csharp
// 靜態工廠
public static class JunkFoodFactory
{
    public static JunkFood Create(string name)
    {
        return new JunkFood(name);  // 內部建構子
    }
}

// 使用 Register 委派
builder.Register<IMeal>(c => JunkFoodFactory.Create("chicken meal"));
```

> **注意**：`Register` 方法是型別安全的，但它會**停用 Auto-Wiring**。使用它意味著你自己負責所有的建構邏輯。

### 3.4 處理多重元件 (Section 13.4)

#### 3.4.1 從多個候選者中選擇

**多重註冊：最後勝出**

```csharp
builder.RegisterType<Steak>().As<IIngredient>();
builder.RegisterType<SauceBéarnaise>().As<IIngredient>();
// Resolve<IIngredient>() → SauceBéarnaise（最後註冊的勝出）
```

**取得所有實作**：

```csharp
IEnumerable<IIngredient> ingredients =
    scope.Resolve<IEnumerable<IIngredient>>();
// 得到 Steak 和 SauceBéarnaise
```

**具名元件 (Named Components)**：

```csharp
builder.RegisterType<Steak>().Named<IIngredient>("meat");
builder.RegisterType<SauceBéarnaise>().Named<IIngredient>("sauce");

IIngredient meat = scope.ResolveNamed<IIngredient>("meat");
IIngredient sauce = scope.ResolveNamed<IIngredient>("sauce");
```

**處理歧義建構子 — WithParameter 的進階用法**：

```csharp
public class ThreeCourseMeal : IMeal
{
    public ThreeCourseMeal(ICourse entree, ICourse mainCourse, ICourse dessert)
    { ... }
}

// 註冊具名 ICourse
builder.RegisterType<Rillettes>().Named<ICourse>("entree");
builder.RegisterType<CordonBleu>().Named<ICourse>("mainCourse");
builder.RegisterType<MousseAuChocolat>().Named<ICourse>("dessert");

// 使用自定義輔助方法簡化註冊
builder.RegisterType<ThreeCourseMeal>().As<IMeal>()
    .WithParameter(Named("entree"), InjectWith<ICourse>("entree"))
    .WithParameter(Named("mainCourse"), InjectWith<ICourse>("mainCourse"))
    .WithParameter(Named("dessert"), InjectWith<ICourse>("dessert"));

// 輔助方法
Func<ParameterInfo, IComponentContext, bool> Named(string name)
    => (p, c) => p.Name == name;

Func<ParameterInfo, IComponentContext, object> InjectWith<T>(string name)
    => (p, c) => c.ResolveNamed<T>(name);
```

**慣例式解析 (Convention-based)**：

```csharp
// 當參數名稱與具名元件名稱相同時，可以一句搞定
builder.RegisterType<ThreeCourseMeal>().As<IMeal>()
    .WithParameter(
        (p, c) => true,
        (p, c) => c.ResolveNamed(p.Name, p.ParameterType));
```

#### 3.4.2 佈線序列 (Wiring Sequences)

**消除歧義的最佳方式——重構為序列**：

```csharp
// 與其 ThreeCourseMeal(ICourse a, ICourse b, ICourse c) 三個同型別參數
// 不如 Meal(IEnumerable<ICourse> courses) 一個序列參數
public class Meal : IMeal
{
    public Meal(IEnumerable<ICourse> courses) { ... }
}
```

**Auto-Wiring 序列**：

```csharp
builder.RegisterType<Rillettes>().As<ICourse>();
builder.RegisterType<CordonBleu>().As<ICourse>();
builder.RegisterType<MousseAuChocolat>().As<ICourse>();
builder.RegisterType<Meal>().As<IMeal>();
// Autofac 自動把三道菜全部注入 Meal
```

**只選部分元件**：

```csharp
builder.RegisterType<Meal>().As<IMeal>()
    .WithParameter(
        (p, c) => true,
        (p, c) => new[]
        {
            c.ResolveNamed<ICourse>("entree"),
            c.ResolveNamed<ICourse>("mainCourse"),
            c.ResolveNamed<ICourse>("dessert")
        });
```

#### 3.4.3 佈線裝飾器 (Wiring Decorators)

**非泛型裝飾器**：

```csharp
// VealCutlet 被 HamCheeseGarlic 裝飾，再被 Breading 裝飾
builder.RegisterType<VealCutlet>()
    .As<IIngredient>();

builder.RegisterDecorator<HamCheeseGarlic, IIngredient>();
builder.RegisterDecorator<Breading, IIngredient>();

// 等同於 Pure DI：
// new Breading(new HamCheeseGarlic(new VealCutlet()))
```

```
┌─────────────────────────────────────┐
│           Breading                  │  ◄── 最外層裝飾器
│  ┌───────────────────────────────┐  │
│  │      HamCheeseGarlic         │  │  ◄── 中間裝飾器
│  │  ┌─────────────────────────┐  │  │
│  │  │      VealCutlet         │  │  │  ◄── 核心實作
│  │  └─────────────────────────┘  │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

> **注意**：Autofac 按照**註冊順序**套用裝飾器。先註冊的先包裹，後註冊的在更外層。

**泛型裝飾器**（搭配 Auto-Registration）：

```csharp
// 1. 自動註冊所有 ICommandService<TCommand> 實作
builder.RegisterAssemblyTypes(assembly)
    .AsClosedTypesOf(typeof(ICommandService<>));

// 2. 依序套用泛型裝飾器
builder.RegisterGenericDecorator(
    typeof(AuditingCommandServiceDecorator<>),
    typeof(ICommandService<>));

builder.RegisterGenericDecorator(
    typeof(TransactionCommandServiceDecorator<>),
    typeof(ICommandService<>));

builder.RegisterGenericDecorator(
    typeof(SecureCommandServiceDecorator<>),
    typeof(ICommandService<>));
```

```
┌───────────────────────────────────────────┐
│      SecureCommandServiceDecorator<T>     │
│  ┌───────────────────────────────────┐    │
│  │ TransactionCommandServiceDecorator│    │
│  │  ┌─────────────────────────────┐  │    │
│  │  │ AuditingCommandService...   │  │    │
│  │  │  ┌───────────────────────┐  │  │    │
│  │  │  │ Actual CommandService │  │  │    │
│  │  │  └───────────────────────┘  │  │    │
│  │  └─────────────────────────────┘  │    │
│  └───────────────────────────────────┘    │
└───────────────────────────────────────────┘
```

#### 3.4.4 佈線組合模式 (Wiring Composites)

**非泛型 Composite**：

```csharp
public class CompositeNotificationService : INotificationService
{
    private readonly IEnumerable<INotificationService> services;

    public CompositeNotificationService(
        IEnumerable<INotificationService> services)
    {
        this.services = services;
    }

    public void OrderApproved(Order order)
    {
        foreach (var service in this.services)
        {
            service.OrderApproved(order);
        }
    }
}
```

```csharp
// 使用具名註冊避免循環依賴
builder.RegisterAssemblyTypes(assembly)
    .Except<CompositeNotificationService>()           // 排除 Composite 自身
    .Where(type => !IsDecoratorFor<INotificationService>(type)) // 排除 Decorator
    .Named<INotificationService>("service");

// 用委派手動建立 Composite
builder.Register(c =>
    new CompositeNotificationService(
        c.ResolveNamed<IEnumerable<INotificationService>>("service")))
    .As<INotificationService>();
```

```
┌─────────────────────────────────────────────┐
│       CompositeNotificationService          │
│                                             │
│   ┌─────────────────────────────────────┐   │
│   │  IEnumerable<INotificationService>  │   │
│   │                                     │   │
│   │  ┌─────────────────────────────┐    │   │
│   │  │ OrderApprovedReceiptSender  │    │   │
│   │  ├─────────────────────────────┤    │   │
│   │  │ AccountingNotifier          │    │   │
│   │  ├─────────────────────────────┤    │   │
│   │  │ OrderFulfillment            │    │   │
│   │  └─────────────────────────────┘    │   │
│   └─────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

> **循環依賴陷阱**：如果 Composite 跟其他實作放在同一個組件，`RegisterAssemblyTypes` 會連 Composite 自身也掃進序列，造成循環依賴。解法是用 `Except<T>()` 排除，或把 Composite 搬到不同的組件（如 Composition Root 所在的組件）。

**泛型 Composite**：

```csharp
// 自動註冊所有 IEventHandler<TEvent> 實作（具名）
builder.RegisterAssemblyTypes(assembly)
    .As(type =>
        from interfaceType in type.GetInterfaces()
        where interfaceType.IsClosedTypeOf(typeof(IEventHandler<>))
        select new KeyedService("handler", interfaceType));

// 註冊泛型 Composite
builder.RegisterGeneric(typeof(CompositeEventHandler<>))
    .As(typeof(IEventHandler<>))
    .WithParameter(
        (p, c) => true,
        (p, c) => c.ResolveNamed("handler", p.ParameterType));
```

> **好消息**：泛型 Composite 和 Decorator 不需要從 `RegisterAssemblyTypes` 中過濾掉，因為 `RegisterAssemblyTypes` 只選取非泛型的實作。

## 4. 關鍵框架與公式 (Key Frameworks)

### 4.1 Autofac API 核心方法速查表

| 方法 | 用途 | Auto-Wiring | 範例 |
|---|---|---|---|
| `RegisterType<T>()` | 註冊單一具體型別 | 是 | `builder.RegisterType<Steak>()` |
| `.As<T>()` | 映射到抽象介面 | - | `.As<IIngredient>()` |
| `RegisterAssemblyTypes()` | 批次掃描組件註冊 | 是 | `builder.RegisterAssemblyTypes(asm)` |
| `.AsClosedTypesOf()` | 泛型抽象的自動註冊 | 是 | `.AsClosedTypesOf(typeof(ICmd<>))` |
| `Register<T>(delegate)` | 委派建立實例 | **否** | `builder.Register(c => ...)` |
| `RegisterInstance<T>()` | 註冊既有實例 | **否** | `builder.RegisterInstance(obj)` |
| `RegisterGeneric()` | 註冊開放泛型 | 是 | `builder.RegisterGeneric(typeof(X<>))` |
| `RegisterDecorator<T,S>()` | 註冊非泛型裝飾器 | 是 | `builder.RegisterDecorator<D,I>()` |
| `RegisterGenericDecorator()` | 註冊泛型裝飾器 | 是 | `builder.RegisterGenericDecorator(...)` |
| `.WithParameter()` | 覆寫特定參數 | 部分 | `.WithParameter("name", value)` |
| `.Named<T>(name)` | 具名註冊 | 是 | `.Named<IIngredient>("meat")` |
| `.Keyed<T>(key)` | 鍵值註冊 | 是 | `.Keyed<IIngredient>(myKey)` |

### 4.2 Lifestyle 選擇決策流程

```
開始
  │
  ▼
該元件是否為 thread-safe？
  │
  ├── 是 ──► 是否有可變狀態需要共享？
  │              │
  │              ├── 否 ──► Singleton (SingleInstance)
  │              │           效能最好，一個實例服務所有請求
  │              │
  │              └── 是 ──► Scoped (InstancePerLifetimeScope)
  │                          每個 Scope 一個實例
  │
  └── 否 ──► Transient (InstancePerDependency) [預設]
              每次請求都建立新實例，最安全
```

### 4.3 Torn Lifestyle 防範公式

```
  錯誤做法（兩次 RegisterType = 兩個快取 = 撕裂）
  ═══════════════════════════════════════════════
  builder.RegisterType<X>().SingleInstance();
  builder.RegisterType<X>().As<IX>().SingleInstance();
  // Resolve<X>() ≠ Resolve<IX>()  ← 兩個不同的 Singleton！

  正確做法（一次 RegisterType + 多個 As = 一個快取）
  ═══════════════════════════════════════════════
  builder.RegisterType<X>()
      .As<X, IX>()      // 或 .As<X>().As<IX>()
      .SingleInstance();
  // Resolve<X>() == Resolve<IX>()  ← 同一個 Singleton
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點一：Lifetime Scope 的記憶體洩漏與並行問題

**為什麼不能直接從根容器 Resolve？**

這是本章最容易被忽略卻最危險的觀念。Autofac 會追蹤幾乎所有元件（包括 Transient），以便在容器 Dispose 時清理它們。

```
從根容器直接 Resolve 的後果：

┌────────────────────────────────────────────────┐
│            Root Container                      │
│                                                │
│  追蹤清單 (永遠不會清理，直到 app 結束):         │
│  ┌──────────┐                                  │
│  │ Transient │ ← Resolve #1 (累積!)            │
│  │ Transient │ ← Resolve #2 (累積!)            │
│  │ Transient │ ← Resolve #3 (累積!)            │
│  │ ...       │ ← 無限累積 = 記憶體洩漏          │
│  └──────────┘                                  │
│                                                │
│  ┌──────────┐                                  │
│  │ Scoped   │ ← 永遠是同一個實例               │
│  └──────────┘     多執行緒共用 = 並行 Bug        │
└────────────────────────────────────────────────┘

正確做法：

┌────────────────────────────────────────────────┐
│            Root Container                      │
│                                                │
│  ┌─────────── Scope (web request) ──────────┐  │
│  │ Transient → 用完就跟 Scope 一起清理       │  │
│  │ Scoped   → 此 Request 專用的實例          │  │
│  └──────────────────────────────────────────┘  │
│  Scope Dispose → 全部釋放，不會洩漏            │
└────────────────────────────────────────────────┘
```

**實戰建議**：在 ASP.NET Core 中，每個 HTTP 請求會自動建立一個 Lifetime Scope（透過 Autofac 的整合套件），你不需要手動管理。但在 Console App 或 Background Service 中，必須自己建立 Scope。

### 難點二：WithParameter 的雙委派模式

`WithParameter` 的兩個委派版本是最容易搞混的 API：

```csharp
.WithParameter(
    // 第一個委派：predicate — 決定「要不要處理這個參數」
    (ParameterInfo p, IComponentContext c) => p.Name == "entree",

    // 第二個委派：valueAccessor — 提供「這個參數的值」
    (ParameterInfo p, IComponentContext c) => c.ResolveNamed<ICourse>("entree")
)
```

> **比喻**：想像你是餐廳的領班。第一個委派是「這張桌子是 VIP 桌嗎？」（判斷條件），第二個委派是「如果是的話，上什麼酒？」（提供值）。Autofac 會對每個建構子參數依序詢問所有 `WithParameter`，直到找到匹配的 predicate。

**進階技巧 — ResolvedParameter 的本質**：

```csharp
// WithParameter 內部其實包裝了 ResolvedParameter
public ResolvedParameter(
    Func<ParameterInfo, IComponentContext, bool> predicate,
    Func<ParameterInfo, IComponentContext, object> valueAccessor);

// predicate 返回 true → 呼叫 valueAccessor 取得值
// predicate 返回 false → 跳過，嘗試下一個 WithParameter 或 Auto-Wiring
```

### 難點三：泛型 Composite 的 WithParameter 與 KeyedService

泛型 Composite 的註冊涉及多個進階概念的交織：

```csharp
// 第一步：用 KeyedService 註冊具名實作
builder.RegisterAssemblyTypes(assembly)
    .As(type =>
        from interfaceType in type.GetInterfaces()
        where interfaceType.IsClosedTypeOf(typeof(IEventHandler<>))
        select new KeyedService("handler", interfaceType));
```

> **拆解**：`KeyedService` 是 Autofac 的內部資料物件，它同時攜帶 key（"handler"）和服務型別（例如 `IEventHandler<OrderApproved>`）。LINQ 查詢會檢查每個型別實作了哪些封閉泛型介面，對於不實作 `IEventHandler<>` 的型別，查詢返回空集合，不會被註冊。

```csharp
// 第二步：註冊泛型 Composite
builder.RegisterGeneric(typeof(CompositeEventHandler<>))
    .As(typeof(IEventHandler<>))
    .WithParameter(
        (p, c) => true,
        (p, c) => c.ResolveNamed("handler", p.ParameterType));
```

> **關鍵洞察**：當你解析 `IEventHandler<OrderApproved>` 時，`p.ParameterType` 會是 `IEnumerable<IEventHandler<OrderApproved>>`（因為 Composite 的建構子接受序列）。將這個封閉型別連同 key "handler" 傳給 `ResolveNamed`，Autofac 就會找到所有以 "handler" 為 key、型別符合的已註冊元件。

```
解析流程：

Resolve<IEventHandler<OrderApproved>>()
    │
    ▼
找到 CompositeEventHandler<OrderApproved> (預設註冊)
    │
    ▼
建構子需要 IEnumerable<IEventHandler<OrderApproved>>
    │
    ▼
WithParameter predicate → true
    │
    ▼
valueAccessor → ResolveNamed("handler", typeof(IEnumerable<IEventHandler<OrderApproved>>))
    │
    ▼
找到所有 key="handler" 且實作 IEventHandler<OrderApproved> 的元件
    │
    ▼
注入 [Handler1, Handler2, Handler3] 到 Composite
```

## 6. 本章小結與架構啟示

### 核心收穫

1. **Autofac 的設計哲學是「明確性」**：它不會猜測你的意圖，而是提供清晰的 API 讓你明確表達。`ContainerBuilder`（配置）和 `IContainer`（消費）的分離就是這種哲學的體現——配置階段不能 Resolve，Resolve 階段不能再註冊。

2. **Lifetime Scope 是 Autofac 的核心概念**：它不僅僅是「Scoped lifestyle」的實現，更是 Autofac 釋放和清理資源的唯一機制。忽略它就等於忽略了整個記憶體管理模型。

3. **Auto-Wiring 是預設、是最佳實踐**：只有在 API 設計不良（多個同型別參數、原始型別依賴、靜態工廠）時才需要 `WithParameter` 或 `Register` 委派。如果你發現自己大量使用這些進階 API，應該回頭審視應用程式的設計。

4. **Decorator 和 Composite 是多重註冊的兩種典型場景**：
   - Decorator：層次關係（A 包 B 包 C），用 `RegisterDecorator` / `RegisterGenericDecorator`
   - Composite：平行關係（A 管理 [B, C, D]），需要具名註冊 + 手動建構

### 架構啟示

```
┌──────────────────────────────────────────────────────────────┐
│              DI 容器是設計品質的放大鏡                         │
│                                                              │
│  如果你的設計遵循 SOLID 原則：                                 │
│  → DI 容器的配置會非常簡潔                                    │
│  → Auto-Wiring + Auto-Registration 就能搞定 90% 的工作        │
│                                                              │
│  如果你的設計違反 SOLID 原則：                                 │
│  → DI 容器的配置會異常複雜                                    │
│  → 到處都是 WithParameter、Register 委派、具名元件             │
│  → 這不是容器的問題，是設計的問題                              │
│                                                              │
│  當配置變得困難時，先檢視設計，再檢視容器用法                   │
└──────────────────────────────────────────────────────────────┘
```

**設計改善方向**：
- 歧義的多參數（如 `ThreeCourseMeal`）→ 重構為接受序列的 `Meal`
- 原始型別依賴 → 抽取為 Parameter Object
- 靜態工廠 / 內部建構子 → 盡量改為公開建構子 + Constructor Injection
- Composition Root 配置越簡單，代表你的應用架構越健康
