## 1. 概述 (Overview)

本章的核心主題是 **攔截 (Interception)** ——在兩個協作物件之間「攔截」呼叫，並在不修改任何一方的前提下，注入額外的行為。作者以烹飪比喻開場：一塊小牛排本身可以吃，但透過裹粉、調味、填入火腿與起司，就變成了精緻的 Cordon Bleu（藍帶排）。軟體也一樣——基於鬆散耦合與面向介面程式設計，你可以用 **Decorator 設計模式** 不斷地為核心實作「加料」，在不動到原始程式碼的情況下，疊加諸如稽核 (Auditing)、斷路器 (Circuit Breaker)、例外處理 (Exception Handling)、授權 (Authorization) 等 **橫切關注點 (Cross-Cutting Concerns)**。

> **一句話摘要**：Interception 就是 Decorator 模式在 DI 世界中的實戰應用，它讓你能以「包裝」的方式為既有元件疊加橫切關注點，而不破壞原始類別的單一職責。

### 名詞白話解釋

| 名詞 | 白話解釋 | 比喻 |
|------|---------|------|
| **Interception (攔截)** | 在呼叫者與服務之間插入一段中間程式碼，可以在呼叫前後做額外的事 | 快遞包裹在送達前經過海關檢查站——海關不改變包裹內容，但可以檢查、記錄、甚至退回 |
| **Decorator Pattern (裝飾器模式)** | 用一個新類別包裝既有類別，兩者實作同一介面，新類別在委派呼叫的前後加入額外行為 | 俄羅斯套娃——每層娃娃外觀一樣，但越往外層越大、越華麗 |
| **Cross-Cutting Concern (橫切關注點)** | 不屬於特定業務功能，但會橫跨整個系統的非功能性需求（如日誌、安全、快取） | 大樓裡的消防系統——它不是任何一個房間的「功能」，但每層樓都需要 |
| **Circuit Breaker (斷路器)** | 當遠端服務連續失敗時，暫時中斷呼叫以「快速失敗」，避免系統資源耗盡 | 家裡的保險絲——電流過大時自動斷電，保護整條電路 |
| **Open/Closed Principle (開放封閉原則)** | 對擴展開放、對修改封閉。透過新增類別來擴展行為，而不是改既有程式碼 | 手機殼——你可以換各種殼來增加防摔或美觀，但不需要拆開手機改硬體 |

## 3. 章節大綱 (Outline)

### 3.1 攔截概論 (Section 9.1 — Introducing Interception)

**核心觀念**：攔截是在消費者 (Consumer) 與服務 (Service) 之間插入中間程式碼，讓你能在服務被呼叫前後執行額外邏輯，且雙方都不需要改動。

```
  ┌──────────┐        ┌──────────────────┐        ┌──────────┐
  │ Consumer │──uses──▶│ Intermediate Code│──uses──▶│ Service  │
  └──────────┘        │  (Decorator)     │        └──────────┘
                      └──────────────────┘
                       在轉發呼叫前後
                       可以做額外的事
```

> **關鍵定義**：Interception 是攔截兩個協作元件之間呼叫的能力，讓你能在不改變這兩個協作者的前提下，豐富或改變 Dependency 的行為。

**致能條件**：鬆散耦合 (Loose Coupling) 與 Liskov 替換原則 (LSP) 是攔截的前提。沒有面向介面程式設計，就無法實現攔截。

### 3.2 Decorator 設計模式 (Section 9.1.1)

Decorator 的結構本質：**一個類別同時「實作」與「包裝」同一個抽象介面**。

```
        ┌─────────────────────┐
        │    <<interface>>    │
        │     IComponent      │
        │  + Operation()      │
        └──────┬──────────────┘
               │ implements
       ┌───────┴────────┐
       │                │
  ┌────▼─────┐   ┌──────▼──────────┐
  │ Concrete │   │   Decorator     │
  │Component │   │                 │
  │          │   │ -decoratee:     │
  │+Operation│   │  IComponent     │──wraps──▶ IComponent
  └──────────┘   │                 │
                 │ +Operation()    │
                 │  { decoratee    │
                 │    .Operation();│
                 │  }              │
                 └─────────────────┘
```

**Decorator 的三種行為模式：**

```csharp
// ===== 模式 1：修改輸入後再委派 =====
public string Greet(string name)
{
    string titledName = "Mr. " + name;          // 修改輸入
    return this.decoratee.Greet(titledName);     // 委派
}

// ===== 模式 2：委派後修改輸出 =====
public string Greet(string name)
{
    string greet = this.decoratee.Greet(name);   // 先委派
    return greet + " Nice to meet you.";          // 修改輸出
}

// ===== 模式 3：短路 (不呼叫 decoratee) =====
public string Greet(string name)
{
    if (name == null)
        return "Hello world!";                    // 直接返回，不委派

    return this.decoratee.Greet(name);
}
```

**巢狀組合（俄羅斯套娃）：**

```csharp
IGreeter greeter =
    new NiceToMeetYouGreeterDecorator(         // 最外層：附加問候語
        new TitledGreeterDecorator(             // 中間層：加上稱謂
            new FormalGreeter()));              // 核心：產生正式問候

string result = greeter.Greet("Samuel L. Jackson");
// 輸出: "Hello, Mr. Samuel L. Jackson. Nice to meet you."
```

呼叫流程示意：

```
呼叫方向 ──────────────────────────────────────────▶

NiceToMeetYou        TitledGreeter        FormalGreeter
  Decorator            Decorator
 ┌─────────┐        ┌─────────────┐      ┌────────────┐
 │ Greet() │──委派──▶│ 加 "Mr."    │─委派─▶│ 產生問候語 │
 │         │        │ Greet()     │      │ Greet()    │
 │ 附加     │◀─回傳──│             │◀回傳──│            │
 │"Nice to" │        └─────────────┘      └────────────┘
 └─────────┘
◀────────────────────────────────────────── 回傳方向
```

### 3.3 實例：Auditing Decorator (Section 9.1.2)

**情境**：為 `IUserRepository` 新增稽核功能，記錄所有「寫入」操作。

```
     ┌───────────────────┐
     │  <<interface>>     │
     │  IUserRepository   │
     │  + GetById()       │
     │  + Update()        │
     └────────┬───────────┘
              │ implements
     ┌────────┴──────────────┐
     │                       │
┌────▼──────────┐  ┌─────────▼──────────────────┐
│ SqlUser       │  │ AuditingUserRepository      │
│ Repository    │  │ Decorator                   │
│               │  │                             │
│               │  │ - decoratee: IUserRepository│
│               │  │ - appender: IAuditTrail     │
│               │  │            Appender         │
└───────────────┘  └─────────────────────────────┘
```

```csharp
public class AuditingUserRepositoryDecorator : IUserRepository
{
    private readonly IAuditTrailAppender appender;
    private readonly IUserRepository decoratee;

    public AuditingUserRepositoryDecorator(
        IAuditTrailAppender appender,
        IUserRepository decoratee)
    {
        this.appender = appender;
        this.decoratee = decoratee;
    }

    // 讀取操作：直接委派，不做稽核
    public User GetById(Guid id)
    {
        return this.decoratee.GetById(id);
    }

    // 寫入操作：委派後追加稽核記錄
    public void Update(User user)
    {
        this.decoratee.Update(user);
        this.appender.Append(user);     // 稽核！
    }
}
```

**Composition Root 中的接線：**

```csharp
private HomeController CreateHomeController()
{
    var context = new CommerceContext();

    IAuditTrailAppender appender =
        new SqlAuditTrailAppender(this.userContext, context);

    // 核心：用 Decorator 包裝真實 Repository
    IUserRepository userRepository =
        new AuditingUserRepositoryDecorator(
            appender,
            new SqlUserRepository(context));   // 被裝飾的原始實作

    IProductService productService =
        new ProductService(
            new SqlProductRepository(context),
            this.userContext,
            userRepository);  // ProductService 只看到 IUserRepository

    return new HomeController(productService);
}
```

> **架構啟示**：`SqlUserRepository` 完全不知道稽核的存在，`ProductService` 也不知道它拿到的是 Decorator。這就是 **Open/Closed Principle** 的威力——你透過新增類別擴展功能，而非修改既有程式碼。

### 3.4 Cross-Cutting Concerns 總覽 (Section 9.2)

以下是常見的橫切關注點：

```
  ┌─────────────────────────────────────────────────────┐
  │              Application Architecture               │
  │  ┌────────────────┐                                 │
  │  │  UI Layer       │  ┌───────┐ ┌────────┐ ┌─────┐ │
  │  ├────────────────┤  │       │ │        │ │     │ │
  │  │  Domain Layer   │  │ 安全  │ │ 日誌   │ │快取 │ │
  │  ├────────────────┤  │       │ │        │ │     │ │
  │  │  Data Access    │  │       │ │        │ │     │ │
  │  └────────────────┘  └───────┘ └────────┘ └─────┘ │
  │       水平分層            垂直的橫切關注點            │
  └─────────────────────────────────────────────────────┘
```

| 橫切關注點 | 說明 |
|-----------|------|
| Auditing（稽核） | 記錄資料異動的時間、使用者、變更內容 |
| Logging（日誌） | 記錄應用程式狀態事件 |
| Performance Monitoring（效能監控） | 追蹤 SLA 指標 |
| Validation（驗證） | 確保操作被呼叫時傳入合法資料 |
| Security（安全） | 確保只有授權使用者能執行特定操作 |
| Caching（快取） | 提升效能，且與資料存取元件解耦 |
| Error Handling（錯誤處理） | 統一例外處理策略 |
| Fault Tolerance（容錯） | 如 Circuit Breaker，處理暫時性故障 |

### 3.5 Circuit Breaker Decorator (Section 9.2.1)

**問題**：當遠端服務（如 WCF、Web API）無回應時，重複呼叫只會讓情況更糟——執行緒被阻塞、服務被壓垮。

**解法**：Circuit Breaker 模式——偵測到故障後「跳閘」，快速失敗 (Fail Fast)，一段時間後再嘗試恢復。

```
  狀態轉換圖：

     呼叫成功                        錯誤發生
   ┌──────────┐                 ┌──────────┐
   │          │    Error        │          │
   │  Closed  │────────────────▶│   Open   │
   │ (正常通行) │                │ (全部拒絕) │
   │          │◀───────┐       │          │
   └──────────┘        │       └─────┬────┘
                       │             │
                  呼叫成功        逾時 (Timeout)
                       │             │
                 ┌─────┴──────┐      │
                 │ Half-Open  │◀─────┘
                 │(允許一個呼叫) │
                 │            │──Error──▶ 回到 Open
                 └────────────┘
```

```csharp
public class CircuitBreakerProductRepositoryDecorator
    : IProductRepository
{
    private readonly ICircuitBreaker breaker;
    private readonly IProductRepository decoratee;

    public CircuitBreakerProductRepositoryDecorator(
        ICircuitBreaker breaker,
        IProductRepository decoratee)
    {
        this.breaker = breaker;
        this.decoratee = decoratee;
    }

    public void Insert(Product product)
    {
        this.breaker.Guard();        // 1. 檢查狀態：Open 則直接拋例外

        try
        {
            this.decoratee.Insert(product);  // 2. 委派呼叫
            this.breaker.Succeed();          // 3. 成功 → 通知 breaker
        }
        catch (Exception ex)
        {
            this.breaker.Trip(ex);           // 4. 失敗 → 跳閘
            throw;
        }
    }

    public IEnumerable<Product> GetAll()
    {
        this.breaker.Guard();

        try
        {
            var products = this.decoratee.GetAll();
            this.breaker.Succeed();
            return products;                 // 有回傳值時需先暫存
        }
        catch (Exception ex)
        {
            this.breaker.Trip(ex);
            throw;
        }
    }
}
```

**替代設計——更精簡的 `ICircuitBreaker` 介面：**

```csharp
// 使用委派 (delegate) 讓介面更精簡
public interface ICircuitBreaker
{
    T Execute<T>(Func<T> action);
}

// 使用起來更簡潔
public IEnumerable<Product> GetAll()
{
    return this.breaker.Execute(() => this.decoratee.GetAll());
}
```

**Composition Root 接線：**

```csharp
var channelFactory = new ChannelFactory<IProductManagementService>("*");
var timeout = TimeSpan.FromMinutes(1);
ICircuitBreaker breaker = new CircuitBreaker(timeout);

IProductRepository repository =
    new CircuitBreakerProductRepositoryDecorator(
        breaker,
        new WcfProductRepository(channelFactory));
```

> **生命週期注意**：`CircuitBreaker` 通常應該是 **Singleton**，因為不論系統建立了多少個 Decorator 實例，它們都應共用同一個 breaker 狀態。但 Singleton 意味著需要 **執行緒安全 (Thread-Safety)**。

### 3.6 Exception Handling Decorator (Section 9.2.2)

**問題**：Circuit Breaker 讓系統快速失敗，但仍然會拋出例外，若未處理會導致應用程式崩潰。

**解法**：再疊一層 Decorator，專門處理特定例外並以友善方式通知使用者。

```csharp
public class ErrorHandlingProductRepositoryDecorator
    : IProductRepository
{
    private readonly IProductRepository decoratee;

    // ... 建構式省略 ...

    public void Insert(Product product)
    {
        try
        {
            this.decoratee.Insert(product);
        }
        catch (CommunicationException ex)
        {
            this.AlertUser(ex.Message);     // 通訊例外 → 提示使用者
        }
        catch (InvalidOperationException ex)
        {
            this.AlertUser(ex.Message);     // 操作例外 → 提示使用者
        }
    }

    private void AlertUser(string message)
    {
        // 格式化錯誤訊息並用 MessageBox 顯示
        MessageBox.Show(message, "Error",
            MessageBoxButton.OK, MessageBoxImage.Warning);
    }
}
```

> **設計原則**：只捕獲特定的已知例外類型，不要做 `catch (Exception)`，因為壓制所有例外是危險的行為。

### 3.7 Authorization Decorator (Section 9.2.3)

**問題**：只有特定角色的使用者能執行寫入操作。

**解法**：用 Decorator 在呼叫前檢查權限，不合格則拋出 `SecurityException`。

```csharp
public class SecureProductRepositoryDecorator
    : IProductRepository
{
    private readonly IUserContext userContext;
    private readonly IProductRepository decoratee;

    public SecureProductRepositoryDecorator(
        IUserContext userContext,
        IProductRepository decoratee)
    {
        this.userContext = userContext;
        this.decoratee = decoratee;
    }

    // 刪除操作需要權限檢查
    public void Delete(Guid id)
    {
        this.CheckAuthorization();          // Guard Clause
        this.decoratee.Delete(id);
    }

    // 查詢操作不需要權限檢查
    public IEnumerable<Product> GetAll()
    {
        return this.decoratee.GetAll();     // 直接委派
    }

    private void CheckAuthorization()
    {
        if (!this.userContext.IsInRole(Role.Administrator))
        {
            throw new SecurityException("Access denied.");
        }
    }
}
```

### 3.8 完整 Decorator 堆疊示意

多個 Decorator 可以層層堆疊，形成一條攔截管線 (Pipeline)：

```
  Consumer
     │
     ▼
  ┌──────────────────────────────────────────┐
  │ SecureProductRepositoryDecorator         │  ← 最外層：權限檢查
  │  ┌───────────────────────────────────┐   │
  │  │ ErrorHandlingProductRepository    │   │  ← 第二層：例外處理
  │  │  Decorator                        │   │
  │  │  ┌────────────────────────────┐   │   │
  │  │  │ CircuitBreakerProduct      │   │   │  ← 第三層：斷路器
  │  │  │ RepositoryDecorator        │   │   │
  │  │  │  ┌─────────────────────┐   │   │   │
  │  │  │  │ WcfProduct          │   │   │   │  ← 核心：真實實作
  │  │  │  │ Repository          │   │   │   │
  │  │  │  └─────────────────────┘   │   │   │
  │  │  └────────────────────────────┘   │   │
  │  └───────────────────────────────────┘   │
  └──────────────────────────────────────────┘
```

在 Composition Root 中的接線：

```csharp
IProductRepository repository =
    new SecureProductRepositoryDecorator(
        userContext,
        new ErrorHandlingProductRepositoryDecorator(
            new CircuitBreakerProductRepositoryDecorator(
                breaker,
                new WcfProductRepository(channelFactory))));
```

> **順序很重要**：Security 應在最外層（先檢查權限），ErrorHandling 在中間（捕獲下層的例外），CircuitBreaker 緊鄰核心（直接保護網路呼叫）。

## 4. 關鍵框架與公式 (Key Frameworks)

### 框架 1：Decorator 的結構公式

```
Decorator = 實作介面 I + 包裝介面 I 的實例 + 在委派前/後加入行為
```

任何 Decorator 都遵循此固定骨架：

```csharp
public class XxxDecorator : ISomeAbstraction
{
    private readonly ISomeAbstraction decoratee;
    // 可能還有其他 Dependencies

    public XxxDecorator(ISomeAbstraction decoratee /*, 其他依賴 */)
    {
        this.decoratee = decoratee;
    }

    public ReturnType SomeMethod(params...)
    {
        // [可選] 前置行為 (Before)
        var result = this.decoratee.SomeMethod(params);  // 委派
        // [可選] 後置行為 (After)
        return result;
    }
}
```

### 框架 2：Interception 決策矩陣

| 你想做什麼？ | 適用的 Decorator 行為 | 範例 |
|-------------|---------------------|------|
| 在呼叫前加入邏輯 | 修改輸入 / Guard Clause | Authorization、Validation |
| 在呼叫後加入邏輯 | 修改輸出 / 附加行為 | Auditing、Logging |
| 包裹整個呼叫 | try/catch 或狀態管理 | Circuit Breaker、Error Handling |
| 取代呼叫 | 短路，不委派 decoratee | Caching（命中快取時直接回傳） |

### 框架 3：Cross-Cutting Concern 與 SOLID 的對應

| SOLID 原則 | 如何被 Interception 實現 |
|-----------|------------------------|
| **S** — Single Responsibility | 每個 Decorator 只負責一個橫切關注點 |
| **O** — Open/Closed | 透過新增 Decorator 擴展行為，不修改既有類別 |
| **L** — Liskov Substitution | Decorator 必須能無縫替換被裝飾的物件 |
| **I** — Interface Segregation | Decorator 基於介面運作，介面越小越好 |
| **D** — Dependency Inversion | 消費者依賴抽象介面，不知道背後疊了幾層 Decorator |

### 框架 4：Decorator vs. 繼承 (Subclassing)

```
  繼承 (Subclassing)             Decorator
  ─────────────────            ──────────
  編譯期綁定                    執行期組合
  單一繼承限制                  無限堆疊
  緊耦合                       鬆耦合
  難以拆卸                     可自由拆裝
  違反 OCP                     遵循 OCP
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：Circuit Breaker 的狀態轉換與生命週期管理

**為什麼難**：Circuit Breaker 不是無狀態的——它有 Closed / Open / Half-Open 三種狀態，並且狀態會隨著時間與呼叫結果而轉換。更麻煩的是，它通常應該是 Singleton（因為同一個遠端服務只需要一個斷路器），而 Singleton 在多執行緒環境下需要執行緒安全。

**突破方式**：

1. 先把狀態轉換圖畫出來，確保你理解每個狀態轉換的觸發條件
2. 把 `ICircuitBreaker` 的三個方法對應到狀態轉換圖上：
   - `Guard()` → 檢查當前狀態，Open 時拋例外
   - `Succeed()` → Closed 保持 Closed；Half-Open 轉 Closed
   - `Trip(ex)` → 任何狀態轉 Open
3. 生命週期：記住 Circuit Breaker 的 Lifestyle 應與被保護的資源對齊，而非與 Repository Decorator 對齊

```
  時間軸：

  t0: Closed ──呼叫失敗──▶ t1: Open ──等待 timeout──▶ t2: Half-Open
                                                           │
                                                    呼叫成功？
                                                    ├── 是 → Closed
                                                    └── 否 → Open (重新計時)
```

---

### 難點 2：Decorator 堆疊順序的設計決策

**為什麼難**：當多個 Decorator 堆疊時，順序不同會導致行為截然不同。例如：ErrorHandling 放在 CircuitBreaker 外面 vs. 裡面，效果完全不同。

**突破方式**：

用「洋蔥模型」思考——呼叫從外到內穿過每一層，回傳從內到外穿過每一層：

```
  請求方向 ─────────────────────────────────────▶

  ┌─ Security ─┐   ┌─ ErrorHandling ─┐   ┌─ CircuitBreaker ─┐   ┌─ Real ─┐
  │ 先檢查權限   │──▶│ 捕獲例外         │──▶│ Guard/Trip       │──▶│ 實際   │
  │             │   │                 │   │                  │   │ 呼叫   │
  │             │◀──│ 顯示訊息         │◀──│ 拋出例外          │◀──│        │
  └─────────────┘   └─────────────────┘   └──────────────────┘   └────────┘

  ◀───────────────────────────────────── 回傳方向
```

**經驗法則**：
- 安全/授權 → 最外層（沒權限就不必進入後續流程）
- 錯誤處理 → 中間層（捕獲下層拋出的例外）
- 容錯/斷路器 → 靠近核心（直接保護實際的 I/O 操作）
- 日誌/稽核 → 依需求放在任何位置

---

### 難點 3：何時用 Decorator vs. 何時需要 AOP

**為什麼難**：本章結尾指出 Decorator 在面對多個不同的 Abstraction 時會產生大量重複程式碼。例如，若你有 `IProductRepository`、`IUserRepository`、`IOrderRepository` 三個介面，都需要 Circuit Breaker，你就得寫三個幾乎一模一樣的 Decorator。

**突破方式**：

理解 Decorator 的「甜蜜點」與「痛點」：

```
  Decorator 的適用範圍
  ─────────────────────────────────────────────────────▶

  │ 少量介面 + 特定邏輯 │  多介面 + 相同模板   │
  │  (用 Decorator 很好) │  (程式碼重複爆炸)    │
  │                     │  → 改用 AOP (Ch10)  │
  │                     │                     │
  ├─────────────────────┼─────────────────────┤

  例：                    例：
  AuditingUserRepo       CircuitBreaker 套用到
  Decorator              所有 Repository
  (只需一個 Decorator)    (需 N 個幾乎相同的 Decorator)
```

**判斷指引**：
- 如果橫切關注點**只套用於一個介面**，或**每個介面的實作邏輯不同** → 用 Decorator
- 如果橫切關注點**需要套用於多個介面**且**邏輯模板完全相同** → 考慮 AOP（第 10 章）

---

## 6. 本章小結與架構啟示

### 核心收穫

1. **Interception = Decorator Pattern + DI**：攔截不是什麼神秘的概念，它就是 Decorator 設計模式在 DI 容器 / Composition Root 中的實際應用。鬆散耦合與面向介面程式設計是它的前提條件。

2. **Decorator 是 Cross-Cutting Concerns 的第一道防線**：無論是 Auditing、Circuit Breaker、Error Handling 還是 Authorization，Decorator 都能讓你在不修改核心業務邏輯的情況下，乾淨地疊加額外行為。

3. **組合勝過繼承**：Decorator 展示了「組合 (Composition)」比「繼承 (Inheritance)」更靈活——你可以在執行期自由拆裝 Decorator，而繼承是在編譯期就被焊死的。

4. **順序決定行為**：Decorator 堆疊的順序直接影響系統行為，架構師必須有意識地設計這個順序。

5. **Decorator 的極限引出 AOP**：當同一套邏輯模板需要套用到大量不同介面時，Decorator 會導致程式碼重複。這正是下一章 (Chapter 10) 將討論的 Aspect-Oriented Programming 要解決的問題。

### 架構師的行動清單

```
  ☐ 檢視現有程式碼中是否有 Cross-Cutting Concerns 混在業務邏輯裡
    （例如 Repository 裡面做日誌、Controller 裡面做權限檢查）
  ☐ 確認系統已面向介面程式設計，具備套用 Decorator 的前提
  ☐ 識別最適合用 Decorator 處理的橫切關注點（通常是少量介面 + 特定邏輯）
  ☐ 在 Composition Root 中統一組裝 Decorator 鏈，而非散落在各處
  ☐ 記錄 Decorator 堆疊的順序及其設計理由
  ☐ 對於跨多介面的重複 Decorator，評估是否需要引入 AOP
```
