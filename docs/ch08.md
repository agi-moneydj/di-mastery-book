## 1. 概述 (Overview)

本章探討依賴注入中最容易被忽略、卻在生產環境最容易出事的主題 —— **物件生命週期管理 (Object Lifetime Management)**。

核心論點是：**當你透過 DI 將控制權交出去（Inversion of Control），你同時也交出了對 Dependency 生命週期的控制權。** 這件事必須由 **Composer**（組合器，通常是 Composition Root 中的 DI Container 或手動組裝的程式碼）來全權負責。

> **Composer（組合器）**：就像餐廳裡的侍酒師，他比你更了解酒窖裡每一支酒的狀態，決定什麼時候開瓶、什麼時候退藏。Composer 就是你系統中決定「何時建立物件、要不要共用、何時釋放」的那位侍酒師。

本章涵蓋三大主軸：

1. **Dependency Lifetime Management 的基本原理** —— 為什麼消費者不該管 Dependency 的生死
2. **Disposable Dependencies 的處理** —— 誰負責 Dispose？怎麼追蹤？
3. **三大 Lifestyle 模式** —— Singleton、Transient、Scoped
4. **Bad Lifestyle Choices** —— Captive Dependencies、Leaky Abstractions、Per-Thread Lifestyle

### 3.1 Managing Dependency Lifetime（管理依賴的生命週期）

#### 核心觀念：Composer 全權負責

當你透過 Constructor Injection 接收 Dependency 時，你只是「借用」它——你把參考存在 private field 中，讓它在你活著的時候也活著。但你無法決定它何時被建立、是否被共用、何時被回收。

> **白話比喻**：你在公司領到一台筆電（Dependency），你可以用它，但你不能決定公司什麼時候買這台筆電、也不能決定它要不要借給別人用、更不能自己把它砸了。IT 部門（Composer）才是管這些事的人。

```
┌─────────────────────────────────────────────────┐
│                   COMPOSER                       │
│          (Composition Root / DI Container)       │
│                                                  │
│   職責：                                          │
│   1. 決定何時「建立」Dependency                     │
│   2. 決定是否「共用」同一個實例                       │
│   3. 決定何時「釋放」(Release) Dependency            │
│   4. 決定何時「Dispose」Dependency                  │
│                                                  │
│   ┌───────────┐    ┌───────────┐                 │
│   │ Singleton │    │ Transient │                  │
│   │ (共用一個)  │    │(每次新建)  │                  │
│   └─────┬─────┘    └─────┬─────┘                 │
│         │                │                       │
│         ▼                ▼                        │
│   ┌───────────────────────────┐                  │
│   │     Consumer A            │                  │
│   │     Consumer B            │                  │
│   │     Consumer C            │                  │
│   └───────────────────────────┘                  │
└─────────────────────────────────────────────────┘
```

#### 兩種共用模式

**獨立實例（各自擁有）**：每個 Consumer 拿到自己的 Dependency 實例。

```csharp
// 每個 Consumer 得到自己的 Repository
var repository1 = new SqlProductRepository(connString);
var repository2 = new SqlProductRepository(connString);

var productService = new ProductService(repository1);
var calculator = new DiscountCalculator(repository2);
```

**共用實例（共享同一個）**：多個 Consumer 共用同一個 Dependency 實例。

```csharp
// 一個 Repository 被兩個 Consumer 共享
var repository = new SqlProductRepository(connString);

var productService = new ProductService(repository);
var calculator = new DiscountCalculator(repository);
```

> **重點**：Consumer 完全不知道自己拿到的 Dependency 是獨享還是共用的——這就是 **Liskov Substitution Principle（里氏替換原則）** 的實踐。

> **Liskov Substitution Principle（LSP，里氏替換原則）**：消費 Abstraction 的程式碼，必須能夠使用任何衍生類別而不察覺差異。白話說就是：你寫好的程式碼，不管背後換了什麼實作，行為都不該壞掉。

#### Pure DI 中的生命週期管理

用 Pure DI 時，Singleton 的做法是在 Composer 的建構子中建立實例，存在 private field 中反覆使用；Transient 則是每次被要求時才建立新實例。

```csharp
public class CommerceControllerActivator : IControllerActivator
{
    // Singleton Dependencies — 建立一次，永遠重用
    private readonly string connectionString;
    private readonly IUserContext userContext;
    private readonly RouteCalculator calculator;

    public CommerceControllerActivator(string connectionString)
    {
        this.connectionString = connectionString;
        this.userContext = new AspNetUserContextAdapter();  // Singleton
        this.calculator = new RouteCalculator(              // Singleton
            this.CreateRouteAlgorithms());
    }

    private HomeController CreateHomeController()
    {
        // Transient Dependencies — 每次請求都建立新的
        return new HomeController(
            new ProductService(
                new SqlProductRepository(
                    new CommerceContext(this.connectionString)),
                this.userContext));  // 注入 Singleton
    }
}
```

```
┌──────────────────────────────────────────────────────┐
│          CommerceControllerActivator (Composer)       │
│                                                      │
│  建構時建立 (Singleton)        每次請求建立 (Transient) │
│  ┌──────────────────────┐     ┌────────────────────┐ │
│  │ AspNetUserContext-    │     │ ProductService     │ │
│  │   Adapter             │     │   └─ SqlProduct-   │ │
│  │ RouteCalculator       │     │       Repository   │ │
│  │   └─ ShortestRoute-  │     │       └─ Commerce- │ │
│  │       Algorithm       │     │           Context  │ │
│  │   └─ ScenicRoute-    │     └────────────────────┘ │
│  │       Algorithm       │                           │
│  │   └─ FastestRoute-   │                            │
│  │       Algorithm       │                            │
│  └──────────────────────┘                            │
└──────────────────────────────────────────────────────┘
```

### 3.2 Working with Disposable Dependencies（處理可釋放的依賴）

#### 核心原則：不要讓 Abstraction 繼承 IDisposable

```csharp
// !! 不好的做法 — Leaky Abstraction !!
public interface IMyDependency : IDisposable
```

這是 **Leaky Abstraction（洩漏抽象）**，因為你讓某個特定實作的記憶體管理需求「洩漏」到了介面設計中。不是所有實作都需要 Dispose（例如 in-memory 實作），強加 IDisposable 會讓其他實作者被迫實作一個空的 Dispose。

> **白話比喻**：就像在「交通工具」的介面上加了「加油」的方法——但電動車不需要加油。加油是汽車的實作細節，不該出現在通用的抽象層。

#### 消費者不該 Dispose 注入的 Dependency

```csharp
// !! 危險的做法 !!
public sealed class OrderService : IDisposable
{
    private readonly IOrderRepository repository;

    public OrderService(IOrderRepository repository)
    {
        this.repository = repository;
    }

    public void Dispose()
    {
        this.repository.Dispose();  // 危險！這個 repository 可能還被別人使用
    }
}
```

如果 `repository` 是被多個 Consumer 共用的，其中一個 Consumer 把它 Dispose 掉，其他 Consumer 就會炸掉。

#### Ephemeral Disposable（短命的 Disposable）

有些 Disposable 物件的生命週期很短——在同一個方法內建立並銷毀。這類物件不是注入進來的，而是在方法內部透過 Factory 建立的。

> **Ephemeral Disposable（短命可釋放物件）**：生命週期不超過單一方法呼叫的 Disposable 物件。就像你在便利商店買的即食便當，吃完就丟，不會帶回家放冰箱。

```csharp
public class WcfProductRepository : IProductRepository
{
    private readonly ChannelFactory<IProductManagementService> factory;

    public WcfProductRepository(
        ChannelFactory<IProductManagementService> factory)
    {
        this.factory = factory;  // Factory 是 Singleton，安全注入
    }

    public void Delete(Guid productId)
    {
        // Channel 是 Ephemeral Disposable — 在方法內建立與銷毀
        using (var channel = this.factory.CreateChannel())
        {
            channel.DeleteProduct(productId);
        }
    }
}
```

#### Composer 負責追蹤與釋放

Composer 必須追蹤它建立的所有 Disposable Dependency，並在適當時機呼叫 `Dispose()`。

```csharp
// 追蹤 Disposable Dependency
private HomeController CreateHomeController(ControllerContext context)
{
    var dbContext = new CommerceContext(this.connectionString);
    TrackDisposable(context, dbContext);  // 把它記下來

    return new HomeController(
        new ProductService(
            new SqlProductRepository(dbContext),
            this.userContext));
}

// 釋放時，以「建立的反序」Dispose
public void Release(ControllerContext context, object controller)
{
    var disposables =
        (List<IDisposable>)context.HttpContext.Items["Disposables"];

    if (disposables != null)
    {
        disposables.Reverse();  // 反序！外層先 Dispose

        foreach (IDisposable disposable in disposables)
        {
            disposable.Dispose();
        }
    }
}
```

```
 Resolve 流程                    Release 流程
 ─────────────                   ──────────────
 1. new CommerceContext()         3. Dispose HomeController
 2. new SqlProductRepository()        (讓它 GC)
 3. new ProductService()         2. Dispose SqlProductRepository
 4. new HomeController()              (讓它 GC)
                                 1. Dispose CommerceContext
     建立順序: 1→2→3→4                 ← 反向 Dispose!
```

> **關鍵規則**：Disposal 必須以建立的**反序**進行（由外而內），因為外層物件在 Dispose 時可能還需要呼叫其 Dependency 的方法。

### 3.3 Lifestyle Catalog（生命週期模式目錄）

#### 三大 Lifestyle 總覽

```
┌──────────────┬────────────────────────────────────────────────────┐
│  Lifestyle   │  說明                                              │
├──────────────┼────────────────────────────────────────────────────┤
│  Singleton   │  整個應用程式生命週期只有一個實例，永遠重用             │
│              │  比喻：公司只有一台印表機，所有人共用                   │
├──────────────┼────────────────────────────────────────────────────┤
│  Transient   │  每次被要求時都建立全新的實例                         │
│              │  比喻：用完即丟的紙杯，每次都拿新的                    │
├──────────────┼────────────────────────────────────────────────────┤
│  Scoped      │  在一個明確定義的範圍（如一個 Web Request）內          │
│              │  只有一個實例，跨範圍則各自獨立                        │
│              │  比喻：每桌客人共用一壺茶，但不跟隔壁桌共用              │
└──────────────┴────────────────────────────────────────────────────┘
```

#### 3.3.1 Singleton Lifestyle

**一句話定義**：在同一個 Composer 的範圍內，元件只有一個實例，每次被請求時都回傳同一個。

**注意**：Singleton Lifestyle 不等於 Singleton Design Pattern！

| | Singleton Lifestyle | Singleton Design Pattern |
|---|---|---|
| 實例數量 | 每個 Composer 一個 | 全域唯一（static） |
| 存取方式 | 透過注入 | 透過靜態成員 |
| 可測試性 | 高 | 低（全域狀態） |
| 靈活性 | 不同 Composer 可有不同實例 | 無法替換 |

**何時使用 Singleton**：
- 元件是**無狀態**的（Stateless）
- 元件是**不可變**的（Immutable）
- 元件是**執行緒安全**的（Thread-safe）
- 效能考量：減少物件建立與 GC 壓力
- 設計需要：Circuit Breaker、In-Memory Cache 等必須共享狀態的模式

**何時不能使用**：
- 元件不是 Thread-safe（例如 EF Core 的 `DbContext`）
- 元件的 Dependency 預期生命週期更短（否則變成 Captive Dependency）

```csharp
public class CommerceControllerActivator : IControllerActivator
{
    private readonly IUserContext userContext;           // Singleton
    private readonly IProductRepository repository;     // Singleton

    public CommerceControllerActivator()
    {
        this.userContext = new FakeUserContext();
        this.repository = new InMemoryProductRepository(); // Thread-safe!
    }

    private HomeController CreateHomeController()
    {
        return new HomeController(
            new ProductService(              // Transient — 每次建立新的
                this.repository,             // 注入 Singleton
                this.userContext));           // 注入 Singleton
    }
}
```

```
 App 啟動
    │
    ▼
 ┌─────────────────────────────┐
 │ CommerceControllerActivator │
 │                             │
 │  userContext ─────────────────── FakeUserContext (Singleton)
 │  repository  ─────────────────── InMemoryProductRepository (Singleton)
 │                             │
 └──────────┬──────────────────┘
            │
    Request 來了
            │
            ▼
 ┌──────────────────────────────────┐
 │ new ProductService(              │  ← Transient
 │     repository,  ← 同一個         │
 │     userContext)  ← 同一個        │
 │                                  │
 │ new HomeController(              │  ← Transient
 │     productService)              │
 └──────────────────────────────────┘
```

#### 3.3.2 Transient Lifestyle

**一句話定義**：每次被要求時，都回傳一個全新的實例。

**何時使用 Transient**：
- **最安全**的選擇（每個 Consumer 有自己的實例，不存在共享問題）
- 但也是**最不高效**的（大量建立和回收物件）
- 對 Thread-safety 有疑慮時，Transient 是保險的選擇
- 很多時候可以用 Scoped 替代，兼顧安全與效率

```csharp
private HomeController CreateHomeController()
{
    return new HomeController(
        new ProductService(
            new SqlProductRepository(this.connStr),
            new AspNetUserContextAdapter(),           // Transient 實例 1
            new SqlUserRepository(
                this.connStr,
                new AspNetUserContextAdapter())));    // Transient 實例 2
    // 即使同一個型別，每個 Consumer 都得到自己的實例
}
```

> **注意**：在桌面應用等只解析一次 object graph 的場景中，Transient 與 Singleton 在行為上沒有差別，因為 graph 只建立一次。

#### 3.3.3 Scoped Lifestyle

**一句話定義**：在一個明確定義的 Scope（通常是一個 Request 或操作）內，行為像 Singleton；跨 Scope 則各自獨立。

> **白話比喻**：你去吃火鍋，同一桌的人共用一鍋湯底（Scoped），但隔壁桌有自己的湯底。而醬料區（Singleton）則是全餐廳共用。

**何時使用 Scoped**：
- Web 應用程式中的 **DbContext** —— 同一個 Request 內共享，跨 Request 獨立
- 長時間運行的應用程式中，每個「操作」需要隔離的狀態
- 背景工作處理佇列中的每個 Item

```
 ┌─────────── Scope 1 (Request 1) ──────────────┐
 │                                               │
 │  HomeController                               │
 │    └─ ProductService ─────┐                   │
 │        └─ SqlProductRepo ─┼── AspNetUserCtx   │
 │                           │   (同一個實例)      │
 │                           │                   │
 └───────────────────────────┘                   │
                                                 │
 ┌─────────── Scope 2 (Request 2) ──────────────┐
 │                                               │
 │  HomeController                               │
 │    └─ ProductService ─────┐                   │
 │        └─ SqlProductRepo ─┼── AspNetUserCtx   │
 │                           │   (另一個實例)      │
 │                           │                   │
 └───────────────────────────┘
```

**Scoped 在長時間運行的 Console 應用的實踐**：

```csharp
public class Composer
{
    private readonly string connectionString;  // Singleton

    public Composer(string connectionString)
    {
        this.connectionString = connectionString;
    }

    public CurrencyRateDisplayer CreateRateDisplayer()
    {
        // Scoped — 每次呼叫建立一個 CommerceContext
        var context = new CommerceContext(this.connectionString);

        return new CurrencyRateDisplayer(
            new SqlCurrencyRepository(context),       // 共用 context
            new CurrencyConverter(
                new SqlExchangeRateProvider(context))); // 共用 context
    }
}

// 呼叫端 — 每分鐘一次，每次一個新的 Scope
private static void DisplayRates(Money money)
{
    CurrencyRateDisplayer displayer =
        composer.CreateRateDisplayer();  // 每次呼叫 = 一個新 Scope
    displayer.DisplayRatesFor(money);
}
```

### 3.4 Bad Lifestyle Choices（壞的生命週期選擇）

#### 3.4.1 Captive Dependency（被俘虜的依賴）

> **Captive Dependency（被俘虜的依賴）**：一個 Dependency 被它的 Consumer 「扣留」了，活得比它本該活的時間還長。就像一個暑期實習生（Scoped），被主管（Singleton）綁住不放，結果待了整年，但這個實習生根本不具備處理全年工作的能力（非 Thread-safe）。

**核心規則**：一個元件的 Dependency，其生命週期必須**大於或等於**該元件本身的生命週期。

```
 ⚠️ Captive Dependency 違反圖

 Dependency 生命週期長度：

  Singleton  ████████████████████████████████████  最長
  Scoped     ████████████████                      中等
  Transient  ████                                  最短

  合法方向（Consumer → Dependency）：
  Transient  → Singleton  ✅
  Transient  → Scoped     ✅
  Transient  → Transient  ✅
  Scoped     → Singleton  ✅
  Scoped     → Scoped     ✅
  Singleton  → Singleton  ✅

  非法方向（會產生 Captive Dependency）：
  Singleton  → Scoped     ❌ 危險！
  Singleton  → Transient  ❌ 危險！
  Scoped     → Transient  ❌ 危險！
```

```csharp
// !! 錯誤示範 — Captive Dependency !!
public class Composer
{
    private readonly IProductRepository repository;  // 意圖當 Singleton

    public Composer(string connectionString)
    {
        // CommerceContext 不是 Thread-safe，不該當 Singleton！
        // 但它被 SqlProductRepository 的 Singleton 生命週期「俘虜」了
        this.repository = new SqlProductRepository(
            new CommerceContext(connectionString));  // Captive!
    }
}
```

**防範方式 —— 三階段建立法**：

```csharp
public class CommerceControllerActivator : IControllerActivator
{
    // 階段 1：Singleton — 應用啟動時建立
    private readonly string connStr;
    private readonly IUserContext userContext;

    public CommerceControllerActivator(string connectionString)
    {
        this.connStr = connectionString;
        this.userContext = new AspNetUserContextAdapter();
    }

    public object Create(ControllerContext ctx)
    {
        // 階段 2：Scoped — 每次 Request 開始時建立
        var context = new CommerceContext(this.connStr);
        var provider = new SqlExchangeRateProvider(context);

        Type type = ctx.ActionDescriptor
            .ControllerTypeInfo.AsType();

        if (type == typeof(HomeController))
            return this.CreateHomeController(context);
        else if (type == typeof(ExchangeController))
            return this.CreateExchangeController(context, provider);
        else
            throw new Exception("Unknown controller " + type.Name);
    }

    // 階段 3：Transient — 組裝 object graph
    private HomeController CreateHomeController(
        CommerceContext context)
    {
        return new HomeController(
            new ProductService(
                new SqlProductRepository(context),  // Scoped
                this.userContext));                  // Singleton
    }
}
```

```
 ┌─────────────────────────────────────────────────────┐
 │              三階段建立法                              │
 │                                                     │
 │  階段 1: App 啟動        → 建立所有 Singleton          │
 │  階段 2: Request 開始    → 建立所有 Scoped             │
 │  階段 3: 組裝 Object Graph → Transient + Scoped       │
 │                              + Singleton 混合注入     │
 │                                                     │
 │  這樣分層後，Captive Dependency 一目了然              │
 └─────────────────────────────────────────────────────┘
```

#### 3.4.2 Leaky Abstraction 洩漏 Lifestyle 選擇

**問題 1：注入 `Lazy<T>` 是 Leaky Abstraction**

```csharp
// !! 不好的做法 !!
public class ProductService : IProductService
{
    private readonly Lazy<IUserContext> userContext;  // Leaky!

    public ProductService(
        IProductRepository repository,
        Lazy<IUserContext> userContext)  // Consumer 被迫知道「延遲建立」這件事
    {
        ...
    }

    public IEnumerable<DiscountedProduct> GetFeaturedProducts()
    {
        return
            from product in this.repository.GetFeaturedProducts()
            select product.ApplyDiscountFor(
                this.userContext.Value);  // 到處都要 .Value
    }
}
```

**為什麼不好？**
1. Consumer 和它的單元測試都變複雜（到處要包 `Lazy<T>`）
2. 事後改成 Lazy 會造成「漣漪式修改」（所有 Consumer + 所有測試都要改）
3. 容易犯錯：新 Consumer 可能忘記用 `Lazy<T>` 而直接用 `IUserContext`

**正確做法 —— Virtual Proxy 模式**：

```csharp
// Virtual Proxy — 放在 Composition Root 中
public class LazyUserContextProxy : IUserContext
{
    private readonly Lazy<IUserContext> userContext;

    public LazyUserContextProxy(Lazy<IUserContext> userContext)
    {
        this.userContext = userContext;
    }

    public bool IsInRole(Role role)
    {
        IUserContext real = this.userContext.Value;  // 延遲建立
        return real.IsInRole(role);
    }
}

// 組裝時：
IUserContext lazyProxy = new LazyUserContextProxy(
    new Lazy<IUserContext>(
        () => new AspNetUserContextAdapter()));

new HomeController(
    new ProductService(
        new SqlProductRepository(new CommerceContext(connectionString)),
        lazyProxy));  // ProductService 依然只知道 IUserContext
```

```
 ┌────────────────────────────────────────────────────────┐
 │             Virtual Proxy 模式                          │
 │                                                        │
 │  ProductService  ──depends on──>  IUserContext          │
 │                                      ▲                 │
 │                                      │ implements      │
 │                              LazyUserContextProxy      │
 │                                      │                 │
 │                                      │ wraps           │
 │                               Lazy<IUserContext>        │
 │                                      │                 │
 │                                      │ creates         │
 │                             AspNetUserContextAdapter    │
 │                                                        │
 │  延遲建立的知識被封裝在 Proxy 裡，                        │
 │  Consumer 完全不知道 Lazy 的存在！                        │
 └────────────────────────────────────────────────────────┘
```

**問題 2：注入 `IEnumerable<T>` 也是 Leaky Abstraction**

同理，讓 Consumer 直接依賴 `IEnumerable<ILogger>` 會讓每個 Consumer 都要自己處理迴圈邏輯、錯誤處理等。

**正確做法 —— Composite 模式**：

```csharp
public class CompositeLogger : ILogger
{
    private readonly IList<ILogger> loggers;

    public CompositeLogger(IList<ILogger> loggers)
    {
        this.loggers = loggers;
    }

    public void Log(LogEntry entry)
    {
        for (int index = 0; index < this.loggers.Count; index++)
        {
            try
            {
                this.loggers[index].Log(entry);
            }
            catch (Exception ex)
            {
                if (loggers.Count > index + 1)
                {
                    loggers[index + 1].Log(new LogEntry(ex));
                }
            }
        }
    }
}

// 組裝：
ILogger composite = new CompositeLogger(new ILogger[]
{
    new SqlLogger(connectionString),
    new WindowsEventLogLogger(source: "MyApp"),
    new FileLogger(directory: "c:\\logs")
});

new Component(composite);  // Consumer 只知道 ILogger，不知道有多個
```

#### 3.4.3 Per-Thread Lifestyle 的並行問題

**為什麼 Per-Thread Lifestyle 是危險的？**

在非同步框架（如 ASP.NET Core）中，一個 Request 可能在 Thread A 上開始，但 I/O 等待後在 Thread B 上繼續。這代表：
- Object graph 帶著 Thread A 的 Dependency 跑到了 Thread B
- 同時，Thread A 可能已經開始處理 Request 2，也使用同一個 Thread A 的 Dependency
- 結果：兩個 Request 同時使用同一個非 Thread-safe 的 Dependency = **並行 Bug**

```
 Thread A          Thread B
 ─────────         ─────────
 Req 1 開始
   │ (使用 Dep-A)
   │
   ├─ async I/O ──────────────> Req 1 繼續
   │                              │ (仍持有 Dep-A!)
 Req 2 開始                       │
   │ (也使用 Dep-A!)              │
   │                              │
   ▼                              ▼
 ⚠️ Dep-A 被兩個 Request 同時使用 = 並行 Bug！
```

**解決方案**：使用 Scoped Lifestyle 而非 Per-Thread Lifestyle。

```csharp
// !! 不好 — Per-Thread !!
[ThreadStatic]
private static CommerceContext context;

static CurrencyParser CreateCurrencyParser(string connectionString)
{
    if (context == null)
        context = new CommerceContext(connectionString);

    return new CurrencyParser(
        new SqlExchangeRateProvider(context), context);
}

// ✅ 好 — Scoped
static CurrencyParser CreateCurrencyParser(string connectionString)
{
    var context = new CommerceContext(connectionString);  // 每次新建

    return new CurrencyParser(
        new SqlExchangeRateProvider(context), context);
}
```

## 4. 關鍵框架與公式 (Key Frameworks)

### Framework 1：Lifestyle 決策流程圖

```
                    這個元件是 Thread-safe 嗎？
                           │
                    ┌──────┴──────┐
                    │             │
                   YES           NO
                    │             │
              它需要被共享嗎？    它需要跨 Consumer
              (Cache/Mediator)   共享狀態嗎？
                    │             │
              ┌─────┴─────┐  ┌───┴───┐
              │           │  │       │
             YES         NO YES     NO
              │           │  │       │
              ▼           ▼  ▼       ▼
          Singleton   Singleton  Scoped  Transient
          (必須)      (建議)     (推薦)   (安全)
```

### Framework 2：Disposable Dependency 責任矩陣

| 角色 | 建立 | 追蹤 | 釋放(Dispose) |
|------|------|------|---------------|
| Consumer | 不能 | 不能 | 不能 |
| Abstraction | 不該繼承 IDisposable | - | - |
| Composer | 全權負責 | 全權負責 | 全權負責 |
| Ephemeral Disposable | 方法內自行建立 | 不需要 | using 語法自行處理 |

### Framework 3：Lifestyle 混搭合法性速查表

```
Consumer 的 Lifestyle  →  Dependency 的 Lifestyle  →  合法？
─────────────────────────────────────────────────────────────
Singleton              →  Singleton                →  ✅
Singleton              →  Scoped                   →  ❌ Captive!
Singleton              →  Transient                →  ❌ Captive!
Scoped                 →  Singleton                →  ✅
Scoped                 →  Scoped (同一 scope)       →  ✅
Scoped                 →  Transient                →  ⚠️ 可能 Captive
Transient              →  Singleton                →  ✅
Transient              →  Scoped                   →  ✅
Transient              →  Transient                →  ✅
```

**口訣**：Dependency 的命要比 Consumer 長（或一樣長），不然就是 Captive！

### Framework 4：三階段建立法（防範 Captive Dependency）

```
Phase 1: Application Startup
    → 建立所有 Singleton Dependencies
    → 儲存在 Composer 的 private readonly fields

Phase 2: Request Start
    → 建立所有 Scoped Dependencies
    → 儲存在 local variables

Phase 3: Object Graph Assembly
    → 建立 Transient Components
    → 將 Phase 1 + Phase 2 + Phase 3 的元件組裝成完整 graph
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：Release ≠ Dispose —— 理解 Release 的語義

**為什麼難？** 很多人直覺認為「釋放物件」就是「Dispose 它」，但實際上 Release 是一個更精細的過程。

**核心概念**：
- **Release** = Composer 判斷哪些 Dependency 可以解除參照、可以 Dispose，哪些要繼續活著
- **Dispose** = 只是 Release 過程中「可能」發生的動作之一

```
 Release 的決策過程：

 Composer 收到 Release(objectGraph) 的通知
     │
     ├── Dependency A (Singleton, 還有其他人在用)
     │       └── 不 Dispose，繼續保留
     │
     ├── Dependency B (Scoped, 是 IDisposable)
     │       └── Dispose 它
     │
     ├── Dependency C (Transient, 不是 IDisposable)
     │       └── 讓 GC 處理
     │
     └── Dependency D (Transient, 是 IDisposable)
             └── Dispose 它
```

**記住**：Release 是「智慧型的清理」，Dispose 只是清理的手段之一。Composer 知道全貌，所以只有它能做正確的 Release 決策。

### 難點 2：Captive Dependency 在非同步環境中的隱蔽性

**為什麼難？** Captive Dependency 的 Bug 往往在開發時看不出來（因為只有你一個人在用），只有在生產環境多人同時使用時才爆發。

**隱蔽的原因**：
1. 開發時通常只有一個 Thread 在跑，Thread-safety 問題不會浮現
2. 測試通常是按順序執行的，不會產生並行存取
3. Bug 的表現可能是「偶爾」的資料錯亂、「偶爾」的 Exception，難以重現

**如何偵測？**
- 使用有 Captive Dependency Detection 功能的 DI Container（本書後續介紹的所有 DI Container 都支援）
- 在 Pure DI 中，嚴格遵守「三階段建立法」
- Code Review 時特別注意：Singleton 的建構子裡是否直接 new 了非 Thread-safe 的物件

### 難點 3：Per-Thread Lifestyle 在 async/await 環境下的陷阱

**為什麼難？** 因為在 ASP.NET Core 中，你的程式碼看起來是「同步」的（或用了 async/await），但底層的 Thread 調度是你看不見的。

**關鍵理解**：
- ASP.NET Core 保證單一 Request 內的程式碼是**循序執行**的（sequential）
- 但它**不保證**是在同一個 Thread 上執行
- `await` 之後，你可能在完全不同的 Thread 上繼續

```csharp
public async Task<IActionResult> GetProducts()
{
    // 可能在 Thread #7 上
    var products = await repository.GetAllAsync();

    // await 之後，可能已經換到 Thread #12 了！
    // 如果 repository 是 Per-Thread 的，
    // 此時 Thread #7 的 Dependency 已經跟著你到 Thread #12，
    // 而 Thread #7 可能已經在處理另一個 Request 了
    return Ok(products);
}
```

**結論**：永遠不要使用 Per-Thread Lifestyle，改用 Scoped Lifestyle。

## 6. 本章小結與架構啟示

### 核心原則總結

1. **Consumer 不管 Dependency 的生死** —— 這是 IoC 的延伸：你不只交出了「選擇實作」的權力，也交出了「管理生命週期」的權力。

2. **Composer 是唯一的生命週期管理者** —— 它知道什麼時候建立、是否共用、什麼時候該 Dispose。

3. **Disposable Abstraction 是 Leaky Abstraction** —— 不要讓你的介面繼承 `IDisposable`。如果某個實作需要 Dispose，那是 Composer 的事。

4. **Disposal 以建立的反序進行** —— 外層先 Dispose，因為它可能在 Dispose 時還需要呼叫內層的方法。

5. **Lifestyle 的黃金法則**：Dependency 的壽命 >= Consumer 的壽命，否則就是 Captive Dependency。

6. **不要把 Lifestyle 的決策洩漏給 Consumer** —— `Lazy<T>`、`Func<T>`、`IEnumerable<T>` 都應該藏在 Proxy 或 Composite 後面。

7. **不要用 Per-Thread Lifestyle** —— 在 async 的世界裡，它是一顆定時炸彈。用 Scoped 取代。

### 架構啟示

```
 ┌─────────────────────────────────────────────────────────────┐
 │                     架構師的行動清單                          │
 │                                                             │
 │  1. 審視你的 DI 註冊：                                       │
 │     - 所有 DbContext → Scoped                               │
 │     - 所有 Stateless Service → Singleton                    │
 │     - 有狀態但非 Thread-safe → Scoped 或 Transient          │
 │                                                             │
 │  2. 檢查 Captive Dependency：                               │
 │     - 用 DI Container 的偵測功能                              │
 │     - 或在 Pure DI 中使用三階段建立法                          │
 │                                                             │
 │  3. 不要在 Interface 上加 IDisposable                        │
 │                                                             │
 │  4. 元件的建構子保持極簡（只有 Guard Clause + 存值）             │
 │     → 這樣 Transient 也不會有效能問題                         │
 │     → Lazy<T> 的需求自然消失                                  │
 │                                                             │
 │  5. 如果 Pure DI 的 Lifetime Management 變得複雜，              │
 │     考慮切換到 DI Container                                   │
 │                                                             │
 │  6. Dispose 的職責交給 Composer，不要在 Consumer 裡 Dispose     │
 └─────────────────────────────────────────────────────────────┘
```

> **最終的架構哲學**：好的生命週期管理，就像好的餐廳管理——客人（Consumer）只需要點菜和享用，食材的採購、保鮮、過期處理，都是廚房（Composer）的事。如果你發現客人需要自己去冰箱拿食材、自己判斷有沒有過期，那就是你的餐廳設計出了問題。
