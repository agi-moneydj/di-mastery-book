## 1. 概述 (Overview)

本章是 **Part 3: Pure DI** 的開篇章節，從「理論」轉向「實戰」。核心命題只有一個：

> **在不同類型的應用程式中，Composition Root 到底該放在哪裡、怎麼寫？**

作者用法式料理的 **mise en place**（萬事就緒）做比喻——所有食材事先備好、切好、預熱好，等客人上桌時只需 5 到 10 分鐘做最後的組合。DI 中的 Object Composition 就是這個「最後組合」動作，而 Composition Root 就是那個「最後組合的廚房」。

本章依序討論三種應用程式類型：

| 應用類型 | Composition Root 位置 | 難度 |
|---|---|---|
| Console Application | `Program.Main()` | 最低 |
| UWP (Rich Client) | `App.OnLaunched()` | 中等 |
| ASP.NET Core MVC | `IControllerActivator` 實作 | 較高 |

每種類型都用電商範例（貫穿全書的 e-commerce 應用）來示範，讓讀者體會同一個領域模型如何在不同框架下被組合起來。

---

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

1. **說明 Object Composition 的定義**：理解它就是「建構相關元件的階層結構」，且此動作發生在 Composition Root 內
2. **在 Console 應用中實作 Composition Root**：在 `Program.Main()` 中載入設定、組合物件圖、呼叫功能
3. **在 UWP 應用中實作 Composition Root**：利用 `App.OnLaunched()` 與 MVVM 模式，將 ViewModel 注入 View
4. **在 ASP.NET Core MVC 中實作 Composition Root**：透過自訂 `IControllerActivator` 來組合 Controller 及其相依
5. **辨識各框架的 Seam（接縫）**：理解不同框架的 IoC 程度不同，找到正確的擴充點是成功實踐 DI 的關鍵
6. **使用 Pure DI 手法組合物件圖**：不依賴 DI Container，純粹用 `new` 來組合相依關係
7. **將自訂 Middleware 整合進 ASP.NET Core 管線**：把中介軟體的邏輯抽出為獨立類別，並透過 DI 注入相依

## 3. 章節大綱 (Outline)

### 3.1 引言：Mise en Place 與 Object Composition

作者以法式料理的「mise en place」（萬事就緒）來類比 DI 的物件組合。在料理中，所有食材提前備好，最後上菜時只需快速組合。軟體開發中，所有元件（類別）事先開發好，最後在 Composition Root 中一次組合。

```
 ┌─────────────────────────────────────────────────────────┐
 │                   時間軸 (Time)                          │
 │                                                         │
 │  ▓▓▓ 熬高湯     ▓▓▓ 清洗貝類                            │
 │  ▓▓▓ 做冰淇淋   ▓▓▓ 預熱烤箱   ▓▓▓ 準備蔬菜             │
 │                                                         │
 │  ◄──────── 事先準備 (Prepare) ────────►  │ 組合上菜 │    │
 │                                          │(Compose) │    │
 │                                          │  5~10 分  │    │
 └─────────────────────────────────────────────────────────┘

 對應到軟體：
 ┌─────────────────────────────────────────────────────────┐
 │                                                         │
 │  ▓▓▓ 開發 Domain Layer                                  │
 │  ▓▓▓ 開發 Data Access Layer                             │
 │  ▓▓▓ 開發 Presentation Layer                            │
 │                                                         │
 │  ◄─── 各模組獨立開發 (鬆耦合) ───►  │ Composition │      │
 │                                      │    Root     │      │
 │                                      │  (組合點)   │      │
 └─────────────────────────────────────────────────────────┘
```

**白話解釋重要名詞：**

- **Object Composition（物件組合）**：就是用 `new` 把一個個物件串起來的動作。比喻：像樂高積木，每塊積木獨立製造，最後一次拼裝成完整模型。
- **Composition Root（組合根）**：應用程式中「唯一負責組合所有物件」的地方。比喻：就像工廠的總裝配線，所有零件在這裡被組裝成成品。
- **Seam（接縫）**：框架提供的擴充點，讓你可以插入自己的邏輯。比喻：衣服的縫線處，你可以在這裡拆開、修改、重新縫合。

**Composition Root 四件事：**

```
┌────────────────────────────────────────────┐
│           Composition Root 職責             │
├────────────────────────────────────────────┤
│  1. 載入設定值 (Load Configuration)         │
│  2. 建構物件圖 (Build Object Graph)         │
│  3. 呼叫功能   (Invoke Functionality)       │
│  4. 釋放物件圖 (Release Object Graph)       │
│                 ↑ 下一章詳述                 │
└────────────────────────────────────────────┘
```

### 3.2 Console Application 的組合 (Section 7.1)

Console 應用是最簡單的 DI 實作場景，因為幾乎沒有外部的 Inversion of Control。框架只是啟動程序，把控制權交給 `Main()` 方法，之後一切由你掌控。

#### 架構總覽

```
┌─────────────────────────────────────────────────────────────┐
│                    Executable (Program)                      │
│  ┌────────────────────────────────────────────────────────┐  │
│  │  Main()                         [Composition Root]     │  │
│  │    1. LoadConnectionString()    載入設定                │  │
│  │    2. CreateCurrencyParser()    建構物件圖              │  │
│  │    3. parser.Parse(args)        解析命令列              │  │
│  │    4. command.Execute()         執行功能                │  │
│  └─────────────────────┬──────────────────────────────────┘  │
└────────────────────────┼────────────────────────────────────┘
                         │ creates
                         ▼
┌─────────────────────────────────────────────────────────────┐
│              Application Services Layer                      │
│                                                              │
│  ┌──────────────┐       ┌──────────────────────┐            │
│  │ CurrencyParser│──uses──▶ UpdateCurrencyCommand │           │
│  └──────┬───────┘       └──────────┬───────────┘            │
│         │                          │                         │
│         │ uses                     │ uses                    │
└─────────┼──────────────────────────┼─────────────────────────┘
          ▼                          ▼
┌─────────────────────────────────────────────────────────────┐
│                      Domain Layer                            │
│  ┌───────────────────────┐    ┌──────────┐                  │
│  │ IExchangeRateProvider │    │ Currency │                   │
│  │      (interface)       │    └──────────┘                  │
│  └───────────┬───────────┘                                   │
└──────────────┼───────────────────────────────────────────────┘
               │ implemented by
               ▼
┌─────────────────────────────────────────────────────────────┐
│                    Data Access Layer                          │
│  ┌──────────────────────────┐   ┌──────────────────┐        │
│  │ SqlExchangeRateProvider  │──▶│  CommerceContext  │        │
│  └──────────────────────────┘   └──────────────────┘        │
└─────────────────────────────────────────────────────────────┘
```

#### 核心程式碼

```csharp
// Composition Root — Program.cs
class Program
{
    static void Main(string[] args)
    {
        // 1. 載入設定
        string connectionString = LoadConnectionString();

        // 2. 建構物件圖
        CurrencyParser parser = CreateCurrencyParser(connectionString);

        // 3. 執行功能
        ICommand command = parser.Parse(args);
        command.Execute();
    }

    // 設定載入與物件組合「分開」，方便測試
    static string LoadConnectionString()
    {
        var configuration = new ConfigurationBuilder()
            .SetBasePath(AppContext.BaseDirectory)
            .AddJsonFile("appsettings.json", optional: false)
            .Build();

        return configuration.GetConnectionString("CommerceConnectionString");
    }

    // Composer：負責組合物件圖
    static CurrencyParser CreateCurrencyParser(string connectionString)
    {
        IExchangeRateProvider provider =
            new SqlExchangeRateProvider(
                new CommerceContext(connectionString));

        return new CurrencyParser(provider);
    }
}
```

**架構啟示：**
- `LoadConnectionString()` 和 `CreateCurrencyParser()` 刻意分開。這樣測試時不需要真的存在設定檔，只要傳入字串即可。
- `CreateCurrencyParser()` 就是 **Composer**（組合者），它用硬接線（hardwired）的方式把 `IExchangeRateProvider` 對應到 `SqlExchangeRateProvider`。
- 其餘程式碼只認識抽象介面，完全不知道實作類別的存在——這就是鬆耦合。

### 3.3 UWP Application 的組合 (Section 7.2)

UWP 應用的組合和 Console 幾乎一樣簡單，但多了一個挑戰：**頁面導航**。UWP 使用 MVVM（Model-View-ViewModel）模式，ViewModels 是有相依性的元件，需要透過 DI 來組合。

#### MVVM 架構圖

```
┌───────────────────────────────────────────────────────┐
│                    MVVM 模式                           │
│                                                        │
│   ┌─────────┐    data binding    ┌───────────┐        │
│   │  View   │◄══════════════════▶│ ViewModel │        │
│   │ (XAML)  │                    │  (C# 類別) │        │
│   └─────────┘                    └─────┬─────┘        │
│                                        │               │
│                                        │ uses          │
│                                        ▼               │
│                                  ┌───────────┐        │
│                                  │   Model   │        │
│                                  │  (POCO)   │        │
│                                  └───────────┘        │
│                                                        │
│  View = 畫面 (只管呈現，不含邏輯)                       │
│  ViewModel = 橋梁 (翻譯 Model 成 UI 能用的格式)         │
│  Model = 純領域物件 (不知道 UI 的存在)                   │
└───────────────────────────────────────────────────────┘
```

**白話解釋：**
- **MVVM**：一種把「畫面」和「邏輯」徹底分離的設計模式。View 只管長什麼樣子（XAML），ViewModel 負責準備資料和處理使用者操作，Model 就是純粹的資料物件。比喻：View 是餐廳的裝潢，ViewModel 是服務生（負責傳遞和翻譯），Model 是廚房出的菜。

```
│                                                          │
│  依賴 ↓ Domain                                           │
├─────────────────────────────────────────────────────────┤
│                    Domain Layer                           │
│                    ════════════                           │
│  Product (Model), IProductRepository (Abstraction)       │
│                                                          │
├─────────────────────────────────────────────────────────┤
│                  Data Access Layer                        │
│                  ════════════════                         │

└─────────────────────────────────────────────────────────┘
```

#### 核心程式碼

```csharp
// App.xaml.cs — 同時擔任 Composition Root 和 INavigationService
public sealed partial class App : Application, INavigationService
{
    // Entry Point
    protected override void OnLaunched(LaunchActivatedEventArgs e)
    {
        if (Window.Current.Content == null)
        {
            Window.Current.Content = new Frame();
            Window.Current.Activate();
            this.NavigateTo<MainViewModel>(null, null);
        }
    }

    // INavigationService 的實作
    public void NavigateTo<TViewModel>(
        Action whenDone, object model)
        where TViewModel : IViewModel
    {
        var page = this.CreatePage(typeof(TViewModel));
        var viewModel = (IViewModel)page.DataContext;
        viewModel.Initialize(whenDone, model);

        var frame = (Frame)Window.Current.Content;
        frame.Content = page;
    }

    // Composer：根據 ViewModel 類型建構 View/ViewModel 配對
    private Page CreatePage(Type vmType)
    {
        var repository = new WcfProductRepository();

        if (vmType == typeof(MainViewModel))
        {
            return new MainPage(
                new MainViewModel(this, repository));
        }
        else if (vmType == typeof(EditProductViewModel))
        {
            return new EditProductPage(
                new EditProductViewModel(repository));
        }
        else if (vmType == typeof(NewProductViewModel))
        {
            return new NewProductPage(
                new NewProductViewModel(repository));
        }
        else
        {
            throw new Exception("Unknown view model.");
        }
    }
}
```

```csharp
// MainPage.xaml.cs — View 透過 Constructor Injection 接收 ViewModel
public sealed partial class MainPage : Page
{
    public MainPage(MainViewModel vm)
    {
        this.InitializeComponent();
        this.DataContext = vm;   // 設定資料繫結來源
    }
}
```

```csharp
// MainViewModel — 使用 Constructor Injection 接收相依
public class MainViewModel : IViewModel, INotifyPropertyChanged
{
    private readonly INavigationService navigator;
    private readonly IProductRepository repository;

    public MainViewModel(
        INavigationService navigator,
        IProductRepository repository)
    {
        this.navigator = navigator;
        this.repository = repository;

        this.AddProductCommand = new RelayCommand(this.AddProduct);
        this.EditProductCommand = new RelayCommand(this.EditProduct);
    }

    public IEnumerable<Product> Model { get; set; }
    public ICommand AddProductCommand { get; }
    public ICommand EditProductCommand { get; }

    public event PropertyChangedEventHandler PropertyChanged = (s, e) => { };

    public void Initialize(object model, Action whenDone)
    {
        this.Model = this.repository.GetAll();
        this.PropertyChanged.Invoke(this,
            new PropertyChangedEventArgs("Model"));
    }

    private void AddProduct()
    {
        this.navigator.NavigateTo<NewProductViewModel>(
            whenDone: this.GoBack);
    }

    private void EditProduct(object product)
    {
        this.navigator.NavigateTo<EditProductViewModel>(
            whenDone: this.GoBack,
            model: product);
    }

    private void GoBack()
    {
        this.navigator.NavigateTo<MainViewModel>();
    }
}
```

**關鍵設計決策：**
- `App` 類別同時實作 `INavigationService`，因此可以用 `this` 注入 ViewModel。
- `MainPage` 移除預設建構子，改用帶參數的建構子接收 ViewModel（Constructor Injection 取代 Property Injection）。
- 繞過 `Frame.Navigate(Type)` 的限制，改用 `frame.Content = page` 手動指定頁面。

### 3.4 ASP.NET Core MVC 的組合 (Section 7.3)

ASP.NET Core MVC 是本章最具挑戰性的框架，因為它有較高程度的 Inversion of Control。不過 ASP.NET Core 從設計之初就考慮了 DI，提供了清晰的擴充點。

#### 請求管線架構

```
                         ASP.NET Core MVC 請求管線
                         ═══════════════════════

    ┌──────────┐      ┌──────────────────────┐      ┌──────────────┐
    │  Request │─────▶│  IControllerActivator │─────▶│  Controller  │
    │  到達    │      │       .Create()       │      │  被建立      │
    └──────────┘      └──────────────────────┘      └──────┬───────┘
                                                           │
                                                           ▼
                                                    ┌──────────────┐
                                                    │ 執行 Action  │
                                                    │   Method     │
                                                    └──────┬───────┘
                                                           │
                                                           ▼
                                                    ┌──────────────┐
    ┌──────────┐      ┌──────────────────────┐      │ ActionResult │
    │ Response │◄─────│  IControllerActivator │◄─────│  回傳        │
    │  送出    │      │       .Release()      │      └──────────────┘
    └──────────┘      └──────────────────────┘

    ※ IControllerActivator 就是 ASP.NET Core MVC 的「Seam（接縫）」
      你可以在這裡插入自己的物件組合邏輯
```

**白話解釋：**
- **IControllerActivator**：ASP.NET Core MVC 用來建立 Controller 的工廠介面。框架會在每次請求到達時呼叫 `Create()`，你可以在這裡決定怎麼 `new` 出 Controller 以及它的所有相依。比喻：就像工廠的生產線管理員——每來一張訂單（請求），他就負責組裝出對應的產品（Controller + Dependencies）。

#### 核心程式碼

```csharp
// IControllerActivator 介面（由 ASP.NET Core 定義）
public interface IControllerActivator
{
    object Create(ControllerContext context);
    void Release(ControllerContext context, object controller);
}
```

```csharp
// 自訂的 CommerceControllerActivator
public class CommerceControllerActivator : IControllerActivator
{
    private readonly string connectionString;

    public CommerceControllerActivator(string connectionString)
    {
        this.connectionString = connectionString;
    }

    public object Create(ControllerContext context)
    {
        Type type = context.ActionDescriptor
            .ControllerTypeInfo.AsType();

        if (type == typeof(HomeController))
        {
            return this.CreateHomeController();
        }
        else if (type == typeof(RouteController))
        {
            return this.CreateRouteController();
        }
        else
        {
            throw new Exception("Unknown controller " + type.Name);
        }
    }

    private HomeController CreateHomeController()
    {
        return new HomeController(
            new ProductService(
                new SqlProductRepository(
                    new CommerceContext(this.connectionString)),
                new AspNetUserContextAdapter()));
    }

    private RouteController CreateRouteController()
    {
        var routeAlgorithms = ...;
        return new RouteController(
            new RouteCalculator(routeAlgorithms));
    }

    public void Release(
        ControllerContext context, object controller)
    {
```

## 4. 關鍵框架與公式 (Key Frameworks)

### Framework 1：Composition Root 四步驟

這是貫穿本章（乃至全書）的核心公式，適用於任何應用類型：

```
┌────────────────────────────────────────────────────────┐
│              Composition Root 標準流程                   │
│                                                         │
│   ┌──────────────┐                                      │
│   │ 1. LOAD      │  載入設定值（連線字串、外部設定等）     │
│   │   Config     │  ※ 與組合邏輯分離，利於測試           │
│   └──────┬───────┘                                      │
│          ▼                                              │
│   ┌──────────────┐                                      │
│   │ 2. BUILD     │  建構完整的物件圖                     │
│   │  Object Graph│  ※ 使用 Pure DI 或 DI Container      │
│   └──────┬───────┘                                      │
│          ▼                                              │
│   ┌──────────────┐                                      │
│   │ 3. INVOKE    │  呼叫物件圖上的方法來執行功能          │
│   │ Functionality│  ※ Composition Root 不做業務邏輯      │
│   └──────┬───────┘                                      │
│          ▼                                              │
│   ┌──────────────┐                                      │
│   │ 4. RELEASE   │  釋放物件圖（處理 IDisposable 等）    │
│   │ Object Graph │  ※ 第 8 章詳述                       │
│   └──────────────┘                                      │
└────────────────────────────────────────────────────────┘
```

### Framework 2：辨識框架 Seam 的思維模型

每個框架的 DI 難度取決於它的 IoC 程度。IoC 越高，你需要找到更精確的 Seam：

```
        低 IoC ◄────────────────────────────────────► 高 IoC

        Console          UWP              ASP.NET Core MVC
          │               │                      │
          ▼               ▼                      ▼
      你全權控制      你大部分控制          框架控制大部分
      Main() 就是     OnLaunched()        必須實作
      Composition     可直接操作           IControllerActivator
      Root                                來「攔截」框架

   難度：★☆☆☆☆    難度：★★☆☆☆       難度：★★★☆☆
```

**決策公式：**
1. 框架有沒有 IoC？ → 沒有 → 直接在 entry point 組合（Console）
2. 有 IoC，但可以手動指定根物件？ → 手動設定即可（UWP: `frame.Content = page`）
3. 有 IoC，且框架控制物件建立？ → 找到框架的 Factory 介面來攔截（ASP.NET Core: `IControllerActivator`）

### Framework 3：Pure DI 的 Composer 模式

不論應用類型，Composer 的實作都遵循同一模式：

```csharp
// 通用模式：根據類型選擇組合策略
object Compose(Type requestedType)
{
    if (requestedType == typeof(TypeA))
    {
        return new TypeA(
            new DependencyX(),
            new DependencyY(
                new SubDependencyZ()));
    }
    else if (requestedType == typeof(TypeB))
    {
        return new TypeB(
            new DependencyW());
    }
    else
    {
        throw new Exception($"Unknown type: {requestedType.Name}");
    }
}
```

這個模式出現在：
- Console 的 `CreateCurrencyParser()`
- UWP 的 `App.CreatePage(Type vmType)`
- ASP.NET Core 的 `CommerceControllerActivator.Create(ControllerContext)`

### Framework 4：設定分離原則

```
  ╔═══════════════════════════════════════════════════════════╗
  ║  設定載入 (Load)  ≠  物件組合 (Compose)                    ║
  ║                                                           ║
  ║  LoadConnectionString()    CreateSomething(connString)    ║
  ║        ↓                           ↓                     ║
  ║  讀取 appsettings.json     接收純粹的 string 參數          ║
  ║  與檔案系統耦合             與檔案系統無關                   ║
  ║  不可單元測試               可單元測試                      ║
  ╚═══════════════════════════════════════════════════════════╝
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：UWP 中 App 同時擔任 Composition Root 和 INavigationService

**為什麼難：** 一個類別同時扮演兩個角色——既是組合所有物件的「工廠」，又是被注入到 ViewModel 中的「導航服務」。這似乎違反了單一職責原則。

**深入解析：**

```
                  App 類別的雙重身份
                  ═══════════════

     身份 1：Composition Root          身份 2：INavigationService
     ┌────────────────────────┐       ┌──────────────────────────┐
     │ OnLaunched()           │       │ NavigateTo<TViewModel>() │
     │ CreatePage(Type)       │       │                          │
     │                        │       │ 被注入到 ViewModel 中     │
     │ 負責「組合」物件        │       │ 提供「導航」功能          │
     └────────────────────────┘       └──────────────────────────┘
                 │                                 ▲
                 │         new MainViewModel(      │
                 └─────────────  this,  ───────────┘
                                 repo);
```

**為什麼可以接受：**
- Composition Root 本來就是一個特殊的地方——它知道所有模組的存在。
- `App` 實作 `INavigationService` 是合理的，因為導航操作（`frame.Content = page`）需要存取 UWP 的 `Frame`，而 `App` 天然擁有這個存取權。
- `CreatePage` 方法在 `NavigateTo` 內被呼叫，每次導航時重新組合物件圖——這正是 Rich Client 應用的特殊需求。

**白話比喻：** 想像一個大樓的管理員。他既是「建築工人」（負責組裝房間裡的家具），又是「電梯操作員」（負責帶住戶到正確的樓層）。在小型建築中，同一個人做這兩件事是合理且高效的。

### 難點 2：ASP.NET Core 中 IControllerActivator 的定位

**為什麼難：** ASP.NET Core 自帶 DI Container，為什麼還要自訂 `IControllerActivator`？Pure DI 和框架內建 DI Container 之間的關係是什麼？

**深入解析：**

```
    ASP.NET Core 的兩條路徑
    ═══════════════════════

    路徑 A：使用內建 DI Container          路徑 B：使用 Pure DI
    ┌──────────────────────────┐          ┌──────────────────────────┐
    │ services.AddTransient    │          │ 實作 IControllerActivator │
    │   <IProductService,      │          │                          │
    │    ProductService>();    │          │ 在 Create() 中手動 new    │
    │                          │          │ 所有物件                  │
    │ 框架自動解析建構子參數    │          │                          │
    │ 自動建立 Controller     │          │ 完全掌控物件的建立        │
    └──────────────────────────┘          └──────────────────────────┘

    本章選擇路徑 B，原因：
    - 第 12 章才會討論何時選 DI Container
    - Pure DI 讓你更清楚看到物件圖的全貌
    - 編譯期就能發現缺少的相依（不會 runtime 炸掉）
```

**作者的建議：** 即使使用第三方 DI Container，也建議保留 ASP.NET Core 內建的 DI Container 來管理框架自身的服務，只用自訂方式（或第三方 Container）來管理你自己的應用元件。

### 難點 3：Middleware 的 DI 整合策略

**為什麼難：** ASP.NET Core 的 Middleware 和 Controller 的建立機制不同。Controller 有 `IControllerActivator` 這個清晰的 Seam，但 Middleware 沒有對等的介面。

**深入解析：**

```
    Controller 的 DI                    Middleware 的 DI
    ════════════════                    ═════════════════

    IControllerActivator               app.Use(lambda)
         │                                  │
         ▼                                  ▼
    框架呼叫 Create()              在 lambda 內手動 new
    你回傳組裝好的 Controller      然後呼叫 Invoke()

    ┌─────────────────────┐       ┌─────────────────────────┐
    │ 有正式的 Seam        │       │ 沒有正式的 Seam          │
    │ 框架知道你的存在     │       │ 用 lambda 作為「接縫」    │
    │ 完整的生命週期管理   │       │ 自己管理生命週期          │
    └─────────────────────┘       └─────────────────────────┘
```

**解決策略：**
1. 先在 lambda 外面建立好相依（如 `ILogger`）
2. 在 lambda 內部用 Pure DI `new` 出 Middleware 實例
3. 呼叫 `middleware.Invoke(context, next)` 來執行

```csharp
// 關鍵：lambda 閉包捕獲了 logger，但每次請求都 new 出新的 middleware
var logger = loggerFactory.CreateLogger("Middleware");  // 共享
app.Use(async (context, next) =>
{
    var middleware = new LoggingMiddleware(logger);      // 每次新建
    await middleware.Invoke(context, next);
});
```

**白話比喻：** Controller 的 DI 像是正式的「訂做西裝服務」——裁縫店（IControllerActivator）有一套標準流程。Middleware 的 DI 更像是「自己動手改衣服」——你得找到布料（相依），自己裁剪（new），自己縫好（Invoke）。

### 架構啟示

#### 啟示 1：Composition Root 是「唯一知道一切」的地方

在整個應用程式中，只有 Composition Root 知道所有的具體類別。其餘的程式碼只認識抽象介面。這意味著：
- 換資料庫？只改 Composition Root。
- 換外部服務？只改 Composition Root。
- 加快取層？只改 Composition Root。
- 其餘所有商業邏輯完全不受影響。

```
┌────────────────────────────────────────────────────────────┐
│                                                             │
│  99% 的程式碼                  1% 的程式碼                   │
│  ┌──────────────────────┐     ┌────────────────────────┐   │
│  │ 只知道 Interface      │     │ Composition Root       │   │
│  │ 不知道誰在實作        │     │ 知道所有 Concrete      │   │
│  │                      │     │ 負責所有接線            │   │
│  │ 高度可測試            │     │ 不需要測試              │   │
│  │ 高度可替換            │     │ （它本身就是設定）      │   │
│  └──────────────────────┘     └────────────────────────┘   │
│                                                             │
└────────────────────────────────────────────────────────────┘
```

#### 啟示 2：框架的 IoC 程度決定 DI 的難度

不是所有框架都對 DI 友善。識別「框架在哪裡搶走你的控制權」是實踐 DI 的第一步。找到 Seam 之後，就能插入你的 Composition Root。

#### 啟示 3：Pure DI 的價值

本章所有範例都使用 Pure DI（直接 `new`），而不是 DI Container。這有幾個好處：
- **編譯期安全**：缺少相依會直接編譯錯誤，不會等到 runtime 才爆
- **物件圖一目了然**：打開 Composer 方法就能看到完整的物件組合
- **無魔法**：不需要學習任何 Container 的 API

#### 啟示 4：同一個領域模型，多種前端

```
                    ┌─────────────────────┐
                    │    Domain Layer      │
                    │  (IProductRepository │
                    │   Product, etc.)     │
                    └─────────┬───────────┘
                              │
              ┌───────────────┼───────────────┐
              │               │               │
              ▼               ▼               ▼
        ┌──────────┐   ┌──────────┐   ┌──────────────┐
        │ Console  │   │   UWP    │   │ ASP.NET Core │
        │ App      │   │  Client  │   │    MVC       │
        └──────────┘   └──────────┘   └──────────────┘

    三個不同的 Composition Root，
    但底下的 Domain 和 Data Access 可以共用！
```

這正是鬆耦合的威力——相同的商業邏輯，可以被不同的 UI 框架組合使用，而商業邏輯本身完全不需要修改。
