## 1. 概述 (Overview)

本章是全書第四部分「DI Containers」的入門篇。作者用了一個精彩的比喻開場：手工做冰淇淋 vs. 用冰淇淋機做冰淇淋。DI（依賴注入）本身是一種**技術（Technique）**，而 DI Container 是幫助你更輕鬆實踐這項技術的**工具（Technology）**。就像你不需要冰淇淋機也能做冰淇淋，你不需要 DI Container 也能實踐 DI——這就是 Pure DI。

本章的核心議題：

- **DI Container 是什麼？** 一個能自動化 Object Composition（物件組合）、Interception（攔截）、Lifetime Management（生命週期管理）的軟體函式庫。
- **Auto-Wiring 如何運作？** 容器利用反射（Reflection）分析建構子，遞迴解析依賴，自動組裝物件圖（Object Graph）。
- **三種配置方式的取捨？** Configuration Files、Configuration as Code、Auto-Registration 各有優劣。
- **何時該用 DI Container？何時用 Pure DI？** 這是架構決策的關鍵判斷。

> **白話說明**：DI Container 就像「自動化工廠」，你告訴它「介面 A 對應實作 B」，它就能幫你把整個物件樹組裝起來，你不必手動 `new` 每一個依賴。

## 2. 學習目標 (Learning Objectives)

讀完本章後，你應該能夠：

1. **解釋 Auto-Wiring 的運作原理**：理解 DI Container 如何透過反射遞迴解析建構子依賴
2. **區分三種配置方式**：清楚 Configuration Files / Configuration as Code / Auto-Registration 的適用場景、優缺點
3. **掌握 Convention over Configuration 的思維**：理解為何慣例優於配置能大幅降低維護成本
4. **做出「Pure DI vs. DI Container」的架構決策**：根據專案規模、團隊熟悉度、風險評估做出正確選擇
5. **理解使用第三方函式庫的隱性成本**：學習曲線、替換風險、團隊知識壁壘

## 3. 章節大綱 (Outline)

### 3.1 DI Container 簡介 (12.1 Introducing DI Containers)

#### 3.1.1 容器的 Resolve API

DI Container 的核心 API 非常簡單——只有一個方法：

```csharp
// 弱型別版本
object Resolve(Type serviceType);

// 泛型版本（編譯時期已知型別時使用）
T Resolve<T>();
```

以 Autofac 為例：

```csharp
var controller = (HomeController)container.Resolve(typeof(HomeController));
```

以 Simple Injector 為例：

```csharp
var controller = (HomeController)container.GetInstance(typeof(HomeController));
```

> **架構警語**：`Resolve` 的方法簽名跟 Service Locator 完全一樣！你必須確保只在 **Composition Root** 中呼叫它，否則就會不小心落入 Service Locator 反模式（詳見第 5 章）。

#### 3.1.2 Auto-Wiring 機制

**Auto-Wiring（自動接線）** 是 DI Container 的核心能力。它的運作原理如下：

```
+-------------------------------------------------------------------+
|                     Auto-Wiring 流程                               |
+-------------------------------------------------------------------+
|                                                                    |
|  要求解析              尋找具體型別             遞迴呼叫            |
|  Abstraction  ------>  對應此 Abstraction  ---->  解析每個           |
|  (介面)                (查字典)                   建構子參數          |
|                                                                    |
|       ^                                              |             |
|       |                                              |             |
|       +---------- 遞迴 (Recursive) ------------------+             |
|                                                                    |
|                         |                                          |
|                         v                                          |
|                  所有參數都解析完畢                                   |
|                         |                                          |
|                         v                                          |
|              用反射建構具體型別                                       |
|              Activator.CreateInstance(type, dependencies)           |
|                         |                                          |
|                         v                                          |
|                    回傳實例                                          |
+-------------------------------------------------------------------+
```

> **白話比喻**：Auto-Wiring 就像「自動組裝生產線」。你只需要提供「零件清單」（Abstraction -> 具體型別的對應表），生產線就會自動查看每個零件需要哪些子零件，一路往下找到最底層，然後從底層開始一層一層往上組裝，最後交出完成品。

#### 3.1.3 範例：簡易 DI Container 實作

作者實作了一個教學用的 `AutoWireContainer`，展示 Auto-Wiring 的本質：

```csharp
public class AutoWireContainer
{
    // 註冊表：Abstraction -> 建立實例的工廠方法
    Dictionary<Type, Func<object>> registrations =
        new Dictionary<Type, Func<object>>();

    // 註冊：告訴容器「遇到 serviceType 時，建立 componentType」
    public void Register(Type serviceType, Type componentType)
    {
        this.registrations[serviceType] =
            () => this.CreateNew(componentType);
    }

    // 註冊：直接提供工廠委派（繞過 Auto-Wiring）
    public void Register(Type serviceType, Func<object> factory)
    {
        this.registrations[serviceType] = factory;
    }

    // 解析：從註冊表找到對應的工廠方法並執行
    public object Resolve(Type type)
    {
        if (this.registrations.ContainsKey(type))
        {
            return this.registrations[type]();
        }
        throw new InvalidOperationException(
            "No registration for " + type);
    }

    // Auto-Wiring 的核心：透過反射分析建構子，遞迴解析依賴
    private object CreateNew(Type componentType)
    {
        var ctor = componentType.GetConstructors()[0];

        var dependencies =
            from p in ctor.GetParameters()
            select this.Resolve(p.ParameterType);  // 遞迴！

        return Activator.CreateInstance(
            componentType, dependencies.ToArray());
    }
}
```

使用方式——以 `HomeController` 的物件圖為例：

```csharp
// Pure DI 的寫法（手動組裝）
new HomeController(
    new ProductService(
        new SqlProductRepository(
            new CommerceContext(connectionString)),
        new AspNetUserContextAdapter()));

// 用 AutoWireContainer（自動組裝）
var container = new AutoWireContainer();

container.Register(typeof(IUserContext), typeof(AspNetUserContextAdapter));
container.Register(typeof(IProductRepository), typeof(SqlProductRepository));
container.Register(typeof(IProductService), typeof(ProductService));
container.Register(typeof(HomeController), typeof(HomeController));

// CommerceContext 含有 primitive 參數，需手動接線
container.Register(
    typeof(CommerceContext),
    () => new CommerceContext(connectionString));

// 一行解析整個物件圖
object controller = container.Resolve(typeof(HomeController));
```

容器解析 `HomeController` 的遞迴過程：

```
Composition Root
      |
      v
  Resolve<HomeController>
      |
      v
  Resolve<IProductService> -> ProductService
      |          |
      v          v
  Resolve       Resolve
  <IProduct     <IUserContext>
   Repository>       |
      |              v
      v         Activator.CreateInstance
  Resolve       <AspNetUserContextAdapter>()
  <Commerce
   Context>
      |
      v
  委派呼叫:
  () => new CommerceContext(connStr)
      |
      v
  Activator.CreateInstance<SqlProductRepository>(commerceContext)
      |
      v
  Activator.CreateInstance<ProductService>(repo, userContext)
      |
      v
  Activator.CreateInstance<HomeController>(productService)
      |
      v
  回傳 HomeController 實例
```

> **Auto-Wiring 的好處**：當你在 `AspNetUserContextAdapter` 新增一個 `CommerceContext` 依賴時，Pure DI 的 Composition Root 必須手動修改，但 Auto-Wiring 的容器什麼都不用改——它會自動發現新的建構子參數並解析。

### 3.2 配置 DI Container (12.2 Configuring DI Containers)

三種配置方式的定位：

```
                       明確 (Explicit)
                           ^
                           |
              Configuration     Configuration
              Files              as Code
                           |
         Late Binding <----+----> Early Binding
                           |
                       Auto-
                       Registration
                           |
                           v
                       隱式 (Implicit)
```

#### 3.2.1 Configuration Files（配置檔）

以 Autofac 的 JSON 配置為例：

```json
{
  "defaultAssembly": "Commerce.Web",
  "components": [
    {
      "services": [{
        "type": "Commerce.Domain.IUserContext, Commerce.Domain"
      }],
      "type": "Commerce.Web.AspNetUserContextAdapter"
    },
    {
      "services": [{
        "type": "Commerce.Domain.IProductRepository, Commerce.Domain"
      }],
      "type": "Commerce.SqlDataAccess.SqlProductRepository, Commerce.SqlDataAccess"
    },
    {
      "type": "Commerce.Web.Controllers.HomeController"
    },
    {
      "type": "Commerce.SqlDataAccess.CommerceContext, Commerce.SqlDataAccess",
      "parameters": {
        "connectionString": "Server=.;Database=MaryCommerce;Trusted_Connection=True;"
      }
    }
  ]
}
```

載入配置：

```csharp
var builder = new Autofac.ContainerBuilder();

IConfigurationRoot configuration =
    new ConfigurationBuilder()
        .AddJsonFile("autofac.json")
        .Build();

builder.RegisterModule(
    new Autofac.Configuration.ConfigurationModule(configuration));
```

**優點**：支援 Late Binding（不重新編譯就能切換實作）
**缺點**：冗長、脆弱、沒有編譯器檢查、不能用 lambda 表達式

> **真實案例警告**：作者 Steven 曾經歷一個百人年的大型專案，完全使用 Spring.NET XML 配置檔。XML 不僅冗長脆弱、沒有 Auto-Wiring 支援，還頻繁造成合併衝突，每天浪費大量開發時間。

**使用時機**：僅用於真正需要 Late Binding 的少數元件（例如：讓客戶可以自訂插件）。

#### 3.2.2 Configuration as Code（程式碼配置）

以 Microsoft.Extensions.DependencyInjection 為例：

```csharp
var services = new ServiceCollection();

services.AddSingleton<IUserContext, AspNetUserContextAdapter>();
services.AddTransient<IProductRepository, SqlProductRepository>();
services.AddTransient<IProductService, ProductService>();
services.AddTransient<HomeController>();  // 對應自己
services.AddScoped<CommerceContext>(
    p => new CommerceContext(connectionString));  // 委派工廠
```

**優點**：
- 比配置檔簡潔得多
- 有編譯器支援（泛型型別檢查）
- 支援 lambda 表達式

**缺點**：
- 不支援 Late Binding
- 每新增一個型別就要記得註冊（線性增長）

> **白話說明**：Configuration as Code 就像「手寫的購物清單」——你一條一條列出需要的東西，清楚明確，但清單會越來越長。

#### 3.2.3 Auto-Registration（自動註冊 / 慣例註冊）

**問題**：當 Configuration as Code 的註冊變得重複：

```csharp
// 這些註冊長得太像了，違反 DRY 原則
services.AddTransient<IProductRepository, SqlProductRepository>();
services.AddTransient<ICustomerRepository, SqlCustomerRepository>();
services.AddTransient<IOrderRepository, SqlOrderRepository>();
services.AddTransient<IShipmentRepository, SqlShipmentRepository>();
services.AddTransient<IImageRepository, SqlImageRepository>();
```

**解法**：用慣例掃描（以 Simple Injector 為例）：

```csharp
var assembly = typeof(SqlProductRepository).Assembly;

// LINQ 查詢：找出所有符合慣例的型別
var repositoryTypes =
    from type in assembly.GetTypes()
    where !type.IsAbstract
    where type.Name.EndsWith("Repository")
    select type;

foreach (Type type in repositoryTypes)
{
    container.Register(
        type.GetInterfaces().Single(), type);
}
```

**泛型 Auto-Registration**——處理開放泛型（Open Generic）更是 Auto-Registration 的殺手級應用：

```csharp
// 一行搞定所有 ICommandService<T> 的實作
Assembly assembly = typeof(AdjustInventoryService).Assembly;
container.Register(typeof(ICommandService<>), assembly);

// 一行搞定泛型 Decorator 的包裝
container.RegisterDecorator(
    typeof(ICommandService<>),
    typeof(AuditingCommandServiceDecorator<>));

container.RegisterDecorator(
    typeof(ICommandService<>),
    typeof(TransactionCommandServiceDecorator<>));

container.RegisterDecorator(
    typeof(ICommandService<>),
    typeof(SecureCommandServiceDecorator<>));
```

如果沒有 Auto-Registration，你得為每一個封閉泛型手動寫上三行 Decorator 註冊——四個實作就是 12 行，而且每新增一個實作就要加 3 行，指數級增長。

> **白話比喻**：Auto-Registration 就像公司的「新人自動入職流程」。你定好規則：「名字結尾是 Repository 的人自動分配到資料存取部門」，之後不管來多少人，都自動安排到位，不需要 HR 一個一個處理。

#### 3.2.4 混合搭配

三種方式並不互斥，最佳實務是：

```
+------------------------------------------------------------------+
|                   配置策略優先順序                                   |
+------------------------------------------------------------------+
|                                                                   |
|   1. Auto-Registration（主力）                                     |
|      - 用慣例處理大部分元件                                         |
|      - 例如：所有 Repository、所有 Service                          |
|                                                                   |
|   2. Configuration as Code（補充）                                  |
|      - 處理無法套用慣例的特殊元件                                    |
|      - 例如：需要 lambda 工廠的元件                                  |
|                                                                   |
|   3. Configuration Files（罕用）                                    |
|      - 僅用於真正需要 Late Binding 的元件                            |
|      - 例如：客戶可替換的插件                                        |
|                                                                   |
+------------------------------------------------------------------+
```

### 3.3 何時使用 DI Container (12.3 When to Use a DI Container)

#### 3.3.1 第三方函式庫的隱性成本

使用任何第三方函式庫（包含 DI Container）都有隱性成本：

| 成本/風險 | 說明 |
|-----------|------|
| **學習曲線** | 團隊成員都必須學習 API、行為、怪癖、限制 |
| **巴士因子** | 如果只有一個人會用，那個人離職就完了 |
| **觀點不合** | 函式庫的設計哲學可能跟你的架構不相容 |
| **棄置風險** | 開發者可能停止維護（例如曾經的 Spring.NET） |
| **替換成本** | 換一個容器 = 重新學習 + 遷移 + 測試 |
| **Yak Shaving** | 為了繞過函式庫的限制而做的一堆無關工作 |

> **名詞解釋**：
> - **巴士因子 (Bus Factor)**：團隊中最少要有幾個人「被巴士撞」（突然離開）才會導致專案停擺。如果只有一個人懂 DI Container，巴士因子就是 1，非常危險。
> - **Yak Shaving（剃犛牛毛）**：為了解決 A 問題，你需要先解決 B，為了 B 又需要 C，一層套一層地做著看似無關的事情。

**重要提醒：不要自己造 DI Container！** 看似 listing 12.1 只有幾十行程式碼，但一個成熟的 DI Container 需要支援 Lifetime Management、Interception、Auto-Registration、循環依賴檢測、清楚的錯誤訊息、擴充點、完整文件......這需要好幾年才能成熟穩定。

#### 3.3.2 Pure DI 的回饋週期更短

Pure DI 的最大優勢：**強型別 (Strongly Typed)**。

```
+-------------------------------------------+
|         回饋週期比較                         |
+-------------------------------------------+
|                                            |
|  Pure DI:                                  |
|  程式碼 -> 編譯器立即報錯 -> 修正            |
|  (最快的回饋)                                |
|                                            |
|  DI Container:                             |
|  程式碼 -> 編譯通過 -> 執行時期才報錯         |
|  (較慢的回饋)                                |
|                                            |
+-------------------------------------------+
```

Pure DI 能在編譯時偵測的問題：
- 缺少的依賴（建構子參數變更時 Composition Root 編譯失敗）
- 部分 Captive Dependencies（生命週期錯誤，如 Scoped 被 Singleton 捕獲）

Pure DI 無法在編譯時偵測的問題：
- 建構子內部拋出的例外
- 可棄置元件是否有被正確 Dispose
- 同一個元件在 Composition Root 不同地方被建立（Torn Lifestyles / Ambiguous Lifestyles）

> DI Container 的優勢正好在此——多數成熟容器能自動偵測可棄置元件、Captive Dependencies 等問題。

#### 3.3.3 最終裁決：決策框架

```
+===================================================================+
|              Pure DI vs. DI Container 決策矩陣                      |
+===================================================================+
|                                                                    |
|                        有價值 (Valuable)                            |
|                             ^                                      |
|                             |                                      |
|         Pure DI  o          |          o  Auto-Registration        |
|                             |                                      |
|    簡單 (Simple) <----------+---------> 精緻 (Sophisticated)        |
|                             |                                      |
|                             |                                      |
|                    Configuration as Code  o                        |
|                             |                                      |
|                             v                                      |
|                        無意義 (Pointless)                           |
|                                                                    |
+===================================================================+
|                                                                    |
|  結論：                                                             |
|  - Pure DI：簡單且有價值，適合小型專案                                |
|  - Auto-Registration：精緻且有價值，適合大型專案                      |
|  - Configuration as Code（單獨使用）：失去強型別又沒減少維護，無意義    |
|                                                                    |
+===================================================================+
```

**決策指南**：

| 情境 | 建議 |
|------|------|
| 小型專案、Composition Root 容易維護 | **Pure DI** |
| 大型專案、元件數量多且可歸納慣例 | **Auto-Registration** |
| 團隊對 DI Container 不熟悉 | **Pure DI**（降低學習成本） |
| 跨多個應用共用程式庫、建構子頻繁重構 | **DI Container**（Auto-Wiring 減少合併衝突） |
| 只用 Configuration as Code 而不用 Auto-Registration | **避免**（成本 > 收益） |
| 少量元件需要 Late Binding | **Pure DI + 手動載入配置檔** |

> **真實案例**：作者 Mark 曾在一個專案中導入 Convention-based Auto-Registration，但團隊其他成員覺得太「自動魔法 (Automagical)」，不願意學習容器 API。Mark 離開後，團隊果然把 DI Container 換回了 Pure DI。

## 4. 關鍵框架與公式 (Key Frameworks)

### 框架一：Auto-Wiring 演算法

```
輸入：serviceType（要解析的型別）
輸出：完全組裝好的物件實例

步驟：
1. 在 registrations 字典中查找 serviceType
2. 如果找到自訂委派 -> 直接呼叫委派回傳
3. 如果找到具體型別映射 -> 進入 CreateNew：
   a. 用反射取得該型別的建構子
   b. 遍歷每個建構子參數
   c. 對每個參數的型別，遞迴呼叫 Resolve（回到步驟 1）
   d. 所有參數解析完畢後，用 Activator.CreateInstance 建構物件
4. 如果找不到 -> 拋出例外
```

### 框架二：三種配置方式比較表

| 維度 | Configuration Files | Configuration as Code | Auto-Registration |
|------|--------------------|-----------------------|-------------------|
| **明確性** | 明確（逐一列出） | 明確（逐一列出） | 隱式（慣例驅動） |
| **綁定時機** | Late Binding | Early Binding | 介於兩者之間 |
| **編譯器檢查** | 無 | 有（泛型約束） | 無 |
| **簡潔程度** | 冗長 | 中等 | 精簡 |
| **擴展性** | 差（線性增長） | 差（線性增長） | 優（慣例自動涵蓋） |
| **適用時機** | 極少數 Late Binding 元件 | 特殊元件補充註冊 | 大量相似元件的主要策略 |

### 框架三：Pure DI vs. DI Container 決策公式

```
決策 =
  如果（Composition Root 小且易維護）
    -> 使用 Pure DI

  如果（Composition Root 大、重複註冊多、可歸納慣例）
    -> 使用 DI Container + Auto-Registration

  如果（團隊不願或無時間學習容器 API）
    -> 使用 Pure DI

  絕對不要：
    -> 僅用 Configuration as Code（不搭配 Auto-Registration）
    -> 自己造 DI Container
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點一：Auto-Wiring 的遞迴本質

**為什麼難？** 遞迴本身就是程式設計中較難理解的概念，加上反射和型別系統的交織，讓初學者容易迷失。

**突破方式**：用具體的物件圖來追蹤。以 `HomeController` 為例，畫出整棵依賴樹，然後從**葉節點**（沒有依賴的類別，如 `AspNetUserContextAdapter`）開始往上組裝：

```
HomeController                    <-- 第 4 步建構（需要 ProductService）
  |
  +-- ProductService              <-- 第 3 步建構（需要 Repo + UserContext）
        |
        +-- SqlProductRepository  <-- 第 2 步建構（需要 CommerceContext）
        |     |
        |     +-- CommerceContext  <-- 第 1a 步：委派直接建立（葉節點）
        |
        +-- AspNetUserContextAdapter  <-- 第 1b 步：無參數建構子（葉節點）
```

重點是理解：容器會**先往下探**到最底層，然後**從底層往上建構**。

### 難點二：Primitive Types 的 Hand Wiring 問題

**為什麼難？** 初學者常困惑：「既然容器能自動接線，為什麼 `CommerceContext` 要手動？」

**核心原因**：DI Container 是靠**型別 (Type)** 來匹配依賴的。`connectionString` 是 `string` 型別，但你的應用可能有很多 `string` 型別的配置值（資料庫連線字串、API Key、檔案路徑......），容器無法僅靠型別分辨它們。

```csharp
// 這行告訴容器：「遇到 CommerceContext 時，別用 Auto-Wiring，
// 直接執行這個委派來建立」
container.Register(
    typeof(CommerceContext),
    () => new CommerceContext(connectionString));
```

> **比喻**：Auto-Wiring 像快遞員看「收件人姓名」（型別）來送包裹。如果有三個人都叫「張三」（都是 `string`），快遞員就不知道該送給誰了。這時你就得親自送（Hand Wiring）。

### 難點三：「Configuration as Code 單獨使用是無意義的」這個論點

**為什麼難？** 很多開發者第一次接觸 DI Container 就是學 `services.AddTransient<IA, A>()`，覺得這樣就夠了，為什麼書中說它「pointless」？

**作者的邏輯鏈**：

1. **Pure DI** 有編譯器檢查的優勢 -> Configuration as Code **失去了**這個優勢（因為用了反射）
2. **Auto-Registration** 能讓維護量趨近於零 -> Configuration as Code **沒有**這個優勢（每新增一個類別仍要手動註冊）
3. 所以 Configuration as Code 是**兩邊的缺點都拿了，但兩邊的優點都沒拿到**的中間地帶

```
Pure DI 的優點：           Configuration as Code：    Auto-Registration 的優點：
- 強型別                   - 失去強型別               - 維護趨近零
- 編譯器即時回饋            - 仍需線性維護             - 慣例驅動
                           = 兩邊都不佔               - 自動發現新元件
```

**並非完全不能用**——它適合作為 Auto-Registration 的「補充」，處理少數無法套用慣例的特殊元件。但如果你的 Composition Root **全部**都是 Configuration as Code，那你不如用 Pure DI（至少有編譯器保護）。

---

## 6. 本章小結與架構啟示

### 核心觀念

1. **DI Container 是可選工具，不是 DI 的先決條件**。DI 是技術，Container 是輔助工具。
2. **Auto-Wiring 是 DI Container 的核心價值**。它讓 Composition Root 對建構子變更有韌性。
3. **三種配置方式的正確搭配**：Auto-Registration 為主、Configuration as Code 為輔、Configuration Files 僅用於 Late Binding。
4. **Pure DI vs. DI Container 不是信仰問題，是工程決策**。小專案用 Pure DI，大專案用 Auto-Registration。
5. **Convention over Configuration 是高階架構思維**。它不只減少配置，更強制團隊遵循一致的程式碼結構。

### 架構啟示

```
+==================================================================+
|                    架構師的五個行動項目                              |
+==================================================================+
|                                                                   |
|  1. 限制 DI Container 於 Composition Root                         |
|     -> 防止 Service Locator 反模式                                 |
|                                                                   |
|  2. 建立命名慣例並嚴格遵守                                          |
|     -> XxxRepository, XxxService, XxxHandler...                   |
|     -> 讓 Auto-Registration 能順利運作                              |
|                                                                   |
|  3. 優先考慮 Pure DI                                               |
|     -> 直到 Composition Root 的維護成為真正的痛點                    |
|                                                                   |
|  4. 警惕「只用一個人懂容器」的陷阱                                    |
|     -> 巴士因子 = 1 是專案風險                                      |
|                                                                   |
|  5. 永遠不要自建 DI Container                                      |
|     -> 要嘛用現成的，要嘛用 Pure DI                                  |
|                                                                   |
+==================================================================+
```

### 一句話總結

> DI Container 的最高境界是「隱形」——透過 Auto-Registration 和 Convention over Configuration，讓容器自動發現並組裝元件，開發者專注於實作業務邏輯，而非基礎設施配置。但如果達不到這個境界，Pure DI 反而是更務實的選擇。
