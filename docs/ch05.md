> **書籍**：Dependency Injection Principles, Practices, and Patterns
> **作者**：Steven van Deursen, Mark Seemann
> **導讀視角**：資深系統架構師

## 1. 概述 (Overview)

本章深入剖析四種最常見的 DI 反模式：**Control Freak**、**Service Locator**、**Ambient Context** 與 **Constrained Construction**。反模式（Anti-Pattern）是一種看似合理卻帶來明顯負面後果的常見做法，就像烹飪時把油加熱到冒煙——看似代表鍋子夠熱，實際上油已經變質產生有害物質。每種反模式都附帶「為什麼是反模式」的分析，以及如何重構回正確 DI 模式的指引。理解這些陷阱，是從「會寫程式」進階到「會做架構設計」的關鍵一步。

---

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

- **辨識** 程式碼中四種 DI 反模式的具體特徵與程式碼氣味（code smell）
- **理解** 每種反模式為何會損害鬆耦合、可測試性、可維護性與可擴展性
- **分析** 為什麼某些「看似解決問題」的做法（如 Static Factory、Service Locator）實際上只是把問題搬到別處
- **區分** DI Container 在 Composition Root 中的合法使用 vs. Service Locator 反模式
- **實作** 從每種反模式重構回正確 DI 模式（主要是 Constructor Injection）的步驟
- **評估** 在遺留系統中何時可以暫時容忍反模式，以及何時必須立即修正

## 3. 章節大綱 (Outline)

### 3.1 反模式概論

- **定義**：反模式是一種常見的問題解決方案，雖然被廣泛採用，卻會產生明顯的負面後果，且已有更好的替代方案存在
- **與設計模式的關係**：模式是「好食譜」，反模式是「常見的烹飪錯誤」
- **四種 DI 反模式總覽**：

| 反模式 | 一句話描述 |
|---|---|
| **Control Freak** | 直接控制 Volatile Dependency 的建立，與 IoC 背道而馳 |
| **Service Locator** | 透過隱式服務提供依賴，但不保證能正確提供 |
| **Ambient Context** | 透過靜態存取器讓單一 Volatile Dependency 全域可用 |
| **Constrained Construction** | 強制所有實作必須有特定的建構函式簽章 |

### 3.2 Control Freak（控制狂）

- **定義與本質**
  - 在 Composition Root 以外的地方依賴 Volatile Dependency
  - 違反 Dependency Inversion Principle（DIP）
  - `new` 關鍵字用於 Volatile Dependency 就是 code smell
- **範例一：直接 new 依賴**
  - 宣告介面型別，卻在建構子裡直接 `new SqlProductRepository()`
  - 變數型別是抽象的，但執行時永遠是固定的具體型別
- **範例二：透過 Factory 嘗試解決**
  - Concrete Factory：只是把問題搬到另一個類別
  - Abstract Factory：陷入無窮遞迴——需要工廠的工廠
  - Static Factory：看似可設定，實則造成循環依賴與模組拖曳
- **範例三：透過多載建構子搭配 Foreign Default**
  - 預設建構子轉呼叫帶參數的建構子並傳入具體實作
  - 看似兩全其美，實則在正式環境中無法攔截依賴
- **分析與負面效應**
  - 無法在編譯後替換依賴
  - 消費模組拖曳不必要的依賴
  - 平行開發困難
  - 可測試性低落
- **重構方向**：提取介面 → 集中建立邏輯 → Constructor Injection

### 3.3 Service Locator（服務定位器）

- **定義與本質**
  - 在 Composition Root 外部提供對無限制 Volatile Dependency 集合的存取
  - 通常實作為可設定的 Static Factory
- **範例：ProductService 使用 Service Locator**
  - 無參數建構子，在方法中呼叫 `Locator.GetService<T>()`
  - 測試時需要 `Locator.Reset()` 與 `Locator.Register()`
- **分析與負面效應**
  - 拖曳冗餘依賴（Locator 本身）
  - 隱藏真正的依賴——API 不誠實
  - 執行期才發現遺漏的依賴
  - 弱型別：泛型只是假象，任意型別都能請求
  - 測試間互相干擾（Interdependent Tests）
- **關鍵區分**：DI Container 封裝在 Composition Root 裡不是 Service Locator，而是基礎設施元件
- **重構方向**：集中 Service Locator 呼叫 → 引入私有欄位 → Constructor Injection

### 3.4 Ambient Context（環境上下文）

- **定義與本質**
  - 透過靜態類別成員全域存取單一 Volatile Dependency
  - 類似 Singleton 模式，但允許替換依賴
- **範例一：ITimeProvider 時間提供者**
  - `TimeProvider.Current` 靜態屬性提供全域存取
  - 內含 Local Default（`DefaultTimeProvider` 使用真實系統時鐘）
- **範例二：Logger 記錄器**
  - `LogManager.GetLogger(typeof(T))` 是最常見的 Ambient Context
  - 開發者因「到處都需要」而合理化此做法
- **分析與負面效應**
  - 隱藏依賴，掩蓋 Constructor Over-injection（暗示違反 SRP）
  - 全域狀態使測試困難（平行測試互相干擾）
  - 難以根據上下文提供不同實作
  - Temporal Coupling（時序耦合）
- **重構方向**：集中靜態呼叫到建構子 → 引入私有欄位 → Constructor Injection

### 3.5 Constrained Construction（受限建構）

- **定義與本質**
  - 強制某個 Abstraction 的所有實作都必須有相同的建構子簽章
  - 通常源於想要實現 Late Binding 的需求
- **範例：使用 Activator.CreateInstance 進行晚期繫結**
  - 從設定檔讀取型別名稱，用 `Activator.CreateInstance` 建立實例
  - 傳入 connection string 作為建構參數——隱含假設所有實作都需要它
- **分析與負面效應**
  - 限制彈性：無法注入不同型別的依賴
  - 難以共享實例（如圖 5.6 所示的共用 CommerceContext）
  - Abstract Factory 嘗試解決卻帶來更多複雜度
- **重構方向**：使用 DI Container（反射分析建構子簽章）或保持 Composition Root 集中化

## 4. 關鍵框架與公式 (Key Frameworks)

### 4.1 反模式辨識決策樹

```
你的程式碼是否在 Composition Root 以外的地方建立 Volatile Dependency？
│
├─ 是 → 使用 new 關鍵字？
│       ├─ 是 → Control Freak
│       └─ 否 → 透過靜態方法取得？
│               ├─ 是 → 可取得任意型別的依賴？
│               │       ├─ 是 → Service Locator
│               │       └─ 否 → 單一依賴透過靜態屬性？
│               │               ├─ 是 → Ambient Context
│               │               └─ 否 → 其他 code smell
│               └─ 否 → 其他情況
│
├─ 否 → 是否強制所有實作有相同建構子簽章？
│       ├─ 是 → Constrained Construction
│       └─ 否 → 恭喜，沒有反模式！
```

### 4.2 Control Freak 辨識與重構框架

**何時使用**：當你發現程式碼在 Composition Root 以外直接 `new` Volatile Dependency 時。

**核心概念**：
- `new` 關鍵字用於 Stable Dependency（如 `List<T>`、`DateTime`）完全沒問題
- `new` 關鍵字用於 Volatile Dependency（如 `SqlProductRepository`）就是 Control Freak
- 三種常見變體：直接 new、Factory 包裝、多載建構子 + Foreign Default

**程式碼範例（反模式）**：

```csharp
// 反模式：Control Freak - 直接 new Volatile Dependency
public class ProductService : IProductService
{
    private readonly IProductRepository repository;

    public ProductService()
    {
        // 在 Composition Root 以外建立 Volatile Dependency
        this.repository = new SqlProductRepository();
    }
}
```

**程式碼範例（重構後）**：

```csharp
// 正確：Constructor Injection
public class ProductService : IProductService
{
    private readonly IProductRepository repository;

    public ProductService(IProductRepository repository)
    {
        this.repository = repository
            ?? throw new ArgumentNullException(nameof(repository));
    }
}
```

### 4.3 Service Locator vs. DI Container 區分框架

**何時使用**：當團隊爭論「這到底是 Service Locator 還是合法的 DI Container 使用」時。

**核心概念**：

```
+-------------------------------------------------------+
|  關鍵問題：誰在呼叫 Resolve / GetService？             |
+-------------------------------------------------------+
|                                                       |
|  Composition Root 呼叫 → 合法的 DI Container 使用     |
|    (基礎設施元件)                                      |
|                                                       |
|  應用程式碼呼叫   → Service Locator 反模式            |
|    (任何非 Composition Root 的地方)                    |
|                                                       |
+-------------------------------------------------------+
|  判斷標準不是 API 的靜態結構（機制），                   |
|  而是 API 在應用中扮演的角色（用途）。                   |
+-------------------------------------------------------+
```

**程式碼範例（反模式 — Service Locator）**：

```csharp
// 反模式：應用程式碼中使用 Service Locator
public class HomeController : Controller
{
    public ViewResult Index()
    {
        // 在應用程式碼中主動「索取」依賴 → Service Locator
        IProductService service =
            Locator.GetService<IProductService>();
        var products = service.GetFeaturedProducts();
        return this.View(products);
    }
}
```

**程式碼範例（正確 — 在 Composition Root 使用 DI Container）**：

```csharp
// 正確：僅在 Composition Root 使用 Container
// Startup.cs (ASP.NET Core)
public void ConfigureServices(IServiceCollection services)
{
    // 這裡是 Composition Root，使用 Container 是合法的
    services.AddScoped<IProductService, ProductService>();
    services.AddScoped<IProductRepository, SqlProductRepository>();
}
```

### 4.4 Ambient Context 辨識與重構框架

**何時使用**：當你看到透過靜態屬性或靜態方法存取 Volatile Dependency 時。

**核心概念**：

```
+--------------------------------------------------+
|  Ambient Context 的典型特徵：                      |
|                                                  |
|  1. 靜態類別或靜態屬性                             |
|  2. 提供對單一 Volatile Dependency 的全域存取       |
|  3. 可替換（與 Singleton 模式的差異）               |
|  4. 通常有 Local Default                          |
+--------------------------------------------------+
```

**程式碼範例（反模式）**：

```csharp
// 反模式：Ambient Context
public static class TimeProvider
{
    private static ITimeProvider current =
        new DefaultTimeProvider(); // Local Default

    public static ITimeProvider Current
    {
        get { return current; }
        set { current = value; }  // 可替換
    }

    private class DefaultTimeProvider : ITimeProvider
    {
        public DateTime Now => DateTime.Now;
    }
}

// 消費端 — 依賴被隱藏了
public class WelcomeMessageGenerator
{
    public string GetWelcomeMessage()
    {
        // 依賴藏在方法內部，從 API 外部看不到
        ITimeProvider provider = TimeProvider.Current;
        DateTime now = provider.Now;
        string partOfDay = now.Hour < 6 ? "night" : "day";
        return $"Good {partOfDay}.";
    }
}
```

**程式碼範例（重構後）**：

```csharp
// 正確：Constructor Injection
public class WelcomeMessageGenerator
{
    private readonly ITimeProvider timeProvider;

    public WelcomeMessageGenerator(ITimeProvider timeProvider)
    {
        this.timeProvider = timeProvider
            ?? throw new ArgumentNullException(nameof(timeProvider));
    }

    public string GetWelcomeMessage()
    {
        DateTime now = this.timeProvider.Now;
        string partOfDay = now.Hour < 6 ? "night" : "day";
        return $"Good {partOfDay}.";
    }
}
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：Service Locator 為什麼是反模式？它不是也能實現鬆耦合嗎？

**為什麼難**：
Service Locator 在表面上滿足了 DI 的大部分目標——支援 Late Binding、平行開發、可測試性。它「幾乎能用」，這正是它危險的地方。很多開發者用了之後覺得「挺好的啊」，難以理解為什麼它是反模式。

**白話解釋**：

想像你去一家餐廳點餐。

- **Constructor Injection** 就像菜單上清楚列出每道菜的食材。你點「番茄義大利麵」，你知道你會得到番茄、麵條、橄欖油、蒜頭。如果廚房缺了番茄，服務生會在你點餐時就告訴你「番茄用完了」。

- **Service Locator** 就像菜單上只有菜名，沒有食材列表。你點了之後，廚房才開始東翻西找食材。如果找不到某個關鍵食材，菜已經做到一半才發現問題，你要等更久才知道「做不了」。更糟的是，你對番茄過敏，但因為菜單上看不到食材，你根本無法事先判斷。

Service Locator 讓元件的 API「說謊」了——它看起來很簡單（無參數建構子），但其實暗藏複雜的依賴關係。這違反了「明確勝於隱含」的設計原則。

**程式碼示範**：

```csharp
// Service Locator: API 看起來很簡單...
var service = new ProductService(); // 看起來不需要任何東西

// 但執行時爆炸！因為忘了 Register
// 丟出 KeyNotFoundException，且錯誤訊息不明確

// ---

// Constructor Injection: API 誠實地告訴你需要什麼
// 編譯器就會提醒你：「嘿，你還沒提供 IProductRepository！」
var service = new ProductService(/* 編譯錯誤：缺少參數 */);
```

```
+-----------------------------------------------------------+
|  Service Locator 的「謊言」                                 |
+-----------------------------------------------------------+
|                                                           |
|  開發者看到的 API：                                        |
|  ┌─────────────────────────────┐                          |
|  │ ProductService()            │  ← 「我不需要任何東西」   |
|  └─────────────────────────────┘                          |
|                                                           |
|  實際的依賴圖：                                            |
|  ┌─────────────────────────────┐                          |
|  │ ProductService              │                          |
|  │   ├── Service Locator       │  ← 冗餘依賴              |
|  │   └── IProductRepository    │  ← 隱藏的真正依賴        |
|  └─────────────────────────────┘                          |
|                                                           |
|  Constructor Injection 的誠實 API：                        |
|  ┌──────────────────────────────────────────────┐         |
|  │ ProductService(IProductRepository repository) │         |
|  └──────────────────────────────────────────────┘         |
|    ← 「我需要一個 IProductRepository 才能工作」             |
+-----------------------------------------------------------+
```

### 難點 2：Static Factory 為什麼解決不了 Control Freak？不是已經可以透過設定檔切換了嗎？

**為什麼難**：
Static Factory 看起來非常合理——可以透過設定檔決定要用哪個實作，不用改程式碼就能切換。很多開發者（包含書中 Mary 和 Jens 的故事）都覺得這是正確答案。要理解為什麼它不行，需要深入理解「依賴的傳遞性」和「循環依賴」的概念。

**白話解釋**：

想像你經營一家手搖飲料店：

- **問題**：你想讓調飲師只知道「要加糖漿」（抽象），不用知道是哪牌的糖漿（具體實作）。
- **Static Factory 的做法**：你開了一個「糖漿倉庫」，裡面存了 A 牌和 B 牌的糖漿，還有一張表寫著「今天用 A 牌」。調飲師去倉庫拿就好。
- **問題所在**：但現在調飲師必須知道「倉庫在哪」，而倉庫必須同時存放所有品牌的糖漿——即使今天只用 A 牌，B 牌也得搬進來。如果未來要加 C 牌，你得改倉庫的結構並重新裝潢。

更嚴重的是，這會造成「循環依賴」——就像 A 部門說「我的工作需要 B 部門的產出」，B 部門又說「我的工作需要 A 部門的產出」，結果誰也動不了。

**程式碼示範**：

```csharp
// Static Factory — 看似可設定，實則問題多多
public static class ProductRepositoryFactory
{
    public static IProductRepository Create()
    {
        var config = new ConfigurationBuilder()
            .SetBasePath(Directory.GetCurrentDirectory())
            .AddJsonFile("appsettings.json")
            .Build();

        string repositoryType = config["productRepository"];

        switch (repositoryType)
        {
            case "sql": return new SqlProductRepository();
            case "azure": return new AzureProductRepository();
            default: throw new InvalidOperationException("...");
        }
    }
}
```

```
  問題：循環依賴
  ====================================

  Domain Library（領域層）
  ┌────────────────────────────────┐
  │  ProductService                │
  │       │ uses                   │
  │       v                       │
  │  IProductRepository           │
  │       │ uses                   │
  │       v                       │
  │  ProductRepositoryFactory ─────┼──── 依賴 SQL 層
  └────────────────────────────────┘
             ^                |
             |  循環依賴！     |
             |                v
  ┌────────────────────────────────┐
  │  SqlProductRepository          │──── 依賴 Domain 層
  │  AzureProductRepository        │     (因為實作 IProductRepository)
  └────────────────────────────────┘
  SQL Data Access Library（資料存取層）

  結果：無論 Factory 放在哪一層，
        都會造成兩個專案之間的循環參照！
```

### 難點 3：Constrained Construction 與 Late Binding 的兩難

**為什麼難**：
Late Binding（晚期繫結）是一個合理的需求——能不重新編譯就替換實作。但要達成這個目標，很容易掉入 Constrained Construction 的陷阱。而用 Abstract Factory 來解決，又會產生「Abstract Composition Root」的複雜度。這是一個需要權衡取捨的設計決策，沒有銀彈。

**白話解釋**：

想像你設計一套「萬用插座轉接頭」：

- **Constrained Construction** 就像規定「所有國家的插頭都必須是兩孔圓形」。這樣轉接頭很好做（`Activator.CreateInstance` 加固定參數），但現實中有的插頭是三孔的、有的是扁的，強制改成圓形會讓它們無法正常供電。

- **正確做法** 是設計一個「智慧型轉接頭」（DI Container），它能自動辨識插頭的形狀（透過反射分析建構子簽章），然後自動接上正確的轉接零件。

但最務實的建議是：如果你不是真的需要「不重新編譯就替換插頭」，就別搞複雜的萬用轉接頭。直接在啟動專案（Composition Root）裡把插頭接好就行了。

**程式碼示範**：

```csharp
// 反模式：Constrained Construction
// 所有 IProductRepository 的實作都被迫接受 string 參數
string connectionString = configuration
    .GetConnectionString("CommerceConnectionString");

// 隱含假設：所有實作都需要 connection string
var constructorArguments = new object[] { connectionString };

IProductRepository repository =
    (IProductRepository)Activator.CreateInstance(
        productRepositoryType,
        constructorArguments);
// 如果換成不需要 connection string 的實作呢？直接爆掉！

// ---

// 正確做法 (1)：不需要 Late Binding 時，直接在 Composition Root 建立
// Startup.cs
public void ConfigureServices(IServiceCollection services)
{
    string connStr = Configuration
        .GetConnectionString("CommerceConnectionString");

    // 每個實作可以有不同的建構子簽章，沒問題
    services.AddScoped<IProductRepository>(sp =>
        new SqlProductRepository(
            sp.GetRequiredService<IUserContext>(),
            new CommerceContext(connStr)));
}

// 正確做法 (2)：需要 Late Binding 時，使用 DI Container 的反射能力
// DI Container 會自動分析建構子參數型別並注入
services.AddScoped<IProductRepository, SqlProductRepository>();
services.AddScoped<IUserContext, AspNetUserContextAdapter>();
services.AddScoped(sp =>
    new CommerceContext(connStr));
```

## 6. 本章小結與架構啟示

### 關鍵結論

1. **Control Freak 是最致命的**：它是所有 DI 反模式中最常見的，因為 `new` 是程式語言的預設行為。每當你在 Composition Root 以外的地方 `new` 一個 Volatile Dependency，你就已經踏入了這個陷阱。

2. **Service Locator 是最危險的**：它之所以危險，是因為它「幾乎能用」。你可以替換依賴、可以測試，但你犧牲了 API 的誠實性。元件變成了「暗箱」——從外部看不出它需要什麼才能正常運作。

3. **Ambient Context 是最容易被合理化的**：「logging 到處都需要」、「時間到處都需要」——這些都是真的，但它不是使用全域靜態存取的藉口。正確的設計（第 10 章的 Cross-Cutting Concerns 處理方式）可以讓你既不汙染每個類別的建構子，又不需要 Ambient Context。

4. **Constrained Construction 是最容易避免的**：除非你真的需要 Late Binding（不重新編譯就替換元件），否則根本不會遇到它。大部分專案只需要在啟動專案的 Composition Root 裡直接組裝物件圖就好。

### 架構師的設計思維

```
  反模式的根源：對「控制」的執著
  ============================================

  開發者的直覺         DI 的思維
  ──────────────      ──────────────
  「我要自己建立」  →  「讓別人幫我組裝」
  「我要知道具體型別」→ 「我只關心抽象介面」
  「方便取用最重要」  →  「明確宣告最重要」
  「全域存取很方便」  →  「依賴注入更安全」

  核心觀念轉換：
  ┌──────────────────────────────────────────┐
  │  Don't call us, we'll call you.          │
  │  (好萊塢原則 / 控制反轉)                  │
  │                                          │
  │  不要主動「索取」依賴，                    │
  │  而是被動「接收」依賴。                    │
  └──────────────────────────────────────────┘
```

### 重構速查表

| 反模式 | 症狀 | 重構目標 | 第一步 |
|---|---|---|---|
| **Control Freak** | `new VolatileDep()` 在 Composition Root 外 | Constructor Injection | 提取介面、集中建立點 |
| **Service Locator** | `Locator.GetService<T>()` 在應用程式碼中 | Constructor Injection | 引入私有 readonly 欄位 |
| **Ambient Context** | `SomeProvider.Current` 靜態存取 | Constructor Injection | 將靜態呼叫集中到建構子 |
| **Constrained Construction** | `Activator.CreateInstance` + 固定參數 | DI Container 或集中化 Composition Root | 移除建構子簽章限制 |

### 給架構師的提醒

> **遺留系統中的務實考量**：在大型遺留系統中，反模式可能是「從完全沒有 DI」到「完整 DI」的過渡階段。即便如此，仍應將反模式視為技術債，有計畫地償還。書中提到的《Working Effectively with Legacy Code》(Michael Feathers) 是處理這類問題的經典參考。
>
> **判斷 Stable vs. Volatile Dependency 是關鍵**：`new List<string>()` 完全沒問題，`new SqlProductRepository()` 才是問題。分不清這兩者的區別，就會過度設計或設計不足。回顧第 1 章和第 4 章對 Stable/Volatile Dependency 的定義，是正確運用本章知識的前提。
