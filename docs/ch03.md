## 1. 概述 (Overview)

本章是全書第一部分的壓軸章節，以第 2 章 Mary 的緊耦合電商應用為對照，**從零開始**重新設計一個鬆耦合版本。透過實作，讀者將看到如何運用 **Constructor Injection**、**Method Injection**、**Composition Root**、**Adapter Pattern** 及 **Dependency Inversion Principle** 來消除 Volatile Dependencies 之間的緊耦合。

核心訊息可以用一句話概括：

> **把「誰來建立物件」的責任，從各個類別手中拿走，集中到應用程式的最頂層（Composition Root），就能讓每一層都只依賴抽象，而非具體實作。**

打個比方：原本 Mary 的程式碼就像家裡每個房間各自拉電線接電廠（緊耦合），重新設計後變成統一由配電箱（Composition Root）管理，每個房間只需要標準插座（介面），想換電器隨時換。

### 原版 vs. 重構版 — 依賴方向對照

```
【Mary 原版 — 緊耦合】              【重構版 — 鬆耦合】

  ┌──────────────┐                    ┌──────────────┐
  │  UI Layer    │                    │  UI Layer    │
  │ (HomeCtrl)   │                    │ (HomeCtrl)   │
  └──────┬───────┘                    └──────┬───────┘
         │ depends on                        │ depends on
         v                                   v
  ┌──────────────┐                    ┌──────────────┐
  │ Domain Layer │                    │ Domain Layer │  <-- 不依賴任何人！
  │ (ProductSvc) │                    │ (interfaces) │
  └──────┬───────┘                    └──────────────┘
         │ depends on                        ^
         v                                   │ depends on
  ┌──────────────┐                    ┌──────────────┐
  │ Data Access  │                    │ Data Access  │
  │  (EF Core)   │                    │ (SqlRepo)    │
  └──────────────┘                    └──────────────┘

  箭頭方向反轉了！                     依賴反轉 (DIP) 成功！
```

### 3.1 重建電商應用 (Rebuilding the E-commerce Application)

#### 3.1.1 建構更可維護的 UI 層

**核心概念**：從 UI 開始（Outside-In），引入 ViewModel 來簡化 View，並透過 Constructor Injection 宣告 HomeController 的依賴。

**Outside-In vs. Inside-Out 開發方式**

```
Inside-Out (Mary 的做法)          Outside-In (本章做法)
─────────────────────────         ─────────────────────────
1. Data Access Layer              1. UI Layer (View + Controller)
2. Domain Layer                   2. Domain Layer (Interfaces + Service)
3. UI Layer                       3. Data Access Layer (Repository)

特點：到最後才能跑程式              特點：一開始就能展示 stub 給利害關係人
```

**View Model 的引入**

原版 Mary 的 View 直接操作 `ViewData` 字典與 `Product` 實體，混雜了顯示邏輯。重構版引入了專屬的 ViewModel：

```csharp
// ViewModel — 純粹的 POCO，容易測試
public class FeaturedProductsViewModel
{
    public FeaturedProductsViewModel(
        IEnumerable<ProductViewModel> products)
    {
        this.Products = products;
    }

    public IEnumerable<ProductViewModel> Products { get; }
}

public class ProductViewModel
{
    private static CultureInfo PriceCulture =
        new CultureInfo("en-US");

    public ProductViewModel(string name, decimal unitPrice)
    {
        // 顯示邏輯封裝在 ViewModel 裡，而非 View 裡
        this.SummaryText = string.Format(
            PriceCulture, "{0} ({1:C})", name, unitPrice);
    }

    public string SummaryText { get; }
}
```

> **白話解釋 — POCO (Plain Old CLR Object)**
> 就是「不依賴任何框架的單純 C# 類別」。就像一個乾淨的白板，上面只寫你需要的東西，不需要繼承某個大框架的基底類別。

> **白話解釋 — DTO (Data Transfer Object)**
> 專門用來「搬運資料」的物件，像是快遞包裹——裡面只裝貨物（資料），包裹本身不會處理任何業務邏輯。

**HomeController 使用 Constructor Injection**

```csharp
public class HomeController : Controller
{
    private readonly IProductService productService;

    // Constructor Injection：明確宣告「我需要一個 IProductService 才能工作」
    public HomeController(IProductService productService)
    {
        // Guard Clause：防止傳入 null
        if (productService == null)
            throw new ArgumentNullException("productService");

        this.productService = productService;
    }

    public ViewResult Index()
    {
        IEnumerable<DiscountedProduct> products =
            this.productService.GetFeaturedProducts();

        var vm = new FeaturedProductsViewModel(
            from product in products
            select new ProductViewModel(product));

        return this.View(vm);
    }
}
```

> **白話解釋 — Constructor Injection（建構式注入）**
> 就像餐廳服務生上班前，必須先領到制服和名牌（依賴），沒有這些東西他根本「無法上工」。把依賴放在建構式參數裡，就是在說：「沒有給我這些東西，我連物件都建不出來。」

> **白話解釋 — Guard Clause（防衛語句）**
> 就像門口的保全——如果你沒帶證件（null），直接攔下來丟例外，絕不放你進門。這確保物件在建立的那一刻起就處於合法狀態。

**重要提醒：POCOs / DTOs / ViewModels 不需要介面**

```
                    需要介面嗎？
                   ┌───────────┐
                   │ 這個類別有 │
                   │ 行為需要   │───── 是 ───> 定義介面
                   │ 被替換、   │              (如 IProductService,
                   │ 攔截或     │               IProductRepository)
                   │ 模擬嗎？   │
                   └─────┬─────┘
                         │ 否
                         v
                   不需要介面
                   (如 Product, DiscountedProduct,
                    FeaturedProductsViewModel)
```

#### 3.1.2 建構獨立的領域模型

**核心概念**：領域層只包含 POCOs 和 Interfaces，不依賴任何外部模組。介面由領域層「擁有」（Dependency Inversion Principle）。

**IProductService — 領域層對 UI 層的契約**

```csharp
public interface IProductService
{
    IEnumerable<DiscountedProduct> GetFeaturedProducts();
}
```

**DiscountedProduct — 領域 POCO**

```csharp
public class DiscountedProduct
{
    public DiscountedProduct(string name, decimal unitPrice)
    {
        if (name == null)
            throw new ArgumentNullException("name");
        this.Name = name;
        this.UnitPrice = unitPrice;
    }

    public string Name { get; }
    public decimal UnitPrice { get; }
}
```

**IProductRepository — 領域層對資料存取層的契約**

```csharp
public interface IProductRepository
{
    IEnumerable<Product> GetFeaturedProducts();
}
```

> **白話解釋 — Repository Pattern（倉儲模式）**
> 把它想像成「資料的倉庫管理員」。業務邏輯跟管理員說「我要精選商品」，管理員去倉庫（資料庫）拿給你，業務邏輯不需要知道倉庫長什麼樣子。

**Product Entity — 含業務方法**

```csharp
public class Product
{
    public string Name { get; set; }
    public decimal UnitPrice { get; set; }
    public bool IsFeatured { get; set; }

    // Method Injection：IUserContext 作為方法參數傳入
    public DiscountedProduct ApplyDiscountFor(IUserContext user)
    {
        bool preferred =
            user.IsInRole(Role.PreferredCustomer);

        decimal discount = preferred ? .95m : 1.00m;

        return new DiscountedProduct(
            name: this.Name,
            unitPrice: this.UnitPrice * discount);
    }
}
```

> **白話解釋 — Entity（實體）**
> Entity 是有「身份證」的領域物件。即使電腦重開機，它在資料庫裡仍然存在、仍然可以被辨認（透過 ID）。就像一個人不管搬了幾次家，身份證號碼不會變。

> **白話解釋 — Method Injection（方法注入）**
> Constructor Injection 是「上班前必須帶好的裝備」，Method Injection 則是「執行某個任務時臨時借用的工具」。Entity 通常是短命物件，不適合在建構時注入所有依賴，所以在需要時才透過方法參數傳入。

**ProductService — 使用 Constructor Injection**

```csharp
public class ProductService : IProductService
{
    private readonly IProductRepository repository;
    private readonly IUserContext userContext;

    public ProductService(
        IProductRepository repository,
        IUserContext userContext)   // Constructor Injection
    {
        if (repository == null)
            throw new ArgumentNullException("repository");
        if (userContext == null)
            throw new ArgumentNullException("userContext");

        this.repository = repository;
        this.userContext = userContext;
    }

    public IEnumerable<DiscountedProduct> GetFeaturedProducts()
    {
        return
            from product in this.repository.GetFeaturedProducts()
            select product.ApplyDiscountFor(this.userContext);
            //     ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
            //     Method Injection：將 userContext 傳給 Entity
    }
}
```

**IUserContext — 將上下文資訊隱藏在抽象背後**

```csharp
public interface IUserContext
{
    bool IsInRole(Role role);
}

public enum Role { PreferredCustomer }
```

> **架構啟示：** 原本 Mary 的做法是讓 Controller 傳一個 `bool isPreferredCustomer` 給 ProductService，這會讓每個 Controller 都要負責判斷使用者角色，既重複又容易出錯。把使用者上下文藏在 `IUserContext` 背後，讓 ProductService 自己去取，Controller 完全不需要知道折扣的判斷邏輯。

**ProductService 的依賴關係圖**

```
┌─────────────────────────────────────────────────────────┐
│                    ProductService                        │
│                                                         │
│  ┌──────────────────┐       ┌──────────────────┐       │
│  │IProductRepository│       │   IUserContext    │       │
│  │─────────────────│       │─────────────────│       │
│  │GetFeaturedProducts()│    │ IsInRole(Role)   │       │
│  └────────┬─────────┘       └────────┬─────────┘       │
│           │ returns                   │ used by          │
│           v                           v                  │
│  ┌────────────────┐         ┌──────────────────┐       │
│  │    Product      │────────>│DiscountedProduct │       │
│  │────────────────│ creates │──────────────────│       │
│  │Name            │         │Name              │       │
│  │UnitPrice       │         │UnitPrice         │       │
│  │IsFeatured      │         └──────────────────┘       │
│  │ApplyDiscountFor()│                                    │
│  └────────────────┘                                     │
└─────────────────────────────────────────────────────────┘
```

#### 3.1.3 建構新的資料存取層

**核心概念**：資料存取層實作領域層定義的介面，Entity Framework 只是內部實作細節，外界完全不知道 EF 的存在。

```csharp
public class SqlProductRepository : IProductRepository
{
    private readonly CommerceContext context;

    public SqlProductRepository(CommerceContext context)
    {
        if (context == null)
            throw new ArgumentNullException("context");
        this.context = context;
    }

    public IEnumerable<Product> GetFeaturedProducts()
    {
        return
            from product in this.context.Products
            where product.IsFeatured
            select product;
    }
}
```

**改良的 CommerceContext — 顯式依賴連線字串**

```csharp
public class CommerceContext : DbContext
{
    private readonly string connectionString;

    // 明確宣告依賴 — 不再偷偷讀設定檔
    public CommerceContext(string connectionString)
    {
        if (string.IsNullOrWhiteSpace(connectionString))
            throw new ArgumentException(
                "connectionString should not be empty.",
                "connectionString");
        this.connectionString = connectionString;
    }

    public DbSet<Product> Products { get; set; }

    protected override void OnConfiguring(
        DbContextOptionsBuilder builder)
    {
        builder.UseSqlServer(this.connectionString);
    }
}
```

> **架構啟示：** Mary 的 CommerceContext 隱式地依賴 `ConfigurationManager` 來讀取連線字串，這是一個「隱藏的依賴」。重構版把連線字串作為建構式參數，讓依賴變得透明且可測試。

#### 3.1.4 實作 ASP.NET Core 專用的 IUserContext Adapter

**核心概念**：使用 Adapter Pattern 將框架特定的 API（`HttpContextAccessor`）轉換為應用專屬的抽象（`IUserContext`）。

```csharp
public class AspNetUserContextAdapter : IUserContext
{
    private static HttpContextAccessor Accessor =
        new HttpContextAccessor();

    public bool IsInRole(Role role)
    {
        return Accessor.HttpContext.User
            .IsInRole(role.ToString());
    }
}
```

**Adapter Pattern 結構圖**

```
┌──────────────────────────────────────────────────────────┐
│                    Adapter Pattern                        │
│                                                          │
│  Client          Target              Adaptee             │
│  (ProductSvc)    (IUserContext)       (HttpContextAccessor)│
│      │               ^                    ^               │
│      │ uses          │ implements         │ calls          │
│      v               │                    │               │
│  ┌─────────┐    ┌────┴──────────────┐    │               │
│  │ Client  │───>│ AspNetUserContext- │────┘               │
│  │         │    │ Adapter           │                     │
│  └─────────┘    │                   │                     │
│                 │ IsInRole(Role)    │                     │
│                 │   => Accessor     │                     │
│                 │     .HttpContext   │                     │
│                 │     .User         │                     │
│                 │     .IsInRole()   │                     │
│                 └───────────────────┘                     │
└──────────────────────────────────────────────────────────┘
```

> **白話解釋 — Adapter Pattern（轉接器模式）**
> 就像出國旅行帶的「萬用插頭轉接器」。你的筆電（Client）只認台灣插頭（IUserContext），但國外插座是不同規格（HttpContextAccessor）。Adapter 負責把介面轉換過來，讓你的筆電照常充電。

#### 3.1.5 在 Composition Root 組合應用程式

**核心概念**：所有物件的建立集中在 Composition Root，它位於應用程式入口點附近。

```csharp
// Composition Root — 整個應用的「配電箱」
// 所有物件圖 (Object Graph) 在此組裝
new HomeController(                          // UI 層
    new ProductService(                      // Domain 層
        new SqlProductRepository(            // Data Access 層
            new CommerceContext(
                connectionString)),
        new AspNetUserContextAdapter()));    // Adapter
```

**物件圖的建構流程**

```
                        Startup (應用啟動)
                            │
                            v
                   讀取 Configuration
                   取得 connectionString
                            │
                            v
              ┌─────────────────────────────┐
              │   Custom Controller Activator│ <── Composition Root
              │   (存放 connectionString)     │
              └─────────────┬───────────────┘
                            │
               Request 到達時呼叫 Create()
                            │
                            v
              new CommerceContext(connStr)
                            │
                            v
              new SqlProductRepository(ctx)
                            │
                            v                         new AspNetUserContextAdapter()
              new ProductService(repo, userCtx) <──────────────┘
                            │
                            v
              new HomeController(productService)
                            │
                            v
                   MVC 呼叫 Index()
                            │
                            v
              GetFeaturedProducts() 一路往下呼叫
                            │
                            v
              回傳 ViewResult + ViewModel
```

> **白話解釋 — Composition Root（組合根）**
> 就像一棟大樓的「總控室」。所有的管線（依賴）在這裡接好，每個房間（類別）只需要知道「插座長什麼樣」（介面），不需要知道電從哪來。Composition Root 盡可能靠近應用程式的進入點（如 `Main` 方法或 `Startup` 類別）。

> **白話解釋 — Object Graph（物件圖）**
> 物件之間透過引用形成的網路結構。就像家族族譜——HomeController 包含 ProductService，ProductService 又包含 SqlProductRepository 和 AspNetUserContextAdapter，層層嵌套形成一棵「物件樹」。

> **白話解釋 — Pure DI**
> 不使用任何 DI Container（如 Autofac、Ninject），純粹用程式碼手動 `new` 出物件圖。好處是一切在編譯期就能檢查，壞處是物件圖複雜時維護成本較高。

### 3.2 分析鬆耦合實作 (Analyzing the Loosely Coupled Implementation)

#### 3.2.1 理解元件之間的互動

完整的請求處理流程：

```
  Web Server
      │
      │ Request arrives
      v
  Controller Activator ─── Create() ──> 建立整個 Object Graph
      │
      v
  HomeController.Index()
      │
      │ 呼叫 productService.GetFeaturedProducts()
      v
  ProductService.GetFeaturedProducts()
      │
      │ 1. 呼叫 repository.GetFeaturedProducts()  -> 取得 Product 列表
      │ 2. 對每個 Product 呼叫 ApplyDiscountFor(userContext) -> Method Injection
      │ 3. 回傳 DiscountedProduct 列表
      v
  HomeController
      │
      │ 將 DiscountedProduct 轉為 ProductViewModel
      │ 包裝進 FeaturedProductsViewModel
      v
  回傳 ViewResult ──> MVC Render View
```

#### 3.2.2 分析新的依賴圖

```
┌────────────────────────────────────────────────────────────┐
│                      依賴圖 (Dependency Graph)              │
│                                                            │
│  ┌─ User Interface Layer ────────────────────────────┐    │
│  │                                                    │    │
│  │  HomeController ──────> IProductService            │    │
│  │  FeaturedProductsViewModel                        │    │
│  │  ProductViewModel                                 │    │
│  │  AspNetUserContextAdapter ──┐                     │    │
│  └────────────────────────────│─────────────────────┘    │
│                               │                           │
│                               │ implements                │
│                               v                           │
│  ┌─ Domain Layer (零依賴！) ─────────────────────────┐    │
│  │                                                    │    │
│  │  IProductService    IProductRepository             │    │
│  │  IUserContext        Product                       │    │
│  │  ProductService      DiscountedProduct             │    │
│  └────────────────────────────────────────────────────┘    │
│                               ^                           │
│                               │ depends on                │
│  ┌─ Data Access Layer ───────│────────────────────────┐  │
│  │                            │                        │  │
│  │  SqlProductRepository ─────┘                       │  │
│  │  CommerceContext                                    │  │
│  └────────────────────────────────────────────────────┘  │
│                                                            │
│  關鍵發現：Domain Layer 不依賴任何人！                       │
│  Data Access Layer 反過來依賴 Domain Layer！                │
└────────────────────────────────────────────────────────────┘
```

**可替換性評估**

| 問題 | Mary 的版本 | 重構版 |
|------|-----------|--------|
| 能否把 Web UI 換成 WPF？ | 可以（原本就可以） | 可以 |
| 能否把 SQL Server 換成 Azure Table？ | 不行（Domain 依賴 Data Access） | **可以！** 只要提供新的 `IProductRepository` 實作 |
| 能否獨立測試 Domain？ | 困難（依賴具體 Data Access） | **容易！** Domain 無外部依賴 |
| 能否並行開發？ | 困難 | **容易！** 各層只依賴介面 |

### 3.2.3 關於 DI Container 的說明

本章刻意只使用 **Pure DI**（手動建構物件圖），不使用 DI Container。作者的立場是：

```
DI Container 的角色定位：

  ┌─────────────────────────────────────────────────────┐
  │  DI Container 是有用的但非必要的工具                   │
  │                                                     │
  │  它能做的：讓 Composition Root 更好維護                │
  │  它不能做的：把緊耦合的程式碼變成鬆耦合                  │
  │                                                     │
  │  正確順序：                                           │
  │  1. 先學會 DI 的原則與模式（本書 Part 2 & 3）          │
  │  2. 再決定是否使用 DI Container（本書 Part 4）         │
  └─────────────────────────────────────────────────────┘
```

## 4. 關鍵框架與公式 (Key Frameworks)

### 框架一：Outside-In 開發流程

```
Step 1: UI Layer
   - 定義 View + ViewModel
   - Controller 用 Constructor Injection 宣告依賴
   - 可以先 hard-code 回傳 stub 資料讓利害關係人看到雛形
              │
              v
Step 2: Domain Layer
   - 定義介面（IProductService, IProductRepository, IUserContext）
   - 實作 ProductService（依賴介面，不依賴具體類別）
   - Entity 中的業務方法使用 Method Injection
              │
              v
Step 3: Data Access Layer
   - 實作 IProductRepository（如 SqlProductRepository）
   - EF Core 只是內部實作細節
              │
              v
Step 4: Composition Root
   - 在入口點附近組裝整個物件圖
   - 決定用 Pure DI 或 DI Container
```

### 框架二：Dependency Inversion Principle (DIP) 實踐法則

```
原則一：高層模組不應依賴低層模組，兩者都應依賴抽象

  ✗ 錯誤                         ✓ 正確
  ProductService                  ProductService
       │                               │
       │ depends on                    │ depends on
       v                               v
  SqlProductRepository            IProductRepository  <-- 抽象
                                       ^
                                       │ implements
                                  SqlProductRepository

原則二：抽象應由使用它的模組「擁有」

  IProductRepository 定義在 Domain Layer（使用者）
  而非 Data Access Layer（實作者）

  這就像：插座標準由住戶（使用者）決定，
  而非電器廠商（實作者）決定
```

### 框架三：何時用介面、何時不用

```
┌─────────────────────────────────┬─────────────────────┐
│         需要介面                 │      不需要介面       │
├─────────────────────────────────┼─────────────────────┤
│ Services (ProductService)       │ POCOs (Product)      │
│ Repositories (IProductRepo)     │ DTOs                 │
│ 上下文提供者 (IUserContext)       │ ViewModels           │
│ 任何 Volatile Dependency        │ Entities (通常)       │
│ 需要被模擬/替換/攔截的類別        │ 純資料物件            │
└─────────────────────────────────┴─────────────────────┘

判斷標準：這個類別有沒有「行為」需要被替換、攔截或模擬？
有 => 定義介面       沒有 => 直接使用具體類別
```

### 框架四：Constructor Injection 實作公式

```csharp
public class 消費者類別
{
    private readonly I依賴介面 _dependency;  // 1. 宣告私有唯讀欄位

    public 消費者類別(I依賴介面 dependency)     // 2. 建構式參數
    {
        if (dependency == null)              // 3. Guard Clause
            throw new ArgumentNullException(nameof(dependency));

        _dependency = dependency;            // 4. 儲存依賴
    }

    public void 某個方法()
    {
        _dependency.做某件事();                // 5. 使用依賴
    }
}
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點一：Dependency Inversion Principle 中「擁有」抽象的含義

**為什麼難？** 很多人以為 DIP 只是「程式寫介面」，但忽略了「誰擁有介面」這個關鍵。

**深入解釋：**

DIP 的第二個要點：**抽象應由使用它的模組擁有，而非實作它的模組。**

```
誰「擁有」介面？

  ┌─ Domain Layer ──────────────────────┐
  │                                     │
  │  ProductService (使用者)             │
  │       │                             │
  │       │ uses                        │
  │       v                             │
  │  IProductRepository (定義在這裡！)    │  <── Domain Layer 擁有這個介面
  │                                     │
  └─────────────────────────────────────┘
           ^
           │ implements
  ┌─ Data Access Layer ─────────────────┐
  │                                     │
  │  SqlProductRepository               │  <── 只是實作者，不擁有介面
  │                                     │
  └─────────────────────────────────────┘
```

**比喻：** 就像一間公司的「職位描述」（介面）應該由用人的部門（高層模組）定義，而非由應徵者（低層模組）定義。用人部門最清楚自己需要什麼。

**本章中的 DIP 違反案例：**

```
IProductService 由 Domain Layer 定義，
但它被 UI Layer 的 HomeController 使用。

嚴格來說 IProductService 應該由 UI Layer 擁有，
但作者基於務實考量選擇不修正：

修正方式：建立額外兩個專案
  - 一個純粹的 UI Layer 專案（不含 Composition Root）
  - 一個 IProductService 定義專案（由 UI Layer 擁有）

代價：增加了方案的複雜度
結論：工程上的務實判斷，不是每個 DIP 違反都值得修正
```

### 難點二：Method Injection 與 Constructor Injection 的使用時機區別

**為什麼難？** 初學者常搞不清楚什麼時候該用建構式注入、什麼時候該用方法注入。

**判斷公式：**

```
該類別是長生命週期物件嗎？（Service、Controller、Repository...）
  ├── 是 ──> 使用 Constructor Injection
  │         依賴在物件建立時注入，整個生命週期都可用
  │
  └── 否（短生命週期物件，如 Entity、Value Object）
        └──> 使用 Method Injection
              依賴只在需要時透過方法參數傳入
```

**本章的實際例子：**

```csharp
// Constructor Injection — ProductService 是長命物件
public class ProductService : IProductService
{
    private readonly IProductRepository repository;   // 建構時注入
    private readonly IUserContext userContext;          // 建構時注入
    // ...
}

// Method Injection — Product 是短命的 Entity
public class Product
{
    // 不在建構式注入 IUserContext，而是在方法呼叫時傳入
    public DiscountedProduct ApplyDiscountFor(IUserContext user)
    {
        // ...
    }
}
```

**為什麼 Product 不用 Constructor Injection？**

```
Product 的生命週期：

  1. 從資料庫查出來（由 EF Core 建立）  <── EF Core 負責建構
  2. 呼叫 ApplyDiscountFor()           <── 使用 IUserContext
  3. 轉換為 DiscountedProduct 後丟棄    <── 生命結束

如果用 Constructor Injection：
  - EF Core 不知道怎麼注入 IUserContext
  - Product 的建構式會變得不必要地複雜
  - 違反 Entity 的「持久性無知」原則

所以用 Method Injection 更合適：
  需要時才傳入，用完就算了。
```

### 難點三：Composition Root 的定位與職責邊界

**為什麼難？** Composition Root 常被誤解為「只是 DI Container 的設定檔」，或與 UI Layer 混為一談。

**關鍵澄清：**

```
┌─────────────────────────────────────────────────────────┐
│  常見誤解                        正確理解                 │
├─────────────────────────────────────────────────────────┤
│  CR 是 UI Layer 的一部分         CR 雖然可能放在同一個     │
│                                 Assembly，但概念上是獨立  │
│                                 的橫切關注點              │
│                                                         │
│  CR 必須使用 DI Container       CR 可以用 Pure DI        │
│                                 （手動 new）也可以用      │
│                                 DI Container             │
│                                                         │
│  CR 散落在各處                   CR 必須集中在一個地方，   │
│                                 盡量靠近入口點             │
│                                                         │
│  CR 負責業務邏輯                 CR 只負責「組裝」，       │
│                                 不包含任何業務邏輯         │
└─────────────────────────────────────────────────────────┘
```

**Composition Root 在不同應用類型中的位置**

```
ASP.NET Core MVC:
  Startup.cs / Program.cs 的 ConfigureServices 方法
  或自訂的 IControllerActivator

Console App:
  Main 方法

WPF:
  App.xaml.cs 的 OnStartup 方法

Windows Service:
  ServiceBase 的子類別建構式

共同點：都在應用程式的「入口」附近
```

**物件建構的責任鏈**

```
❌ 錯誤做法：每個類別自己 new 依賴

  HomeController
    └── new ProductService(
            new SqlProductRepository(
                new CommerceContext("...")),  // Controller 知道太多了！
            new AspNetUserContextAdapter())

✅ 正確做法：Composition Root 統一建構

  Composition Root（入口點附近）
    └── 建構完整的 Object Graph
        └── 交給框架去使用

  HomeController 只知道 IProductService，
  不知道 ProductService 的存在，
  更不知道 SqlProductRepository 和 CommerceContext。
```

### 從 Mary 的 4 個類別到 9 + 3 — 值得嗎？

```
投入：更多的類別和介面（9 classes + 3 interfaces vs. 4 classes）

回報：
  ✅ 領域層零依賴 — 可獨立測試、獨立部署
  ✅ 資料存取層可替換 — 換 DB 不改業務邏輯
  ✅ UI 可替換 — Web 換 WPF 不動核心
  ✅ 並行開發 — 團隊成員各做各的層
  ✅ 可攔截 — 加 logging、caching 不改原始碼
  ✅ Outside-In 更快得到回饋 — 利害關係人早期就能看到東西

結論：鬆耦合的「額外成本」在真實專案中很快就會回本。
     系統越複雜，DI 的價值越明顯。
```

### 架構師的實戰建議

1. **介面定義權歸使用者**：設計 Repository 介面時，讓 Domain Layer 決定它需要什麼，而非讓 Data Access Layer 決定它能提供什麼。

2. **隱藏上下文資訊**：像 `IUserContext` 這樣的設計，避免讓 Controller 傳遞使用者角色資訊，減少出錯的機會。

3. **務實勝過完美**：本章示範了一個 DIP 違反（`IProductService` 的歸屬），但作者選擇不修正，因為修正的成本大於收益。架構設計中，trade-off 無處不在。

4. **ViewModel 是你的朋友**：把顯示邏輯從 View 移到 ViewModel，View 越笨越好。Mary 的貨幣格式化 bug（丹麥使用者看到錯誤幣值）就是 View 太聰明的後果。

5. **Entity Framework 只是實作細節**：不要讓 EF 的 `DbContext` 洩漏到 Domain Layer。透過 Repository 模式隔離，未來換 ORM 或換資料庫都不影響核心業務。
