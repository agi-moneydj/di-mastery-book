## 1. 概述 (Overview)

本章是全書最具架構洞察力的章節之一。作者以專業廚房的「Mise en place」（備料就緒）為比喻開場：一個高效的廚房需要的不只是好食材，更需要精心規劃的動線與專用工位，軟體開發也是如此。程式碼庫就是我們的廚房，而 **Aspect-Oriented Programming (AOP)** 就是讓這個廚房高效運作的關鍵佈局策略。

本章的核心論點：**AOP 不需要特殊工具，透過 SOLID 原則驅動的軟體設計本身就能實現 AOP**。作者從一個逐漸膨脹的 `IProductService` 介面出發，一步步透過 SOLID 原則重構，最終得到以 `ICommandService<TCommand>` 為核心的泛型訊息傳遞架構，讓橫切關注點（Cross-Cutting Concerns）可以用單一 Decorator 優雅地套用到所有業務操作上。

> **白話說明**：想像你在一家餐廳工作。如果每道菜都要個別記錄「誰點的、什麼時候做的、花了多少錢」，那每個廚師都要重複寫同樣的紀錄——這就是橫切關注點（Cross-Cutting Concerns）帶來的重複問題。本章教你如何設計一套統一的「出菜系統」，讓所有菜品自動通過同一套流程（安全檢查、交易記錄、稽核記錄），而不需要每道菜都寫一遍。

## 2. 學習目標 (Learning Objectives)

完成本章學習後，你應該能夠：

1. **理解 AOP 的本質**：不是工具問題，而是設計問題。認識三種 AOP 方法（SOLID 設計、動態攔截、編譯時織入）的優劣。
2. **融會貫通 SOLID 五大原則**：SRP、OCP、LSP、ISP、DIP 如何各司其職又相互配合，共同驅動可維護的架構。
3. **掌握 CQS（命令查詢分離）**：理解為何將讀寫操作分開是重構的第一步。
4. **運用 Parameter Object 模式**：將方法參數提取為獨立的命令物件，使其成為使用案例的定義。
5. **設計泛型抽象 `ICommandService<TCommand>`**：用單一泛型介面統一所有命令操作，實現一個 Decorator 解決一個橫切關注點。
6. **實作通用 Decorator**：包含 Transaction、Auditing、Security 等橫切關注點的泛型 Decorator。
7. **使用被動屬性（Passive Attributes）**：以中繼資料驅動行為，而非在屬性中嵌入行為。
8. **從 RPC 風格轉向訊息傳遞模型**：理解為什麼訊息傳遞（message-passing）比方法呼叫（method-calling）更利於維護與擴展。

## 3. 章節大綱 (Outline)

### 3.1 AOP 導論 (10.1)

**AOP 的定義**：AOP 旨在減少實作橫切關注點所需的樣板程式碼。它在單一位置實作這些模式，然後以宣告式或基於慣例的方式套用到程式碼庫，而不修改程式碼本身。

**三種 AOP 方法比較**：

| 方法 | 優點 | 缺點 |
|------|------|------|
| **SOLID 設計** | 不需任何工具；容易實作；專注設計；系統更易維護 | 對舊系統不易套用 |
| **動態攔截 (Dynamic Interception)** | 對既有系統容易加入；良好的免費工具 | 強耦合於 AOP 工具；失去編譯時支援 |
| **編譯時織入 (Compile-time Weaving)** | 對既有系統容易加入 | 造成 Temporal Coupling；織入後無法脫離；是 DI 的反面 |

**DRY 原則的重複問題**：以 `CircuitBreakerProductRepositoryDecorator` 為例，`Delete` 和 `Insert` 方法中只有一行呼叫不同，其餘完全相同——這就是 AOP 要解決的重複問題。

```csharp
// 重複的 Circuit Breaker 樣板程式碼
public void Delete(Product product)
{
    this.breaker.Guard();
    try
    {
        this.decoratee.Delete(product);  // 唯一的差異
        this.breaker.Succeed();
    }
    catch (Exception ex)
    {
        this.breaker.Trip(ex);
        throw;
    }
}

public void Insert(Product product)
{
    this.breaker.Guard();
    try
    {
        this.decoratee.Insert(product);  // 唯一的差異
        this.breaker.Succeed();
    }
    catch (Exception ex)
    {
        this.breaker.Trip(ex);
        throw;
    }
}
```

> **白話說明**：想像你有 20 道菜，每道菜出餐流程都一樣（檢查品質 -> 擺盤 -> 記錄），唯一不同的只是「哪道菜」。你不會寫 20 份出餐 SOP，你會寫 1 份通用 SOP，然後「套用」到每道菜上。

### 3.2 SOLID 原則回顧 (10.2)

| 原則 | 核心概念 | 比喻 |
|------|----------|------|
| **SRP** (單一職責) | 每個類別只有一個改變的理由 | 一位廚師只負責一個料理站（冷盤站、烤爐站），不該什麼都做 |
| **OCP** (開放封閉) | 對擴展開放，對修改封閉 | 新菜品上架不應該改動原有的點餐系統，只需「新增」一份菜單 |
| **LSP** (里氏替換) | 任何子型別都能替換父型別而不破壞行為 | 任何合格廚師都能按照食譜做出同樣的菜，不管是誰在該工位 |
| **ISP** (介面隔離) | 使用細粒度的介面，消費者不應被迫依賴不使用的方法 | 點甜點的客人不需要看到整本菜單，只需要甜點頁 |
| **DIP** (依賴反轉) | 程式應依賴抽象，且消費端決定抽象的形狀 | 廚師定義「我需要什麼食材」，而不是被倉庫決定「我能給你什麼」 |

### 3.3 SOLID 驅動 AOP 的核心重構 (10.3) — 本章精華

這是全章最重要的部分。作者以 `IProductService` 介面為例，展示了一段逐步重構的旅程。

#### 起點：膨脹的 IProductService

```csharp
public interface IProductService
{
    IEnumerable<DiscountedProduct> GetFeaturedProducts();
    void DeleteProduct(Guid productId);
    Product GetProductById(Guid productId);
    void InsertProduct(Product product);
    void UpdateProduct(Product product);
    Paged<Product> SearchProducts(
        int pageIndex, int pageSize,
        Guid? manufacturerId, string searchText);
    void UpdateProductReviewTotals(
        Guid productId, ProductReview[] reviews);
    void AdjustInventory(
        Guid productId, bool decrease, int quantity);
    void UpdateHasTierPricesProperty(Product product);
    void UpdateHasDiscountsApplied(
        Guid productId, string discountDescription);
}
```

**SOLID 違規分析**：

```
+---------------------------+-----------------------------------------------+
|  違反的原則                |  問題描述                                      |
+---------------------------+-----------------------------------------------+
|  ISP (介面隔離)           |  消費者被迫依賴不使用的方法。                    |
|                           |  例如 HomeController 只用 GetFeaturedProducts  |
|                           |  卻要面對 10 個方法的介面。                     |
+---------------------------+-----------------------------------------------+
|  SRP (單一職責)           |  ProductService 有多個改變的理由：              |
|                           |  折扣變更、庫存調整、搜尋條件變更...            |
+---------------------------+-----------------------------------------------+
|  OCP (開放封閉)           |  新增功能導致 sweeping changes：               |
|                           |  介面改變 -> 所有實作、Decorator、              |
|                           |  Test Double 都得跟著改。                      |
+---------------------------+-----------------------------------------------+
```

#### Step 1：分離讀寫 (CQS)

```
      IProductService                    IProductQueryServices
  +---------------------+              +------------------------+
  | GetFeaturedProducts |   拆分為     | GetFeaturedProducts    |
  | GetProductById      |  ========>   | GetProductById         |
  | SearchProducts      |              | SearchProducts         |
  | DeleteProduct       |              +------------------------+
  | InsertProduct       |
  | UpdateProduct       |              IProductCommandServices
  | AdjustInventory     |              +------------------------+
  | UpdateReviewTotals  |              | DeleteProduct          |
  | UpdateTierPrices    |              | InsertProduct          |
  | UpdateDiscounts     |              | UpdateProduct          |
  +---------------------+              | AdjustInventory        |
                                       | UpdateReviewTotals     |
                                       | UpdateTierPrices       |
                                       | UpdateDiscounts        |
                                       +------------------------+
```

> **CQS 白話說明**：Query（查詢）只看不動，Command（命令）只動不看。像圖書館：「查詢藏書」是 Query，「借出歸還」是 Command，兩者的處理流程（如是否需要交易、稽核）完全不同，放在一起只會互相干擾。

#### Step 2：拆分為一個方法一個介面 (ISP + SRP)

```
 IProductCommandServices              7 個獨立的介面 + 7 個實作
+------------------------+
| DeleteProduct          |           IAdjustInventoryService
| InsertProduct          |           +---------------------------+
| UpdateProduct          |  拆分為   | AdjustInventory(...)      |
| AdjustInventory        | ======>   +---------------------------+
| UpdateReviewTotals     |
| UpdateTierPrices       |           IUpdateProductReviewTotalsService
| UpdateDiscounts        |           +-------------------------------+
+------------------------+           | UpdateProductReviewTotals(...)  |
                                     +-------------------------------+
                                     ... (其餘 5 個介面)
```

```csharp
public interface IAdjustInventoryService
{
    void AdjustInventory(Guid productId, bool decrease, int quantity);
}

public interface IUpdateProductReviewTotalsService
{
    void UpdateProductReviewTotals(Guid productId, ProductReview[] reviews);
}
```

**優點**：每個類別只有一個職責，新增功能只需新增類別。
**問題**：7 個介面 x 10 種橫切關注點 = 70 個 Decorator！

#### Step 3：Parameter Object + 統一介面 (OCP)

```csharp
// 將方法參數提取為 Parameter Object（命令物件）
public class AdjustInventory  // 這就是「命令」本身
{
    public Guid ProductId { get; set; }
    public bool Decrease { get; set; }
    public int Quantity { get; set; }
}

// 統一介面：所有命令服務都實作 ICommandService
public interface ICommandService
{
    void Execute(object command);  // 一個介面統治一切
}
```

```
  ICommandService (非泛型版)

  +-------------------------------+        +-----------------------+
  | AdjustInventoryService        | -----> | AdjustInventory       |
  +-------------------------------+        +-----------------------+
  | DeleteProductService          | -----> | DeleteProduct         |
  +-------------------------------+        +-----------------------+
  | InsertProductService          | -----> | InsertProduct         |
  +-------------------------------+        +-----------------------+
  | UpdateHasDiscountAppliedSvc   | -----> | UpdateHasDiscount...  |
  +-------------------------------+        +-----------------------+
  | UpdateHasTierPricesSvc        | -----> | UpdateHasTierPrices   |
  +-------------------------------+        +-----------------------+
  | UpdateProductService          | -----> | UpdateProduct         |
  +-------------------------------+        +-----------------------+
  | UpdateProductReviewTotalsSvc  | -----> | UpdateProductReview...|
  +-------------------------------+        +-----------------------+

  全部實作同一個 ICommandService 介面
  每個 Service cast 其 object 參數為對應的 Parameter Object
```

> **白話說明**：Parameter Object 就像快遞的「包裹」。不管裡面裝什麼（書、衣服、電子產品），快遞系統只看到「一個包裹」。統一介面就是快遞公司的標準流程：收件、分揀、運送、簽收——對每個包裹都一樣。

**問題**：`object` 參數導致 LSP 違規！

```csharp
// LSP 違規範例
public class AdjustInventoryService : ICommandService
{
    public void Execute(object cmd)
    {
        var command = (AdjustInventory)cmd;  // 如果傳入錯誤型別就崩潰！
        // ...
    }
}
```

#### Step 4：泛型抽象修復 LSP

```csharp
// 最終版：泛型 ICommandService<TCommand>
public interface ICommandService<TCommand>
{
    void Execute(TCommand command);
}

// 實作時明確指定處理的命令型別
public class AdjustInventoryService
    : ICommandService<AdjustInventory>
{
    private readonly IInventoryRepository repository;

    public AdjustInventoryService(IInventoryRepository repository)
    {
        this.repository = repository;
    }

    public void Execute(AdjustInventory command)
    {
        // 不需要 cast！型別安全！
        var productId = command.ProductId;
        // ...
    }
}
```

**消費端也變得型別安全**：

**完整重構旅程的架構圖**：

```
  ===================================================================
  重構前：1 個肥大介面
  ===================================================================

  IProductService (10 個方法)
         |
         v
  ProductService (10 個方法的實作)
         |
  若要加橫切關注點 => 每個 Decorator 都得實作 10 個方法

  ===================================================================
  重構後：泛型訊息傳遞架構
  ===================================================================

                    ICommandService<TCommand>
                           |
            +--------------+--------------+
            |              |              |
   ICommandService   ICommandService   ICommandService
   <AdjustInventory> <DeleteProduct>   <UpdateProduct>
            |              |              |
   AdjustInventory-  DeleteProduct-   UpdateProduct-
   Service           Service          Service

  加一個橫切關注點 => 只寫 1 個泛型 Decorator
  加一個新功能     => 只加 1 個 Command + 1 個 Service
  ===================================================================
```

#### 橫切關注點的 Decorator 實作

**交易管理 (Transaction)**：

```csharp
public class TransactionCommandServiceDecorator<TCommand>
    : ICommandService<TCommand>
{
    private readonly ICommandService<TCommand> decoratee;

    public TransactionCommandServiceDecorator(
        ICommandService<TCommand> decoratee)
    {
        this.decoratee = decoratee;
    }

    public void Execute(TCommand command)
    {
        using (var scope = new TransactionScope())
        {
            this.decoratee.Execute(command);
            scope.Complete();
        }
    }
}
```

**稽核 (Auditing)**：

```csharp
public class AuditingCommandServiceDecorator<TCommand>
    : ICommandService<TCommand>
{
    private readonly IUserContext userContext;
    private readonly ITimeProvider timeProvider;
    private readonly CommerceContext context;
    private readonly ICommandService<TCommand> decoratee;

    public AuditingCommandServiceDecorator(
        IUserContext userContext,
        ITimeProvider timeProvider,
        CommerceContext context,
        ICommandService<TCommand> decoratee)
    {
        this.userContext = userContext;
        this.timeProvider = timeProvider;
        this.context = context;
        this.decoratee = decoratee;
    }

    public void Execute(TCommand command)
    {
        this.decoratee.Execute(command);
        this.AppendToAuditTrail(command);
    }

    private void AppendToAuditTrail(TCommand command)
    {
        var entry = new AuditEntry
        {
            UserId = this.userContext.CurrentUser.Id,
            TimeOfExecution = this.timeProvider.Now,
            Operation = command.GetType().Name,       // 用反射取得命令名稱
            Data = JsonConvert.SerializeObject(command) // 序列化命令資料
        };

        this.context.AuditEntries.Add(entry);
        this.context.SaveChanges();
    }
}
```

> **關鍵洞察**：因為命令物件包含了所有使用案例的資料，稽核 Decorator 可以自動擷取「做了什麼操作」和「帶了什麼資料」。這是從 RPC 風格轉向訊息傳遞模型最大的紅利之一。

**安全 (Security) — 使用被動屬性**：

```csharp
// 被動屬性：只提供中繼資料，不包含行為
[AttributeUsage(AttributeTargets.Class)]
public class PermittedRoleAttribute : Attribute
{
    public readonly Role Role;
    public PermittedRoleAttribute(Role role)
    {
        this.Role = role;
    }
}

// 在命令物件上標記允許的角色
[PermittedRole(Role.InventoryManager)]
public class AdjustInventory
{
    public Guid ProductId { get; set; }
    public bool Decrease { get; set; }
    public int Quantity { get; set; }
}

[PermittedRole(Role.Administrator)]
public class UpdateProductReviewTotals
{
    public Guid ProductId { get; set; }
    public ProductReview[] Reviews { get; set; }
}
```

```csharp
public class SecureCommandServiceDecorator<TCommand>
    : ICommandService<TCommand>
{
    private static readonly Role PermittedRole = GetPermittedRole();
    private readonly IUserContext userContext;
    private readonly ICommandService<TCommand> decoratee;

    public SecureCommandServiceDecorator(
        IUserContext userContext,
        ICommandService<TCommand> decoratee)
    {
        this.decoratee = decoratee;
        this.userContext = userContext;
    }

    public void Execute(TCommand command)
    {
        this.CheckAuthorization();
        this.decoratee.Execute(command);
    }

    private void CheckAuthorization()
    {
        if (!this.userContext.IsInRole(PermittedRole))
            throw new SecurityException();
    }

    private static Role GetPermittedRole()
    {
        var attribute = typeof(TCommand)
            .GetCustomAttribute<PermittedRoleAttribute>();

        if (attribute == null)
            throw new InvalidOperationException(
                "[PermittedRole] missing.");

        return attribute.Role;
    }
}
```

> **被動屬性 vs 行為屬性**：被動屬性只是「標籤」，不帶任何行為邏輯。就像商品上的「易碎品」標籤——標籤不會自己保護商品，但快遞員看到標籤後會小心搬運。行為和標籤分離，各自可以獨立變化。

**提取可重用的 Decorate 方法消除重複**：

```csharp
private ICommandService<TCommand> Decorate<TCommand>(
    ICommandService<TCommand> decoratee, CommerceContext context)
{
    return
        new SecureCommandServiceDecorator<TCommand>(
            this.userContext,
            new TransactionCommandServiceDecorator<TCommand>(
                new AuditingCommandServiceDecorator<TCommand>(
                    this.userContext,
                    this.timeProvider,
                    context,
                    decoratee)));
}

// 使用時只需一行
var service = Decorate(new AdjustInventoryService(repository), context);
return new InventoryController(service);
```

### 3.4 橫切關注點的最佳實踐 (10.3.4)

作者強調橫切關注點應該套用在**正確的粒度層級**。對多數非純 CRUD 的應用程式而言，Repository 層級粒度太細。一個業務操作可能呼叫多個 Repository，若在 Repository 層級加 Transaction，就會產生多個交易，危害資料正確性。

**適合在 `ICommandService<TCommand>` 層級套用的橫切關注點**：

| 關注點 | 說明 |
|--------|------|
| **稽核 (Auditing)** | 記錄命令名稱、資料、使用者、時間，而非個別 Entity 變更 |
| **日誌 (Logging)** | 單一 Decorator 記錄所有業務操作，減少散佈各處的 log 語句 |
| **效能監控 (Performance)** | 99% 時間花在業務操作，此處是理想的監控邊界 |
| **安全 (Security)** | 在業務操作層級限制存取，而非在 Repository 層級 |
| **容錯 (Fault Tolerance)** | 資料庫死鎖重試等機制應包裹在交易之外 |
| **驗證 (Validation)** | 命令物件可搭配 Data Annotations 進行驗證 |

**關於 Logging 的架構智慧**：

良好設計的 SOLID 應用程式只需要極少的手動日誌語句。作者提出三個實踐：
1. 遇到意外情況拋出例外，而非記錄後繼續執行。
2. 不要在操作中途 catch 例外、記錄、繼續——讓例外冒泡到頂層統一記錄。
3. 讓方法保持小巧，stack trace 本身就會提供足夠的偵錯資訊。

### 3.5 結論 (10.3.5)

本章的重構從 **RPC 風格的方法呼叫模型**轉向了**訊息傳遞模型**。命令物件可以被序列化、排入佇列、記錄、重播——這些能力在傳統的方法呼叫模型中難以實現。

```
  RPC 風格 (方法呼叫模型)           訊息傳遞模型
  ─────────────────────            ─────────────────────
  service.AdjustInventory(         service.Execute(
    productId,                         new AdjustInventory {
    decrease,                              ProductId = ...,
    quantity);                             Decrease = ...,
                                           Quantity = ...
                                       });

  - 參數散落，難以序列化            - 命令物件可序列化
  - 難以統一處理                    - 可排入佇列
  - 每個方法簽章不同                - 統一的 Execute 介面
  - 橫切關注點要逐方法套用          - 一個 Decorator 套用所有
```

## 4. 關鍵框架與公式 (Key Frameworks)

### 框架一：SOLID 違規診斷矩陣

當你看到一個「大介面」時，使用以下矩陣檢查：

```
+----------+---------------------------+------------------------------------+
| 原則     | 診斷問題                   | 症狀                                |
+----------+---------------------------+------------------------------------+
| ISP      | 消費者是否使用所有方法？    | 測試時需要 Mock 不使用的方法         |
| SRP      | 類別有幾個改變的理由？      | 方法間缺乏內聚性，容易拆分          |
| OCP      | 新增功能是否導致連鎖修改？  | 新增一個方法 => 修改介面 +            |
|          |                            | 所有實作 + 所有 Decorator + Test Double|
| LSP      | 替換實作是否破壞消費端？    | 需要 cast、型別檢查                  |
| DIP      | 介面形狀由誰決定？         | 為了特定實作而新增介面成員            |
+----------+---------------------------+------------------------------------+
```

### 框架二：四步重構公式

```
Step 1: CQS 分離
    IXxxService  -->  IXxxQueryService + IXxxCommandService

Step 2: ISP + SRP 拆分
    IXxxCommandService (N methods)  -->  N 個單一方法介面 + N 個實作

Step 3: Parameter Object + 統一介面
    N 個不同介面  -->  1 個 ICommandService + N 個 Parameter Object

Step 4: 泛型修復 LSP
    ICommandService (object)  -->  ICommandService<TCommand>
```

### 框架三：泛型 Decorator 組合公式

```
新增功能     = 1 個 Command class + 1 個 Service class（零修改既有程式碼）
新增橫切關注點 = 1 個泛型 Decorator class（套用到所有 Command）
修改橫切關注點 = 修改 1 個 Decorator class
```

### 框架四：判斷橫切關注點的正確粒度層級

```
┌──────────────────────────┐
│   Controller / API 層    │  <-- 認證、速率限制
├──────────────────────────┤
│   業務操作層             │  <-- 授權、交易、稽核、驗證、
│   (ICommandService<T>)   │      日誌、效能監控、容錯   <== 大多數橫切關注點的最佳位置
├──────────────────────────┤
│   Repository 層          │  <-- 快取（查詢）、連線管理
├──────────────────────────┤
│   基礎設施層             │  <-- 重試（HTTP 呼叫）
└──────────────────────────┘
```

### 框架五：被動屬性設計原則

```
  錯誤做法：行為屬性（Aspect Attribute）
  +----------------------------------+
  | [TransactionAttribute]           |  <-- 屬性自帶行為，強耦合於框架
  | public class MyService { ... }   |
  +----------------------------------+

  正確做法：被動屬性（Passive Attribute）
  +----------------------------------+
  | [PermittedRole(Role.Admin)]      |  <-- 純中繼資料，無行為
  | public class AdjustInventory     |
  | { ... }                          |
  +----------------------------------+
          |
          | Decorator 在執行時讀取屬性
          v
  +----------------------------------+
  | SecureCommandServiceDecorator<T> |  <-- 行為在 Decorator 中
  | {                                |
  |   typeof(TCommand)               |
  |     .GetCustomAttribute<...>();  |
  | }                                |
  +----------------------------------+
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點一：為什麼 `object` 參數導致 LSP 違規？

**困難所在**：很多人覺得 `ICommandService` 用 `object` 當參數很方便，看不出問題在哪裡。

**深入解析**：

LSP 要求你能用任意實作替換同一個抽象而不破壞消費端。問題來了：

```csharp
// 消費端依賴 ICommandService
public class InventoryController
{
    private readonly ICommandService service;

    public InventoryController(ICommandService service)
    {
        this.service = service;
    }

    public ActionResult AdjustInventory(...)
    {
        this.service.Execute(adjustInventoryCommand);
        // ...
    }
}
```

Composition Root 如果不小心注入了錯誤的實作：

```csharp
// 這段程式碼可以編譯通過！但執行時會崩潰
ICommandService service = new UpdateProductReviewTotalsService(repo);
new InventoryController(service);  // 編譯器不會報錯

// 但當 InventoryController 呼叫 service.Execute(adjustInventory) 時
// UpdateProductReviewTotalsService 會嘗試：
//   var command = (UpdateProductReviewTotals)cmd;  // InvalidCastException!
```

**泛型版本的解決**：

```csharp
// 編譯器直接阻止錯誤組合
ICommandService<AdjustInventory> service =
    new UpdateProductReviewTotalsService(repo);  // 編譯錯誤！型別不匹配！
```

> **比喻**：`object` 就像一個「萬用插座」，任何插頭都能插，但如果你把 220V 的電器插到 110V 的插座上，雖然插得進去，但用的時候就炸了。泛型就像「專用插座」，錯誤的插頭根本插不進去。

### 難點二：從 7 個介面 → 1 個泛型介面的心智跳躍

**困難所在**：Step 2 拆出了 7 個各自不同的介面（不同方法名、不同參數），Step 3 卻要把它們合併成一個介面，這在直覺上是矛盾的。

**解析**：關鍵在於 **Parameter Object 的提取讓 pattern 浮現**。

```
重構前：
  IAdjustInventoryService.AdjustInventory(Guid, bool, int)
  IUpdateReviewTotalsService.UpdateProductReviewTotals(Guid, Review[])
  IDeleteProductService.DeleteProduct(Guid)

  -> 方法名不同、參數不同、看不出共通點

重構後（提取 Parameter Object）：
  IAdjustInventoryService.Execute(AdjustInventory)
  IUpdateReviewTotalsService.Execute(UpdateProductReviewTotals)
  IDeleteProductService.Execute(DeleteProduct)

  -> 方法名相同（Execute）、參數數量相同（1 個）、回傳值相同（void）
  -> Pattern 浮現！可以抽取共同介面！
```

> **比喻**：想像你有不同口味的飲料機——可樂機、雪碧機、果汁機。每台操作方式都不一樣。但如果你把所有飲料都裝進標準的「杯子」裡（Parameter Object），那你只需要一台「出杯機」（ICommandService），放上任何杯子按下按鈕就好。差異被封裝進杯子（命令物件）裡了。

### 難點三：為什麼類別數量增加反而提升了可維護性？

**困難所在**：從 2 個型別（IProductService + ProductService）變成 15 個型別（1 介面 + 7 Parameter Objects + 7 Services），直覺上覺得更複雜了。

**解析**：可維護性的衡量標準不是類別數量，而是**改變的代價**。

```
重構前（2 個型別）：
  - 新增功能：修改 IProductService + ProductService + 所有 Decorator + 所有 Test Double
    => 改 N 個檔案

  - 新增橫切關注點：為 IProductService 寫 1 個 Decorator
    但這個 Decorator 要實作所有 10 個方法！
    而且對每個類似介面（ICustomerService, IOrderService...）都要重複！
    => 寫 M(介面數) x K(方法數) 段幾乎一樣的程式碼

重構後（15 個型別）：
  - 新增功能：新增 1 個 Command + 1 個 Service
    => 改 0 個既有檔案，加 2 個新檔案

  - 新增橫切關注點：寫 1 個泛型 Decorator
    => 加 1 個新檔案，自動套用到所有命令
```

**量化比較（假設 5 個類似介面，每個 10 個方法，10 種橫切關注點）**：

```
                          重構前                    重構後
新增 1 個功能需修改       ~12 個檔案                0 個（新增 2 個）
新增 1 個橫切關注點       5 個 Decorator x 10 方法   1 個泛型 Decorator
                        = 50 段重複程式碼            = 1 段程式碼
橫切關注點總程式碼量      10 x 50 = 500 段           10 段
```

> **比喻**：一座有 100 間房間的大樓。重構前的設計是每間房間都有自己的空調、暖氣、灑水系統——看起來「簡單」（只有 100 個房間），但要加裝煙霧偵測器就得進 100 間房間施工。重構後的設計是有中央空調、中央暖氣、中央灑水系統——看起來「複雜」（多了管線間、控制室），但加裝煙霧偵測器只需要在中央系統加一個模組就好。

## 6. 本章小結與架構啟示

### 核心啟示

1. **AOP 是設計問題，不是工具問題**。你的第一反應應該是改善設計，只有在設計無法解決時才考慮 AOP 工具。

2. **SOLID 原則是一個有機的整體**。本章完美展示了五個原則如何環環相扣：ISP 催生了介面拆分，SRP 給了每個類別單一職責，OCP 透過 Parameter Object 和泛型介面消除了 sweeping changes，LSP 透過泛型型別參數保證了型別安全，DIP 確保了消費端決定抽象的形狀。

3. **訊息傳遞優於方法呼叫**。當你把 `service.DoSomething(a, b, c)` 轉變為 `service.Execute(new DoSomethingCommand { A = a, B = b, C = c })`，你得到了序列化、佇列化、日誌記錄、重播等一系列免費能力。

4. **類別數量不等於複雜度**。真正的複雜度指標是「改變的代價」——新增功能或橫切關注點時，需要修改多少個既有檔案。

5. **正確的粒度層級至關重要**。多數橫切關注點應該在業務操作層級（`ICommandService<TCommand>`）套用，而非 Repository 層級。一個業務操作可能跨越多個 Repository 呼叫，在 Repository 層級套用 Transaction 會導致多個不必要的交易。

6. **被動屬性分離了中繼資料與行為**。用屬性標記「誰能做這件事」，用 Decorator 決定「怎麼檢查」——兩者可以獨立演化。

### 架構師的行動指南

```
當你面對一個「大介面」時：

  1. 問自己：消費者是否都用到所有方法？（ISP 檢查）
  2. 問自己：實作類別有幾個改變的理由？（SRP 檢查）
  3. 問自己：新增功能是否造成連鎖修改？（OCP 檢查）
  4. 如果以上任一答案不理想 => 啟動四步重構
     Step 1: CQS 分離讀寫
     Step 2: 拆分為單一方法介面
     Step 3: Parameter Object + 統一 Execute
     Step 4: 泛型化消除 LSP 風險
  5. 在正確的粒度層級套用橫切關注點
  6. 用 Composition Root 中的 Decorate<T> 方法統一裝飾
```

### 延伸思考

本章只處理了 Command 端。Query 端可以用類似的方式設計：

```csharp
// Query 的泛型抽象
public interface IQueryService<TQuery, TResult>
{
    TResult Execute(TQuery query);
}
```

這讓你可以對查詢操作套用快取、日誌、效能監控等橫切關注點，同樣只需要一個泛型 Decorator。這種 Command/Query 的對稱設計，正是 CQRS（命令查詢職責分離）架構模式的基礎。
