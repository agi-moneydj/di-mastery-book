## 1. 概述 (Overview)

本章是 DI 實踐的核心武器庫。如果說前三章是「為什麼要用 DI」和「DI 是什麼」，那麼第四章就是「DI 怎麼做」。作者以廚師的料理術語做類比開場——就像「折疊」(folding) 和「收汁」(reduction) 讓廚師能精準溝通，DI 模式也是軟體工程師的共通語言，讓我們能簡潔且有效地討論物件組合的結構方式。

本章系統性地介紹四種 DI 模式，每一種都有明確的適用場景、優缺點和程式碼範例：

| 模式 | 一句話定義 | 使用頻率 |
|------|-----------|---------|
| **Composition Root** | 在應用程式進入點附近，集中組裝所有物件圖 | 每個啟動專案必用 |
| **Constructor Injection** | 透過建構函式靜態宣告必要的相依 | 最常用，應為預設選擇 |
| **Method Injection** | 透過方法參數傳入隨操作變化的相依 | 適用於特定場景 |
| **Property Injection** | 透過可寫屬性覆寫 Local Default 行為 | 僅用於可重用函式庫 |

> **架構師觀點**：Constructor Injection 與 Composition Root 是你每天都會用到的模式。Method Injection 在 DDD 或外掛場景下偶爾出現。Property Injection 幾乎只在打造可重用函式庫時才有意義。把精力集中在前兩者上。

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

1. **辨識並實作 Composition Root**——知道在 Console App、ASP.NET Core 等不同應用類型中，Composition Root 應放在哪裡
2. **熟練使用 Constructor Injection**——理解它為何是預設選擇、如何搭配 Guard Clause、readonly 欄位正確實作
3. **區分四種模式的適用場景**——面對一個依賴需求時，能迅速判斷該用哪種注入方式
4. **理解 Local Default vs Foreign Default**——知道為什麼「預設實作在同一個模組」與「預設實作在不同模組」的差別如此關鍵
5. **避免 Temporal Coupling**——理解為什麼 Method Injection 不該在 Composition Root 中使用，以及為什麼 Property Injection 需要額外防護
6. **理解 Null Object 模式**——知道如何用 Null Object 取代可選依賴的 null 檢查
7. **破解「依賴爆炸」迷思**——能向同事解釋為什麼 Composition Root 看似依賴所有模組，實際上反而降低了整體耦合度

## 3. 章節大綱 (Outline)

### 3.1 Composition Root（組合根）

#### 3.1.1 定義與核心觀念

**Composition Root 是什麼？**
它是應用程式中一個單一的、邏輯上的位置，用來把所有模組組裝在一起。它應該盡可能靠近應用程式的進入點（Entry Point）。

> **白話比喻**：想像你在組裝一台電腦。Composition Root 就是你的「組裝工作台」。所有零件（主機板、CPU、記憶體、硬碟）都從各自的包裝盒拿出來，在這張工作台上被組裝成一台完整的電腦。你不會在客廳裝 CPU、在臥室裝記憶體——你在一個地方完成所有組裝。

```
                    應用程式架構圖

    +--------------------------------------------------+
    |              Entry Point (Main)                    |
    |  +--------------------------------------------+  |
    |  |         COMPOSITION ROOT                    |  |
    |  |                                            |  |
    |  |   在此組裝所有物件圖                         |  |
    |  |   - 讀取設定                                |  |
    |  |   - 建立依賴關係                            |  |
    |  |   - 組裝物件樹                              |  |
    |  +------+----------+----------+---------------+  |
    |         |          |          |                   |
    |         v          v          v                   |
    |   +---------+ +---------+ +---------+            |
    |   |   UI    | | Domain  | |  Data   |            |
    |   |  Layer  | |  Layer  | | Access  |            |
    |   +---------+ +---------+ +---------+            |
    +--------------------------------------------------+

    Composition Root 直接依賴所有模組，
    但 UI/Domain/Data Access 層之間不互相直接依賴。
```

#### 3.1.2 Console Application 範例

```csharp
public static class Program
{
    // 應用程式進入點
    public static void Main(string[] args)
    {
        string connectionString = args[0];

        // 請 Composition Root 建立物件圖
        HomeController controller = CreateController(connectionString);

        var result = controller.Index();
        var vm = (FeaturedProductsViewModel)result.Model;

        Console.WriteLine("Featured products:");
        foreach (var product in vm.Products)
        {
            Console.WriteLine(product.SummaryText);
        }
    }

    // 這就是 Composition Root —— 集中建立物件圖的地方
    private static HomeController CreateController(string connectionString)
    {
        var userContext = new ConsoleUserContext();
        return new HomeController(
            new ProductService(
                new SqlProductRepository(
                    new CommerceContext(connectionString)),
                userContext));
    }
}
```

> **重點**：Composition Root 不是一個方法或一個類別——它是一個「概念」。它可以是 Main 方法的一部分，也可以橫跨多個類別，只要它們都在同一個模組（assembly）裡。

#### 3.1.3 ASP.NET Core 範例（Pure DI）

在 ASP.NET Core 中，Composition Root 透過 `Startup` 類別和自訂的 `IControllerActivator` 實現：

```csharp
// Startup.cs —— Composition Root 的第一部分
public class Startup
{
    public Startup(IConfiguration configuration)
    {
        this.Configuration = configuration;
    }

    public IConfiguration Configuration { get; }

    public void ConfigureServices(IServiceCollection services)
    {
        services.AddMvc();
        services.AddHttpContextAccessor();

        // 載入設定值
        var connectionString =
            this.Configuration.GetConnectionString("CommerceConnection");

        // 用自訂的 ControllerActivator 替換預設的
        services.AddSingleton<IControllerActivator>(
            new CommerceControllerActivator(connectionString));
    }
}

// CommerceControllerActivator.cs —— Composition Root 的第二部分
public class CommerceControllerActivator : IControllerActivator
{
    private readonly string connectionString;

    public CommerceControllerActivator(string connectionString)
    {
        this.connectionString = connectionString;
    }

    // 每個 HTTP 請求都會呼叫此方法來建立 Controller
    public object Create(ControllerContext ctx)
    {
        Type type = ctx.ActionDescriptor.ControllerTypeInfo.AsType();

        if (type == typeof(HomeController))
        {
            return new HomeController(
                new ProductService(
                    new SqlProductRepository(
                        new CommerceContext(this.connectionString)),
                    new AspNetUserContextAdapter()));
        }
        else
        {
            throw new Exception("Unknown controller.");
        }
    }
}
```

```
    Composition Root 跨越兩個類別

    +----------------------------------+
    |       COMPOSITION ROOT           |
    |                                  |
    |  +------------+  +------------+  |
    |  |  Startup   |  | Commerce-  |  |
    |  |  (載入設定) |  | Controller-|  |
    |  |            |  | Activator  |  |
    |  |            |  | (組裝物件圖)|  |
    |  +------------+  +------------+  |
    |                                  |
    |  兩個類別在同一個模組（assembly）中  |
    +----------------------------------+
```

**關鍵設計決策**：Startup 負責載入設定（configuration），CommerceControllerActivator 只依賴設定值（connection string），不依賴設定系統。這個分離讓物件組裝可以不需要有效的設定檔就能測試。

#### 3.1.4 「依賴爆炸」的迷思

許多開發者抱怨：「Composition Root 讓我的進入點依賴了所有的 assembly，這不是反過來增加依賴了嗎？」

這是一個常見的誤解。原因是 **assembly 的依賴具有傳遞性（transitivity）**。

> **白話解釋**：如果 A 依賴 B，B 依賴 C，那麼 A 其實也依賴 C。就像你如果依賴電力公司（B），電力公司依賴天然氣供應商（C），那停電時你也連帶受到天然氣供應的影響。

```
    緊密耦合 (Mary's App)              鬆散耦合 (DI App)

    Entry Point                        COMPOSITION ROOT
      |                                  |     |     |
      v                                  v     v     v
    UI Layer ----+                     UI    Domain  Data
      |          |                     Layer  Layer  Access
      v          |
    Domain ------+                     看似 5 條依賴線
      |          |
      v          |                     但 Mary 的 App
    Data Access -+                     因為傳遞性，
                                       實際上有 6 條依賴！
    看似 3 條，
    但傳遞性讓實際上有 6 條

    結論：鬆散耦合 + Composition Root
         反而減少了整體依賴數量！
```

#### 3.1.5 重要規則摘要

| 規則 | 說明 |
|------|------|
| Composition Root 不是 UI 層 | 即使放在同一個 assembly，邏輯上它不屬於 UI 層 |
| 只有啟動專案有 Composition Root | 類別庫（library）不應該有 Composition Root |
| DI Container 只能在 Composition Root 使用 | 在其他地方使用等同於 Service Locator 反模式 |
| 不要擔心效能 | 建立物件圖通常非常快速，幾乎不會是效能瓶頸 |
| 並行操作需要新物件圖 | 當開啟新執行緒時，Composition Root 應為其建立新的物件圖 |

### 3.2 Constructor Injection（建構函式注入）

#### 3.2.1 定義與核心觀念

**Constructor Injection** 是透過建構函式參數來靜態宣告一個類別所需的依賴。

> **白話比喻**：就像買一台新車時，引擎是必須的——你在下訂單（建構函式）的時候就要指定引擎型號。車子不可能沒有引擎就出廠。Constructor Injection 就是確保你的類別「出廠」時一定配備了必要零件。

```
    Constructor Injection 流程

    Composition Root (客戶端/組裝者)
         |
         | 1. 建立 Dependency
         v
    IProductService dependency = new ProductService(...)
         |
         | 2. 注入到 Consumer 的建構函式
         v
    new HomeController(dependency)
         |
         | 3. 使用 Consumer
         v
    controller.Index()
         |         |
         |         | 4. Consumer 內部使用 Dependency
         |         v
         |    service.GetFeaturedProducts()
         v
    回傳結果
```

#### 3.2.2 標準實作模板

```csharp
public class HomeController
{
    // 1. 用 readonly 欄位儲存依賴
    private readonly IProductService service;

    // 2. 建構函式接收依賴
    public HomeController(IProductService service)
    {
        // 3. Guard Clause 防止 null
        if (service == null)
            throw new ArgumentNullException(nameof(service));

        // 4. 儲存依賴供日後使用
        this.service = service;
    }

    // 5. 其他方法直接使用，不需再檢查 null
    public ViewResult Index()
    {
        var products = this.service.GetFeaturedProducts();
        // ... 使用 products
    }
}
```

**三個關鍵紀律**：

| 紀律 | 原因 |
|------|------|
| **只用一個 public 建構函式** | 多個建構函式造成歧義——呼叫端或 DI Container 不知道該用哪一個 |
| **Guard Clause 檢查 null** | 編譯器無法阻止傳入 null，需要執行時期保護 |
| **欄位標記為 readonly** | 防止意外在其他方法中修改依賴引用 |

> **重要原則**：建構函式裡除了 Guard Clause 和儲存依賴之外，不該有其他邏輯。根據單一職責原則，建構函式的工作就是「接收並儲存依賴」，不要讓它做額外的事。這讓物件的建構既快速又可靠。

#### 3.2.3 Local Default vs Foreign Default

**Local Default（本地預設）**：預設實作與消費者在同一個模組或層中。

**Foreign Default（外部預設）**：預設實作在不同的 assembly 中。

> **白話比喻**：Local Default 就像你家廚房裡的鹽巴——隨手可取，不需要依賴外部。Foreign Default 就像你需要去隔壁鄰居家借醬油——你必須「依賴」鄰居家有醬油，而且鄰居搬走了你就沒醬油用了。

```
    Local Default vs Foreign Default

    +-------------------+          +-------------------+
    | Module A          |          | Module B          |
    |                   |          |                   |
    | [Consumer]        |          | [SqlRepository]   |
    |    |              |          |  (Foreign Default) |
    |    v              |          +-------------------+
    | [IStrategy]       |
    |    ^              |      <-- 如果 Consumer 直接
    |    |              |          new SqlRepository()
    | [LocalStrategy]   |          就會變成 Foreign Default
    |  (Local Default)  |          = Control Freak 反模式
    +-------------------+
```

**規則**：
- 如果有 Local Default，可以考慮 Property Injection
- 如果沒有合理的 Local Default（最常見的情況），用 Constructor Injection
- 注意傳遞性：一個 Local Default 如果自身的依賴是 Foreign Default，那它其實也是 Foreign Default

#### 3.2.4 完整範例：加入貨幣轉換功能

Mary 的電商應用需要支援多幣別轉換。`ICurrencyConverter` 的具體實作可能是 Web Service 或資料庫（典型的跨 assembly 依賴），所以不存在合理的 Local Default，應該使用 Constructor Injection。

```csharp
// 抽象與值物件
public interface ICurrencyConverter
{
    Money Exchange(Money money, Currency targetCurrency);
}

public class Currency
{
    public readonly string Code;
    public Currency(string code)
    {
        if (code == null) throw new ArgumentNullException(nameof(code));
        this.Code = code;
    }
}

public class Money
{
    public readonly decimal Amount;
    public readonly Currency Currency;
    public Money(decimal amount, Currency currency)
    {
        if (currency == null) throw new ArgumentNullException(nameof(currency));
        this.Amount = amount;
        this.Currency = currency;
    }
}
```

```csharp
// 在 ProductService 中透過 Constructor Injection 注入 ICurrencyConverter
public class ProductService : IProductService
{
    private readonly IProductRepository repository;
    private readonly IUserContext userContext;
    private readonly ICurrencyConverter converter; // 新增的依賴

    public ProductService(
        IProductRepository repository,
        IUserContext userContext,
        ICurrencyConverter converter)   // 加入第三個依賴
    {
        if (repository == null) throw new ArgumentNullException(nameof(repository));
        if (userContext == null) throw new ArgumentNullException(nameof(userContext));
        if (converter == null)  throw new ArgumentNullException(nameof(converter));

        this.repository = repository;
        this.userContext = userContext;
        this.converter = converter;
    }

    public IEnumerable<DiscountedProduct> GetFeaturedProducts()
    {
        Currency userCurrency = this.userContext.Currency;
        var products = this.repository.GetFeaturedProducts();
        return
            from product in products
            let unitPrice = product.UnitPrice
            let amount = this.converter.Exchange(
                money: unitPrice,
                targetCurrency: userCurrency)
            select product
                .WithUnitPrice(amount)
                .ApplyDiscountFor(this.userContext);
    }
}
```

> **注意**：在 `GetFeaturedProducts` 中可以直接使用 `this.converter`，不需要先檢查是否為 null，因為 Guard Clause 在建構時就已經保證了它一定存在。

#### 3.2.5 Null Object 模式

**問題**：有時候消費者有一個依賴（例如 `ILogger`），但在某些部署環境中不需要實際的日誌功能。這時該怎麼辦？

**錯誤做法**：把依賴設為可選（nullable），然後到處加 null 檢查。

**正確做法**：使用 Null Object 模式——建立一個「什麼都不做」的實作，注入給消費者。

```csharp
// 抽象
public interface ILogger
{
    void Log(LogEntry entry);
}

// 真正的實作：寫入檔案
public class FileLogger : ILogger
{
    public string Path { get; }
    public void Log(LogEntry entry)
    {
        // 實際寫入檔案的邏輯
    }
}

// Null Object：什麼都不做
public class NullLogger : ILogger
{
    public void Log(LogEntry entry)
    {
        // 故意留空——這就是 Null Object 的精髓
    }
}
```

```
    Null Object 模式結構

    Client ---------> ILogger
                        ^   ^
                       /     \
              FileLogger    NullLogger
              (寫檔案)      (什麼都不做)

    在 Composition Root 中：
    - 需要日誌 => 注入 FileLogger
    - 不需要   => 注入 NullLogger
    - Consumer 完全不用改，也不用做 null 檢查
```

> **白話比喻**：Null Object 就像一個沉默的傾聽者。你對它說話（呼叫方法），它微笑點頭（接受呼叫），但不採取任何行動。消費者不需要知道或在意對方是否真的有在「做事」。

#### 3.2.6 優缺點

| 優點 | 缺點 |
|------|------|
| **保證注入**：編譯器 + Guard Clause 雙重保護 | 某些舊框架要求無參數建構函式（Constrained Construction 反模式） |
| **容易實作**：模式固定且簡單 | |
| **靜態宣告**：一眼就能看出類別需要哪些依賴 | |

### 3.3 Method Injection（方法注入）

#### 3.3.1 定義與核心觀念

**Method Injection** 是透過方法參數傳入依賴，發生在 Composition Root 之外。

> **白話比喻**：Constructor Injection 就像你把常備工具（螺絲起子、扳手）放在工具箱裡隨身攜帶。Method Injection 則像是你到了特定工地，有人遞給你一個特殊工具來完成當下這個任務——你用完就還回去，不帶走。

```
    Method Injection 流程

    ProductService
         |
         | 1. 建立 Product 實體（或從 Repository 取得）
         v
      Product 實體
         |
         | 2. 呼叫方法，同時傳入 IUserContext 依賴
         v
    product.ApplyDiscountFor(userContext)
         |
         | 3. Product 使用 IUserContext 計算折扣
         v
      回傳 DiscountedProduct
```

#### 3.3.2 兩種典型場景

**場景一：消費者（Consumer）隨每次呼叫而變化**

這是最常見的情況。例如，多個不同的 `Customer` 實體都需要使用同一個 `IVoucherRedemptionService`。

```csharp
// DDD Entity 使用 Method Injection
public class Customer
{
    public Guid Id { get; private set; }
    public string Name { get; private set; }

    public Customer(Guid id, string name)
    {
        // ... 初始化
    }

    // 每個方法只接收自己需要的依賴
    public void RedeemVoucher(
        Voucher voucher,
        IVoucherRedemptionService service)
    {
        if (voucher == null) throw new ArgumentNullException(nameof(voucher));
        if (service == null) throw new ArgumentNullException(nameof(service));

        service.ApplyRedemptionForCustomer(voucher, this.Id);
    }

    public void MakePreferred(IEventHandler handler)
    {
        if (handler == null) throw new ArgumentNullException(nameof(handler));
        handler.Publish(new CustomerMadePreferred(this.Id));
    }
}
```

```csharp
// 上層服務使用 Constructor Injection 接收依賴，
// 再透過 Method Injection 傳給 Entity
public class CustomerServices : ICustomerServices
{
    private readonly ICustomerRepository repository;
    private readonly IVoucherRedemptionService service;

    public CustomerServices(
        ICustomerRepository repository,
        IVoucherRedemptionService service)  // Constructor Injection
    {
        this.repository = repository;
        this.service = service;
    }

    public void RedeemVoucher(Guid customerId, Voucher voucher)
    {
        // Customer 是從 Repository 取出的，不是在 Composition Root 建立的
        var customer = this.repository.GetById(customerId);

        // 透過 Method Injection 把依賴傳給 Entity
        customer.RedeemVoucher(voucher, this.service);

        this.repository.Save(customer);
    }
}
```

> **為什麼 Entity 不用 Constructor Injection？** 因為 Entity 混合了「執行時資料」（Id, Name）和「行為」（RedeemVoucher），而且不是在 Composition Root 中建立的。如果用 Constructor Injection 注入所有依賴，每次建立 Entity 都得提供所有依賴，即使當次操作只用到其中一個。這會使測試和建構變得複雜。

**場景二：依賴（Dependency）隨每次呼叫而變化**

適用於外掛（add-in）系統，例如圖片處理應用的特效外掛：

```csharp
// 外掛介面：每次呼叫時，應用程式傳入不同的 context
public interface IImageEffectAddIn
{
    Bitmap Apply(
        Bitmap source,
        IApplicationContext context);  // Method Injection
}

// 客戶端逐一呼叫每個外掛
public Bitmap ApplyEffects(Bitmap source)
{
    if (source == null) throw new ArgumentNullException(nameof(source));

    Bitmap result = source;
    foreach (IImageEffectAddIn effect in this.effects)
    {
        // 每次呼叫都傳入當前的 context
        result = effect.Apply(result, this.context);
    }
    return result;
}
```

#### 3.3.3 完整範例：貨幣轉換移入 Product Entity

Mary 發現貨幣轉換會在應用程式的多處使用，所以把轉換邏輯集中到 Product Entity 中：

```csharp
public class Product
{
    public string Name { get; set; }
    public Money UnitPrice { get; set; }
    public bool IsFeatured { get; set; }

    // 透過 Method Injection 接收 ICurrencyConverter
    public Product ConvertTo(
        Currency currency,
        ICurrencyConverter converter)
    {
        if (currency == null)
            throw new ArgumentNullException(nameof(currency));
        if (converter == null)
            throw new ArgumentNullException(nameof(converter));

        var newUnitPrice = converter.Exchange(this.UnitPrice, currency);
        return this.WithUnitPrice(newUnitPrice);
    }

    public Product WithUnitPrice(Money unitPrice)
    {
        return new Product
        {
            Name = this.Name,
            UnitPrice = unitPrice,
            IsFeatured = this.IsFeatured
        };
    }
}
```

```csharp
// 重構後的 GetFeaturedProducts —— 更簡潔
public IEnumerable<DiscountedProduct> GetFeaturedProducts()
{
    Currency currency = this.userContext.Currency;
    return
        from product in this.repository.GetFeaturedProducts()
        select product
            .ConvertTo(currency, this.converter)  // Method Injection
            .ApplyDiscountFor(this.userContext);    // Method Injection
}
```

#### 3.3.4 Temporal Coupling（時序耦合）

**嚴禁**在 Composition Root 中使用 Method Injection 來初始化依賴。這會造成 Temporal Coupling——方法必須以特定順序呼叫，但 API 不會告訴你這個順序。

```csharp
// 反面教材：Temporal Coupling
public class Component
{
    private ISomeInterface dependency;

    public void Initialize(ISomeInterface dependency) // 必須先呼叫這個
    {
        this.dependency = dependency;
    }

    public void DoSomething() // 才能呼叫這個
    {
        if (this.dependency == null)
            throw new InvalidOperationException("Call Initialize first.");
        this.dependency.DoStuff();
    }
}

// 這段程式碼可以編譯，但執行時爆炸：
var c = new Component();
c.DoSomething();  // 忘記呼叫 Initialize => 例外！
```

```
    Temporal Coupling 問題

    正確順序:                    常犯的錯誤:
    1. new Component()          1. new Component()
    2. Initialize(dep)          2. DoSomething()  <-- 炸了!
    3. DoSomething()

    API 沒有任何提示告訴你必須先呼叫 Initialize。
    解決方案：改用 Constructor Injection。
```

**解法**：改用 Constructor Injection。

```csharp
// 正確做法：Constructor Injection 消除 Temporal Coupling
public class Component
{
    private readonly ISomeInterface dependency;

    public Component(ISomeInterface dependency)
    {
        if (dependency == null)
            throw new ArgumentNullException(nameof(dependency));
        this.dependency = dependency;
    }

    public void DoSomething()
    {
        this.dependency.DoStuff();  // 保證可用，不需 null 檢查
    }
}
```

> **重要警告**：透過 Method Injection 接收的依賴不應該被儲存。只能使用或傳遞，不能存起來。存起來會導致 Temporal Coupling、Captive Dependencies 或隱藏的副作用。

#### 3.3.5 優缺點

| 優點 | 缺點 |
|------|------|
| 允許呼叫者提供操作特定的 context | 適用場景有限 |
| 允許將依賴注入不在 Composition Root 建立的物件（如 DDD Entity）| 依賴變成公開 API 的一部分 |

---

### 3.4 Property Injection（屬性注入）

#### 3.4.1 定義與核心觀念

**Property Injection** 是透過可寫屬性讓呼叫者可以覆寫類別的預設行為（Local Default）。也稱為 Setter Injection。

> **白話比喻**：就像一台電視出廠時已經設定好預設頻道（Local Default），但你可以透過遙控器（Property）換台。如果你懶得換，預設頻道也能用。但如果電視沒有預設頻道（沒有 Local Default），那出廠就是黑畫面——這種情況就不該用 Property Injection。

```
    Property Injection 結構

    +----------------------+
    |      Consumer        |
    |                      |
    | - Dependency {get;set}  <--- 呼叫者可選擇性地設定
    |                      |
    | - creates ---------->  LocalDefault (預設實作)
    |                      |
    +----------------------+

    使用時：
    var consumer = new Consumer();
    // 如果不設定，就用 LocalDefault
    // 如果想換：
    consumer.Dependency = new CustomImplementation();
```

#### 3.4.2 使用時機——幾乎只用於可重用函式庫

**明確的規則**：

1. **撰寫應用程式時**：永遠不要用 Property Injection。用 Constructor Injection。
2. **撰寫可重用函式庫時**：如果有好的 Local Default 且依賴是可選的，可以考慮 Property Injection。

> **為什麼應用程式碼不該用？** 在應用程式中，你有 Composition Root 來集中管理所有依賴。即使有 Local Default，也應該在 Composition Root 中顯式注入，讓 Composition Root 完全掌控。Property Injection 在應用程式碼中沒有優勢，反而帶來 Temporal Coupling 的風險。

#### 3.4.3 完整範例：Simple Injector 的擴展點

以 Simple Injector（一個 DI Container 函式庫）為例，它用 Property Injection 讓使用者覆寫預設的建構函式解析行為：

```csharp
public class ContainerOptions
{
    // Local Default：預設實作在同一個函式庫中
    IConstructorResolutionBehavior resolutionBehavior =
        new DefaultConstructorResolutionBehavior();

    public IConstructorResolutionBehavior ConstructorResolutionBehavior
    {
        get
        {
            return this.resolutionBehavior;
        }
        set
        {
            // Guard Clause 1：防止 null
            if (value == null)
                throw new ArgumentNullException(nameof(value));

            // Guard Clause 2：Popsicle Immutability
            // 一旦開始使用就不能再改
            if (this.Container.HasRegistrations)
            {
                throw new InvalidOperationException(
                    "The ConstructorResolutionBehavior property " +
                    "cannot be changed after the first registration " +
                    "has been made to the container.");
            }

            this.resolutionBehavior = value;
        }
    }
}
```

```csharp
// 使用方式
var container = new Container();

// 可選：覆寫預設行為
container.Options.ConstructorResolutionBehavior =
    new CustomConstructorResolutionBehavior();

// 如果不設定，就用 DefaultConstructorResolutionBehavior
```

> **名詞解釋 — Popsicle Immutability（冰棒不可變性）**：像冰棒一樣，在凍結之前可以塑形，一旦凍結就不能再改變。Simple Injector 允許你在第一次註冊之前修改設定，之後就「凍結」了。

#### 3.4.4 Open/Closed Principle（開放/封閉原則）

Property Injection 與 Open/Closed Principle 密切相關：

- **對擴展開放**：你可以透過替換屬性來改變行為
- **對修改封閉**：不需要修改消費者的原始碼

> **白話解釋**：軟體實體應該像 USB 接口一樣——你可以插入各種不同的 USB 設備（開放擴展），但不需要改動電腦主機板的電路（封閉修改）。DI 讓你可以替換或攔截類別來新增或變更行為，而消費者和被依賴者都不需要知道這件事。

#### 3.4.5 優缺點

| 優點 | 缺點 |
|------|------|
| 容易理解 | 正確實作不簡單（需要 Guard Clause + Popsicle Immutability） |
| | 適用範圍有限（幾乎只用於可重用函式庫） |
| | 造成 Temporal Coupling |

## 4. 關鍵框架與公式 (Key Frameworks)

### 4.1 DI 模式完整對照表

| 維度 | Composition Root | Constructor Injection | Method Injection | Property Injection |
|------|-----------------|----------------------|------------------|--------------------|
| **注入時機** | 應用啟動時 | 物件建立時 | 方法呼叫時 | 物件建立後、使用前 |
| **發生位置** | 進入點附近 | Composition Root 內 | Composition Root 外 | Composition Root 內 |
| **依賴性質** | N/A | 必要（Required） | 隨操作變化 | 可選（Optional） |
| **適用對象** | 啟動專案 | 幾乎所有元件 | Entity、外掛 | 可重用函式庫 |
| **Local Default** | N/A | 不需要 | 不需要 | 必須有 |
| **Temporal Coupling** | N/A | 無 | 有（如果儲存依賴）| 有 |

### 4.2 Constructor Injection 實作清單

```
    Constructor Injection 必做清單

    [x] 只有一個 public 建構函式
    [x] 每個依賴都是建構函式參數
    [x] 每個參數都有 Guard Clause（null 檢查）
    [x] 依賴存入 private readonly 欄位
    [x] 建構函式不包含其他邏輯
    [x] 其他方法不需要檢查依賴是否為 null
```

### 4.3 核心名詞對照

| 英文術語 | 中文翻譯 | 白話說明 |
|----------|---------|---------|
| **Composition Root** | 組合根 | 應用程式中集中組裝物件的唯一地方 |
| **Object Graph** | 物件圖 | 一個根物件以及它所有依賴組成的樹狀結構 |
| **Guard Clause** | 守衛子句 | 在方法開頭檢查參數有效性的防禦性程式碼 |
| **Local Default** | 本地預設 | 與消費者在同一個模組中的預設實作 |
| **Foreign Default** | 外部預設 | 在不同模組中的預設實作，會造成緊密耦合 |
| **Temporal Coupling** | 時序耦合 | 必須以特定順序呼叫方法，但 API 沒告訴你順序 |
| **Null Object** | 空物件 | 實作介面但什麼都不做的物件，取代 null |
| **Popsicle Immutability** | 冰棒不可變性 | 初始化期間可以修改，之後就凍結不可變 |

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 5.1 難點一：Composition Root 「不是 UI 層」

**為什麼難？**
很多開發者看到 Composition Root 放在 ASP.NET Core 的啟動專案中（通常也是 UI 層所在的 assembly），就認為 Composition Root 是 UI 層的一部分。然後就困惑：「UI 層為什麼要依賴 Data Access 層？這不是違反了分層架構嗎？」

**釐清**：

```
    Assembly 是部署單位，Layer 是邏輯單位

    +----- 一個 Assembly (.dll) -----+
    |                                |
    |  +-- Composition Root ---+     |
    |  | (邏輯上獨立)          |     |
    |  | 依賴所有模組           |     |
    |  +-----------------------+     |
    |                                |
    |  +-- UI Layer -----------+     |
    |  | (邏輯上獨立)          |     |
    |  | 只依賴 Domain Layer   |     |
    |  +-----------------------+     |
    |                                |
    +--------------------------------+

    在同一個 Assembly 裡，但邏輯上是兩個不同的東西。
    UI Layer 的程式碼不知道也不該知道 Data Access 的存在。
```

> **比喻**：一棟大樓的同一層樓可以同時有「管理室」和「員工辦公區」。管理室（Composition Root）需要知道整棟大樓的所有樓層和部門。員工辦公區（UI Layer）只需要知道自己的工作範圍。它們在同一層樓（同一個 assembly），但職責完全不同。

### 5.2 難點二：Method Injection 何時該用、何時不該用

**為什麼難？**
Method Injection 的使用時機比較微妙。它看起來跟「傳參數」沒什麼不同，但卻有明確的使用邊界。

**決策框架**：

| 情境 | 該用 Method Injection 嗎？ |
|------|--------------------------|
| DDD Entity 需要一個 Service 來完成某個操作 | 是 —— Entity 不在 Composition Root 建立，且每次是不同的 Entity 實例 |
| 外掛需要接收 runtime context | 是 —— context 可能每次不同 |
| 在 Composition Root 中用 Initialize 方法設定依賴 | **不是** —— 改用 Constructor Injection，否則會有 Temporal Coupling |
| 一個 Service 需要另一個 Service | **不是** —— 用 Constructor Injection |
| 方法接收依賴後把它存進欄位 | **不是** —— 絕對不要這樣做 |

**核心原則**：Method Injection 注入的依賴只能「使用」或「傳遞」，不能「儲存」。

### 5.3 難點三：Property Injection 的正確實作

**為什麼難？**
Property Injection 表面上最簡單（只要一個 get/set 屬性），但正確實作需要處理多個邊界情況。很多人一開始就用 Property Injection（因為看起來簡單），結果寫出脆弱的程式碼。

**脆弱版本 vs 健壯版本**：

```csharp
// 脆弱版本 —— 三大問題
public class Consumer
{
    public IDependency Dependency { get; set; }  // 可能是 null！

    public void DoSomething()
    {
        this.Dependency.DoStuff();  // NullReferenceException！
    }
}

// 問題 1：忘記設定 => NullReferenceException
var c = new Consumer();
c.DoSomething();  // 爆炸

// 問題 2：設成 null
c.Dependency = null;

// 問題 3：使用途中被換掉
c.Dependency = new ImplA();
c.DoSomething();
c.Dependency = new ImplB();  // 中途換掉，可能造成不一致
c.DoSomething();
```

```csharp
// 健壯版本 —— 具備三層防護
public class ContainerOptions
{
    // 1. Local Default 在建構時就設定好
    private IConstructorResolutionBehavior resolutionBehavior =
        new DefaultConstructorResolutionBehavior();

    public IConstructorResolutionBehavior ConstructorResolutionBehavior
    {
        get => this.resolutionBehavior;
        set
        {
            // 2. Guard Clause：不接受 null
            if (value == null)
                throw new ArgumentNullException(nameof(value));

            // 3. Popsicle Immutability：凍結後不可變
            if (this.Container.HasRegistrations)
                throw new InvalidOperationException(
                    "Cannot be changed after first registration.");

            this.resolutionBehavior = value;
        }
    }
}
```

```
    Property Injection 三層防護

    +------------------------------------+
    |        Property Setter             |
    |                                    |
    |  [1] null 檢查                     |
    |       value == null? => 丟例外     |
    |                                    |
    |  [2] Popsicle Immutability         |
    |       已凍結? => 丟例外            |
    |                                    |
    |  [3] 賦值                          |
    |       this.field = value;          |
    +------------------------------------+

    + 建構函式中設定 Local Default
    = 永遠不會是 null
```

**結論**：Property Injection 的「簡單」是假象。如果你不是在寫可重用函式庫，請直接用 Constructor Injection。

## 6. 本章小結與架構啟示

### 6.1 四大模式的定位

```
    DI 模式的重要性光譜

    必用                                      罕用
    |============|============|==========|===========|
    Composition   Constructor   Method     Property
    Root          Injection     Injection  Injection

    每個專案        90% 的       DDD Entity   只在寫
    都需要          依賴注入      和外掛場景    函式庫時
                   場景
```

### 6.2 架構啟示

1. **集中化是力量**：Composition Root 模式告訴我們，把物件組裝集中到一個地方，比分散到各處更好。這不只適用於 DI——任何「配置」、「接線」的工作都應該盡量集中。

2. **靜態宣告優於動態發現**：Constructor Injection 讓依賴關係成為編譯時期的宣告，而不是執行時期的驚喜。這讓系統更可預測、更容易理解。

3. **越少選擇越好**：四種模式中，90% 的時候用 Constructor Injection 就對了。在架構設計中，減少決策點反而能提升一致性和品質。

4. **Guard Clause 是契約精神的體現**：每個建構函式的 Guard Clause 就是在說「我需要這些東西才能正常工作」。這是 Design by Contract 思想的簡化實踐。

5. **不要追求「聰明」的設計**：Property Injection 看起來更「靈活」，但 Constructor Injection 更「可靠」。在架構選擇中，可靠永遠優先於靈活。

本章學會了「正確的做法」。下一章（第 5 章）將從反面出發，介紹 DI 的反模式（anti-patterns）——那些看起來像 DI 但實際上有害的做法，包括 Service Locator 和 Control Freak。了解什麼不該做，與了解什麼該做同樣重要。
