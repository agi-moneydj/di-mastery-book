## 章節精通指南 | Chapter Mastery Guide

> **書籍**：Dependency Injection Principles, Practices, and Patterns
> **作者**：Steven van Deursen, Mark Seemann
> **導讀視角**：資深系統架構師
> **程式碼範例**：C#

## 1. 概述 (Overview)

本章聚焦於三種在實務中套用依賴注入 (DI) 時最常遇到的**程式碼異味 (Code Smells)**：

1. **Constructor Over-injection** -- 建構式參數過多，暗示違反了單一職責原則 (SRP)。
2. **Abstract Factory 濫用** -- 不恰當地使用抽象工廠，導致洩漏抽象 (Leaky Abstraction) 與不必要的複雜度。
3. **Cyclic Dependencies** -- 循環依賴讓物件圖譜 (Object Graph) 無法被建構，根因通常是 SRP 違規。

> **Code Smell（程式碼異味）** 是什麼？用白話說，它就像家裡的瓦斯味道 -- 不一定代表已經爆炸了，但它在告訴你「這裡值得好好檢查一下」。跟 Anti-pattern（反模式）不同，Anti-pattern 是「已知一定會導致災難」的做法，而 Code Smell 是「可能有問題」的提示訊號。

本章的核心啟示：**Constructor Injection 不是問題的根源，它是「讓問題浮出水面」的偵測器。** 當你的建構式參數越來越多，不要怪 DI，而是要感謝 DI 讓你提早看見設計缺陷。

## 2. 學習目標 (Learning Objectives)

讀完本章後，你應該能夠：

- [ ] 辨識 Constructor Over-injection 的特徵，知道參數多少算「太多」
- [ ] 使用 **Facade Service 重構**，將自然聚合的依賴群組藏在單一抽象後方
- [ ] 使用 **Composite 模式** 搭配 Facade Service，進一步縮減依賴數量
- [ ] 理解 **Domain Event（領域事件）** 如何將通知機制泛型化，達到開放封閉原則 (OCP)
- [ ] 辨識 Abstract Factory 被濫用的兩種場景：(a) 用無參數工廠管理生命週期 (b) 用工廠選擇依賴
- [ ] 理解 **Leaky Abstraction** 的概念，特別是 `IDisposable` 出現在抽象層時的危害
- [ ] 使用 **Proxy 模式** 取代無參數的 Abstract Factory
- [ ] 使用 **Adapter 模式** 取代「依執行期資料選擇依賴」的 Abstract Factory
- [ ] 診斷 Dependency Cycle 的根因（通常是 SRP 違規），並透過拆分類別來打破循環
- [ ] 在萬不得已時，用 **Property Injection** 或 **Virtual Proxy** 打破循環

## 3. 章節大綱 (Outline)

### 3.1 Constructor Over-injection（建構式過度注入）

#### 3.1.1 辨識 Constructor Over-injection

當建構式的參數清單越長越多，代表這個類別承擔了太多責任。作者的經驗法則是**以四個參數為上限**，超過就該調查是否有 SRP 違規。

```
這像什麼？想像一位餐廳主廚，他同時負責洗菜、切菜、煮湯、烤肉、擺盤。
當你列出他需要的所有工具時（菜刀、鍋子、烤箱、盤子、調味料...），
清單越來越長 -- 這不是工具的問題，而是他一個人做太多事了。
```

原始的 `OrderService` 有五個依賴：

```csharp
public class OrderService : IOrderService
{
    public OrderService(
        IOrderRepository orderRepository,       // 存取訂單
        IMessageService messageService,          // 寄送回條
        IBillingSystem billingSystem,            // 通知會計
        ILocationService locationService,        // 找最近倉庫
        IInventoryManagement inventoryManagement) // 通知倉庫出貨
    {
        // ... 儲存依賴
    }

    public void ApproveOrder(Order order)
    {
        this.UpdateOrder(order);   // 更新資料庫
        this.Notify(order);        // 通知外部系統
    }
}
```

依賴關係圖：

```
                        OrderService
                 ┌────┬────┬────┬────┐
                 v    v    v    v    v
           IOrder  IMessage IBilling ILocation IInventory
           Repo    Service  System   Service   Mgmt

           *** 五個直接依賴 = SRP 違規的警訊 ***
```

> **重要觀念**：有人會想「那我用 Property Injection 把幾個參數移出建構式」。**千萬不要這樣做**！這只是把症狀藏起來，類別的複雜度並沒有降低。

#### 3.1.2 使用 Facade Service 重構

**Facade Service（門面服務）** 是把一組「自然聚合」的依賴，連同它們的行為一起，藏在一個新的抽象後方。

> 白話說：Facade Service 就像一個「部門經理」-- 外面的人只需要找經理交代事情，不需要認識部門裡的每個員工。

**第一步：找到自然聚合**

`ILocationService` 和 `IInventoryManagement` 總是一起協作，負責「訂單履行」：

```csharp
// 新的抽象 -- 隱藏了兩個底層依賴
public interface IOrderFulfillment
{
    void Fulfill(Order order);
}

// Facade Service 實作
public class OrderFulfillment : IOrderFulfillment
{
    private readonly ILocationService locationService;
    private readonly IInventoryManagement inventoryManagement;

    public OrderFulfillment(
        ILocationService locationService,
        IInventoryManagement inventoryManagement)
    {
        this.locationService = locationService;
        this.inventoryManagement = inventoryManagement;
    }

    public void Fulfill(Order order)
    {
        this.locationService.FindWarehouses(/* ... */);
        this.inventoryManagement.NotifyWarehouses(/* ... */);
    }
}
```

```
      重構前 (5 個依賴)                 重構後 (4 個依賴)
      ──────────────                   ──────────────
        OrderService                     OrderService
     ┌──┬──┬──┬──┐                    ┌──┬──┬──┐
     v  v  v  v  v                    v  v  v  v
    OR MS BS LS IM                   OR MS BS  IOrderFulfillment
                                                    |
                                              ┌─────┴─────┐
                                              v           v
                                         ILocation  IInventory
                                          Service      Mgmt

    OR = IOrderRepository
    MS = IMessageService
    BS = IBillingSystem
```

**第二步：統一通知介面 + Composite 模式**

剩下的 `IMessageService`、`IBillingSystem`、`IOrderFulfillment` 都在做「通知」的事情，可以用共同介面統一：

```csharp
public interface INotificationService
{
    void OrderApproved(Order order);
}
```

然後用 **Composite 模式** 將多個實作包裝成一個：

```csharp
public class CompositeNotificationService : INotificationService
{
    private readonly IEnumerable<INotificationService> services;

    public CompositeNotificationService(
        IEnumerable<INotificationService> services)
    {
        this.services = services;
    }

    public void OrderApproved(Order order)
    {
        foreach (var service in this.services)
        {
            service.OrderApproved(order);  // 逐一轉發
        }
    }
}
```

> **Composite 模式**白話說：就像群組訊息。你只發一次訊息，群組裡每個人都收到。消費者不需要知道群組裡有幾個人，也不需要知道他們是誰。

最終的 `OrderService` 只剩兩個依賴：

```csharp
public class OrderService : IOrderService
{
    private readonly IOrderRepository orderRepository;
    private readonly INotificationService notificationService;

    public OrderService(
        IOrderRepository orderRepository,
        INotificationService notificationService)
    {
        this.orderRepository = orderRepository;
        this.notificationService = notificationService;
    }

    public void ApproveOrder(Order order)
    {
        this.UpdateOrder(order);
        this.notificationService.OrderApproved(order);
    }
}
```

最終架構：

```
                    OrderService
                   ┌─────┴─────┐
                   v           v
            IOrderRepo   INotificationService
                               |
                         Composite (0..*)
                        ┌──────┼──────┐
                        v      v      v
                   Receipt  Billing  Fulfillment
                   Sender   Notifier
                                      ┌────┴────┐
                                      v         v
                                 ILocation  IInventory
                                  Service     Mgmt
```

Composition Root 的組裝方式：

```csharp
var repository = new SqlOrderRepository(connectionString);

var notificationService = new CompositeNotificationService(
    new INotificationService[]
    {
        new OrderApprovedReceiptSender(messageService),
        new AccountingNotifier(billingSystem),
        new OrderFulfillment(locationService, inventoryManagement)
    });

var orderService = new OrderService(repository, notificationService);
```

#### 3.1.3 使用 Domain Events 重構

當系統逐漸成長，`INotificationService` 的方法會越來越多（`OrderApproved`、`OrderCancelled`、`OrderShipped`...），這會違反**介面隔離原則 (ISP)**。解法是把每個事件提升為獨立的型別：

```csharp
// 領域事件 -- 簡單的資料物件
public class OrderApproved
{
    public readonly Guid OrderId;
    public OrderApproved(Guid orderId)
    {
        this.OrderId = orderId;
    }
}

public class OrderCancelled
{
    public readonly Guid OrderId;
    public OrderCancelled(Guid orderId)
    {
        this.OrderId = orderId;
    }
}
```

> **Domain Event（領域事件）** 白話說：就像公司裡的「廣播系統」。當「訂單核准」這件事發生了，不是由訂單處理人員一個一個去通知會計、倉庫、客服，而是廣播一句「訂單 #12345 已核准」，有興趣的部門自己去處理。

然後定義泛型的事件處理器介面：

```csharp
public interface IEventHandler<TEvent>
{
    void Handle(TEvent e);
}
```

每個處理器實作針對特定事件型別：

```csharp
public class OrderFulfillment : IEventHandler<OrderApproved>
{
    private readonly ILocationService locationService;
    private readonly IInventoryManagement inventoryManagement;

    public OrderFulfillment(
        ILocationService locationService,
        IInventoryManagement inventoryManagement)
    {
        this.locationService = locationService;
        this.inventoryManagement = inventoryManagement;
    }

    public void Handle(OrderApproved e)
    {
        this.locationService.FindWarehouses(/* ... */);
        this.inventoryManagement.NotifyWarehouses(/* ... */);
    }
}
```

泛型的 Composite：

```csharp
public class CompositeEventHandler<TEvent> : IEventHandler<TEvent>
{
    private readonly IEnumerable<IEventHandler<TEvent>> handlers;

    public CompositeEventHandler(
        IEnumerable<IEventHandler<TEvent>> handlers)
    {
        this.handlers = handlers;
    }

    public void Handle(TEvent e)
    {
        foreach (var handler in this.handlers)
        {
            handler.Handle(e);
        }
    }
}
```

```
  Domain Events 架構全貌
  ========================

  OrderService
       |
       | ApproveOrder()
       v
  IEventHandler<OrderApproved>
       |
       v
  CompositeEventHandler<OrderApproved>  ─── "廣播站"
       |
       ├──> OrderApprovedReceiptSender   ─── 寄回條給客戶
       ├──> AccountingNotifier           ─── 通知會計系統
       └──> OrderFulfillment             ─── 安排倉庫出貨

  *** 新增 handler 只需要：
      1. 建立新類別，實作 IEventHandler<OrderApproved>
      2. 在 Composition Root 註冊
      不需要改動任何既有程式碼 ── 完美的 OCP! ***
```

**Domain Events 的額外好處 -- 可靠訊息 (Reliable Messaging)**：

將事件序列化後放入持久化佇列（如 MSMQ、Azure Queue），可以實現：
- 每個 handler 在獨立交易中執行
- 失敗的 handler 可以單獨重試
- 多個 handler 可以平行處理
- 系統崩潰後可以從佇列恢復

### 3.2 Abstract Factory 濫用

#### 3.2.1 用無參數工廠管理生命週期 -- Leaky Abstraction

**問題場景**：開發者建立一個無參數的工廠來讓消費者控制依賴的生命週期。

```csharp
// 有問題的做法
public interface IProductRepositoryFactory
{
    IProductRepository Create();  // 無參數 = 紅旗!
}

public class HomeController : Controller
{
    private readonly IProductRepositoryFactory factory;

    public HomeController(IProductRepositoryFactory factory)
    {
        this.factory = factory;
    }

    public ViewResult Index()
    {
        // 消費者被迫管理依賴的生命週期 -- 這不是它的責任!
        using (IProductRepository repository = this.factory.Create())
        {
            var products = repository.GetFeaturedProducts();
            return this.View(products);
        }
    }
}
```

```
  問題的互動流程
  ===============

  HomeController ──── ProductRepositoryFactory
       |                       |
       | Create()              |
       |<──────────────────────|
       |    new SqlProductRepository()
       |                       |
       | GetFeaturedProducts() |
       |──────────────────────>|
       |                       |
       | Dispose()             |   <── 消費者不該管這件事!
       |──────────────────────>|
```

> **Leaky Abstraction（洩漏抽象）** 白話說：就像你租車時，租車公司要求你自己換機油、自己修引擎。你只是想開車（使用服務），不需要知道引擎的保養細節（生命週期管理）。當抽象層強迫消費者處理實作細節時，就是在「洩漏」。

**核心原則**：

> 概念上，一個服務抽象只存在一個實例。消費者不應該操心可能存在多個依賴實例這件事。

**為什麼 `IDisposable` 不該出現在抽象層？**

- 並非所有實作都需要處置資源（例如記憶體內的 Mock 就不需要）
- 把 `IDisposable` 放在抽象層，等於假設「所有實作都需要資源清理」
- 生命週期管理應該集中在 **Composition Root**，而不是散佈在應用程式碼中

**解法：用 Proxy 模式取代工廠**

```csharp
// 乾淨的消費者 -- 只依賴 IProductRepository
public class HomeController : Controller
{
    private readonly IProductRepository repository;

    public HomeController(IProductRepository repository)
    {
        this.repository = repository;
    }

    public ViewResult Index()
    {
        var products = this.repository.GetFeaturedProducts();
        return this.View(products);  // 乾淨、簡單
    }
}

// Proxy 封裝了生命週期管理（放在 Composition Root）
public class SqlProductRepositoryProxy : IProductRepository
{
    private readonly string connectionString;

    public SqlProductRepositoryProxy(string connectionString)
    {
        this.connectionString = connectionString;
    }

    public IEnumerable<Product> GetFeaturedProducts()
    {
        // 生命週期管理被封裝在 Proxy 內部
        using (var repository = this.Create())
        {
            return repository.GetFeaturedProducts();
        }
    }

    private SqlProductRepository Create()
    {
        return new SqlProductRepository(this.connectionString);
    }
}
```

```
  Proxy 模式的互動流程（大幅簡化）
  ================================

  HomeController ───── SqlProductRepositoryProxy
       |                        |
       | GetFeaturedProducts()  |
       |───────────────────────>|
       |                        |── new SqlProductRepository()
       |                        |── GetFeaturedProducts()
       |                        |── Dispose()  (內部處理!)
       |<───────────────────────|
       |     Products           |
```

> **Proxy 模式**白話說：就像飯店的禮賓服務。你（消費者）只需要說「幫我叫計程車」，禮賓（Proxy）負責去哪裡叫、怎麼叫、叫完怎麼付款。你不需要知道這些細節。

#### 3.2.2 用 Abstract Factory 依執行期資料選擇依賴 -- 用 Adapter 取代

**問題場景**：線上地圖網站讓使用者選擇路線類型（最短、最快、最美），需要依據執行期的選擇來取得不同演算法。

```csharp
// 有問題的做法 -- 使用 Abstract Factory
public interface IRouteAlgorithmFactory
{
    IRouteAlgorithm CreateAlgorithm(RouteType routeType);
}

public class RouteController : Controller
{
    private readonly IRouteAlgorithmFactory factory;

    public RouteController(IRouteAlgorithmFactory factory)
    {
        this.factory = factory;
    }

    public ViewResult GetRoute(RouteSpecification spec, RouteType routeType)
    {
        // 看起來只有一個依賴，實際上有兩個!
        IRouteAlgorithm algorithm = this.factory.CreateAlgorithm(routeType);
        var route = algorithm.CalculateRoute(spec);
        // ...
    }
}
```

**問題分析**：

```
  表面上的依賴圖                    實際上的依賴圖
  ==============                   ==============

  RouteController                  RouteController
       |                              |         \
       v                              v          v
  IRouteAlgorithm              IRouteAlgorithm  IRouteAlgorithm
      Factory                      Factory
                                      |
                                      v (creates/returns)
                                 IRouteAlgorithm

  看起來 1 個依賴                  實際上有 2 個依賴!
```

> **為什麼多一個依賴很嚴重？** 因為寫單元測試時，你需要同時 mock `IRouteAlgorithmFactory` 和 `IRouteAlgorithm`，測試邏輯變得更複雜。更重要的是，這違反了**最少知識原則** -- 服務抽象不應該在定義中暴露其他服務抽象。

**解法：用 Adapter 模式合併**

```csharp
// 新的抽象 -- 同時接受規格和路線類型
public interface IRouteCalculator
{
    RouteResult Calculate(RouteSpecification spec, RouteType routeType);
}

// 簡化後的 Controller -- 只有一個真正的依賴
public class RouteController : Controller
{
    private readonly IRouteCalculator calculator;

    public RouteController(IRouteCalculator calculator)
    {
        this.calculator = calculator;
    }

    public ViewResult GetRoute(RouteSpecification spec, RouteType routeType)
    {
        var route = this.calculator.Calculate(spec, routeType);
        var vm = new RouteViewModel { /* ... */ };
        return this.View(vm);
    }
}

// Adapter 內部封裝了「選擇演算法」的邏輯
public class RouteCalculator : IRouteCalculator
{
    private readonly IDictionary<RouteType, IRouteAlgorithm> algorithms;

    public RouteCalculator(
        IDictionary<RouteType, IRouteAlgorithm> algorithms)
    {
        this.algorithms = algorithms;
    }

    public RouteResult Calculate(RouteSpecification spec, RouteType type)
    {
        return this.algorithms[type].CalculateRoute(spec);
    }
}
```

Composition Root 的組裝：

```csharp
var algorithms = new Dictionary<RouteType, IRouteAlgorithm>
{
    { RouteType.Shortest, new ShortestRouteAlgorithm() },
    { RouteType.Fastest,  new FastestRouteAlgorithm() },
    { RouteType.Scenic,   new ScenicRouteAlgorithm() }
};

new RouteController(new RouteCalculator(algorithms));
```

```
  重構前                          重構後
  ======                         ======

  RouteController                RouteController
     |         \                      |
     v          v                     v
  IRouteAlg   IRouteAlg         IRouteCalculator
  Factory                            |
                                RouteCalculator
                                     |
                                     v
                              IRouteAlgorithm (internal)
```

### 3.3 Cyclic Dependencies（循環依賴）

#### 3.3.1 經典的雞與蛋問題

```csharp
public class Chicken : IChicken
{
    public Chicken(IEgg egg) { }   // 雞需要蛋
}

public class Egg : IEgg
{
    public Egg(IChicken chicken) { } // 蛋需要雞
}

// 嘗試建構...
new Chicken(
    new Egg(
        ???   // 要先有 Chicken 才能建立 Egg，
              // 但要先有 Egg 才能建立 Chicken
    )
);
```

```
    循環依賴示意圖
    ==============

    Chicken ──(needs)──> IEgg
       ^                   |
       |              Egg (implements IEgg)
       |                   |
       └──(needs)──────────┘
             IChicken

    *** 兩者互相需要對方先存在，這個圖無法被滿足 ***
```

#### 3.3.2 實務範例：稽核追蹤引發的循環

Mary 開發一個稽核追蹤功能。三個相關類別：

```
  SqlAuditTrailAppender（稽核記錄器）
    依賴：IUserContext（取得當前使用者）

  AspNetUserContextAdapter（實作 IUserContext）
    依賴：IUserRepository（從資料庫取得使用者資訊）

  SqlUserRepository（實作 IUserRepository）
    依賴：IAuditTrailAppender（寫入稽核記錄）  <── 新增的依賴造成循環!
```

```
    循環依賴的完整圖譜
    ==================

    AspNetUserContext ──(uses)──> IUserRepository
    Adapter                          |
         ^                     SqlUserRepository
         |                           |
    IUserContext              (uses) IAuditTrail
         |                         Appender
    SqlAuditTrail                    |
    Appender <───────────────────────┘

    *** 繞了一圈又回到起點 ***
```

**方法呼叫圖**（注意：雖然物件圖是循環的，但方法呼叫並不是遞迴的）：

```
  UserService.UpdateMailAddress(userId, newMailAddress)
    └──> SqlUserRepository.Update(user)
           └──> SqlAuditTrailAppender.Append(entity)
                  └──> AspNetUserContextAdapter.CurrentUser
                         └──> SqlUserRepository.GetByName(name)
                                (這裡結束，沒有再呼叫 Append)
```

#### 3.3.3 透過修正 SRP 違規來打破循環

根因分析：`IUserRepository` 太胖了！它同時包含 `Update`、`GetById`、`GetByName` 三個方法，但並非所有消費者都需要全部方法。

**解法：拆分介面**

```csharp
// 拆出來的新介面 -- 只負責「依名稱查詢使用者」
public interface IUserByNameRetriever
{
    User GetByName(string name);
}

public class SqlUserByNameRetriever : IUserByNameRetriever
{
    public SqlUserByNameRetriever(CommerceContext context)
    {
        this.context = context;
    }

    public User GetByName(string name) { /* ... */ }
    // 注意：不依賴 IAuditTrailAppender!
}

// 瘦身後的 IUserRepository
public interface IUserRepository
{
    void Update(User user);
    User GetById(Guid id);
}

public class SqlUserRepository : IUserRepository
{
    public SqlUserRepository(
        CommerceContext context,
        IAuditTrailAppender appender)
    {
        this.context = context;
        this.appender = appender;
    }

    public void Update(User user) { /* 包含稽核邏輯 */ }
    public User GetById(Guid id) { /* ... */ }
}
```

```
    重構前（循環）                     重構後（無循環）
    ============                     ==============

    AspNetUserContext                 AspNetUserContext
    Adapter                          Adapter
       |                                |
    IUserRepository                 IUserByName       IUserRepository
       |                            Retriever              |
    SqlUserRepository                  |              SqlUserRepository
       |                          SqlUserByName            |
    IAuditTrail                    Retriever          IAuditTrail
    Appender                          |               Appender
       |                         CommerceContext          |
    SqlAuditTrail                 (無循環!)         SqlAuditTrail
    Appender                                         Appender
       |                                                  |
    IUserContext ─> 回到起點 (循環!)                  IUserContext
                                                          |
                                                    AspNetUserContext
                                                    Adapter
                                                    (不再依賴 IUserRepository
                                                     而是 IUserByNameRetriever)

                                                    *** 循環被打破! ***
```

打破循環後的 Composition Root：

```csharp
var userContext = new AspNetUserContextAdapter(
    new SqlUserByNameRetriever(context));  // 不再依賴完整的 Repository

var appender = new SqlAuditTrailAppender(
    userContext, context, timeProvider);

var repository = new SqlUserRepository(context, appender);
// 一切都能正常編譯了!
```

#### 3.3.4 打破循環的策略總覽

| 策略 | 說明 | 優先度 |
|------|------|--------|
| **拆分類別** | 最常見也最推薦。將過胖的類別依 ISP/SRP 拆成更小的類別 | 最高 |
| **.NET Events** | 將「通知」改為事件機制，特別適用於一方只呼叫 `void` 方法的情況 | 中 |
| **Property Injection** | 最後手段。建立不完整的物件，事後再注入缺少的依賴 | 最低（治標不治本） |

#### 3.3.5 最後手段：Property Injection / Virtual Proxy

```csharp
// 方式一：直接在原類別使用 Property Injection
var appender = new SqlAuditTrailAppender(context, timeProvider);
// 此時 appender 是「部分初始化」的狀態

var repository = new SqlUserRepository(context, appender);

var userContext = new AspNetUserContextAdapter(
    new SqlUserByNameRetriever(context));

appender.UserContext = userContext;  // 事後注入，完成初始化


// 方式二：Virtual Proxy -- 不動原類別
var lazyAppender = new LazyAuditTrailAppender();  // 空殼

var repository = new SqlUserRepository(context, lazyAppender);

var userContext = new AspNetUserContextAdapter(
    new SqlUserByNameRetriever(context));

lazyAppender.Appender = new SqlAuditTrailAppender(
    userContext, context, timeProvider);  // 注入真正的實作
```

```csharp
// Virtual Proxy 實作
public class LazyAuditTrailAppender : IAuditTrailAppender
{
    public IAuditTrailAppender Appender { get; set; }

    public void Append(Entity changedEntity)
    {
        if (this.Appender == null)
            throw new InvalidOperationException("Appender was not set.");

        this.Appender.Append(changedEntity);  // 轉發給真正的實作
    }
}
```

> **Virtual Proxy** 白話說：就像一張「代理人委託書」。委託書上先蓋好章（建立 Proxy），等本人出現後再填入本人的資料（注入真正的依賴）。在本人出現前，如果有人拿著委託書去辦事，就會被拒絕（拋出例外）。

## 4. 關鍵框架與公式 (Key Frameworks)

### 4.1 Code Smell 判斷決策樹

```
  建構式參數是否超過 4 個？
  ├── 是 ──> Constructor Over-injection
  │         ├── 是否有「自然聚合」的依賴群組？
  │         │   ├── 是 ──> 使用 Facade Service 重構
  │         │   └── 否 ──> 考慮 Domain Events 泛型化
  │         └── 通知/事件類操作是否越來越多？
  │             └── 是 ──> 引入 IEventHandler<T> + Composite
  └── 否 ──> 暫時安全

  是否有注入 Abstract Factory？
  ├── Factory 的 Create 方法無參數？
  │   └── 是 ──> Leaky Abstraction! 改用 Proxy 模式
  └── Factory 接受執行期參數來「選擇」回傳哪個實作？
      └── 是 ──> 改用 Adapter，將 Factory + 產品合併成單一抽象

  程式碼無法編譯（循環參考）？
  ├── 是 ──> Cyclic Dependency
  │         ├── 首先：找出 SRP 違規，拆分類別（推薦）
  │         ├── 其次：使用 .NET Events 解耦
  │         └── 最後手段：Property Injection / Virtual Proxy
  └── 否 ──> 沒有循環問題
```

### 4.2 Anti-pattern vs Code Smell 對照

| 特性 | Anti-pattern (Ch5) | Code Smell (Ch6) |
|------|-------------------|-------------------|
| **確定性** | 確定會產生負面後果 | 可能有問題，值得調查 |
| **嚴重程度** | 高 -- 必須修正 | 中 -- 需要進一步判斷 |
| **範例** | Control Freak, Service Locator | Constructor Over-injection, Factory 濫用 |

### 4.3 SOLID 原則在本章的體現

| 原則 | 本章的體現 |
|------|-----------|
| **SRP** | Constructor Over-injection 是 SRP 違規的信號；Cyclic Dependency 的根因通常也是 SRP 違規 |
| **OCP** | Domain Events + Composite 讓新增 handler 不需修改既有程式碼 |
| **LSP** | Proxy 和 Adapter 實作必須能替代原本的抽象 |
| **ISP** | 過胖的介面（如 `INotificationService` 方法越來越多）需要拆分；`IUserRepository` 拆分為更小的介面 |
| **DIP** | 抽象由消費者定義（`IRouteCalculator` 而非 `IRouteAlgorithmFactory`）；抽象不應暴露其他服務抽象 |

### 4.4 設計模式速查表

| 模式 | 本章的用途 | 一句話描述 |
|------|-----------|-----------|
| **Facade** | 把多個依賴藏在單一抽象後方 | 部門經理 -- 外人只找經理 |
| **Composite** | 讓多個通知服務看起來像一個 | 群組訊息 -- 發一次，所有人收到 |
| **Proxy** | 封裝生命週期管理，取代無參數工廠 | 飯店禮賓 -- 幫你處理所有細節 |
| **Adapter** | 合併 Factory + Product 為單一介面 | 萬用插頭 -- 把兩種不同接口統一 |
| **Virtual Proxy** | 打破循環依賴的最後手段 | 代理人委託書 -- 先蓋章後補人 |

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 5.1 難點一：從 Facade Service 到 Composite 到 Domain Events 的漸進演化

**為什麼難**：這三步重構是連貫的，每一步都建立在前一步的基礎上。讀者容易在中途迷失方向，不清楚「為什麼前面那步不夠好，還要繼續改」。

**突破方式**：把它想成餐廳的進化 --

1. **Facade Service**：原本主廚一個人做五道菜（5 個依賴），現在把「甜點」外包給甜點師傅（Facade 封裝 2 個依賴 = 1 個新抽象）。主廚的負擔從 5 降到 4。

2. **Composite**：發現剩下的三道菜都是「配菜」性質，統一由「配菜組」處理。主廚只需要跟配菜組長說一聲。負擔從 4 降到 2。

3. **Domain Events**：餐廳越來越大，配菜組的菜色越來越多（介面方法越來越多）。改成「點菜系統」-- 主廚只需要發一張單子（事件），廚房裡誰看到跟自己有關的單子就動手做。新菜色 = 新增一個廚師，不影響任何既有流程。

```
  演化歷程摘要
  ===========

  第一步              第二步                第三步
  Facade Service      + Composite           + Domain Events
  ──────────          ──────────            ──────────
  5 -> 4 依賴         4 -> 2 依賴           固定 2 依賴
  手動聚合            統一介面              泛型介面
  特定名稱            特定名稱              通用模板
  新增需改介面        新增需改介面          新增不改任何東西
```

### 5.2 難點二：Abstract Factory 到底什麼時候該用、什麼時候不該用

**為什麼難**：Abstract Factory 是 GoF 經典模式，很多人學了就想用。但本章說「在 DI 的情境下，工廠常常被濫用」，這跟傳統教科書的說法產生衝突。

**判斷準則**：

```
  問自己這個問題：「工廠的 Create 方法是否需要執行期值作為參數？」

  不需要 ──> 濫用! 用 Proxy 取代
  │
  └── 即使需要，再問：「工廠回傳的是否是另一個服務抽象？」
      │
      是 ──> 可能濫用! 服務抽象不應暴露其他服務抽象
      │      考慮用 Adapter 合併成單一抽象
      │
      否 ──> 合法使用（例如：工廠回傳的是資料物件而非服務）
```

**核心原則**：
- 消費者不應該知道依賴可能有多個實例
- 消費者不應該管理依賴的生命週期
- 服務抽象不應該在定義中暴露其他服務抽象

### 5.3 難點三：循環依賴的根因分析與 ISP/SRP 的關聯

**為什麼難**：表面上看，循環依賴是「A 需要 B，B 需要 A」的簡單問題。但實際根因往往藏在「其中一個類別做了太多事」。從看到循環到找出 SRP 違規，需要分析物件圖和方法呼叫圖兩個維度。

**分析步驟**：

```
  Step 1: 畫出物件依賴圖，標示循環路徑
  Step 2: 畫出方法呼叫圖，確認是否遞迴（遞迴 = 更嚴重的問題）
  Step 3: 檢查循環中每個類別的「介面方法數量」
  Step 4: 找到方法最多的那個介面 -- 它最可能是 SRP 違規的元凶
  Step 5: 把該介面拆分，讓消費者只依賴它們真正需要的那部分
  Step 6: 驗證拆分後的圖是否已經無循環（acyclic）
```

在 Mary 的例子中，`IUserRepository` 有 3 個方法（`Update`、`GetById`、`GetByName`），但 `AspNetUserContextAdapter` 只需要 `GetByName`。拆出 `IUserByNameRetriever` 後，循環就消失了。

**記住**：一個類別的方法越多，它越可能違反 SRP；介面越胖，越難實作 Proxy、Decorator、Test Double。

## 6. 本章小結與架構啟示

### 核心收穫

1. **Constructor Injection 是設計品質的溫度計**。參數越多 = 溫度越高 = 該看醫生了。不要因為溫度計顯示發燒就把溫度計丟掉（改用 Property Injection 或 Service Locator）。

2. **Facade Service + Composite + Domain Events 是一套漸進式的重構路徑**。從「減少直接依賴」到「統一通知介面」到「泛型化事件處理」，每一步都在提升系統的可維護性和擴展性。

3. **Abstract Factory 在 DI 的世界裡經常被濫用**。在引入工廠之前，先問自己：「我是因為某個具體實作才需要工廠，還是概念上真的需要執行期建立？」

4. **循環依賴的根因幾乎都是 SRP 違規**。解法不是用 Property Injection 硬接（那只是止痛藥），而是用 ISP 將過胖的介面拆分成更聚焦的小介面。

5. **服務抽象不應暴露其他服務抽象**。這條規則與最少知識原則相關 -- 你的消費者知道得越少，你的架構彈性越大。

### 架構師的行動清單

```
  在 Code Review 時檢查：
  ┌─────────────────────────────────────────────────────────┐
  │ [ ] 是否有建構式超過 4 個參數的類別？                      │
  │ [ ] 是否有介面繼承 IDisposable？                          │
  │ [ ] 是否有無參數 Create 方法的 Abstract Factory？          │
  │ [ ] 是否有 Factory 回傳服務抽象（而非資料物件）？           │
  │ [ ] 是否有一個介面的方法數量持續增長？                      │
  │ [ ] 是否有循環依賴（編譯錯誤或執行期 StackOverflow）？     │
  └─────────────────────────────────────────────────────────┘
```
