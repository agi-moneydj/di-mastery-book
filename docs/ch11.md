## 1. 概述 (Overview)

本章是第 10 章「以設計實踐 AOP」的延續。第 10 章展示了用純粹 SOLID 設計原則來實作 AOP（Aspect-Oriented Programming，面向切面程式設計），而本章則切換到**工具導向**的角度，探討兩種常見的工具型 AOP 技術：

| 技術 | 代表工具 | 作者立場 |
|------|----------|----------|
| **Dynamic Interception**（動態攔截） | Castle Dynamic Proxy | 可接受的臨時方案 |
| **Compile-time Weaving**（編譯期織入） | PostSharp | **反模式（Anti-pattern）** |

核心訊息非常明確：

```
優先順序：SOLID 設計 AOP  >  Dynamic Interception  >  Compile-time Weaving（避免使用）
```

> **白話解釋**：想像你有一棟房子（你的程式碼），需要加裝保全系統（橫切關注點）。最好的做法是在設計藍圖階段就規劃好（SOLID 設計 AOP）。次好的方案是在房子蓋好後外掛一套獨立的保全系統（Dynamic Interception）。最差的做法是把保全線路直接灌進牆壁裡，日後想拆掉或更換都極度困難（Compile-time Weaving）。

---

## 2. 學習目標 (Learning Objectives)

完成本章學習後，你應當能夠：

1. **理解 Dynamic Interception 的運作原理**——工具在執行期自動產生 Decorator，省去手動撰寫重複模板程式碼的工夫
2. **使用 Castle Dynamic Proxy** 在 Composition Root 中套用 Interceptor
3. **精確列舉 Dynamic Interception 的四大缺點**——喪失編譯期檢查、與工具強耦合、無法普遍適用、未解決底層設計問題
4. **理解 Compile-time Weaving 的運作機制**——後置編譯器修改 IL 組件
5. **解釋為何 Compile-time Weaving 是 DI 的反模式**——無法使用 Constructor Injection、編譯期耦合破壞可測試性
6. **在真實專案中做出正確的 AOP 策略選擇**

## 3. 章節大綱 (Outline)

### 3.1 從重複程式碼看 Dynamic Interception 的動機

第 10 章的 `CircuitBreakerProductRepositoryDecorator` 違反了 DRY（Don't Repeat Yourself）原則——每個方法的 Circuit Breaker 模板幾乎一模一樣：

```csharp
// 這段「模板」在 Delete、Insert 等方法中不斷重複
public void Delete(Product product)
{
    this.breaker.Guard();       // 1. 前置檢查
    try
    {
        this.decoratee.Delete(product); // 2. 呼叫被裝飾者
        this.breaker.Succeed();         // 3. 成功通知
    }
    catch (Exception ex)
    {
        this.breaker.Trip(ex);          // 4. 失敗跳脫
        throw;
    }
}
```

手動建立 Decorator 的步驟極其機械化：(1) 建新類別、(2) 實作介面、(3) 對每個方法套用模板。既然如此，為什麼不讓工具自動完成？

> **關鍵字：DRY 原則** — 「不要重複自己」。同一段邏輯只應存在一處。如果你發現自己在多個地方寫了幾乎一樣的程式碼，這就是違反 DRY 的訊號。

### 3.2 Dynamic Interception 的運作原理

Dynamic Interception 利用 .NET 在執行期動態產生型別（emit types）的能力，自動產生 Decorator 類別。這個 Decorator 沒有對應的原始碼檔案，而是直接從抽象模型編譯而成。

```
+--------------------------------------------------+
|            Dynamic Interception 流程               |
+--------------------------------------------------+

  手動撰寫                     執行期自動產生
  ┌──────────────┐            ┌──────────────────────────────┐
  │  Interceptor │            │  RuntimeGeneratedDecorator   │
  │  (攔截器)     │            │  : IProductRepository        │
  │              │            │                              │
  │  Intercept(  │◄───────────│  Insert(Product p)           │
  │    invocation│            │  {                           │
  │  ) {         │            │    var inv = new Invocation();│
  │    // 前置   │            │    inv.Proceed = ()          │
  │    Proceed() │            │      => target.Insert(p);    │
  │    // 後置   │            │    interceptor.Intercept(inv);│
  │  }           │            │  }                           │
  └──────────────┘            └──────────────────────────────┘
                                        │
                                        ▼
                              ┌──────────────────┐
                              │ SomeRepository   │
                              │ (真正的實作)      │
                              │ : IProductRepo   │
                              └──────────────────┘
```

**方法呼叫流程：**

```
┌────────┐  呼叫方法  ┌─────────────┐  Intercept()  ┌───────────┐  Proceed()  ┌────────────┐
│ Client │──────────►│ Generated   │──────────────►│ Logging   │───────────►│ Some       │
│        │           │ Decorator   │               │ Aspect    │            │ Repository │
│        │◄──────────│             │◄──────────────│           │◄───────────│            │
└────────┘  回傳值   └─────────────┘   回傳值       └───────────┘   回傳值    └────────────┘
```

> **白話比喻**：Dynamic Interception 就像一位「萬能秘書」。你告訴她一套處理來電的 SOP（Interceptor），她就會自動幫每一通進線電話（方法呼叫）套用這套 SOP，不管來電是找業務部還是技術部。你不需要為每個部門各寫一套接線流程。

### 3.3 Castle Dynamic Proxy 實作範例

#### 3.3.1 實作 Circuit Breaker Interceptor

```csharp
public class CircuitBreakerInterceptor
    : Castle.DynamicProxy.IInterceptor  // 實作 Castle 的 IInterceptor 介面
{
    private readonly ICircuitBreaker breaker;

    // 使用 Constructor Injection —— Interceptor 也是 DI 友善的
    public CircuitBreakerInterceptor(ICircuitBreaker breaker)
    {
        this.breaker = breaker;
    }

    public void Intercept(IInvocation invocation)
    {
        this.breaker.Guard();           // 前置：斷路器檢查
        try
        {
            invocation.Proceed();       // 核心：讓呼叫繼續往下走
            this.breaker.Succeed();     // 後置（成功）：通知斷路器
        }
        catch (Exception ex)
        {
            this.breaker.Trip(ex);      // 後置（失敗）：觸發斷路器
            throw;
        }
    }
}
```

**關鍵差異**：與手動 Decorator 不同，這裡不直接呼叫 `decoratee.Delete(product)` 這種具體方法，而是透過 `invocation.Proceed()` 進行通用轉發。`IInvocation` 介面提供了方法名稱、參數值等執行期資訊。

> **名詞解釋：IInvocation（調用物件）** — 代表「正在進行中的方法呼叫」的抽象。就像法院裡的「案件卷宗」，包含案件的所有資訊（方法名稱、參數、回傳值），你可以查閱它，也可以決定是否讓案件繼續審理（Proceed）。

#### 3.3.2 在 Composition Root 中組裝

```csharp
// 建立 Castle 的 ProxyGenerator（建議整個應用程式生命週期只建一次）
var generator = new Castle.DynamicProxy.ProxyGenerator();

// 準備 Circuit Breaker 的依賴
var timeout = TimeSpan.FromMinutes(1);
var breaker = new CircuitBreaker(timeout);
var interceptor = new CircuitBreakerInterceptor(breaker);

// 建立真正的 Repository
var wcfRepository = new WcfProductRepository();

// 請 Castle 產生 Decorator（Castle 稱之為 Proxy）
IProductRepository repository = generator
    .CreateInterfaceProxyWithTarget<IProductRepository>(
        wcfRepository,      // 被裝飾的目標
        interceptor);       // 要套用的攔截器

// repository 現在是一個被 Circuit Breaker 保護的 IProductRepository
```

> **重點提醒**：`ProxyGenerator` 應該建立一次並快取，因為它在背後進行的型別產生是昂貴的操作。即使使用了 Castle，你依然可以保持 Pure DI——Castle 只負責產生 Decorator，物件圖的組裝仍在你的掌控之中。

### 3.4 Dynamic Interception 的優缺點分析

#### 優點（與 SOLID 設計 AOP 的共通之處）

| 優點 | 說明 |
|------|------|
| 處理橫切關注點 | 只要程式碼面向抽象（Abstraction）就能套用 |
| DI 友善 | Interceptor 可使用 Constructor Injection，易於測試 |
| 集中於 Composition Root | 避免程式碼重複，不同應用程式可選擇性套用不同切面 |

#### 四大缺點

```
┌─────────────────────────────────────────────────────────────────────┐
│              Dynamic Interception 的四大缺點                         │
├─────────────────────────┬───────────────────────────────────────────┤
│ 1. 喪失編譯期支援       │ 大量反射（reflection）操作，需手動處理      │
│    (Loss of compile-    │ 型別轉換與裝箱。Interceptor 比 Decorator   │
│     time support)       │ 更複雜、更難讀、更難維護。                  │
├─────────────────────────┼───────────────────────────────────────────┤
│ 2. 與工具強耦合          │ Interceptor 直接依賴 Castle 的介面         │
│    (Coupled to tooling) │ （IInterceptor、IInvocation）。引入額外     │
│                         │ 的外部程式庫依賴與學習成本。               │
├─────────────────────────┼───────────────────────────────────────────┤
│ 3. 無法普遍適用          │ 只能攔截介面上的方法（virtual/abstract）。  │
│    (Not universally     │ 私有方法無法被攔截。                       │
│     applicable)         │                                           │
├─────────────────────────┼───────────────────────────────────────────┤
│ 4. 未解決底層設計問題     │ 只解決了橫切關注點的套用問題，系統整體      │
│    (Doesn't fix design  │ 可維護性僅略有提升，遠不如 SOLID 設計。     │
│     problems)           │                                           │
└─────────────────────────┴───────────────────────────────────────────┘
```

> **白話比喻（喪失編譯期支援）**：手動 Decorator 就像一份簽好的正式合約，條款清清楚楚、型別明明白白。Interceptor 則像是一份「空白委任書」——什麼方法都能代理，但如果參數型別搞錯了，要到執行的時候才會爆炸。

### 3.5 Compile-time Weaving（編譯期織入）

#### 3.5.1 運作原理

Compile-time Weaving 修改正常的編譯流程。後置編譯器（Post-compiler）讀取 C# 編譯器產出的 IL 組件，將切面程式碼「織入」原始 IL 中，再寫回磁碟覆蓋原始組件。

```
  編譯期織入流程
  ═══════════════

  ┌──────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
  │ C# 原始碼 │────►│  C# 編譯器    │────►│ 原始 IL 組件  │────►│  後置編譯器   │
  └──────────┘     └──────────────┘     └──────────────┘     │  (PostSharp) │
                                                              └──────┬───────┘
  ┌──────────────┐                                                   │
  │ 切面類別      │───────────────────────────────────────────────────┘
  │ (Aspect)     │                        ▼
  └──────────────┘                  ┌──────────────┐
                                    │ 修改後 IL 組件│
                                    │ (切面已織入)  │
                                    └──────────────┘
```

**織入前後對比：**

```csharp
// ===== 織入前 =====
public class SomeRepository : IRepo
{
    public void Insert(Product p)
    {
        // 核心操作
    }
}

// ===== 織入後（IL 層面的等效程式碼）=====
public class SomeRepository : IRepo
{
    // 切面的資料成員也被注入
    public void Insert(Product p)
    {
        // [自動加入] 開始記錄日誌
        // 核心操作
        // [自動加入] 結束記錄日誌
    }
}
```

#### 3.5.2 PostSharp 範例：Transaction Aspect

使用 Attribute 宣告式套用切面：

```csharp
public class SqlProductRepository : IProductRepository
{
    [Transaction]  // 一行就套用了交易切面
    public void Insert(Product product) { ... }

    [Transaction]
    public void Update(Product product) { ... }

    [Transaction]
    public void Delete(Guid id) { ... }

    // GetAll 不需要交易，所以不加 Attribute
    public IEnumerable<Product> GetAll() { ... }
}
```

TransactionAttribute 的實作：

```csharp
[AttributeUsage(AttributeTargets.Method | AttributeTargets.Class | AttributeTargets.Assembly,
    AllowMultiple = false)]
[PostSharp.Serialization.PSerializable]
[PostSharp.Extensibility.MulticastAttributeUsage(
    MulticastTargets.Method,
    TargetMemberAttributes = MulticastAttributes.Instance | MulticastAttributes.Static)]
public class TransactionAttribute
    : PostSharp.Aspects.OnMethodBoundaryAspect
{
    public override void OnEntry(MethodExecutionArgs args)
    {
        // 方法進入時：建立 TransactionScope
        args.MethodExecutionTag = new TransactionScope();
    }

    public override void OnSuccess(MethodExecutionArgs args)
    {
        // 方法成功時：提交交易
        var scope = (TransactionScope)args.MethodExecutionTag;
        scope.Complete();
    }

    public override void OnExit(MethodExecutionArgs args)
    {
        // 方法結束時（保證執行，如 finally）：釋放資源
        var scope = (TransactionScope)args.MethodExecutionTag;
        scope.Dispose();
    }
}
```

注意這裡的冗餘——需要覆寫多個方法、套用多個 Attribute、透過 `MethodExecutionTag` 在方法間傳遞狀態。相比之下，第 10 章的 `TransactionCommandServiceDecorator<TCommand>` Decorator 可以直接用 `using` 語句，程式碼更自然、更清晰。

### 3.6 為何 Compile-time Weaving 是反模式

#### 問題一：DI 不友善（無法使用 Constructor Injection）

```csharp
// 嘗試在 Attribute 中使用 Constructor Injection —— 這行不通！
public class CircuitBreakerAttribute : OnMethodBoundaryAspect
{
    private readonly ICircuitBreaker breaker;

    public CircuitBreakerAttribute(ICircuitBreaker breaker) // 編譯錯誤！
    {                                                        // .NET Attribute 只允許
        this.breaker = breaker;                              // 基本型別（string, int 等）
    }                                                        // 作為建構子參數
}

// 使用時根本無法注入：
[CircuitBreakerAttribute(???)]  // <-- 這裡該放什麼？怎麼放？
```

**根本原因**：Attribute 是在編譯期決定的，而物件圖是在執行期（Composition Root 中）組裝的。這兩個時間點完全不同，無法銜接。

> **白話比喻**：這就像在蓋房子的藍圖上標註「此處安裝進口冷氣」，但冷氣要等房子蓋好、住進去之後才會送到。藍圖上根本沒辦法寫死一台還不存在的冷氣機。

**被迫使用的替代方案及其問題：**

```
┌────────────────────────────────────────────────────────────────────┐
│               替代方案                    引發的問題                │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  Ambient Context（環境上下文）            Temporal Coupling         │
│  ┌──────────────────────────────┐       （時序耦合：忘了設定就     │
│  │ public static ICircuitBreaker│        NullReferenceException）  │
│  │   Breaker { get; set; }     │                                  │
│  └──────────────────────────────┘       Captive Dependency         │
│                                         （囚禁依賴：static 的      │
│                                          生命週期問題）            │
│                                                                    │
│                                         Interdependent Tests       │
│                                         （測試互相干擾：static     │
│                                          狀態殘留）               │
│                                                                    │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  Service Locator（服務定位器）           以上所有問題 ＋            │
│                                         隱藏依賴關係 ＋            │
│                                         額外的冗餘依賴             │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

> **名詞解釋：**
> - **Temporal Coupling（時序耦合）**：程式碼的正確運作取決於特定的呼叫順序。好比你必須先插上電源才能開電視，但程式碼裡沒有任何機制強制你這麼做——忘了插電就直接爆炸。
> - **Captive Dependency（囚禁依賴）**：一個短命的元件被一個長命的元件持有，導致它永遠無法被釋放或更新。就像把鮮魚放進冰櫃後把冰櫃焊死了。
> - **Interdependent Tests（互相依賴的測試）**：測試 A 的副作用影響測試 B 的結果。每個測試都必須記得清理 static 狀態，忘了就會造成隨機失敗——最惱人的那種 bug。

#### 問題二：編譯期耦合破壞可測試性

切面在編譯期就被織入 IL，這意味著：

```
┌─────────────────────────────────────────────────────────────────┐
│                    編譯期耦合的影響                               │
│                                                                 │
│  1. 單元測試無法隔離                                             │
│     你想測試 SqlProductRepository 的邏輯，                       │
│     但 [Transaction] 和 [CheckAuthorization] 切面               │
│     已經被焊死在 IL 裡面，無法移除。                              │
│                                                                 │
│  2. 整合測試被干擾                                               │
│     你想測試 Repository 與資料庫的互動，                          │
│     但 [CheckAuthorization] 強迫你先走登入流程，                  │
│     拖慢測試速度、增加維護負擔。                                  │
│                                                                 │
│  3. 快取切面導致荒謬結果                                         │
│     你寫了一個測試想查詢資料庫，結果卻命中快取，                  │
│     根本沒碰到資料庫——測試結果毫無意義。                         │
│                                                                 │
│  結論：你無法控制哪些切面在哪些測試中生效。                       │
└─────────────────────────────────────────────────────────────────┘
```

#### 核心結論：Compile-time Weaving 是 DI 的反面

```
  DI 的目標                          Compile-time Weaving 的效果
  ═══════════                        ═══════════════════════════
  在執行期（Composition Root）        在編譯期就把依賴焊死
  集中組裝物件圖                      在程式碼中

  引入 Seam（接縫）以便替換           消除接縫，使替換不可能
  Volatile Dependency

  ──────────────────────────────────────────────────────────
  結論：兩者完全互斥，Compile-time Weaving 是 DI 的反模式
  ──────────────────────────────────────────────────────────
```

> **重要但書**：作者明確指出，這裡討論的是針對 **Volatile Dependencies（易變依賴）** 使用 Compile-time Weaving 的情況。對於 **Stable Dependencies（穩定依賴）**，從 DI 的角度來看並不特別有趣，使用 Compile-time Weaving 也未必有害。

## 4. 關鍵框架與公式 (Key Frameworks)

### 4.1 AOP 策略選擇決策樹

```
                    需要套用橫切關注點？
                           │
                           ▼
                 涉及 Volatile Dependency？
                    /              \
                  是                否
                  │                 │
                  ▼                 ▼
           能夠重構為               可考慮 Compile-time
           SOLID 設計嗎？           Weaving（無害）
            /         \
          能           不能（時間/預算不允許）
          │                  │
          ▼                  ▼
    SOLID 設計 AOP      Dynamic Interception
    （第 10 章方法）     （本章方法，臨時方案）

    ┌──────────────────────────────────────────┐
    │ 優先順序：                                │
    │ 1. SOLID 設計 AOP（最佳）                 │
    │ 2. Dynamic Interception（可接受的暫時方案）│
    │ 3. Compile-time Weaving（反模式，避免）    │
    └──────────────────────────────────────────┘
```

### 4.2 三種 AOP 方法完整比較表

| 維度 | SOLID 設計 AOP | Dynamic Interception | Compile-time Weaving |
|------|---------------|---------------------|---------------------|
| **編譯期安全性** | 完整 | 部分喪失（反射） | 完整但被鎖死 |
| **Constructor Injection** | 完全支持 | 完全支持 | 不可能 |
| **與工具的耦合度** | 無 | 中（Castle 等） | 高（PostSharp 等） |
| **適用範圍** | 介面方法 | 介面方法 | 任何方法（含私有） |
| **可測試性** | 優秀 | 良好 | 差 |
| **Composition Root 集中管理** | 是 | 是 | 否 |
| **解決底層設計問題** | 是 | 否 | 否 |
| **對 Volatile Dependencies** | 完全友善 | 友善 | 反模式 |

### 4.3 Castle Dynamic Proxy 核心 API

```csharp
// 1. 建立 ProxyGenerator（應用程式生命週期建一次）
var generator = new ProxyGenerator();

// 2. 基於介面建立代理（包裹目標物件 + 攔截器）
IMyService proxy = generator.CreateInterfaceProxyWithTarget<IMyService>(
    target,        // 被代理的目標物件
    interceptor1,  // 可傳入多個 IInterceptor
    interceptor2
);

// 3. IInterceptor 介面（唯一需要實作的方法）
public interface IInterceptor
{
    void Intercept(IInvocation invocation);
}

// 4. IInvocation 介面的關鍵成員
//    - Proceed()          讓呼叫繼續往下走
//    - Arguments          方法參數陣列（object[]）
//    - ReturnValue        回傳值
//    - Method             方法的反射資訊
```

### 4.4 Interceptor 模板結構

```csharp
public class YourAspectInterceptor : IInterceptor
{
    // 透過 Constructor Injection 注入依賴
    private readonly ISomeDependency dependency;

    public YourAspectInterceptor(ISomeDependency dependency)
    {
        this.dependency = dependency;
    }

    public void Intercept(IInvocation invocation)
    {
        // ┌──── 前置邏輯（Before）────┐
        //   例如：權限檢查、日誌記錄、快取查詢
        // └────────────────────────────┘

        invocation.Proceed();  // 呼叫原始方法

        // ┌──── 後置邏輯（After）─────┐
        //   例如：記錄結果、快取寫入
        // └────────────────────────────┘

        // 也可以用 try/catch/finally 包裹 Proceed()
        // 來處理例外狀況
    }
}
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 5.1 難點一：為什麼 Attribute 無法搭配 Constructor Injection

**困惑來源**：初學者可能覺得 `[CircuitBreaker]` 看起來很優雅，為什麼不能用？

**深層解釋**：

這個問題的根源在於「時間軸不對」：

```
  時間軸
  ═══════

  編譯期 ──────────────────────────────── 執行期
     │                                      │
     ▼                                      ▼
  Attribute 被建立                     Composition Root 組裝物件圖
  (由 .NET Runtime 控制,               (由開發者控制,
   只能用基本型別常數)                   可以 new 任何物件)

  你在這個時間點                        你在這個時間點
  拿不到 ICircuitBreaker               才能建立 ICircuitBreaker
  的實體                                的實體
```

.NET 的 Attribute 建構子只能接受編譯期常數（`string`、`int`、`bool`、`Type` 等）。這是語言層級的限制，不是框架的問題。而 Volatile Dependencies 需要在執行期透過 Composition Root 來決定用哪個實作——這兩件事在時間軸上就是對不上。

**記憶口訣**：「Attribute 活在編譯期，DI 活在執行期，兩個世界無法交匯。」

---

### 5.2 難點二：Dynamic Interception 為什麼「未解決底層設計問題」

**困惑來源**：Dynamic Interception 明明也能套用切面、也支援 DI，為什麼說它沒解決問題？

**深層解釋**：

回想第 10 章的核心論點：一個肥大的 `IProductService` 介面（違反 ISP）導致了各種維護困難。SOLID 設計 AOP 的做法是先重構為 `ICommandHandler<T>` / `IQueryHandler<T>` 這樣的通用介面，然後用泛型 Decorator 優雅地套用切面。

Dynamic Interception 跳過了這個重構步驟——它讓你在不改變設計的情況下套用切面：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  SOLID 設計 AOP                                             │
│  ══════════════                                             │
│  IProductService (肥大介面)                                  │
│       │                                                     │
│       ▼ 重構                                                │
│  ICommandHandler<InsertProduct>                             │
│  ICommandHandler<DeleteProduct>                             │
│  IQueryHandler<GetAllProducts, IEnumerable<Product>>        │
│       │                                                     │
│       ▼ 套用泛型 Decorator                                  │
│  TransactionCommandServiceDecorator<T>                      │
│  CircuitBreakerCommandServiceDecorator<T>                   │
│                                                             │
│  結果：設計改善 ＋ 切面套用 ＝ 整體可維護性大幅提升          │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Dynamic Interception                                       │
│  ════════════════════                                       │
│  IProductService (肥大介面，原封不動)                        │
│       │                                                     │
│       ▼ 直接用 Castle 套用 Interceptor                      │
│  CircuitBreakerInterceptor                                  │
│                                                             │
│  結果：切面套用 ＋ 設計不變 ＝ 可維護性僅略有提升            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

Dynamic Interception 是「治標不治本」的方案——它解決了橫切關注點的重複問題，但系統中違反 SOLID 原則的其他問題依然存在。

---

### 5.3 難點三：Compile-time 耦合如何破壞測試

**困惑來源**：「原始碼沒被改動」聽起來很無害，為什麼說它造成了耦合？

**深層解釋**：

關鍵在於理解「原始碼解耦 ≠ 執行期解耦」：

```csharp
// 原始碼看起來很乾淨
public class SqlProductRepository : IProductRepository
{
    [Transaction]
    [CheckAuthorization]
    [Caching]
    public IEnumerable<Product> GetAll() { /* 查詢資料庫 */ }
}
```

但編譯後的 IL 實際上變成了：

```csharp
// 等效的 IL（概念示意）
public class SqlProductRepository : IProductRepository
{
    public IEnumerable<Product> GetAll()
    {
        // [CheckAuthorization 的 IL] —— 需要連線到認證服務
        CheckAuth();

        // [Caching 的 IL] —— 可能直接回傳快取結果
        if (cache.Has("GetAll")) return cache.Get("GetAll");

        // [Transaction 的 IL]
        using (var scope = new TransactionScope())
        {
            var result = /* 查詢資料庫 */;
            scope.Complete();

            // [Caching 的 IL]
            cache.Set("GetAll", result);

            return result;
        }
    }
}
```

寫整合測試時，你想測的是「Repository 能不能正確查詢資料庫」，但你無法關掉認證切面（測試要先登入）、無法關掉快取切面（查詢可能命中快取而非資料庫）。所有切面都被「焊死」在 IL 裡。

用 Dynamic Interception 或 SOLID Decorator 就沒有這個問題，因為切面是在 Composition Root 組裝的——測試時不組裝就好了。

## 6. 本章小結與架構啟示

### 核心結論

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│   1. SOLID 設計 AOP 是王道                                          │
│      透過重構為通用介面 + 泛型 Decorator，                           │
│      同時提升設計品質和切面套用能力。                                 │
│                                                                     │
│   2. Dynamic Interception 是務實的退路                               │
│      當團隊尚未準備好進行大規模重構時，                               │
│      它是一個可接受的臨時方案，等待時機成熟再升級。                    │
│                                                                     │
│   3. Compile-time Weaving 對 Volatile Dependencies 是反模式          │
│      它與 DI 的核心理念（執行期組裝、接縫、可替換性）完全對立。       │
│                                                                     │
│   4. 判斷標準：是否為 Volatile Dependency                            │
│      如果切面涉及外部系統（資料庫、網路服務、檔案系統），              │
│      絕對不要使用 Compile-time Weaving。                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 架構師的實戰心得

1. **漸進式改善策略**：在遺留系統中，可以先引入 Dynamic Interception 來快速處理橫切關注點（如日誌、Circuit Breaker），同時逐步將核心介面重構為 CQRS 風格的通用介面。這是一條務實的演進路徑。

2. **工具依賴的代價**：Castle Dynamic Proxy 雖然是成熟的開源工具，但它仍然是一個額外的外部依賴。團隊需要學習它的 API，且 Interceptor 程式碼與 Castle 強耦合。在做技術選型時，要把這個「隱性成本」算進去。

3. **可測試性是最高指導原則**：Compile-time Weaving 最致命的問題不是技術上的，而是它破壞了測試的隔離性。一個無法被有效測試的系統，長期來看就是一個不可維護的系統。

4. **編譯期 vs 執行期的思維框架**：DI 的精髓在於將「什麼時候決定用哪個實作」這件事推遲到執行期。任何試圖在編譯期就鎖死依賴關係的技術，都與 DI 的核心精神相悖。這個思維框架不只適用於 AOP，也適用於所有架構決策。

5. **從本章邁向 Part 4**：本章結束了 Pure DI 的討論。接下來的 Part 4 將探討 DI Container（Autofac、Simple Injector、MS.DI），它們可以幫助自動化 Composition Root 中的物件圖組裝工作。
