> **書籍**：Dependency Injection Principles, Practices, and Patterns
> **作者**：Steven van Deursen, Mark Seemann
> **導讀視角**：資深系統架構師
> **程式碼範例**：C#

---

## 1. 概述 (Overview)

本章深入介紹 **Simple Injector** — 一個由本書作者之一 Steven van Deursen 於 2010 年建立的現代化 DI Container。Simple Injector 的核心設計哲學是「**嚴格 (Strictness)**」：它不會猜測你的意圖，而是透過 API 設計與診斷機制（Diagnostic Facility）主動偵測並阻止配置錯誤。

Simple Injector 與其他 DI Container（如 Autofac）最顯著的差異在於以下幾點：

| 特性 | Simple Injector 作法 | 其他容器常見作法 |
|------|----------------------|-----------------|
| 同一抽象多次 Register | 直接拋出例外，要求改用 `Collection.Register` | 靜默覆蓋或堆疊 |
| Captive Dependency 偵測 | `Verify()` 自動掃描並報錯 | 多數不偵測或僅在執行期才暴露 |
| Sequence 注入 | 以 **Stream（串流）** 形式惰性解析 | 預先建立完整集合 |
| Scope 運作方式 | **Ambient Scope（環境範疇）**，永遠從 Container 解析 | 必須從 Scope 實例解析 |

> **白話比喻**：如果把 DI Container 比作一座自動化倉庫，大多數容器在你放錯貨物時只是默默接受，等到出貨才爆炸。Simple Injector 則像一位嚴格的品管人員 — 你一放貨它就檢查，不合格直接退回。

---

## 2. 學習目標 (Learning Objectives)

完成本章學習後，你應該能夠：

1. **掌握 Register / Resolve 兩步驟模式**：理解 Simple Injector 的「先配置、再鎖定」的容器生命週期。
2. **運用三大 Lifestyle**：正確配置 Transient、Singleton、Scoped，並理解 Ambient Scope 的運作原理。
3. **善用 Verify + Diagnostic**：在應用程式啟動時自動偵測 Captive Dependency 等常見配置錯誤。
4. **設計與註冊 Sequences、Decorators、Composites**：用 `Collection.Register`、`RegisterDecorator`、泛型開放型別完成進階 wiring。
5. **理解 Sequences are Streams**：明白 Simple Injector 中 `IEnumerable<T>` 是惰性串流，以及這對 Lifestyle 與效能的影響。
6. **處理 Difficult API**：使用 Parameter Object 或 delegate 處理含有原始型別參數的建構式。

## 3. 章節大綱 (Outline)

### 3.1 Simple Injector 簡介 (Section 14.1)

#### 3.1.1 Container 的兩步驟流程：Register → Resolve

Simple Injector 的使用遵循一個明確的兩階段模式：

```
┌─────────────────────────────────────────────────────────┐
│                    Configure Phase                       │
│  ┌──────────┐    ┌──────────┐    ┌──────────────────┐   │
│  │ Register  │───>│  Verify  │───>│ Container Locked │   │
│  │ 註冊元件  │    │ 驗證配置  │    │   容器鎖定       │   │
│  └──────────┘    └──────────┘    └──────────────────┘   │
│                                          │               │
└──────────────────────────────────────────│───────────────┘
                                           │
                                           v
                              ┌──────────────────┐
                              │     Resolve      │
                              │   解析元件實例    │
                              └──────────────────┘
```

**白話說明**：你先把所有「配方」（註冊）告訴容器，然後叫它做一次「品管檢查」（Verify），通過之後容器就被「鎖住」，接下來只能從中取東西（Resolve），不能再修改配方。

```csharp
// 步驟 1: 建立容器
var container = new Container();

// 步驟 2: 註冊抽象到具體的映射
container.Register<IIngredient, SauceBéarnaise>();
container.Register<ICourse, Course>();

// 步驟 3: 驗證 — 一旦呼叫就鎖定容器
container.Verify();

// 步驟 4: 解析
IIngredient ingredient = container.GetInstance<IIngredient>();
```

#### 3.1.2 三種配置方式

```
        明確 (Explicit)
          ▲
          │
  ┌───────┴───────┐
  │  Config Files  │  ← 設定檔（Simple Injector 不內建支援，但可自行實作）
  └───────┬───────┘
          │
  ┌───────┴────────────┐
  │ Configuration      │  ← 程式碼配置（最常用）
  │ as Code            │
  └───────┬────────────┘
          │
  ┌───────┴────────────┐
  │ Auto-Registration  │  ← 慣例式掃描（最省力）
  └────────────────────┘
          │
          ▼
        隱含 (Implicit)
```

**Configuration as Code（程式碼配置）**：

```csharp
container.Register<IIngredient, SauceBéarnaise>();
container.Register<ICourse, Course>();
```

**Auto-Registration（慣例掃描）**：

```csharp
// 掃描整個 Assembly，找到所有 IIngredient 的實作並註冊
Assembly assembly = typeof(Steak).Assembly;
container.Collection.Register<IIngredient>(assembly);
```

**泛型抽象的 Auto-Registration**：

```csharp
// 掃描 Assembly，自動找到所有 ICommandService<TCommand> 的實作
Assembly assembly = typeof(AdjustInventoryService).Assembly;
container.Register(typeof(ICommandService<>), assembly);
```

> **白話比喻**：Configuration as Code 像是手寫菜單，每道菜逐一列出；Auto-Registration 像是跟廚房說「冰箱裡有什麼食材，都做成沙拉」，靠慣例自動匹配。

#### 3.1.3 同一抽象不可重複 Register

```csharp
container.Register<IIngredient, SauceBéarnaise>();
container.Register<IIngredient, Steak>();  // 拋出例外！
```

Simple Injector 會拋出例外，提示你應該改用 `Collection.Register`。這是它「嚴格」哲學的體現。

```csharp
// 正確做法：使用 Collection.Register 註冊 Sequence
container.Collection.Register<IIngredient>(
    typeof(SauceBéarnaise),
    typeof(Steak));

// 或逐一 Append
container.Collection.Append<IIngredient, SauceBéarnaise>();
container.Collection.Append<IIngredient, Steak>();
```

#### 3.1.4 設定檔配置（自行實作）

Simple Injector 不內建設定檔支援，但你可以利用 .NET Core Configuration 自行實作：

```csharp
// simpleinjector.json
// {
//   "registrations": [
//     {
//       "service": "Namespace.IIngredient, AssemblyName",
//       "implementation": "Namespace.Steak, AssemblyName"
//     }
//   ]
// }

var config = new ConfigurationBuilder()
    .AddJsonFile("simpleinjector.json")
    .Build();

var registrations = config.GetSection("registrations").GetChildren();

foreach (var reg in registrations)
{
    container.Register(
        Type.GetType(reg["service"]),
        Type.GetType(reg["implementation"]));
}
```

### 3.2 生命週期管理 (Section 14.2 — Managing Lifetime)

#### 3.2.1 三大 Lifestyle

| Lifestyle | 說明 | 白話比喻 |
|-----------|------|---------|
| **Transient** | 每次請求建立新實例，不追蹤、不 Dispose。**這是預設值** | 用完即丟的免洗餐具 |
| **Singleton** | 整個容器生命期只有一個實例，容器 Dispose 時才釋放 | 餐廳裡唯一的一套銀餐具，從開店用到關店 |
| **Scoped** | 在一個 Scope 中是 Singleton，跨 Scope 則建立新實例 | 每桌客人共用同一套餐具，換桌就換新的 |

```
  ┌──────────────────── Container ─────────────────────┐
  │                                                     │
  │  Singleton ──────────────────────────────────────── │
  │  (整個容器共享一份)                                  │
  │                                                     │
  │  ┌──── Scope A ────┐    ┌──── Scope B ────┐        │
  │  │  Scoped (共享)   │    │  Scoped (共享)   │        │
  │  │                  │    │                  │        │
  │  │  Transient (新)  │    │  Transient (新)  │        │
  │  │  Transient (新)  │    │  Transient (新)  │        │
  │  └─────────────────┘    └─────────────────┘        │
  └─────────────────────────────────────────────────────┘
```

```csharp
// Singleton
container.Register<IIngredient, SauceBéarnaise>(Lifestyle.Singleton);

// Transient（預設，可省略）
container.Register<IIngredient, SauceBéarnaise>(Lifestyle.Transient);
container.Register<IIngredient, SauceBéarnaise>(); // 等同上面

// Scoped — 需要先設定 DefaultScopedLifestyle
container.Options.DefaultScopedLifestyle = new AsyncScopedLifestyle();
container.Register<IIngredient, SauceBéarnaise>(Lifestyle.Scoped);
```

#### 3.2.2 Releasing Components（釋放元件）

Simple Injector 沒有顯式的 `Release` 方法，而是透過 Scope 的 Dispose 來釋放：

```csharp
using (AsyncScopedLifestyle.BeginScope(container))
{
    IMeal meal = container.GetInstance<IMeal>(); // 注意：從 container 解析，不是從 scope
    meal.Consume();
} // Scope 結束，Scoped 元件被 Dispose
```

> **關鍵觀念**：Releasing（釋放）不等於 Disposing（銷毀）。Release 只是告知容器「我用完了」，容器根據 Lifestyle 決定是否銷毀。Scoped 元件在 Scope 結束時銷毀；Singleton 在容器 Dispose 時銷毀。

#### 3.2.3 Ambient Scope（環境範疇）

**這是 Simple Injector 最獨特的特性之一。**

Ambient Scope 意味著 Scope 是「漂浮在當前執行上下文中的」，你不需要持有 Scope 的參考就能從中解析。

```csharp
var container = new Container();
container.Options.DefaultScopedLifestyle = new AsyncScopedLifestyle();

// scope1 = null（目前沒有 active scope）
Scope scope1 = Lifestyle.Scoped.GetCurrentScope(container);

using (Scope scope2 = AsyncScopedLifestyle.BeginScope(container))
{
    // scope3 == scope2（自動偵測到當前 active scope）
    Scope scope3 = Lifestyle.Scoped.GetCurrentScope(container);
}

// scope4 = null（scope 已結束）
Scope scope4 = Lifestyle.Scoped.GetCurrentScope(container);
```

```
  時間軸 ──────────────────────────────────────────>

  GetCurrentScope → null
                          ┌── BeginScope ──┐
                          │  scope 生效     │
                          │  GetCurrentScope│
                          │  → 同一個 scope │
                          └── Dispose ─────┘
                                            GetCurrentScope → null
```

> **白話比喻**：Ambient Scope 就像「環境溫度」。你不需要拿著溫度計到處跑，只要在房間裡，溫度計就自動讀到當前房間的溫度。`AsyncScopedLifestyle` 透過 `AsyncLocal<T>` 存放 Scope，所以即使 async/await 跳到不同執行緒，Scope 仍然跟著走。

**Ambient Scope 帶來的好處**：
- 永遠從 `Container` 解析，不會忘記要從哪個 Scope 解析
- 不會因為誤從根容器解析而產生記憶體洩漏（其他容器如 Autofac 會有此問題）
- 沒有 active scope 時解析 Scoped 元件，Simple Injector 會直接拋出描述性例外

#### 3.2.4 Verify + Diagnostic（容器驗證與診斷）

這是 Simple Injector 的殺手級功能：

```
  ┌────────────────────────────────────────────────────┐
  │                 Configure Phase                     │
  │                                                     │
  │  Register ─────> Verify ─────> Container Locked     │
  │                    │                                 │
  │                    ├── 嘗試解析所有物件圖              │
  │                    ├── 偵測 Captive Dependency        │
  │                    ├── 偵測 Disposable Transient      │
  │                    └── 偵測其他常見配置問題             │
  └────────────────────────────────────────────────────┘
```

**Captive Dependency（被俘虜的相依）偵測範例**：

> **白話說明**：「Captive Dependency」就是一個短命元件被長命元件抓住（capture）了。想像一顆新鮮蛋黃（Transient，應該很快用掉）被放進了一罐永遠不會開的美乃滋（Singleton）裡 — 蛋黃會在裡面腐爛，但你永遠不會知道。

```csharp
var container = new Container();

// EggYolk 是 Transient（短命）
container.Register<EggYolk>(Lifestyle.Transient);

// Mayonnaise 是 Singleton（長命），但依賴 EggYolk
container.Register<Mayonnaise>(Lifestyle.Singleton);
container.Register<SunflowerOil>(Lifestyle.Singleton);

// Verify 會拋出例外！
container.Verify();
// 例外訊息：
// [Lifestyle Mismatch] Mayonnaise (Singleton) depends on EggYolk (Transient).
```

> **注意**：Simple Injector 稱 Captive Dependency 為 **Lifestyle Mismatch**。與 Autofac 不同，Simple Injector 不允許 Transient 被注入 Singleton（Autofac 則允許，認為 Transient 會「活得跟消費者一樣久」）。

**Suppressing Warnings（壓制警告）**：

當你確定某個 Lifestyle Mismatch 是刻意的，可以壓制特定警告：

```csharp
var container = new Container();

Registration reg = Lifestyle.Transient
    .CreateRegistration<EggYolk>(container);

reg.SuppressDiagnosticWarning(
    DiagnosticType.LifestyleMismatch,
    justification: "我喜歡吃過期蛋黃。"); // justification 是必填的

container.AddRegistration(typeof(EggYolk), reg);
container.Register<Mayonnaise>(Lifestyle.Singleton);
container.Register<SunflowerOil>(Lifestyle.Singleton);
container.Verify(); // 這次不會拋例外
```

### 3.3 註冊困難的 API (Section 14.3 — Registering Difficult APIs)

#### 3.3.1 原始型別參數的問題

Simple Injector 的 Auto-Wiring 依靠型別來匹配相依，但原始型別（string、int、enum）會造成歧義：

```csharp
public class ChiliConCarne
{
    public ChiliConCarne(Spiciness spiciness) { }
}

public enum Spiciness { Mild, Medium, Hot }

// 這樣會拋例外！
container.Register<ICourse, ChiliConCarne>();
// 例外：parameter 'spiciness' of type Spiciness cannot be used for constructor injection
```

#### 3.3.2 解法一：Parameter Object（參數物件）— 推薦

> **白話說明**：Parameter Object 就是把一堆零散的基本型別包裝成一個有意義的物件。就像你不會把鹽巴、辣椒粉散落在廚房各處，而是放進一個叫「調味料罐」的容器裡。

```csharp
// 抽取為 Parameter Object
public class Flavoring
{
    public readonly Spiciness Spiciness;
    public readonly bool ExtraSalty;

    public Flavoring(Spiciness spiciness, bool extraSalty)
    {
        this.Spiciness = spiciness;
        this.ExtraSalty = extraSalty;
    }
}

// 現在可以正常註冊了
var flavoring = new Flavoring(Spiciness.Medium, extraSalty: true);
container.RegisterInstance<Flavoring>(flavoring);
container.Register<ICourse, ChiliConCarne>();
```

#### 3.3.3 解法二：Delegate / Code Block

```csharp
// 使用 delegate 手動建構
container.Register<ICourse>(
    () => new ChiliConCarne(Spiciness.Hot));

// 如果 ChiliConCarne 還有其他相依，需要回呼容器
container.Register<ICourse>(
    () => new ChiliConCarne(
        Spiciness.Medium,
        container.GetInstance<IIngredient>()));
```

> **注意**：使用 delegate 會讓 Simple Injector 的診斷功能「看不到」內部的相依關係，因此應優先使用 Parameter Object。

#### 3.3.4 靜態工廠（Static Factory）

```csharp
public static class JunkFoodFactory
{
    public static JunkFood Create(string name) => new JunkFood(name);
}

// 用 delegate 包裝靜態工廠
container.Register<IMeal>(() => JunkFoodFactory.Create("chicken meal"));
```

### 3.4 多元件操作 (Section 14.4 — Working with Multiple Components)

#### 3.4.1 條件式註冊（Conditional Registration）

當同一抽象有多個實作，且需要根據消費者來選擇時：

```csharp
public class ThreeCourseMeal : IMeal
{
    public ThreeCourseMeal(ICourse entrée, ICourse mainCourse, ICourse dessert) { }
}

// 根據建構式參數名稱做條件式注入
container.Register<IMeal, ThreeCourseMeal>();

container.RegisterConditional<ICourse, Rillettes>(
    c => c.Consumer.Target.Name == "entrée");

container.RegisterConditional<ICourse, CordonBleu>(
    c => c.Consumer.Target.Name == "mainCourse");

container.RegisterConditional<ICourse, MousseAuChocolat>(
    c => c.Consumer.Target.Name == "dessert");
```

```
  ┌─────────────────────────┐
  │    ThreeCourseMeal      │
  │                         │
  │  entrée ──────> Rillettes          (條件: Name == "entrée")
  │  mainCourse ──> CordonBleu         (條件: Name == "mainCourse")
  │  dessert ─────> MousseAuChocolat   (條件: Name == "dessert")
  └─────────────────────────┘
```

> **注意**：根據參數名稱做配對不具備重構安全性（重新命名參數可能導致配置失效）。

#### 3.4.2 Wiring Sequences（序列配線）

**移除歧義的更好設計 — 從固定參數改為 `IEnumerable<T>`**：

```csharp
// 改良後的 Meal 類別
public class Meal : IMeal
{
    public Meal(IEnumerable<ICourse> courses) { }
}

// 註冊序列
container.Collection.Register<ICourse>(assembly);

// 註冊 Meal — Auto-Wiring 自動注入所有 ICourse
container.Register<IMeal, Meal>();
```

> **設計啟示**：如果你發現容器配置很困難，這往往是設計有問題的信號。把 `ThreeCourseMeal` 重構為 `Meal(IEnumerable<ICourse>)` 不僅讓配置更簡單，也讓類別更通用（從 0 道菜到 N 道菜都支援）。

**挑選子集合**：

```csharp
// 建立子集合（不會立即解析，是 Stream）
IEnumerable<ICourse> frenchMenu = container.Collection.Create<ICourse>(
    typeof(Rillettes),
    typeof(CordonBleu),
    typeof(MousseAuChocolat));

IEnumerable<ICourse> texMexMenu = container.Collection.Create<ICourse>(
    typeof(CeasarSalad),
    typeof(ChiliConCarne));

// 將子集合注入特定的 Meal 實例
container.RegisterInstance<IMeal>(new Meal(frenchMenu));
```

#### 3.4.3 Wiring Decorators（裝飾者配線）

> **白話說明**：Decorator 就像「一層一層的包裝紙」。比如一塊小牛排（VealCutlet），先塞進火腿和起司（HamCheeseGarlic），再裹上麵包粉（Breading），就變成了維也納炸肉排。

**非泛型 Decorator**：

```csharp
var c = new Container();
c.Register<IIngredient, VealCutlet>();
c.RegisterDecorator<IIngredient, HamCheeseGarlic>();
c.RegisterDecorator<IIngredient, Breading>();

// 結果等同於：
// new Breading(
//     new HamCheeseGarlic(
//         new VealCutlet()));
```

```
  ┌─────────────────────────┐
  │       Breading          │  ← 最外層（最後註冊的 Decorator）
  │  ┌──────────────────┐   │
  │  │ HamCheeseGarlic  │   │  ← 中間層
  │  │  ┌────────────┐  │   │
  │  │  │ VealCutlet  │  │   │  ← 核心（原始元件）
  │  │  └────────────┘  │   │
  │  └──────────────────┘   │
  └─────────────────────────┘
```

**泛型 Decorator（最強大的用法）**：

```csharp
// Auto-Register 所有 ICommandService<T> 實作
container.Register(typeof(ICommandService<>), assembly);

// 依序套上三層 Decorator
container.RegisterDecorator(
    typeof(ICommandService<>),
    typeof(AuditingCommandServiceDecorator<>));

container.RegisterDecorator(
    typeof(ICommandService<>),
    typeof(TransactionCommandServiceDecorator<>));

container.RegisterDecorator(
    typeof(ICommandService<>),
    typeof(SecureCommandServiceDecorator<>));
```

```
  ┌──────────────────────────────────────┐
  │  SecureCommandServiceDecorator<T>    │  ← 安全檢查
  │  ┌────────────────────────────────┐  │
  │  │ TransactionCommandService...   │  │  ← 交易管理
  │  │  ┌──────────────────────────┐  │  │
  │  │  │ AuditingCommandService.. │  │  │  ← 稽核日誌
  │  │  │  ┌────────────────────┐  │  │  │
  │  │  │  │ 實際的 Command     │  │  │  │  ← 業務邏輯
  │  │  │  │ Service 實作       │  │  │  │
  │  │  │  └────────────────────┘  │  │  │
  │  │  └──────────────────────────┘  │  │
  │  └────────────────────────────────┘  │
  └──────────────────────────────────────┘
```

> **重要原則**：Decorator 按照**註冊順序**套用。先註冊的在內層，後註冊的在外層。

#### 3.4.4 Wiring Composites（組合配線）

> **白話說明**：Composite 是一個「幕後老大」，它實作了跟手下一樣的介面，但實際上把工作分派給一群手下去做。對外看起來是一個人，內部是整個團隊。

**非泛型 Composite**：

```csharp
// 手下們：註冊為 Sequence
container.Collection.Register<INotificationService>(
    typeof(OrderApprovedReceiptSender),
    typeof(AccountingNotifier),
    typeof(OrderFulfillment));

// 幕後老大：註冊為單一 mapping
container.Register<INotificationService, CompositeNotificationService>();
```

```
  解析 INotificationService 得到：

  ┌───────────────────────────────────┐
  │  CompositeNotificationService     │  ← 對外只有一個
  │                                   │
  │  ┌─ OrderApprovedReceiptSender ─┐ │
  │  ├─ AccountingNotifier ─────────┤ │  ← 內部委派給多個
  │  └─ OrderFulfillment ───────────┘ │
  └───────────────────────────────────┘
```

**Simple Injector 的智慧**：`Collection.Register` 在掃描 Assembly 時，會**自動過濾掉 Composite 和 Decorator 類別**，不會把它們混進 Sequence 裡。這點比 Autofac 更聰明。

**泛型 Composite**：

```csharp
// 序列：所有 IEventHandler<T> 的實作
container.Collection.Register(typeof(IEventHandler<>), assembly);

// Composite：將序列包裝起來
container.Register(
    typeof(IEventHandler<>),
    typeof(CompositeEventHandler<>));
```

#### 3.4.5 Sequences are Streams（序列即串流）

**這是 Simple Injector 最獨特的設計決策之一。**

> **白話說明**：在大多數 DI Container 中，當你注入一個 `IEnumerable<IIngredient>` 時，容器會**立刻**把所有食材都做好放進一個陣列給你。Simple Injector 則不同 — 它給你的是一張「食材清單」（Stream），只有當你真正要用某個食材時，它才即時製作。就像串流影片，你按播放才開始傳輸，不是一次把整部電影下載好。

```csharp
IEnumerable<IIngredient> stream =
    container.GetAllInstances<IIngredient>();

// 此時沒有任何 IIngredient 實例被建立！

IIngredient first1 = stream.First();  // 現在才建立第一個
IIngredient first2 = stream.First();  // 又建立一個新的（Transient）

object.ReferenceEquals(first1, first2); // false
```

**Stream 搭配 Lifestyle**：

```csharp
var c = new Container();
c.Collection.Append<IIngredient, SauceBéarnaise>();            // Transient（預設）
c.Collection.Append<IIngredient, Steak>(Lifestyle.Singleton);  // Singleton

var s = c.GetInstance<IEnumerable<IIngredient>>();

object.ReferenceEquals(s.First(), s.First()); // false（Transient 每次新建）
object.ReferenceEquals(s.Last(), s.Last());   // true （Singleton 同一個）
```

**Stream 的兩大優勢**：

1. **效能優勢**：注入 Stream 本身幾乎零成本（Stream 本身是 Singleton），只有在迭代時才建立元素。適合元素很多但不一定全部需要的場景。

2. **避免 Captive Dependency**：因為 Stream 是惰性的，即使 Singleton 消費者持有 `IEnumerable<T>` 的參考，Stream 中的 Transient 元素也不會被「俘虜」— 每次迭代都是重新向容器請求。

```csharp
// 這是安全的！即使 CompositeLogger 是 Singleton
container.Collection.Register<ILogger>(assembly);
container.Register<ILogger, CompositeLogger>(Lifestyle.Singleton);
// CompositeLogger 持有 IEnumerable<ILogger>（Stream），
// 每次迭代都會重新解析，不會 capture 任何 Transient logger
```

## 4. 關鍵框架與公式 (Key Frameworks)

### 4.1 Simple Injector API 速查表

```
┌────────────────────────────────────────────────────────────────┐
│                    Simple Injector API 核心方法                  │
├──────────────────────────┬─────────────────────────────────────┤
│ 方法                      │ 用途                                │
├──────────────────────────┼─────────────────────────────────────┤
│ Register<TServ, TImpl>() │ 一對一映射（不可重複同一抽象）        │
│ Register<T>(Func<T>)     │ 用 delegate 建構（失去 Auto-Wiring） │
│ RegisterInstance<T>(obj)  │ 註冊已存在的實例                    │
│ RegisterConditional<>()   │ 條件式註冊（依消費者決定注入哪個）    │
│ RegisterDecorator<>()     │ 註冊 Decorator（按順序套用）         │
├──────────────────────────┼─────────────────────────────────────┤
│ Collection.Register<T>() │ 註冊序列（多個實作）                  │
│ Collection.Append<T>()   │ 逐一追加至序列                       │
│ Collection.Create<T>()   │ 建立序列子集合（不註冊進容器）        │
├──────────────────────────┼─────────────────────────────────────┤
│ GetInstance<T>()          │ 解析單一實例                         │
│ GetAllInstances<T>()     │ 解析序列（等同 GetInstance<IEnum..>）│
│ Verify()                 │ 驗證 + 診斷所有配置                   │
│ Dispose()                │ 釋放容器及所有 Singleton              │
└──────────────────────────┴─────────────────────────────────────┘
```

### 4.2 Lifestyle 選擇決策樹

```
  你的元件是否為 Thread-safe？
        │
        ├── 否 ──> 它需要在 Web Request 之間共享嗎？
        │              │
        │              ├── 否 ──> 使用 Transient（預設）
        │              └── 是 ──> 使用 Scoped
        │
        └── 是 ──> 它是否持有狀態或昂貴資源？
                       │
                       ├── 否 ──> 使用 Singleton（最高效）
                       └── 是 ──> 狀態是否與 Request 相關？
                                      │
                                      ├── 是 ──> 使用 Scoped
                                      └── 否 ──> 使用 Singleton
                                                  （確保 Thread-safe）
```

### 4.3 Register vs Collection.Register 決策

```
  你有多個同一抽象的實作嗎？
        │
        ├── 否 ──> 使用 Register<TService, TImpl>()
        │
        └── 是 ──> 這些實作是「同級」（peers）嗎？
                       │
                       ├── 是 ──> 使用 Collection.Register<T>()
                       │          消費者依賴 IEnumerable<T>
                       │
                       └── 否 ──> 它們是層層包裝嗎？
                                      │
                                      ├── 是 ──> 使用 RegisterDecorator<T>()
                                      │
                                      └── 它是「一對多」嗎？
                                             │
                                             └── 是 ──> 用 Collection.Register 註冊子元素
                                                        用 Register 註冊 Composite
```

### 4.4 Decorator + Composite + Sequence 整合模式

以下是一個完整的真實場景 wiring 範例：

```csharp
var container = new Container();
container.Options.DefaultScopedLifestyle = new AsyncScopedLifestyle();

// 1. Auto-Register 所有 ICommandService<T> 實作
Assembly assembly = typeof(AdjustInventoryService).Assembly;
container.Register(typeof(ICommandService<>), assembly);

// 2. 套上 Cross-Cutting Concern Decorators
container.RegisterDecorator(
    typeof(ICommandService<>),
    typeof(AuditingCommandServiceDecorator<>));
container.RegisterDecorator(
    typeof(ICommandService<>),
    typeof(TransactionCommandServiceDecorator<>));
container.RegisterDecorator(
    typeof(ICommandService<>),
    typeof(SecureCommandServiceDecorator<>));

// 3. 註冊 Event Handler 序列 + Composite
container.Collection.Register(typeof(IEventHandler<>), assembly);
container.Register(
    typeof(IEventHandler<>),
    typeof(CompositeEventHandler<>));

// 4. 驗證一切配置
container.Verify();
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 5.1 Ambient Scope 與 AsyncLocal 的運作機制

**難點**：為什麼可以從 `Container`（而非 Scope）解析 Scoped 元件？這跟其他容器完全不同。

**解析**：

`AsyncScopedLifestyle` 底層使用 `AsyncLocal<T>` 儲存當前的 Scope 實例。`AsyncLocal<T>` 是 .NET 提供的機制，可以在非同步呼叫鏈中「隨身攜帶」資料，即使 `await` 之後切換了執行緒，資料仍然存在。

```
  Thread 1                    Thread 2
  ─────────                   ─────────
  BeginScope()
       │
       ▼
  AsyncLocal<Scope> = scope
       │
  await SomeAsync()  ──────>  繼續執行
       │                          │
       ▼                          ▼
  （不再使用）              AsyncLocal<Scope> 仍然 = scope
                                  │
                                  ▼
                            container.GetInstance<T>()
                            → 透過 AsyncLocal 找到 scope
                            → 正確解析 Scoped 元件
```

**類比**：就像 .NET 的 `TransactionScope` — 你只要 `new TransactionScope()` 包住一段程式碼，裡面的資料庫操作就自動加入同一筆交易，不需要把 Transaction 物件傳來傳去。

**為何重要**：因為你永遠從 Container 解析，所以：
- 不可能因為「從錯的 Scope 解析」而產生 bug
- 不可能因為「從根容器解析 Scoped 元件」而造成記憶體洩漏
- 降低了 DI 程式碼的複雜度

### 5.2 Sequences are Streams — 惰性解析的深層影響

**難點**：Stream 行為違反了對 `IEnumerable<T>` 的直覺認知，且對 Lifestyle 有深遠影響。

**解析**：

大多數開發者認為拿到一個 `IEnumerable<T>` 就是拿到了一組已經存在的物件。但在 Simple Injector 中，`IEnumerable<T>` 是一個**生產者**（producer），而不是一個**集合**（collection）。

```
  傳統 DI Container:
  ┌─────────────────────────────────────────────────┐
  │  GetInstance<IEnumerable<ILogger>>()             │
  │                                                  │
  │  → new FileLogger()    ┐                         │
  │  → new DbLogger()      ├──> new List<ILogger>()  │
  │  → new ConsoleLogger() ┘    (已建立好的清單)      │
  └─────────────────────────────────────────────────┘

  Simple Injector:
  ┌─────────────────────────────────────────────────┐
  │  GetInstance<IEnumerable<ILogger>>()             │
  │                                                  │
  │  → Stream 物件 (本身是 Singleton)                 │
  │     │                                            │
  │     ├── .First() → 此時才 new FileLogger()       │
  │     ├── .Skip(1).First() → new DbLogger()        │
  │     └── 未迭代 → ConsoleLogger 從未被建立         │
  └─────────────────────────────────────────────────┘
```

**Captive Dependency 問題的解決**：

```csharp
// 在傳統 DI Container 中，這是 Captive Dependency：
// Singleton 消費者持有 Transient 元素的參考
// → Transient 被困在 Singleton 裡，永遠不會被回收

// 在 Simple Injector 中，這是安全的：
container.Register<ILogger, CompositeLogger>(Lifestyle.Singleton);
container.Collection.Register<ILogger>(assembly);
// CompositeLogger（Singleton）持有 IEnumerable<ILogger>（Stream）
// 每次 foreach 都重新向容器解析 → Transient 元素不會被俘虜
```

**但要注意**：如果你呼叫 `.ToList()` 把 Stream 物質化為固定清單，就失去了 Stream 的保護，可能重新引入 Captive Dependency 問題。

### 5.3 Diagnostic 如何偵測 Captive Dependency 以及 Transient Disposable 警告

**難點**：Verify 不只是「試著建立物件圖」，它還進行靜態分析。很多開發者只知道呼叫 `Verify()`，但不理解它內部做了什麼。

**解析**：

`Verify()` 實際上執行兩個步驟：

```
  Verify() 內部流程
  ━━━━━━━━━━━━━━━━

  步驟 1: 實例化驗證
  ┌─────────────────────────────────────────┐
  │  遍歷所有註冊                            │
  │  → 嘗試建立每個物件圖                    │
  │  → 確認每個相依都能被解析                │
  │  → 檢查建構式參數是否合法                │
  └─────────────────────────────────────────┘
                    │
                    ▼
  步驟 2: 診斷分析（Diagnostic Analysis）
  ┌─────────────────────────────────────────┐
  │  分析每個物件圖的 Lifestyle 關係          │
  │                                          │
  │  ✗ Lifestyle Mismatch                    │
  │    Singleton → Transient (被俘虜)         │
  │    Singleton → Scoped   (被俘虜)         │
  │    Scoped   → Transient (被俘虜)         │
  │                                          │
  │  ✗ Disposable Transient                  │
  │    Transient 元件實作 IDisposable         │
  │    但 Transient 不被追蹤/Dispose          │
  │    → 可能的資源洩漏                       │
  │                                          │
  │  ✗ 其他診斷項目...                        │
  └─────────────────────────────────────────┘
```

**Lifestyle Mismatch 規則表**：

```
  消費者 Lifestyle    依賴 Lifestyle    結果
  ─────────────────   ───────────────   ──────────
  Singleton           Singleton         OK
  Singleton           Scoped            MISMATCH
  Singleton           Transient         MISMATCH
  Scoped              Singleton         OK
  Scoped              Scoped            OK
  Scoped              Transient         MISMATCH
  Transient           Singleton         OK
  Transient           Scoped            OK
  Transient           Transient         OK

  規則：依賴的生命不可以比消費者短
  （短命元件不能被長命元件俘虜）
```

> **白話口訣**：「短命不可寄居長命之下」。如果你的房客（Dependency）簽的是月租約（Transient），但房東（Consumer）買了整棟（Singleton），房客走不了就變成「被俘虜的依賴」。

## 6. 本章小結與架構啟示

### 核心總結

1. **Simple Injector 的「嚴格」是一種設計力量**，而非限制。它強制你做出明確的決策（Register vs Collection.Register、Register vs RegisterDecorator），這種明確性在大型系統中是極有價值的。

2. **Verify() 是你的安全網**。在應用程式啟動時呼叫它，能在第一個 HTTP Request 進來之前就發現配置錯誤。這是其他大多數 DI Container 做不到的。

3. **Ambient Scope 消除了一整類 bug**。你不需要在程式碼中傳遞 Scope 實例，也不會因為從錯誤的地方解析而產生記憶體洩漏。

4. **Sequences are Streams 是一個深思熟慮的設計**。它不僅提高了效能，更根本性地解決了 Captive Dependency 在集合注入場景中的問題。

### 架構師啟示

| 啟示 | 說明 |
|------|------|
| **容器配置困難 = 設計有問題** | 如果你發現很難用 Simple Injector 配置某個元件，先回頭檢查設計。`ThreeCourseMeal` → `Meal(IEnumerable<ICourse>)` 的重構就是最好的例子。 |
| **Register/Resolve 分離原則** | Container 在第一次使用後被鎖定。這迫使你將所有配置集中在 Composition Root，符合依賴注入的最佳實踐。 |
| **優先使用 Auto-Registration + 泛型** | `container.Register(typeof(ICommandService<>), assembly)` 一行搞定上百個 Use Case 的註冊。這是 SOLID 原則（特別是 OCP）在容器配置層面的體現。 |
| **Diagnostic-Driven Development** | 把 `Verify()` 當作類似於編譯器的角色 — 如果通過了，你對配置的正確性就有了基本保證。 |
| **Parameter Object 優於 Delegate** | 使用 delegate 會讓容器「失明」（喪失診斷能力）。Parameter Object 讓容器能完整追蹤物件圖。 |
| **Stream 設計的啟發** | Simple Injector 的 Stream 設計告訴我們：惰性求值（Lazy Evaluation）不只是效能優化，更可以是一種安全機制（防止 Captive Dependency）。 |

### 與其他容器的比較摘要

```
  ┌──────────────────┬───────────────┬───────────────┬────────────────┐
  │ 特性              │ Simple Injector│ Autofac       │ MS DI          │
  ├──────────────────┼───────────────┼───────────────┼────────────────┤
  │ 重複註冊同一抽象   │ 拋出例外       │ 後者覆蓋前者   │ 後者覆蓋前者    │
  │ Captive Dep 偵測  │ Verify 自動偵測│ 不偵測        │ 不偵測（需手動）│
  │ Sequence 行為     │ Stream（惰性） │ 預先建立      │ 預先建立        │
  │ Scope 模型        │ Ambient       │ 顯式 Scope    │ 顯式 Scope     │
  │ Decorator 支援    │ 原生支援       │ 原生支援      │ 不支援         │
  │ Composite 支援    │ 自動過濾       │ 需手動處理    │ 不支援         │
  └──────────────────┴───────────────┴───────────────┴────────────────┘
```

> **最終體悟**：Simple Injector 的設計哲學體現了一個重要的軟體工程原則 —— **Fail Fast**（快速失敗）。與其在執行期的深夜三點鐘收到生產環境的錯誤告警，不如在應用程式啟動的第一秒就讓配置錯誤浮現。這種「嚴格到近乎固執」的設計，恰恰是大型企業系統最需要的品質。
