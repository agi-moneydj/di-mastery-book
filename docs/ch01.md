# 第 1 章：DI 基礎：是什麼、為什麼、怎麼做 (The Basics of Dependency Injection: What, Why, and How)

> **作者**：Steven van Deursen, Mark Seemann
> **導讀視角**：資深系統架構師
> **程式碼範例**：C#

## 1. 概述 (Overview)

Dependency Injection（DI）不是目的，而是達成「鬆散耦合（Loose Coupling）」的手段，而鬆散耦合的終極目標是讓程式碼更具可維護性。本章以「荷蘭醬（Sauce Bearnaise）」作為比喻開場：DI 就像做荷蘭醬一樣，看似困難，其實只要掌握核心原理，任何人都能成功運用。本章從破除四大迷思出發，接著透過「電器插座與插頭」的類比闡述鬆散耦合如何對應經典設計模式，再以一個完整的 Hello DI! 範例展示 Constructor Injection 的實作方式，最後定義了 DI 的三大維度：Object Composition（物件組合）、Lifetime Management（生命週期管理）與 Interception（攔截）。

---

## 2. 學習目標 (Learning Objectives)

完成本章後，你應該能夠：

- **辨識** DI 的四大常見迷思（Late Binding Only、Unit Testing Only、Abstract Factory on Steroids、需要 DI Container），並清楚說明為何它們是錯的
- **理解** 鬆散耦合的核心目的：可維護性（Maintainability），以及它與 Liskov Substitution Principle、Open/Closed Principle、Single Responsibility Principle 之間的關聯
- **區分** Stable Dependency（穩定相依）和 Volatile Dependency（易變相依），並知道何時該引入 Seam
- **實作** 一個基於 Constructor Injection 的簡單 DI 範例，包含介面定義、具體實作、Decorator 模式擴展及單元測試
- **描述** DI 的三大維度：Object Composition、Lifetime Management、Interception，以及它們彼此的關係

## 3. 章節大綱 (Outline)

### 3.1 撰寫可維護的程式碼 (Writing Maintainable Code)

#### 3.1.1 DI 的四大迷思 (Common Myths about DI)

| 迷思 | 錯誤認知 | 正確理解 |
|------|----------|----------|
| **Late Binding Only** | DI 只在需要抽換實作（如換資料庫）時才有用 | Late Binding 只是 DI 眾多好處之一 |
| **Unit Testing Only** | DI 只是為了支援單元測試 | 即使不寫測試，DI 仍提供其他巨大好處 |
| **Abstract Factory on Steroids** | DI 就是一個通用的服務查詢工廠 | DI 是 Service Locator 的**反面**，不是主動要求依賴，而是要求外部供給 |
| **需要 DI Container** | 沒有 DI Container 就無法做 DI | DI Container 是可選工具，不用容器的做法叫做 Pure DI |

> **白話解釋**：很多人把 DI 想成「一個神奇的工具箱」，以為一定要有那個工具箱才能做事。但 DI 其實是一種「設計思維」——就像你不需要買全套專業廚具也能煮好菜一樣，關鍵在於你的烹飪方法（設計原則），而不是鍋碗瓢盆（工具）。

#### 3.1.2 理解 DI 的目的 (Understanding the Purpose of DI)

**核心鏈條**：

```
DI (手段) → Loose Coupling (途徑) → Maintainability (目標)
```

**電器插座類比**：

```
+--------------------------------------------------+
|  飯店房間（應用程式）                               |
|                                                    |
|  ┌──────────┐     ┌──────────┐                    |
|  │ 插座     │     │ 插座     │   ← 介面 (Interface)|
|  │ (Socket) │     │ (Socket) │                    |
|  └────┬─────┘     └────┬─────┘                    |
|       │                │                           |
|  ┌────┴─────┐     ┌────┴─────┐                    |
|  │ 吹風機   │     │ 電腦     │   ← 實作 (Implementation)
|  │ (Plug)   │     │ (Plug)   │                    |
|  └──────────┘     └──────────┘                    |
+--------------------------------------------------+

緊密耦合 = 把電線直接焊在牆壁裡（壞了要拆牆）
鬆散耦合 = 用插座與插頭（壞了直接拔掉換新的）
```

#### 3.1.3 四種設計模式與電器類比

| 電器場景 | 設計模式 | 說明 |
|----------|----------|------|
| 把吹風機換成電腦，插座不用改 | **Liskov Substitution Principle** | 可以用任何符合介面的實作替換，不影響消費者 |
| 插上兒童安全插座蓋（無功能） | **Null Object Pattern** | 一個「什麼都不做」的實作，避免 NullReferenceException |
| 接上 UPS 不斷電系統再接電腦 | **Decorator Pattern** | 在原有實作外面包一層，增加額外功能（如安全檢查、日誌） |
| 用延長線接多個電器 | **Composite Pattern** | 將多個實作聚合成一個，統一對外提供服務 |
| 出國旅行用轉接頭 | **Adapter Pattern** | 讓不相容的介面能夠協同工作 |

### 3.2 一個簡單範例：Hello DI! (A Simple Example)

#### 3.2.1 程式碼結構

```
Main() ──creates──► ConsoleMessageWriter : IMessageWriter
  │
  └──creates/uses──► Salutation ──uses──► IMessageWriter
```

#### 3.2.2 DI 帶來的五大好處

| 好處 | 描述 | 何時有價值 |
|------|------|------------|
| **Late Binding** | 不重新編譯就能抽換服務 | 標準軟體產品 |
| **Extensibility** | 以未預期的方式擴展和重用程式碼 | **永遠有價值** |
| **Parallel Development** | 程式碼可以平行開發 | 大型複雜應用程式 |
| **Maintainability** | 職責清晰的類別更容易維護 | **永遠有價值** |
| **Testability** | 類別可以被單元測試 | **永遠有價值** |

### 3.3 什麼該注入、什麼不該注入 (What to Inject and What Not to Inject)

#### 3.3.1 Stable Dependencies（穩定相依）

滿足以下條件即為穩定相依：
- 已經存在且可用
- 新版本不會有破壞性變更
- 包含確定性（deterministic）演算法
- 你永遠不需要替換、包裝或攔截它

> **例如**：`System.String`、`System.Math`、`System.Collections.Generic.List<T>` 等 BCL 類別

#### 3.3.2 Volatile Dependencies（易變相依）

符合以下任一條件即為易變相依：
- 需要設定執行期環境（如資料庫、訊息佇列、Web Service、檔案系統）
- 尚未存在或仍在開發中
- 並非所有開發機器都有安裝
- 包含非確定性行為（如亂數、當前時間）

> **白話說明**：穩定相依就像「自來水」——打開水龍頭一定有水，你不需要為它額外準備什麼。易變相依則像「瓦斯」——你需要特別配管、需要安全檢查、不同地方可能用不同的燃氣規格，所以必須透過標準化的接口（Seam）來使用它。

#### 3.3.3 Seam（接縫）

**Seam** 是應用程式從各個組成部分組裝在一起的地方，如同衣服的縫線。在程式碼中，每當你決定針對 Abstraction 而非 concrete type 來程式設計時，就引入了一個 Seam。

```
Main()
  │
  └── creates/uses ──► Salutation
                           │
                        ← SEAM →    （IMessageWriter 介面就是 Seam）
                           │
                       ConsoleMessageWriter
```

### 3.4 DI 的範疇 (DI Scope)

#### 3.4.1 Object Composition（物件組合）

將各個類別組裝成完整的應用程式，就像把電器插到插座上。這是 DI 最初也是最核心的關注點。

#### 3.4.2 Object Lifetime（物件生命週期）

當類別交出對 Dependency 的控制權後，它也放棄了控制實例何時建立、何時銷毀的能力。兩個消費者可以共享同一個 Dependency 實例，或各自擁有獨立實例。

#### 3.4.3 Interception（攔截）

在 Dependency 傳遞給消費者之前進行修改或包裝。這是 Decorator 模式的應用，也是通向 AOP（Aspect-Oriented Programming）的路徑，能優雅地處理 Cross-Cutting Concerns（橫切關注點），如日誌、驗證、快取等。

#### 3.4.4 三個維度的關係

```
                ┌─────────────────────────────────────────────┐
                │            DI 的三大維度                      │
                │                                              │
                │  ┌──────────────────┐                        │
                │  │ Object           │  ← 基礎：沒有它就      │
                │  │ Composition      │     沒有其他兩個        │
                │  │ (物件組合)        │                        │
                │  └────────┬─────────┘                        │
                │           │                                  │
                │     ┌─────┴──────┐                           │
                │     │            │                           │
                │  ┌──┴───────┐ ┌──┴──────────┐               │
                │  │ Lifetime │ │ Interception │               │
                │  │ Mgmt     │ │ (攔截)        │              │
                │  │(生命週期) │ │              │               │
                │  └──────────┘ └──────────────┘               │
                │                                              │
                │  Composition 是地基                           │
                │  Lifetime 處理副作用                          │
                │  Interception 是真正收穫好處的地方             │
                └─────────────────────────────────────────────┘
```

## 4. 關鍵框架與公式 (Key Frameworks)

### 框架一：Constructor Injection（建構式注入）

- **何時使用**：當一個類別有 Volatile Dependency 時，透過建構式參數宣告它所需要的依賴
- **核心概念**：類別不自己建立依賴，而是「要求」外部提供。就像小孩不自己開冰箱，而是告訴父母「我午餐需要喝點東西」
- **程式碼範例**：

```csharp
// 1. 定義抽象（介面）
public interface IMessageWriter
{
    void Write(string message);
}

// 2. 建立具體實作
public class ConsoleMessageWriter : IMessageWriter
{
    public void Write(string message)
    {
        Console.WriteLine(message);
    }
}

// 3. 消費者透過建構式接收依賴（Constructor Injection）
public class Salutation
{
    private readonly IMessageWriter writer;

    public Salutation(IMessageWriter writer) // ← 注入點
    {
        this.writer = writer
            ?? throw new ArgumentNullException(nameof(writer)); // Guard Clause
    }

    public void Exclaim()
    {
        this.writer.Write("Hello DI!");
    }
}

// 4. 在應用程式入口點（Composition Root）進行組裝
private static void Main()
{
    IMessageWriter writer = new ConsoleMessageWriter();
    var salutation = new Salutation(writer);
    salutation.Exclaim();
}
```

> **關鍵觀念**：`Salutation` 完全不知道 `ConsoleMessageWriter` 的存在，它只認識 `IMessageWriter` 介面。這就是「針對介面程式設計，而非針對實作」。

### 框架二：Decorator Pattern（裝飾者模式）用於擴展功能

- **何時使用**：需要在不修改現有類別的情況下新增功能時（例如加入安全檢查、日誌記錄、快取等）
- **核心概念**：像俄羅斯套娃一樣，一層包一層，每層負責一個職責
- **程式碼範例**：

```csharp
// Decorator：在原有的 IMessageWriter 外面包一層安全檢查
public class SecureMessageWriter : IMessageWriter
{
    private readonly IMessageWriter writer;  // 被裝飾的物件
    private readonly IIdentity identity;

    public SecureMessageWriter(
        IMessageWriter writer,  // ← 同時實作並消費同一個介面
        IIdentity identity)
    {
        this.writer = writer
            ?? throw new ArgumentNullException(nameof(writer));
        this.identity = identity
            ?? throw new ArgumentNullException(nameof(identity));
    }

    public void Write(string message)
    {
        if (this.identity.IsAuthenticated)
        {
            this.writer.Write(message); // 認證通過才放行
        }
        // 未認證則靜默忽略（或可拋例外）
    }
}

// 在 Composition Root 重新組裝，不需要改 Salutation 的任何一行
private static void Main()
{
    IMessageWriter writer =
        new SecureMessageWriter(
            new ConsoleMessageWriter(),           // 真正做事的
            WindowsIdentity.GetCurrent());        // 身分驗證來源

    var salutation = new Salutation(writer);       // Salutation 完全不知道變化
    salutation.Exclaim();
}
```

```
組裝結構示意：

    Salutation
        │
        └──► IMessageWriter
                 │
            SecureMessageWriter  (Decorator - 安全檢查)
                 │
            ConsoleMessageWriter (真正輸出到 Console)
```

### 框架三：Stable vs. Volatile Dependency 決策流程

- **何時使用**：每當你要決定是否為某個依賴引入 Seam（抽象層）時
- **核心概念**：不是所有依賴都需要抽象化，只有 Volatile Dependency 才需要

```
               你的 Dependency 是否...
                      │
         ┌────────────┼─────────────┐
         │            │             │
    需要外部環境？  非確定性？    仍在開發中？
    (DB/API/MQ)   (隨機/時間)   (未完成)
         │            │             │
         ▼            ▼             ▼
     ┌───┴────────────┴─────────────┴───┐
     │  以上任一為「是」                   │
     │  → Volatile Dependency            │
     │  → 必須引入 Seam（介面）           │
     │  → 透過 Constructor Injection 注入 │
     └──────────────────────────────────┘
                      │
                 全部為「否」
                      │
     ┌────────────────┴────────────────┐
     │  Stable Dependency               │
     │  → 可以直接使用，不需要抽象化     │
     │  → 例如 String, List<T>, Math    │
     └─────────────────────────────────┘
```

### 框架四：Pure DI vs. DI Container

- **Pure DI**：不使用任何 DI Container，直接在程式入口點手動 `new` 並組裝所有物件
- **DI Container**：使用第三方程式庫（如 Microsoft.Extensions.DependencyInjection、Autofac）自動化物件的建立與組裝
- **重要觀念**：DI Container 是可選的工具，不是 DI 的必要條件

```csharp
// Pure DI（手動組裝）
private static void Main()
{
    // 你親手把每個零件組裝起來
    IMessageWriter writer = new ConsoleMessageWriter();
    var salutation = new Salutation(writer);
    salutation.Exclaim();
}

// 使用 DI Container（以 Microsoft DI 為例，概念示意）
// var services = new ServiceCollection();
// services.AddTransient<IMessageWriter, ConsoleMessageWriter>();
// services.AddTransient<Salutation>();
// var provider = services.BuildServiceProvider();
// var salutation = provider.GetRequiredService<Salutation>();
// salutation.Exclaim();
```

## 5. 最難理解的三個部分 (3 Hardest Sections)

### 難點 1：DI 不是 Service Locator，而是它的反面

- **為什麼難**：很多開發者第一次接觸 DI 時，直覺會把 DI Container 當成一個「全能工廠」——需要什麼就跟它要什麼。這種思維恰好就是 Service Locator 反模式，與 DI 的精神完全相反。兩者在表面上看起來都「幫你取得依賴」，但控制方向完全不同。
- **白話解釋**：
  - **Service Locator**（錯誤做法）：像是你走進自助餐廳，自己去取餐檯挑菜。你必須知道餐檯在哪裡、有什麼菜、怎麼取。你**主動去找**依賴。
  - **Dependency Injection**（正確做法）：像是你坐在餐廳裡，服務生把菜端到你面前。你只需要告訴服務生你要什麼（透過建構式參數宣告），服務生就會送來。你**被動接收**依賴。

```
Service Locator（反模式）：
┌──────────────┐         ┌──────────────┐
│  Consumer    │───問──►  │  Locator     │
│  （你）      │◄──拿──── │  （自助餐檯） │
└──────────────┘         └──────────────┘
   Consumer 知道 Locator 的存在，且主動去查找

Dependency Injection（正確做法）：
┌──────────────┐         ┌──────────────┐
│  Composition │───注入──►│  Consumer    │
│  Root        │         │  （你）       │
│  （服務生）   │         │  只宣告需要   │
└──────────────┘         └──────────────┘
   Consumer 不知道誰給的，只知道自己需要什麼
```

- **程式碼示範**：

```csharp
// ❌ Service Locator 反模式 —— 類別自己去「要」依賴
public class Salutation
{
    private readonly IMessageWriter writer;

    public Salutation()
    {
        // 主動向 Service Locator 索取 → 緊密耦合於 Locator 本身
        this.writer = ServiceLocator.GetService<IMessageWriter>();
    }
}

// ✅ Dependency Injection —— 類別被動「接收」依賴
public class Salutation
{
    private readonly IMessageWriter writer;

    public Salutation(IMessageWriter writer) // 被動宣告需求
    {
        this.writer = writer
            ?? throw new ArgumentNullException(nameof(writer));
    }
}
```

### 難點 2：Volatile vs. Stable Dependency 的判斷邊界

- **為什麼難**：現實世界中，某些依賴的「穩定度」並不明顯。例如 `DateTime.Now` 是 BCL 類別，直覺上應該是 Stable，但因為它的行為是非確定性的（每次呼叫結果不同），所以其實是 Volatile。又例如，SQL Server Client 是 BCL 的一部分，但因為它隱含了一個「需要有 SQL Server 正在執行」的前提，所以也是 Volatile。
- **白話解釋**：你可以把 Dependency 想像成料理的食材：
  - **Stable Dependency**（穩定食材）= 鹽巴、胡椒、油 → 每個廚房都有，不會壞，結果可預測
  - **Volatile Dependency**（易變食材）= 新鮮海鮮 → 不是每個地方都買得到，品質每天不同，需要特別保存處理

  判斷關鍵不在於「它是不是 .NET 內建的」，而在於「它的行為是否可預測、是否需要外部環境、是否可能需要替換」。

- **程式碼示範**：

```csharp
// ❌ 直接使用 DateTime.Now → 非確定性，無法測試
public class PromotionService
{
    public bool IsBlackFriday()
    {
        return DateTime.Now.Month == 11
            && DateTime.Now.Day == 29; // 結果隨時間變化
    }
}

// ✅ 將時間抽象化，變成可注入的 Volatile Dependency
public interface ITimeProvider
{
    DateTime Now { get; }
}

public class SystemTimeProvider : ITimeProvider
{
    public DateTime Now => DateTime.Now;
}

public class PromotionService
{
    private readonly ITimeProvider timeProvider;

    public PromotionService(ITimeProvider timeProvider) // 注入
    {
        this.timeProvider = timeProvider
            ?? throw new ArgumentNullException(nameof(timeProvider));
    }

    public bool IsBlackFriday()
    {
        var now = this.timeProvider.Now;
        return now.Month == 11 && now.Day == 29;
    }
}

// 測試時就可以控制時間
public class FixedTimeProvider : ITimeProvider
{
    public DateTime Now { get; set; } // 可以隨意設定
}
```

### 難點 3：為什麼簡單範例看起來像過度工程（Over-Engineering）

- **為什麼難**：Hello DI! 範例把一行 `Console.WriteLine("Hello DI!")` 膨脹成兩個類別加一個介面共 28 行程式碼。這讓很多新手覺得 DI 是「把簡單的事情複雜化」，進而產生抗拒心理。
- **白話解釋**：這就像蓋房子的地基。蓋一間狗屋不需要打地基，直接釘幾塊木板就好。但如果你要蓋一棟十層大樓，沒有地基是不行的。Hello DI! 就是那間「狗屋」——DI 看起來多此一舉。但在真實的企業級應用中（那棟十層大樓），DI 的基礎架構讓你可以：
  - 輕鬆替換資料庫實作（從 SQL Server 換成 PostgreSQL）
  - 不改現有程式就新增功能（加上快取層、日誌層、安全層）
  - 讓 10 個團隊同時開發不同模組而不互相干擾
  - 用 Test Double 進行隔離測試

  DI 的開銷是固定的，但好處隨著複雜度成長而**倍增**。

- **程式碼示範**（展示複雜度增長後 DI 的價值）：

```csharp
// 隨著需求增長，只需在 Composition Root 調整組裝方式
// 而所有已存在的類別都不需要修改（Open/Closed Principle）

private static void Main()
{
    // 需求 v1：寫到 Console
    IMessageWriter writer = new ConsoleMessageWriter();

    // 需求 v2：加上安全檢查 → 只需「包」一層
    writer = new SecureMessageWriter(
        writer,
        WindowsIdentity.GetCurrent());

    // 需求 v3：加上日誌記錄 → 再「包」一層
    // writer = new LoggingMessageWriter(writer, logger);

    // 需求 v4：加上效能監控 → 再「包」一層
    // writer = new PerformanceMessageWriter(writer, metrics);

    // Salutation 從頭到尾一行都不用改！
    var salutation = new Salutation(writer);
    salutation.Exclaim();
}
```

```
需求越疊越多，但核心邏輯 Salutation 完全不受影響：

v1:  Salutation → ConsoleMessageWriter
v2:  Salutation → SecureMessageWriter → ConsoleMessageWriter
v3:  Salutation → LoggingWriter → SecureWriter → ConsoleWriter
v4:  Salutation → PerfWriter → LoggingWriter → SecureWriter → ConsoleWriter

每一層都只負責一件事（Single Responsibility Principle）
每一層都可以獨立測試
每一層的加入都不需要改動其他層（Open/Closed Principle）
```

## 6. 本章小結與架構啟示

### 核心要點回顧

1. **DI 是手段，不是目的**。最終目標是可維護的程式碼，DI 透過實現鬆散耦合來達成這個目標。

2. **針對介面程式設計**是一切的基礎。Gang of Four 在 1994 年就告訴我們：「Program to an interface, not an implementation.」DI 就是將這個原則付諸實踐的系統化方法。

3. **Constructor Injection 是首選的注入方式**。透過建構式參數靜態宣告依賴，讓類別的需求一目了然。

4. **只對 Volatile Dependency 引入 Seam**。不是所有東西都要抽象化，過度抽象反而會增加不必要的複雜度。

5. **DI 必須是全面性的（Pervasive）**。你無法輕易地把鬆散耦合「補丁式」地加到現有的緊密耦合程式碼上。從專案初期就要有意識地應用。

### 架構啟示

```
┌─────────────────────────────────────────────────┐
│              架構師的思考模型                      │
│                                                  │
│  問自己三個問題：                                  │
│                                                  │
│  1. 這個依賴是 Stable 還是 Volatile？             │
│     → Volatile 就要引入 Seam                     │
│                                                  │
│  2. 新功能能否只透過新增類別來實現，               │
│     而不修改既有程式碼？                           │
│     → 這是 Open/Closed Principle 的試金石         │
│                                                  │
│  3. 這個類別是否只負責一件事？                     │
│     → 如果不是，考慮拆解並用 Decorator/Composite  │
│       來組合                                      │
│                                                  │
│  當你持續這樣思考，你的系統就會自然走向             │
│  鬆散耦合、高度可維護的架構。                      │
└─────────────────────────────────────────────────┘
```

### 重要名詞索引

| 術語 | 白話解釋 |
|------|----------|
| **Dependency Injection (DI)** | 一組讓程式碼鬆散耦合的設計原則與模式。類別不自己建立依賴，而是由外部提供。 |
| **Loose Coupling（鬆散耦合）** | 元件之間的依賴關係透過抽象來維持，修改一個元件不會連帶影響其他元件。就像用插座和插頭，換電器不用拆牆。 |
| **Constructor Injection（建構式注入）** | 透過類別的建構式參數來傳入依賴。這是最常見、最推薦的 DI 方式。 |
| **Pure DI** | 不使用任何 DI Container，手動在程式入口點組裝所有物件的做法。 |
| **Seam（接縫）** | 應用程式中可以拆開並重新組裝的地方，通常就是引入介面的位置。 |
| **Stable Dependency（穩定相依）** | 已存在、不會破壞性變更、行為可預測、不需要替換的依賴。 |
| **Volatile Dependency（易變相依）** | 需要外部環境、行為不確定、可能需要替換的依賴。這才是 DI 的重點對象。 |
| **Composition Root（組合根）** | 應用程式的入口點，是唯一應該知道所有具體實作並負責組裝的地方。 |
| **Service Locator（服務定位器）** | 一種反模式。類別主動向一個全域工廠查詢依賴，與 DI 的被動接收精神相反。 |
| **Cross-Cutting Concerns（橫切關注點）** | 跨越多個模組的通用功能，如日誌、安全、交易管理。透過 Interception 可以優雅處理。 |
| **Liskov Substitution Principle（LSP）** | 任何使用基底類別（或介面）的地方，都應該能透明地替換成其衍生類別（或其他實作），而不影響程式的正確性。 |
| **Open/Closed Principle（OCP）** | 軟體實體應該對擴展開放，對修改封閉。新功能應透過新增程式碼實現，而非修改現有程式碼。 |
| **Single Responsibility Principle（SRP）** | 每個類別只應有一個改變的理由，只負責一件事。 |
